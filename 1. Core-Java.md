# Core Java (Fundamentals & OOP) — Questions & Answers

## 1. Explain the concept of Object-Oriented Programming in Java

**Answer:**

OOP is a programming paradigm that models software around **objects** — bundles of state (fields) and behaviour (methods). Java is built on four core OOP pillars:

```text
         ┌──────────────┐
         │  Abstraction │  "What it does"
         └──────┬───────┘
                │ defines interface
         ┌──────▼───────┐
         │ Encapsulation│  "How it hides"
         └──────┬───────┘
                │ shared via
         ┌──────▼───────┐
         │  Inheritance │  "What it is"
         └──────┬───────┘
                │ enables
         ┌──────▼───────┐
         │ Polymorphism │  "How it behaves"
         └──────────────┘
```

---

### a. Encapsulation — Hide State, Expose Behaviour

Bundle data and the methods that operate on it into one unit. Hide internal state behind a controlled interface.

```java
public class BankAccount {
    private double balance;   // Hidden — cannot be accessed directly
    private String ownerId;

    public BankAccount(String ownerId, double initialBalance) {
        this.ownerId = ownerId;
        this.balance = initialBalance;
    }

    // Controlled write -- validation is enforced here
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Amount must be positive");
        balance += amount;
    }

    public boolean withdraw(double amount) {
        if (amount > balance) return false;   // Business rule enforced
        balance -= amount;
        return true;
    }

    public double getBalance() { return balance; }  // Controlled read
}

BankAccount acc = new BankAccount("alice", 1000);
// acc.balance = -999;  // ❌ Private -- cannot break invariant
acc.deposit(500);       // ✅ Goes through validation
```

> **Real-world analogy:** An ATM machine — you can deposit/withdraw through buttons (public API) but cannot directly touch the cash inside (private state).

---

### b. Abstraction — Hide Complexity, Show Essentials

Define **what** a type must do without specifying **how**. Callers depend on the abstraction, not the implementation.

```java
// Abstract definition -- "what" a payment must do
public abstract class Payment {
    protected double amount;

    public Payment(double amount) { this.amount = amount; }

    public abstract void process();    // Each subclass decides HOW

    public void printReceipt() {       // Shared behaviour built on the abstraction
        System.out.printf("Paid %.2f via %s%n", amount, getClass().getSimpleName());
    }
}

// Concrete implementations -- "how"
public class CreditCardPayment extends Payment {
    private String cardNumber;

    public CreditCardPayment(double amount, String cardNumber) {
        super(amount);
        this.cardNumber = cardNumber;
    }

    @Override
    public void process() {
        System.out.println("Charging card ending in " + cardNumber.substring(cardNumber.length() - 4));
    }
}

public class CryptoPayment extends Payment {
    private String walletAddress;

    public CryptoPayment(double amount, String walletAddress) {
        super(amount);
        this.walletAddress = walletAddress;
    }

    @Override
    public void process() {
        System.out.println("Sending " + amount + " BTC to " + walletAddress);
    }
}
```

> **Real-world analogy:** A remote control — you press "volume up" without knowing the electronics inside the TV.

---

### c. Inheritance — Reuse and Extend

A subclass **inherits** the fields and methods of its parent and can extend or override them. Models "IS-A" relationships.

```java
public class Vehicle {
    protected String brand;
    protected int speed;

    public Vehicle(String brand) { this.brand = brand; }

    public void accelerate(int amount) { speed += amount; }
    public void brake(int amount)      { speed = Math.max(0, speed - amount); }

    public String status() {
        return brand + " going " + speed + " km/h";
    }
}

public class ElectricCar extends Vehicle {
    private int batteryLevel;

    public ElectricCar(String brand, int battery) {
        super(brand);              // Call parent constructor
        this.batteryLevel = battery;
    }

    @Override
    public void accelerate(int amount) {
        super.accelerate(amount);  // Reuse parent logic
        batteryLevel -= amount / 10;  // Add EV-specific behaviour
    }

    @Override
    public String status() {
        return super.status() + " | Battery: " + batteryLevel + "%";
    }
}

ElectricCar tesla = new ElectricCar("Tesla", 100);
tesla.accelerate(50);
System.out.println(tesla.status());  // "Tesla going 50 km/h | Battery: 95%"
```

> **Real-world analogy:** A smartphone IS-A phone — it inherits calling ability and adds camera, apps, and more.

---

### d. Polymorphism — One Interface, Many Behaviours

The same method call behaves differently depending on the **actual object type** at runtime.

```java
public abstract class Shape {
    public abstract double area();
    public abstract String name();

    public void describe() {     // Common behaviour using the abstraction
        System.out.printf("%s has area %.2f%n", name(), area());
    }
}

public class Circle extends Shape {
    private final double radius;
    public Circle(double r) { this.radius = r; }

    @Override public double area()   { return Math.PI * radius * radius; }
    @Override public String name()   { return "Circle(r=" + radius + ")"; }
}

public class Rectangle extends Shape {
    private final double w, h;
    public Rectangle(double w, double h) { this.w = w; this.h = h; }

    @Override public double area()   { return w * h; }
    @Override public String name()   { return "Rectangle(" + w + "x" + h + ")"; }
}

// Polymorphism in action -- same reference type, different runtime behaviour
List<Shape> shapes = List.of(new Circle(5), new Rectangle(4, 6), new Circle(3));

for (Shape s : shapes) {
    s.describe();   // Each calls its own area() and name()
}
// Circle(r=5.0) has area 78.54
// Rectangle(4.0x6.0) has area 24.00
// Circle(r=3.0) has area 28.27

double totalArea = shapes.stream()
        .mapToDouble(Shape::area)
        .sum();   // Works for any Shape -- new shapes added without changing this code
```

> **Real-world analogy:** A `draw()` call on a drawing app works differently for a circle, rectangle, or line — same button, different result.

---

### Benefits of OOP

| Benefit | Pillar | Explanation |
|---|---|---|
| **Modularity** | Encapsulation | Classes hide internals; changes stay local |
| **Reusability** | Inheritance | Shared logic written once in the parent |
| **Flexibility** | Polymorphism | Swap implementations without changing callers |
| **Clarity** | Abstraction | Expose only what's needed; hide complexity |
| **Maintainability** | All | Easier to test, refactor, and extend |

## 2. What is the difference between `==` and `.equals()` in Java?

**Answer:**

| | `==` | `.equals()` |
|---|---|---|
| **Primitives** | Compares **values** | Not applicable (primitives have no methods) |
| **Objects** | Compares **references** (memory addresses) | Compares **content** (as defined by the class) |
| **Default (`Object`)** | — | Same as `==` unless overridden |
| **`String`, wrappers** | May surprise you (see below) | Content comparison ✅ |

---

### a. Primitives — `==` Always Compares Values

```java
int    a = 5,   b = 5;
double x = 1.5, y = 1.5;
char   c = 'A', d = 'A';

a == b;   // true  -- value comparison
x == y;   // true
c == d;   // true
```

---

### b. Objects — `==` Compares References (Memory Addresses)

```java
String s1 = new String("Hello");   // New object on heap
String s2 = new String("Hello");   // Another new object on heap
String s3 = "Hello";               // From String Pool
String s4 = "Hello";               // Same String Pool entry as s3

// Memory model:
// Heap:         [s1:"Hello"] [s2:"Hello"]
// String Pool:  [s3/s4: "Hello"]  <-- one shared object

s1 == s2;        // false -- different heap objects (different addresses)
s1.equals(s2);   // true  -- same content

s3 == s4;        // true  -- same object in String Pool
s3.equals(s4);   // true  -- same content

s1 == s3;        // false -- heap object vs pool object
s1.equals(s3);   // true  -- same content
```

---

### c. Integer Cache Gotcha

Java caches `Integer` objects for values **-128 to 127**. This catches many developers off-guard:

```java
Integer x = 127;
Integer y = 127;
x == y;          // true  -- same cached object

Integer a = 128;
Integer b = 128;
a == b;          // false -- outside cache, two new heap objects!
a.equals(b);     // true  -- content comparison always works

// Rule: NEVER use == to compare Integer (or any wrapper) objects
```

---

### d. Custom Classes — Override `equals()` for Content Comparison

Without overriding, `.equals()` falls back to `==` (reference check):

```java
public class Point {
    private final int x, y;
    public Point(int x, int y) { this.x = x; this.y = y; }
}

Point p1 = new Point(3, 4);
Point p2 = new Point(3, 4);

p1 == p2;        // false -- different objects
p1.equals(p2);   // false -- Object.equals() defaults to ==

// Override to get meaningful content equality:
public class Point {
    private final int x, y;
    public Point(int x, int y) { this.x = x; this.y = y; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;                          // Same reference fast path
        if (o == null || getClass() != o.getClass()) return false;
        Point other = (Point) o;
        return x == other.x && y == other.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);  // Always override hashCode together with equals!
    }
}

p1.equals(p2);   // true now
```

---

### e. Null Safety — `Objects.equals()`

```java
String name = null;

// ❌ Throws NullPointerException
name.equals("Alice");

// ❌ Still risky if both could be null
"Alice".equals(name);   // safe only if the literal is not null

// ✅ Null-safe comparison
Objects.equals(name, "Alice");  // false -- no NPE
Objects.equals(null, null);     // true
Objects.equals("Alice", null);  // false
```

---

### Best Practices

```java
// 1. Primitives --> use ==
int a = 5, b = 5;
a == b;   // ✅

// 2. Objects (String, Integer, custom) --> use .equals()
String s = "hello";
s.equals("hello");          // ✅

// 3. Null-safe comparison --> use Objects.equals()
Objects.equals(maybeNull, "expected");  // ✅ No NPE risk

// 4. Null check itself --> use ==
if (obj == null) { ... }    // ✅

// 5. "Yoda condition" to avoid NPE on unknown variables
"literal".equals(variable); // ✅ Safe even if variable is null

// 6. Never use == on wrapper types (Integer, Long, Double...)
Integer x = 200, y = 200;
x.equals(y);                // ✅  (not x == y)
```

## 3. What are `hashCode()` and `equals()` methods used for?

```text
// THE CONTRACT:
1. If a.equals(b) == true  ->  a.hashCode() == b.hashCode()  MUST
2. If a.hashCode() == b.hashCode()  ->  a.equals(b) may be true or false
3. If a.equals(b) == false  ->  hashCodes MAY differ (recommended for performance)

// CONSEQUENCE:
[!] Override BOTH or NEITHER -- never just one!
```

#### How HashMap Uses Both

```text
// When you do: map.put(key, value)
1. Calculate hashCode() -> determines bucket index
2. If bucket has entries -> use equals() to find exact match

// Visual:
HashMap Buckets:
[0] -> null
[1] -> [Entry("A")] -> [Entry("K")]  // Same hashCode, different objects
[2] -> [Entry("B")]
...
```

#### Proper Implementation

```java
public class Employee {
	private Long id;
	private String name;
    
	@Override
	public boolean equals(Object o) {
		if (this == o) return true;
		if (o == null || getClass() != o.getClass()) return false;
		Employee employee = (Employee) o;
		return Objects.equals(id, employee.id);  // Based on business key
	}
    
	@Override
	public int hashCode() {
		return Objects.hash(id);  // Same fields as equals()
	}
}

// Modern alternative: Use records (Java 14+)
public record Employee(Long id, String name) {}  // Auto-generates both!
```

## 4. Explain method overloading vs method overriding

**Answer:**

| | **Overloading** | **Overriding** |
|---|---|---|
| **Where** | Same class (or subclass) | Subclass overrides parent method |
| **Signature** | Different parameter list | Identical signature |
| **Return type** | Can differ | Must be same or covariant |
| **Resolved at** | Compile-time (static dispatch) | Runtime (dynamic dispatch) |
| **Polymorphism** | Ad-hoc / compile-time | Runtime polymorphism |
| **`@Override`** | Not applicable | Always use it |
| **`static` methods** | Can be overloaded | **Cannot** be overridden (hidden instead) |
| **`private`/`final`** | Can be overloaded | **Cannot** be overridden |

---

### a. Method Overloading — Compile-time

Same method name, **different parameter list** (type, count, or order). The compiler resolves which version to call.

```java
public class Calculator {

    // Different number of parameters
    public int add(int a, int b)           { return a + b; }
    public int add(int a, int b, int c)    { return a + b + c; }

    // Different parameter types
    public double add(double a, double b)  { return a + b; }
    public String add(String a, String b)  { return a + b; }

    // Different parameter order (use carefully -- easy to confuse)
    public void print(String label, int value) { System.out.println(label + ": " + value); }
    public void print(int value, String label) { System.out.println(value + " " + label); }
}

Calculator calc = new Calculator();
calc.add(1, 2);          // int add(int, int)    -- resolved at compile-time
calc.add(1.0, 2.0);      // double add(double, double)
calc.add("Hello", "!");  // String add(String, String)
```

#### Overloading — Rules and Pitfalls

```java
// ✅ Valid overloads: differ by parameter list
void process(int x)      { }
void process(double x)   { }
void process(int x, int y) { }

// ❌ NOT valid overloads: return type alone does NOT count
int  getValue()    { return 1; }
// double getValue() { return 1.0; } // Compile error: duplicate method

// ⚠️ Widening pitfall: compiler picks the "closest" type
void show(long x)   { System.out.println("long");   }
void show(double x) { System.out.println("double"); }

show(42);       // "long"   -- int widens to long first (closer match)
show(42L);      // "long"
show(42.0);     // "double"

// ⚠️ Autoboxing comes after widening
void show(Integer x) { System.out.println("Integer"); }
show(42);       // Still "long" -- widening beats autoboxing
```

---

### b. Method Overriding — Runtime

Subclass provides its own implementation for an **inherited method**. The JVM decides which version to call based on the **actual object type** at runtime.

```java
public class Animal {
    public String sound() { return "..."; }
    public void describe() {
        System.out.println("I am a " + getClass().getSimpleName() + " and I say: " + sound());
    }
}

public class Dog extends Animal {
    @Override                              // Always annotate -- compiler catches mistakes
    public String sound() { return "Woof!"; }
}

public class Cat extends Animal {
    @Override
    public String sound() { return "Meow!"; }
}

// Runtime polymorphism -- actual type drives method selection
Animal a = new Dog();
a.sound();      // "Woof!"  -- Dog's version, resolved at runtime
a.describe();   // "I am a Dog and I say: Woof!"

List<Animal> animals = List.of(new Dog(), new Cat(), new Animal());
animals.forEach(Animal::describe);
// I am a Dog and I say: Woof!
// I am a Cat and I say: Meow!
// I am a Animal and I say: ...
```

#### Overriding — Rules

```java
public class Parent {
    protected Number compute(int x) { return x; }  // returns Number
}

public class Child extends Parent {
    @Override
    public Integer compute(int x) { return x * 2; }  // ✅ Covariant return (Integer extends Number)
    // Access modifier can be WIDENED: protected -> public ✅
    // Access modifier CANNOT be NARROWED: public -> protected ❌
}
```

```text
Rules for valid overriding:
  ✅ Same method name and parameter list
  ✅ Return type: same OR a subtype (covariant return type)
  ✅ Access modifier: same or wider (protected -> public is OK)
  ✅ Checked exceptions: same or narrower (cannot add new checked exceptions)
  ❌ Cannot override: static, final, or private methods
  ❌ Cannot narrow access modifier (public -> protected is a compile error)
```

#### `static` methods — Hidden, not Overridden

```java
public class Base {
    public static void staticMethod() { System.out.println("Base.static"); }
    public void instanceMethod()      { System.out.println("Base.instance"); }
}

public class Sub extends Base {
    public static void staticMethod() { System.out.println("Sub.static"); }  // HIDES, not overrides
    @Override
    public void instanceMethod()      { System.out.println("Sub.instance"); }
}

Base obj = new Sub();
obj.staticMethod();   // "Base.static"  -- resolved at COMPILE-TIME (reference type)
obj.instanceMethod(); // "Sub.instance" -- resolved at RUNTIME (actual type)
```

---

### Quick Recap

```text
Overloading  --> same name, different params --> compile-time --> "which signature?"
Overriding   --> same signature, subclass    --> runtime     --> "which implementation?"
```

## 5. What is the difference between an abstract class and an interface?

```java
// ABSTRACT CLASS -- Shared state + behavior
public abstract class Animal {
	protected String name;  // Instance state
    
	public Animal(String name) { this.name = name; }  // Constructor
    
	public void eat() { System.out.println(name + " eats"); }  // Concrete
	public abstract void speak();  // Abstract -- must implement
}

// INTERFACE -- Contract (capability)
public interface Flyable {
	int MAX_ALTITUDE = 10000;  // public static final
    
	void fly();  // Abstract
    
	default void land() {  // Default implementation (Java 8+)
		System.out.println("Landing...");
	}
}

// A class can extend ONE abstract class but implement MULTIPLE interfaces
public class Bird extends Animal implements Flyable, Comparable<Bird> {
	public Bird(String name) { super(name); }
    
	@Override public void speak() { System.out.println("Chirp!"); }
	@Override public void fly() { System.out.println("Flying!"); }
	@Override public int compareTo(Bird o) { return name.compareTo(o.name); }
}
```

#### When to Use

- **Abstract Class:** "IS-A" relationship with shared state/behavior (e.g., `Animal`)
- **Interface:** "CAN-DO" capability across unrelated classes (e.g., `Comparable`, `Serializable`)

### What is the difference between Interface and Abstract Class in Java (with use case)?

**Answer:**

- **Definition & Purpose**
	- **Abstract Class:** A base class that can contain fields, constructors, concrete methods, and abstract methods. Use when you want to provide shared state/behavior for a family of related classes.
	- **Interface:** A contract describing capabilities that classes can implement. Interfaces focus on behavior and (since Java 8) may include default and static methods.

- **Key Differences**
	- **Multiple Inheritance:** Classes can implement many interfaces but can extend only one class (abstract or concrete).
	- **State:** Abstract classes can have instance fields; interfaces cannot (only constants/static fields).
	- **Constructors:** Abstract classes can define constructors; interfaces cannot.
	- **Versioning:** Adding methods to abstract classes is easier for subclass reuse; adding methods to interfaces should use default methods carefully to avoid breaking implementors.

- **When to Use (Use Cases)**
	- **Abstract Class Use Case:** Shared implementation + state. Example: `AbstractRepository` that holds a `DataSource`, common CRUD code, and an abstract `mapRow()` for subclasses to implement.

```java
public abstract class AbstractRepository<T> {
	protected final DataSource ds;
	protected AbstractRepository(DataSource ds) { this.ds = ds; }
	public void save(T entity) { /* common save logic */ }
	protected abstract T mapRow(ResultSet rs);
}
```

	- **Interface Use Case:** Shared capability across unrelated types. Example: `Auditable` capability added to domain objects; multiple classes can implement it regardless of their class hierarchy.

```java
public interface Auditable {
	void setCreatedBy(String user);
	void setCreatedAt(Instant ts);
	default void markCreated(String user) {
		setCreatedBy(user);
		setCreatedAt(Instant.now());
	}
}
```

- **Quick Decision Checklist**
	- Need shared state/implementation -> choose **abstract class**.
	- Need multiple, orthogonal capabilities or broad public API -> choose **interface**.
	- Prefer composition over inheritance when appropriate.

## 6. What are access modifiers in Java?

**Answer:**

Access modifiers control the **visibility** of classes, fields, methods, and constructors. Java has four levels:

---

### Visibility Matrix

| Modifier | Same Class | Same Package | Subclass (other pkg) | Any Class |
|---|:---:|:---:|:---:|:---:|
| `private` | ✅ | ❌ | ❌ | ❌ |
| *(default)* | ✅ | ✅ | ❌ | ❌ |
| `protected` | ✅ | ✅ | ✅ | ❌ |
| `public` | ✅ | ✅ | ✅ | ✅ |

```text
+--------------------------------------------------+
|                   WORLD (public)                 |
|  +--------------------------------------------+  |
|  |       OTHER PACKAGE SUBCLASS (protected)   |  |
|  |  +--------------------------------------+  |  |
|  |  |       SAME PACKAGE (default)         |  |  |
|  |  |  +--------------------------------+  |  |  |
|  |  |  |     SAME CLASS (private)       |  |  |  |
|  |  |  +--------------------------------+  |  |  |
|  |  +--------------------------------------+  |  |
|  +--------------------------------------------+  |
+--------------------------------------------------+
```

---

### a. `private` — Same Class Only

```java
public class BankAccount {
    private double balance;        // Only BankAccount methods can touch this
    private String pin;

    public double getBalance() {   // Controlled access through public method
        return balance;
    }

    public void deposit(double amount) {
        if (amount > 0) balance += amount;  // Validation enforced here
    }

    private boolean isValidPin(String input) {  // Internal helper, hidden from outside
        return pin.equals(input);
    }
}

BankAccount acc = new BankAccount();
// acc.balance = 500;      // ❌ Compile error: balance has private access
acc.deposit(500);          // ✅ Only through the public API
```

---

### b. *(default / package-private)* — Same Package Only

No keyword — the absence of a modifier means package-private. Commonly used for internal implementation classes.

```java
// com.example.internal
class QueryBuilder {             // package-private class: only visible within com.example.internal
    String build(String table) { // package-private method
        return "SELECT * FROM " + table;
    }
}

// com.example.internal (same package) -- OK
class UserRepository {
    void loadUsers() {
        QueryBuilder qb = new QueryBuilder();   // ✅ Same package
        String query = qb.build("users");
    }
}

// com.example.api (different package) -- NOT OK
class UserService {
    // QueryBuilder qb = new QueryBuilder();  // ❌ Not visible outside package
}
```

---

### c. `protected` — Package + Subclasses

```java
// com.example.base
public abstract class Shape {
    protected String color;      // Accessible in subclasses anywhere

    protected double validate(double value) {    // Utility for subclasses
        if (value <= 0) throw new IllegalArgumentException("Must be positive");
        return value;
    }

    public abstract double area();
}

// com.example.shapes (different package, but extends Shape)
public class Circle extends Shape {
    private final double radius;

    public Circle(double radius, String color) {
        this.color  = color;            // ✅ Inherited protected field
        this.radius = validate(radius); // ✅ Inherited protected method
    }

    @Override
    public double area() { return Math.PI * radius * radius; }
}

// ⚠️ protected access via a non-subclass reference is still restricted
Shape s = new Circle(5, "red");
// s.color = "blue";   // ❌ Not accessible outside the package through a Shape reference
```

---

### d. `public` — Accessible Everywhere

Use for API surface — anything you intend external consumers to call.

```java
// Public API class
public class OrderService {
    public Order createOrder(String productId, int qty) { ... }   // ✅ Public API
    public void cancelOrder(String orderId)              { ... }   // ✅ Public API

    private Order buildOrder(String productId, int qty)  { ... }  // Internal logic
    private void notifyWarehouse(Order order)            { ... }   // Internal logic
}
```

---

### Access Modifiers on Classes (Top-level)

| Modifier | Allowed? | Meaning |
|---|:---:|---|
| `public` | ✅ | Visible everywhere; filename must match class name |
| *(default)* | ✅ | Visible only within the same package |
| `protected` | ❌ | Not allowed on top-level classes |
| `private` | ❌ | Not allowed on top-level classes |

```java
// File: com/example/internal/Helper.java
class Helper { }          // package-private -- hides internal utility from consumers

// File: com/example/api/UserApi.java
public class UserApi { }  // public -- part of the published API surface
```

> Nested / inner classes **can** use all four modifiers.

---

### Best Practices

- **Fields:** Always `private` — expose state only through methods (getters/setters or records)
- **Methods:** Start with the most restrictive; widen only when a caller genuinely needs access
- **Classes:** `public` for your API; package-private for internal implementation details
- **Constructors:** `private` for singletons/factories; `protected` for abstract base classes
- Apply the **Principle of Least Privilege** — grant only the minimum access required

## 7. What is the difference between `final`, `finally`, and `finalize()`?

**Answer:**

These three keywords share the word "final" but serve completely different purposes.

| Keyword | Type | Purpose |
|---|---|---|
| `final` | Modifier | Restricts change — on variables, methods, or classes |
| `finally` | Block | Guarantees cleanup code always runs after `try/catch` |
| `finalize()` | Method | Called by GC before object is collected (**deprecated**) |

---

### a. `final` — Prevents Modification

`final` can be applied to **variables**, **methods**, and **classes**.

```java
// 1. final VARIABLE -- cannot be reassigned
final int MAX_SIZE = 100;
MAX_SIZE = 200;  // ❌ Compile error: cannot assign to final variable

// Reference is final, but the object content can still change!
final List<String> list = new ArrayList<>();
list = new ArrayList<>();  // ❌ Compile error: cannot reassign reference
list.add("item");          // ✅ OK -- mutating the object, not the reference

// Blank final -- must be assigned exactly once (often in constructor)
public class Config {
    private final String apiKey;  // Blank final field

    public Config(String key) {
        this.apiKey = key;  // ✅ Assigned once in constructor
    }
}
```

```java
// 2. final METHOD -- cannot be overridden
public class Template {
    public final void process() {  // Subclasses CANNOT override this
        validate();
        execute();
        cleanup();
    }
    protected void execute() { }   // This CAN be overridden
}

// 3. final CLASS -- cannot be extended (no subclasses)
public final class String { }      // That's why String can't be subclassed!
public final class Integer { }     // Same for all wrapper classes
```

---

### b. `finally` — Guaranteed Cleanup Block

The `finally` block **always executes** after `try/catch`, regardless of whether an exception was thrown or a `return` was hit.

```java
Connection connection = null;
try {
    connection = getConnection();
    // ... risky DB operations
    return processResult(connection);  // finally STILL runs before returning
} catch (SQLException e) {
    log.error("DB error", e);
    throw new RuntimeException(e);    // finally STILL runs before re-throwing
} finally {
    // Runs in ALL cases: normal, exception, or return
    if (connection != null) {
        try { connection.close(); } catch (SQLException ignored) { }
    }
}
```

#### Edge Case — When `finally` does NOT run

```java
// Only two cases where finally is skipped:
System.exit(0);       // JVM terminates entirely
Thread.stop();        // Thread killed (deprecated/legacy)
// ... or JVM crash / power loss
```

#### Modern Alternative — `try-with-resources` (Java 7+)

```java
// Preferred: AutoCloseable resources are closed automatically
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {

    ps.setString(1, userId);
    ResultSet rs = ps.executeQuery();
    // ... process rs

}  // conn and ps are closed here automatically, even on exception
// No finally block needed!
```

> Any class implementing `AutoCloseable` / `Closeable` can be used in try-with-resources.

---

### c. `finalize()` — Deprecated ⚠️

`finalize()` was a method on `Object` called by the **Garbage Collector** just before reclaiming an object's memory. It was meant for native resource cleanup, but is **unreliable and dangerous**.

```java
// ❌ DON'T USE -- Deprecated since Java 9, removed in Java 18+
@Override
protected void finalize() throws Throwable {
    // Problems:
    // - Timing is unpredictable (or it may NEVER run)
    // - Can resurrect objects, causing memory leaks
    // - Slows down GC significantly
    // - Exceptions thrown here are silently ignored
    super.finalize();
}
```

#### Modern Replacements

```java
// ✅ 1. Try-with-resources (best for IO / DB resources)
try (FileInputStream fis = new FileInputStream("file.txt")) {
    // fis.close() called automatically
}

// ✅ 2. Implement AutoCloseable for your own classes
public class NativeResource implements AutoCloseable {
    private final long nativeHandle;

    public NativeResource() {
        this.nativeHandle = allocateNative();  // Native call
    }

    @Override
    public void close() {
        freeNative(nativeHandle);  // Explicit, deterministic cleanup
    }
}
// Usage: try (NativeResource r = new NativeResource()) { ... }

// ✅ 3. Cleaner API (Java 9+) -- safe finalizer alternative
import java.lang.ref.Cleaner;

public class ManagedResource implements AutoCloseable {
    private static final Cleaner CLEANER = Cleaner.create();
    private final Cleaner.Cleanable cleanable;
    private final long handle;

    public ManagedResource(long handle) {
        this.handle = handle;
        this.cleanable = CLEANER.register(this, () -> freeNative(handle));
    }

    @Override
    public void close() {
        cleanable.clean();  // Explicitly clean when done
    }
}
```

---

### Quick Comparison

```text
final     --> modifier    --> compile-time enforcement of immutability/non-overridability
finally   --> block       --> runtime guarantee of cleanup code execution
finalize  --> method      --> GC hook, unreliable -- AVOID, use AutoCloseable instead
```

## 8. Explain the concept of immutability in Java

**Answer:**

An **immutable object** is one whose state **cannot be changed after construction**. Every field is set once in the constructor and never modified again. The object is thread-safe by nature and safe to share freely.

---

### The 5 Rules for Building an Immutable Class

```java
// Rule 1: Declare the class as final (prevent subclasses from breaking immutability)
public final class ImmutablePerson {

    // Rule 2: All fields are private and final
    private final String name;
    private final int age;
    private final List<String> hobbies;    // mutable field -- needs extra care
    private final Address address;         // mutable field -- needs extra care

    // Rule 3: Set all fields only via constructor; apply defensive copies for mutable params
    public ImmutablePerson(String name, int age, List<String> hobbies, Address address) {
        this.name    = name;
        this.age     = age;
        this.hobbies = new ArrayList<>(hobbies);      // Defensive copy of incoming list
        this.address = new Address(address);           // Defensive copy of incoming object
    }

    // Rule 4: No setters -- expose only getters
    public String getName() { return name; }
    public int    getAge()  { return age; }

    // Rule 5: Return defensive copies / unmodifiable views of mutable fields
    public List<String> getHobbies() {
        return Collections.unmodifiableList(hobbies);  // Callers cannot mutate the list
    }

    public Address getAddress() {
        return new Address(address);                   // Return a copy, not the original
    }
}
```

---

### Why Defensive Copies Matter

Without them, callers can mutate your "immutable" object's internals:

```java
// ❌ BROKEN -- no defensive copy (field exposed directly)
public final class BadImmutable {
    private final List<String> tags;

    public BadImmutable(List<String> tags) {
        this.tags = tags;           // Stores the original reference!
    }

    public List<String> getTags() {
        return tags;                // Returns the internal reference!
    }
}

List<String> original = new ArrayList<>(List.of("java", "oop"));
BadImmutable obj = new BadImmutable(original);

original.add("hacked");            // Mutates obj.tags through the original list!
obj.getTags().add("also hacked");  // Mutates obj.tags through the returned reference!
```

```java
// ✅ CORRECT -- defensive copies in constructor AND getter
public final class GoodImmutable {
    private final List<String> tags;

    public GoodImmutable(List<String> tags) {
        this.tags = List.copyOf(tags);          // Immutable copy (Java 10+)
    }

    public List<String> getTags() {
        return tags;                            // Safe: List.copyOf is already unmodifiable
    }
}

List<String> original = new ArrayList<>(List.of("java", "oop"));
GoodImmutable obj = new GoodImmutable(original);

original.add("hacked");            // Has NO effect on obj.tags
// obj.getTags().add("x");         // ❌ Throws UnsupportedOperationException
```

---

### Why Immutability Matters

| Benefit | Explanation |
|---|---|
| **Thread-safe** | No synchronization needed — object state never changes |
| **Safe to share** | Can be passed anywhere without defensive copies at the call site |
| **Cache / hash keys** | `hashCode()` is stable — safe as `HashMap` key or in `Set` |
| **Reasoning** | Easier to predict behaviour — no hidden mutations |
| **Failure atomicity** | A half-constructed immutable object is never observable |

```java
// Thread safety -- no synchronized needed
private static final ImmutablePerson GUEST = new ImmutablePerson("Guest", 0, List.of(), new Address());

// Safe HashMap key -- hashCode never changes
Map<ImmutablePerson, String> roles = new HashMap<>();
roles.put(new ImmutablePerson("Alice", 30, List.of(), addr), "ADMIN");
```

---

### Built-in Immutable Classes in Java

```text
String                  -- most famous example; every "modification" returns a new object
Integer, Long, Double   -- all primitive wrapper classes
BigInteger, BigDecimal  -- arbitrary-precision numbers
LocalDate, LocalTime    -- java.time (Java 8+); all date/time types are immutable
Instant, Duration       -- same package
List.of(), Map.of()     -- unmodifiable collection factory methods (Java 9+)
List.copyOf()           -- immutable copy of existing collection (Java 10+)
```

---

### Modern Alternative — Records (Java 16+)

Java Records implement all 5 rules **automatically**:

```java
// Equivalent to the full ImmutablePerson class above (for simple value types)
public record Person(String name, int age) { }

Person p = new Person("Alice", 30);
p.name();   // "Alice"  (accessor, not getName())
p.age();    // 30

// Records are: final, fields are final, no setters, equals/hashCode/toString generated
// Note: for fields with mutable types, you still need a compact constructor
public record TaggedItem(String id, List<String> tags) {
    public TaggedItem {
        tags = List.copyOf(tags);  // Compact constructor -- defensive copy
    }
}
```

---

### Immutability Checklist

- `[x]` Class declared `final`
- `[x]` All fields `private final`
- `[x]` No setters
- `[x]` Defensive copy of **mutable inputs** in the constructor
- `[x]` Return **copies or unmodifiable views** of mutable fields in getters
- `[x]` Consider `List.copyOf()` / `Map.copyOf()` (Java 10+) for cleaner defensive copies

## 9. What is the difference between `String`, `StringBuilder`, and `StringBuffer`?

**Answer:**

---

### Core Differences at a Glance

| Feature | `String` | `StringBuilder` | `StringBuffer` |
|---|---|---|---|
| **Mutability** | Immutable | Mutable | Mutable |
| **Thread-safe** | ✅ Yes (immutable) | ❌ No | ✅ Yes (synchronized) |
| **Performance** | Slow (new object per change) | Fast (single buffer) | Slower than StringBuilder (lock overhead) |
| **Storage** | String Pool / Heap | Heap | Heap |
| **Since** | Java 1.0 | Java 1.5 | Java 1.0 |
| **Use case** | Constants, keys, shared data | Single-threaded building | Multi-threaded building (rare) |

---

### a. `String` — Immutable

Every modification creates a **new object**. The original string is never changed.

```java
String s = "Hello";
s = s + " World";   // New object: "Hello World" (old "Hello" stays in pool)
s = s + "!";        // New object: "Hello World!"

// String Pool: literal strings are cached and reused
String a = "Java";
String b = "Java";
a == b;             // true -- same object from pool

String c = new String("Java");
a == c;             // false -- c is a separate heap object
a.equals(c);        // true  -- same content
```

```text
// Memory model:
String Pool:  ["Hello"] ["World"] ["Java"]
Heap:         ["Hello World"] ["Hello World!"]  (created on each modification)
```

---

### b. `StringBuilder` — Mutable, NOT thread-safe

Uses an internal `char[]` buffer. Modifications happen **in place** — no new objects.

```java
StringBuilder sb = new StringBuilder(16);  // Initial capacity: 16 chars
sb.append("Hello");         // "Hello"        -- same buffer
sb.append(" World");        // "Hello World"  -- same buffer
sb.insert(5, ",");          // "Hello, World" -- same buffer
sb.replace(7, 12, "Java");  // "Hello, Java"  -- same buffer
sb.delete(5, 7);            // "HelloJava"    -- same buffer
sb.reverse();               // "avaJolleH"    -- same buffer

String result = sb.toString();  // Convert to String when done

// Fluent chaining
String result2 = new StringBuilder()
        .append("Name: ").append("Alice")
        .append(", Age: ").append(30)
        .toString();
// "Name: Alice, Age: 30"
```

```java
// BAD: String concatenation in loops (creates N temporary objects)
String result = "";
for (int i = 0; i < 1000; i++) {
    result += i;    // 1000 String objects created and discarded!
}

// GOOD: StringBuilder accumulates in one buffer
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
String result = sb.toString();
```

> **Note:** The Java compiler automatically rewrites `"a" + "b" + "c"` into a `StringBuilder` chain — but NOT concatenation inside loops.

---

### c. `StringBuffer` — Mutable, thread-safe (synchronized)

Identical API to `StringBuilder`, but every method is `synchronized`. Only needed when **multiple threads** share the same buffer instance.

```java
StringBuffer sharedBuffer = new StringBuffer();

Runnable task = () -> {
    for (int i = 0; i < 100; i++) {
        sharedBuffer.append("x");  // Safe: synchronized internally
    }
};

Thread t1 = new Thread(task);
Thread t2 = new Thread(task);
t1.start(); t2.start();
t1.join();  t2.join();

System.out.println(sharedBuffer.length());  // Always 200 (no race condition)
```

> **Modern alternative:** Prefer `String.join()`, `Collectors.joining()`, or external synchronization over `StringBuffer`.

---

### Practical Patterns

```java
// 1. Fixed text, constants, map keys --> String
private static final String PREFIX = "user_";
map.put("name", value);

// 2. Building in a single thread --> StringBuilder
public String buildQuery(String base, Map<String, String> params) {
    StringBuilder sb = new StringBuilder(base).append("?");
    params.forEach((k, v) -> sb.append(k).append("=").append(v).append("&"));
    sb.deleteCharAt(sb.length() - 1);  // Remove trailing &
    return sb.toString();
}

// 3. Joining a collection --> String.join() or Collectors.joining()
String csv  = String.join(", ", List.of("Alice", "Bob", "Carol"));
String csv2 = stream.collect(Collectors.joining(", ", "[", "]"));

// 4. Formatted strings --> formatted() (Java 15+)
String msg = "Hello, %s! You are %d years old.".formatted("Alice", 30);
```

#### When to Use

- **`String`** — Constants, map keys, method parameters, shared / thread-safe text
- **`StringBuilder`** — Building strings in loops or multi-step assembly (single-threaded)
- **`StringBuffer`** — Shared mutable buffer across threads (rare — prefer `String.join` or `Collectors.joining` instead)

## 13. What is the `this` keyword in Java?

**Answer:**

`this` is a reference to the **current object instance** inside a non-static context. It has three main uses:

---

### a. Disambiguate instance fields from local variables / parameters

```java
public class Person {
    private String name;
    private int age;

    public Person(String name, int age) {
        this.name = name;  // this.name = field; name = parameter
        this.age  = age;
    }

    public void setName(String name) {
        this.name = name;  // Without 'this', 'name' would hide the field
    }
}
```

---

### b. Constructor chaining with `this()`

Delegate one constructor to another in the same class. `this()` **must be the first statement**.

```java
public class Rectangle {
    private int width;
    private int height;
    private String color;

    // Primary constructor
    public Rectangle(int width, int height, String color) {
        this.width  = width;
        this.height = height;
        this.color  = color;
    }

    // Delegating constructors
    public Rectangle(int width, int height) {
        this(width, height, "white");  // Calls primary constructor
    }

    public Rectangle(int side) {
        this(side, side);  // Calls 2-arg constructor -> then primary
    }
}

// Usage
Rectangle r1 = new Rectangle(4, 6, "blue");
Rectangle r2 = new Rectangle(4, 6);   // color = "white"
Rectangle r3 = new Rectangle(5);      // 5x5 square, color = "white"
```

---

### c. Pass the current instance as an argument

```java
public class EventButton {
    private String label;

    public EventButton(String label) {
        this.label = label;
    }

    public void register(EventListener listener) {
        listener.onEvent(this);  // Pass this button to the listener
    }

    public String getLabel() { return label; }
}

// Listener receives the actual EventButton object
EventListener logger = (btn) -> System.out.println("Clicked: " + btn.getLabel());
new EventButton("Submit").register(logger);  // "Clicked: Submit"
```

---

### Key Rules

- `this` is only available in **non-static** methods and constructors (no object instance in static context).
- `this()` must be the **first statement** in a constructor — you cannot call both `this()` and `super()`.
- Compare with `super`: `this` → current instance / same-class constructor; `super` → parent class.

```text
                    +------------------------------------+
                    |          Inside a Class            |
                    |                                    |
  this.field        |  --> current instance field        |
  this.method()     |  --> current instance method       |
  this(args)        |  --> another constructor (same)    |
  return this       |  --> fluent / builder pattern      |
                    +------------------------------------+
```

> **Fluent Builder tip:** Returning `this` from setters enables method chaining.
>
> ```java
> public class Builder {
>     private String name;
>     private int age;
>
>     public Builder name(String name) { this.name = name; return this; }
>     public Builder age(int age)      { this.age  = age;  return this; }
>     public Person  build()           { return new Person(name, age); }
> }
>
> Person p = new Builder().name("Alice").age(30).build();
> ```

## 14. What is Java Project Lombok?

**Answer:**

**Project Lombok** is a **compile-time annotation processor** that automatically generates boilerplate Java code — constructors, getters/setters, `equals()`, `hashCode()`, `toString()`, and more — so you never have to write it by hand.

```text
Source code with @Lombok annotations
         │
         ▼  (javac annotation processing)
Generated bytecode = as if you wrote all the boilerplate
         │
         ▼
Clean, short source + full-featured .class file
```

---

### Setup (Maven)

```xml
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.32</version>
    <scope>provided</scope>   <!-- compile-time only, not in runtime JAR -->
</dependency>
```

> Also install the **Lombok plugin** in IntelliJ IDEA or Eclipse so the IDE understands generated methods.

---

### Core Annotations

#### `@Getter` / `@Setter`

```java
@Getter
@Setter
public class User {
    private Long   id;
    private String name;
    private String email;
}

// Lombok generates:
// public Long   getId()             { return id; }
// public String getName()           { return name; }
// public String getEmail()          { return email; }
// public void   setId(Long id)      { this.id = id; }
// public void   setName(String name){ this.name = name; }
// public void   setEmail(...)       { ... }

User u = new User();
u.setName("Alice");
System.out.println(u.getName());  // "Alice"
```

#### `@NoArgsConstructor`, `@AllArgsConstructor`, `@RequiredArgsConstructor`

```java
@NoArgsConstructor               // public User() {}
@AllArgsConstructor              // public User(Long id, String name, String email) {}
@RequiredArgsConstructor         // constructor for final / @NonNull fields only
public class User {
    private Long   id;
    private final String name;   // included in @RequiredArgsConstructor
    private String email;
}
```

#### `@ToString` / `@EqualsAndHashCode`

```java
@ToString(exclude = "passwordHash")       // Omit sensitive fields
@EqualsAndHashCode(of = "id")             // Use only 'id' for equality
public class User {
    private Long   id;
    private String name;
    private String passwordHash;
}

// Generates: equals(), hashCode() based on id only
// toString() → "User(id=1, name=Alice)"  (passwordHash excluded)
```

#### `@Data` — All-in-one for mutable classes

```java
// Equivalent to: @Getter + @Setter + @RequiredArgsConstructor + @ToString + @EqualsAndHashCode
@Data
public class ProductDto {
    private Long   id;
    private String name;
    private double price;
}
```

#### `@Value` — Immutable class

```java
// Equivalent to: @Getter + @AllArgsConstructor + @ToString + @EqualsAndHashCode
//                + makes class final + makes all fields private final
@Value
public class Money {
    double amount;
    String currency;
}

Money m = new Money(9.99, "USD");
// m.amount = 5;  // ❌ No setter, field is final
```

#### `@Builder` — Builder pattern generated automatically

```java
@Builder
@Value
public class Order {
    String  id;
    String  product;
    int     quantity;

Order order = Order.builder()

```

#### `@Slf4j` — Logger injection

```java
@Slf4j                     // Injects: private static final Logger log = LoggerFactory.getLogger(...)
@Service
public class OrderService {
    public void process(Order order) {
        log.info("Processing order: {}", order.getId());
        // ...
    }
}
// Also: @Log4j2, @Log, @CommonsLog for other logging frameworks
```

#### Common JPA pattern

```java
@Entity
@Table(name = "orders")
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@EqualsAndHashCode(of = "id")
@ToString(exclude = "items")
public class OrderEntity {
    @Id @GeneratedValue
    private Long id;
    private String status;

    @OneToMany(mappedBy = "order")
    private List<OrderItem> items;
}
```

### Tradeoffs

| Pro | Con |
|---|---|
| Drastically reduces boilerplate | Requires IDE plugin to work properly |
| Annotations are explicit and readable | Hides generated code — harder to debug |
| Works with any Java version | Magic at compile time can surprise teammates |
| Fine-grained control per annotation | Not part of the JDK — external dependency |
| `@Builder` is very powerful | `@Data` on JPA entities can cause issues (avoid!) |

> **Avoid `@Data` on `@Entity` classes** — it generates `equals()`/`hashCode()` using all fields, which breaks JPA lazy-loading proxies. Use `@EqualsAndHashCode(of = "id")` instead.

---

### Quick Reference

```text
@Getter / @Setter          --> generate getters / setters
@NoArgsConstructor         --> no-arg constructor (needed by JPA, Jackson)
@AllArgsConstructor        --> all-fields constructor
@RequiredArgsConstructor   --> constructor for final / @NonNull fields
@ToString                  --> toString() with optional exclusions
@EqualsAndHashCode         --> equals() & hashCode() with optional field selection
@Data                      --> @Getter + @Setter + @RequiredArgsConstructor + @ToString + @EqualsAndHashCode
@Value                     --> immutable class (all final)
@Builder                   --> builder pattern
@Slf4j / @Log4j2           --> logger field injection
```

## 15. What is Java Records, and how is it different from Project Lombok?

**Answer:**

---

### Java Records (Java 16+)

A **record** is a special, concise class declaration for **immutable data carriers** — objects whose sole purpose is to hold data. The compiler auto-generates everything boilerplate.

```java
// Everything below is generated automatically by the compiler:
//  - private final fields
//  - canonical (all-args) constructor
//  - accessor methods: name(), age(), email()
//  - equals(), hashCode(), toString()

public record Person(String name, int age, String email) { }

// Usage
Person p = new Person("Alice", 30, "alice@example.com");
System.out.println(p.name());     // "Alice"   (accessor, NOT getName())
System.out.println(p.age());      // 30
System.out.println(p);            // Person[name=Alice, age=30, email=alice@example.com]

Person p2 = new Person("Alice", 30, "alice@example.com");
p.equals(p2);   // true  -- structural equality built-in
```

#### Customising Records

```java
public record Person(String name, int age, String email) {

    // Compact canonical constructor -- validate/normalise inputs
    public Person {
        if (age < 0) throw new IllegalArgumentException("Age must be non-negative");
        name = name.strip();   // normalise before assignment
    }

    // Extra instance methods are allowed
    public String displayName() {
        return name + " (" + age + ")";
    }

    // Static factory / helpers
    public static Person guest() {
        return new Person("Guest", 0, "guest@example.com");
    }
}
```

#### Record limitations

- **Always `final`** — cannot be extended (can implement interfaces).
- **All fields are `final`** — no mutability; use `with`-style copies instead.
- **Cannot add instance fields** beyond the record components.
- **Not a replacement for full domain objects** with complex logic/state.

```java
// Simulating a "wither" (copy with one field changed)
Person updated = new Person(p.name(), p.age(), "newemail@example.com");
```

### Project Lombok

Lombok is a **third-party annotation processor** (compile-time) that eliminates boilerplate for *any* class — mutable or immutable, simple or complex.

```java
// Lombok annotations on a mutable class
@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
@EqualsAndHashCode
@ToString
public class PersonDto {
    private String name;
    private int    age;
    private String email;
}

// Lombok's equivalent of a record: @Value (immutable) or @Data (mutable)
@Value   // final class, final fields, all-args constructor, getters, equals, hashCode, toString
public class PersonValue {
    String name;
    int    age;
    String email;
}

// Builder pattern (no manual builder class needed)
@Builder
@Value
public class Order {
    String  id;
    String  product;
    int     quantity;
}

Order o = Order.builder().id("001").product("Book").quantity(2).build();
```

### Records vs Lombok — Side-by-Side Comparison

| Feature | Java Records | Lombok |
|---|---|---|
| **Origin** | Built into JDK (Java 16+) | Third-party library |
| **Activation** | Language keyword `record` | Annotation + build plugin |
| **Mutability** | Always immutable | Both (`@Value` = immutable, `@Data` = mutable) |
| **Accessor style** | `name()` (no `get` prefix) | `getName()` (JavaBean style) |
| **Builder pattern** | Manual or external | `@Builder` auto-generates |
| **Inheritance** | Cannot extend classes | Works with normal class hierarchy |
| **Extra fields** | Not allowed beyond components | Allowed freely |
| **IDE support** | Native (no plugin needed) | Requires Lombok plugin |
| **Reflection / bytecode** | Clean class file | Compiler-modified bytecode |
| **`@ToString` exclusions** | Manual override | `@ToString(exclude = "password")` |
| **Null-safety validation** | Manual compact constructor | `@NonNull` annotation |
| **Java version required** | Java 16+ (preview: 14–15) | Any Java version |

---

### When to Choose Which

```text
Use Java Records when:
  ✅ You need a simple, immutable data holder (DTOs, API responses, value objects)
  ✅ You want zero external dependencies
  ✅ You're on Java 16+
  ✅ Immutability is a design goal (domain events, query results)

Use Lombok when:
  ✅ You need mutable entities (JPA @Entity cannot be records)
  ✅ You need @Builder for complex object construction
  ✅ You're on Java 8–15 (pre-records)
  ✅ You need fine-grained control (@ToString.Exclude, @EqualsAndHashCode.Include)
  ✅ You mix immutable + mutable requirements in the same codebase
```

#### Practical Example — JPA Entity (must use Lombok, NOT Record)

```java
// ❌ Records don't work as JPA entities -- no no-args constructor, fields are final
// public record UserEntity(Long id, String username) { }  // WRONG

// ✅ Use Lombok with JPA
@Entity
@Table(name = "users")
@Getter @Setter
@NoArgsConstructor
@AllArgsConstructor
@EqualsAndHashCode(of = "id")
@ToString(exclude = "passwordHash")
public class UserEntity {
    @Id @GeneratedValue
    private Long   id;
    private String username;
    private String passwordHash;
}
```

> **Summary:** Records are the idiomatic Java choice for pure data containers. Lombok covers everything else — mutable entities, builders, and legacy Java support. They are **complementary**, not mutually exclusive.
