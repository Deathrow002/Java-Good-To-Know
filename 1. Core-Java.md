# Core Java (Fundamentals & OOP) — Questions & Answers

## 1. What are the main features of Java?
**Answer:**

Java is a general-purpose, object-oriented programming language designed for portability, robustness, and ease of development. Key features include:

- **Platform independence (Write Once, Run Anywhere):** Java code runs on the Java Virtual Machine (JVM), allowing the same bytecode to run on different operating systems.
- **Object-Oriented:** Encapsulation, inheritance, polymorphism, and abstraction are core to Java's design, promoting modular, reusable code.
- **Automatic Memory Management:** Garbage collection relieves developers from manual memory deallocation and helps prevent memory leaks.
- **Strong Standard Library (JDK):** Rich APIs for I/O, networking, concurrency, collections, JDBC, XML/JSON processing, and more.
- **Multithreading & Concurrency:** Built-in language and library support for threads, synchronization, executors, and high-level concurrency utilities.
- **Robustness & Security:** Strong type system, exception handling, runtime checks, and a security manager (when needed) improve reliability and safety.
- **Performance (JIT & Optimizations):** Just-In-Time (JIT) compilation and hotspot optimizations deliver competitive performance.
- **Portability & Backward Compatibility:** Emphasis on stable APIs and backward compatibility across Java versions.
- **Large Ecosystem & Tooling:** Mature tooling (Maven/Gradle, IDEs), frameworks (Spring), and wide library support.

Quick example: a simple `HelloWorld` program

```java
public class HelloWorld {
	public static void main(String[] args) {
		System.out.println("Hello, Java!");
	}
}
```

These features make Java suitable for everything from small utilities to large-scale enterprise applications, backend services, and Android development.

## 2. Explain the concept of Object-Oriented Programming in Java

```java
// Encapsulation: private data, controlled access
public class BankAccount {
	private double balance;  // Hidden state

	public void deposit(double amount) {
		if (amount > 0) balance += amount;  // Validation
	}
}

// Inheritance + Polymorphism
public abstract class Shape {
	public abstract double area();  // Abstraction
}

public class Circle extends Shape {
	private double radius;

	@Override
	public double area() {  // Polymorphism
		return Math.PI * radius * radius;
	}
}
```

#### Benefits of OOP

- **Modularity:** Classes can be written and maintained independently
- **Reusability:** Inheritance allows code reuse across classes
- **Flexibility:** Polymorphism enables interchangeable implementations
- **Maintainability:** Encapsulation isolates changes to specific classes

## 3. What is the difference between `==` and `.equals()` in Java?

```java
// String comparison
String s1 = new String("Hello");
String s2 = new String("Hello");
String s3 = "Hello";
String s4 = "Hello";

s1 == s2       // false -- different objects in heap
s1.equals(s2)  // true  -- same content

s3 == s4       // true  -- same object in String Pool (interned)
s3.equals(s4)  // true  -- same content

// Primitive comparison
int a = 5, b = 5;
a == b         // true  -- compares values directly
```

#### Best Practices

- Use `.equals()` for object content comparison
- Use `==` for primitive types and null checks
- Use `Objects.equals(a, b)` for null-safe comparison
- For strings, prefer `"literal".equals(variable)` to avoid NPE

## 4. What are `hashCode()` and `equals()` methods used for?

```text
// THE CONTRACT:
1. If a.equals(b) == true  ->  a.hashCode() == b.hashCode()  MUST
2. If a.hashCode() == b.hashCode()  ->  a.equals(b) may be true or false
3. If a.equals(b) == false  ->  hashCodes MAY differ (recommended for performance)

// CONSEQUENCE:
[!] Override BOTH or NEITHER -- never just one!
```

#### How HashMap Uses Both

```text
// When you do: map.put(key, value)
1. Calculate hashCode() -> determines bucket index
2. If bucket has entries -> use equals() to find exact match

// Visual:
HashMap Buckets:
[0] -> null
[1] -> [Entry("A")] -> [Entry("K")]  // Same hashCode, different objects
[2] -> [Entry("B")]
...
```

#### Proper Implementation

```java
public class Employee {
	private Long id;
	private String name;
    
	@Override
	public boolean equals(Object o) {
		if (this == o) return true;
		if (o == null || getClass() != o.getClass()) return false;
		Employee employee = (Employee) o;
		return Objects.equals(id, employee.id);  // Based on business key
	}
    
	@Override
	public int hashCode() {
		return Objects.hash(id);  // Same fields as equals()
	}
}

// Modern alternative: Use records (Java 14+)
public record Employee(Long id, String name) {}  // Auto-generates both!
```

## 5. Explain method overloading vs method overriding

```java
// OVERLOADING -- Same class, different parameters
public class Calculator {
	public int add(int a, int b) { return a + b; }
	public int add(int a, int b, int c) { return a + b + c; }  // Different params
	public double add(double a, double b) { return a + b; }    // Different types
}

// OVERRIDING -- Subclass provides specific implementation
public class Animal {
	public void speak() { System.out.println("Some sound"); }
}

public class Dog extends Animal {
	@Override  // Always use this annotation!
	public void speak() { System.out.println("Woof!"); }
}

// Runtime polymorphism
Animal animal = new Dog();
animal.speak();  // Output: "Woof!" -- Dog's version is called
```

#### Key Points

- **Overloading:** Provides multiple ways to call the same method with different inputs
- **Overriding:** Enables runtime polymorphism -- actual object type determines method
- `static`, `final`, and `private` methods cannot be overridden

## 6. What is the difference between an abstract class and an interface?

```java
// ABSTRACT CLASS -- Shared state + behavior
public abstract class Animal {
	protected String name;  // Instance state
    
	public Animal(String name) { this.name = name; }  // Constructor
    
	public void eat() { System.out.println(name + " eats"); }  // Concrete
	public abstract void speak();  // Abstract -- must implement
}

// INTERFACE -- Contract (capability)
public interface Flyable {
	int MAX_ALTITUDE = 10000;  // public static final
    
	void fly();  // Abstract
    
	default void land() {  // Default implementation (Java 8+)
		System.out.println("Landing...");
	}
}

// A class can extend ONE abstract class but implement MULTIPLE interfaces
public class Bird extends Animal implements Flyable, Comparable<Bird> {
	public Bird(String name) { super(name); }
    
	@Override public void speak() { System.out.println("Chirp!"); }
	@Override public void fly() { System.out.println("Flying!"); }
	@Override public int compareTo(Bird o) { return name.compareTo(o.name); }
}
```

#### When to Use

- **Abstract Class:** "IS-A" relationship with shared state/behavior (e.g., `Animal`)
- **Interface:** "CAN-DO" capability across unrelated classes (e.g., `Comparable`, `Serializable`)

### What is the difference between Interface and Abstract Class in Java (with use case)?

**Answer:**

- **Definition & Purpose**
	- **Abstract Class:** A base class that can contain fields, constructors, concrete methods, and abstract methods. Use when you want to provide shared state/behavior for a family of related classes.
	- **Interface:** A contract describing capabilities that classes can implement. Interfaces focus on behavior and (since Java 8) may include default and static methods.

- **Key Differences**
	- **Multiple Inheritance:** Classes can implement many interfaces but can extend only one class (abstract or concrete).
	- **State:** Abstract classes can have instance fields; interfaces cannot (only constants/static fields).
	- **Constructors:** Abstract classes can define constructors; interfaces cannot.
	- **Versioning:** Adding methods to abstract classes is easier for subclass reuse; adding methods to interfaces should use default methods carefully to avoid breaking implementors.

- **When to Use (Use Cases)**
	- **Abstract Class Use Case:** Shared implementation + state. Example: `AbstractRepository` that holds a `DataSource`, common CRUD code, and an abstract `mapRow()` for subclasses to implement.

```java
public abstract class AbstractRepository<T> {
	protected final DataSource ds;
	protected AbstractRepository(DataSource ds) { this.ds = ds; }
	public void save(T entity) { /* common save logic */ }
	protected abstract T mapRow(ResultSet rs);
}
```

	- **Interface Use Case:** Shared capability across unrelated types. Example: `Auditable` capability added to domain objects; multiple classes can implement it regardless of their class hierarchy.

```java
public interface Auditable {
	void setCreatedBy(String user);
	void setCreatedAt(Instant ts);
	default void markCreated(String user) {
		setCreatedBy(user);
		setCreatedAt(Instant.now());
	}
}
```

- **Quick Decision Checklist**
	- Need shared state/implementation -> choose **abstract class**.
	- Need multiple, orthogonal capabilities or broad public API -> choose **interface**.
	- Prefer composition over inheritance when appropriate.

## 7. What are access modifiers in Java?

```text
+--------------------------------------------------+
|                   WORLD (public)                 |
|  +--------------------------------------------+  |
|  |           SUBCLASS (protected)             |  |
|  |  +--------------------------------------+  |  |
|  |  |        PACKAGE (default)             |  |  |
|  |  |  +--------------------------------+  |  |  |
|  |  |  |        CLASS (private)         |  |  |  |
|  |  |  +--------------------------------+  |  |  |
|  |  +--------------------------------------+  |  |
|  +--------------------------------------------+  |
+--------------------------------------------------+
```

```java
public class BankAccount {
	private double balance;           // Only this class
	String accountType;               // Package-private (default)
	protected String ownerName;       // Package + subclasses
	public String accountNumber;      // Everyone
    
	private void updateBalance() { }  // Internal implementation
	public double getBalance() {      // Public API
		return balance;
	}
}
```

#### Best Practices

- **Fields:** Always `private` -- expose via methods if needed
- **Methods:** Start with most restrictive, widen only when required
- **Classes:** `public` for APIs, package-private for internal implementations
- Follow the principle of **least privilege**

## 8. What is the difference between `final`, `finally`, and `finalize()`?

```java
// FINAL -- Prevents modification
final int MAX_SIZE = 100;       // Constant variable (cannot reassign)
final List<String> list = new ArrayList<>();  // Reference can't change
list.add("item");               // But object content CAN change!

final class ImmutableClass { }  // Cannot be extended
class Parent {
	final void method() { }     // Cannot be overridden
}

// FINALLY -- Cleanup block
try {
	connection = getConnection();
	// risky operations
} catch (SQLException e) {
	log.error("DB error", e);
} finally {
	// ALWAYS runs -- even if exception thrown or return in try
	if (connection != null) connection.close();
}

// Modern alternative: try-with-resources
try (Connection conn = getConnection()) {
	// Auto-closes conn, no finally needed
}
```

#### About `finalize()` -- Deprecated

```java
// DON'T USE -- Deprecated since Java 9, removed in Java 18
@Override
protected void finalize() throws Throwable {
	// Unpredictable timing, may never run!
}

// INSTEAD USE:
1. try-with-resources (AutoCloseable)
2. Cleaner API (Java 9+)
3. Explicit close() methods
```

## 9. Explain the concept of immutability in Java

```java
public final class ImmutablePerson {  // 1. Make class final
	private final String name;         // 2. Make fields private and final
	private final List<String> hobbies;
    
	public ImmutablePerson(String name, List<String> hobbies) {
		this.name = name;
		this.hobbies = new ArrayList<>(hobbies);  // 3. Defensive copy
	}
    
	// 4. No setters -- only getters
	public String getName() { return name; }
    
	public List<String> getHobbies() {
		return Collections.unmodifiableList(hobbies);  // 5. Return copy/view
	}
}

// Modern alternative: Records (Java 14+)
public record Person(String name, int age) { }  // Automatically immutable!
```

#### Immutable Classes in Java

- `String` — most famous example
- `Integer`, `Double`, all wrapper classes
- `LocalDate`, `LocalTime`, `Instant` (java.time)
- `BigInteger`, `BigDecimal`

#### Immutability Checklist

- Class declared `final` (or all methods final)
- All fields `private final`
- No setters
- Defensive copies of mutable objects (in constructor and getters)
- No methods that modify internal state

## 10. What is the difference between `String`, `StringBuilder`, and `StringBuffer`?

```java
// String -- Creates new objects on each modification
String s = "Hello";      // ["Hello"]
s = s + " World";        // ["Hello"] + ["Hello World"] -- 2 objects!
s = s + "!";             // ["Hello"] + ["Hello World"] + ["Hello World!"] -- 3 objects!

// StringBuilder -- Modifies same buffer
StringBuilder sb = new StringBuilder("Hello");  // ["Hello_____"] (capacity)
sb.append(" World");     // ["Hello World_"] -- same buffer
sb.append("!");          // ["Hello World!"] -- same buffer
```

#### Code Examples

```java
// [OK] BAD: String concatenation in loop (creates many objects)
String result = "";
for (int i = 0; i < 1000; i++) {
	result += i;  // Creates 1000 String objects!
}

// GOOD: StringBuilder for building strings
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
	sb.append(i);  // Same buffer, efficient
}
String result = sb.toString();

// Modern alternatives
String result = String.join(", ", list);  // For joining
// String templates (Java 21+) example (note: syntax varies by JDK feature)
```

#### When to Use

- **String:** Constants, method parameters, HashMap keys, shared data
- **StringBuilder:** Building strings in loops, single-threaded scenarios
- **StringBuffer:** Multi-threaded string building (rare -- usually better alternatives)

## 11. What is the `extends` keyword in Java?

```java
// Class inheritance -- single parent only
public class Animal {
	protected String name;
    
	public void eat() {
		System.out.println(name + " is eating");
	}
}

public class Dog extends Animal {  // Dog IS-A Animal
	public void bark() {
		System.out.println(name + " says Woof!");
	}
}

// Interface extending multiple interfaces
public interface Swimmer { void swim(); }
public interface Runner { void run(); }

public interface Triathlete extends Swimmer, Runner {
	void cycle();  // Now has swim(), run(), and cycle()
}

// Generic type bound
public <T extends Comparable<T>> T findMax(List<T> list) {
	return Collections.max(list);  // T must be Comparable
}
```

#### What Gets Inherited

- `public` and `protected` members
- Package-private members (if same package)
- `private` members (not accessible, but still exist)
- Constructors (not inherited, but can be called via `super()`)

> Note: Java doesn't support multiple class inheritance to avoid the "diamond problem" — use interfaces instead.

## 12. What is the `implements` keyword in Java?

```java
// Define interfaces (contracts)
public interface Flyable {
	void fly();
	default void land() { System.out.println("Landing..."); }
}

public interface Swimmable {
	void swim();
}

// Implement multiple interfaces
public class Duck implements Flyable, Swimmable {
	@Override
	public void fly() {
		System.out.println("Duck is flying");
	}
    
	@Override
	public void swim() {
		System.out.println("Duck is swimming");
	}
	// land() is inherited from Flyable's default method
}

// Combine extends and implements
public class Penguin extends Bird implements Swimmable {
	@Override
	public void swim() {
		System.out.println("Penguin swims gracefully");
	}
}

// Using interfaces for polymorphism
Flyable flyer = new Duck();     // Interface reference
flyer.fly();                     // "Duck is flying"

List<Swimmable> swimmers = List.of(new Duck(), new Penguin());
swimmers.forEach(Swimmable::swim);  // Both swim!
```

## 13. What is the `super` keyword in Java?

```java
public class Animal {
	protected String name;
    
	public Animal(String name) {
		this.name = name;
		System.out.println("Animal constructor called");
	}
    
	public void speak() {
		System.out.println("Animal speaks");
	}
}

public class Dog extends Animal {
	private String breed;
    
	public Dog(String name, String breed) {
		super(name);  // 1. MUST be first line -- calls Animal(name)
		this.breed = breed;
		System.out.println("Dog constructor called");
	}
    
	@Override
	public void speak() {
		super.speak();  // 2. Call parent's speak() first
		System.out.println(name + " says Woof!");  // Then add behavior
	}
}

// Usage
Dog dog = new Dog("Buddy", "Golden Retriever");
// Output:
// Animal constructor called
// Dog constructor called

dog.speak();
// Output:
// Animal speaks
// Buddy says Woof!
```

#### Important Rules

- `super()` must be the **first statement** in a constructor
