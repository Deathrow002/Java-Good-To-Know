## Q1. What is SQL and what are its main categories of commands?

**Answer:**
SQL (Structured Query Language) is a standard language for managing and manipulating relational databases.

| Category | Full Form | Commands | Purpose |
|----------|-----------|----------|---------|
| **DDL** | Data Definition Language | `CREATE`, `ALTER`, `DROP`, `TRUNCATE` | Define/modify schema |
| **DML** | Data Manipulation Language | `SELECT`, `INSERT`, `UPDATE`, `DELETE` | Manipulate data |
| **DCL** | Data Control Language | `GRANT`, `REVOKE` | Control access |
| **TCL** | Transaction Control Language | `COMMIT`, `ROLLBACK`, `SAVEPOINT` | Manage transactions |

## Q2. What is the difference between `DELETE`, `TRUNCATE`, and `DROP`?

**Answer:**

| Feature | DELETE | TRUNCATE | DROP |
|---------|--------|----------|------|
| Type | DML | DDL | DDL |
| Removes | Specific rows (with `WHERE`) | All rows | Entire table/object |
| Can rollback? | Yes | No (in most DBs) | No |
| Triggers fired? | Yes | No | No |
| Resets identity? | No | Yes | N/A |
| Slower/Faster | Slower (row-by-row) | Faster | Fastest |

```sql
DELETE FROM employees WHERE id = 10;   -- removes one row
TRUNCATE TABLE employees;              -- removes all rows, keeps structure
DROP TABLE employees;                  -- removes table entirely
```

## Q3. What are Joins in SQL? Explain all types with examples.

**Answer:**
A JOIN combines rows from two or more tables based on a related column.

```sql
-- INNER JOIN: only matching rows from both tables
SELECT e.name, d.department_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;

-- LEFT JOIN: all rows from left table + matching from right
SELECT e.name, d.department_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;

-- RIGHT JOIN: all rows from right table + matching from left
SELECT e.name, d.department_name
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.id;

-- FULL OUTER JOIN: all rows from both tables
SELECT e.name, d.department_name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.id;

-- CROSS JOIN: cartesian product of both tables
SELECT e.name, d.department_name
FROM employees e
CROSS JOIN departments d;

-- SELF JOIN: join a table with itself
SELECT a.name AS employee, b.name AS manager
FROM employees a
JOIN employees b ON a.manager_id = b.id;
```

## Q4. What is the difference between `WHERE` and `HAVING`?

**Answer:**

| Feature | WHERE | HAVING |
|---------|-------|--------|
| Filters | Individual rows | Grouped results |
| Used with | Any query | `GROUP BY` clause |
| Aggregate functions | Not allowed | Allowed |
| Execution order | Before `GROUP BY` | After `GROUP BY` |

```sql
-- WHERE: filters rows before grouping
SELECT dept_id, COUNT(*) 
FROM employees
WHERE salary > 50000
GROUP BY dept_id;

-- HAVING: filters groups after grouping
SELECT dept_id, COUNT(*) AS total
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 5;
```

## Q5. What are Primary Key, Foreign Key, and Unique Key? How do they differ?

**Answer:**

| Feature | Primary Key | Foreign Key | Unique Key |
|---------|-------------|-------------|------------|
| Uniqueness | Must be unique | Not necessarily unique | Must be unique |
| NULL allowed | No | Yes (depends on DB) | Yes (one NULL allowed) |
| Per table | Only one | Multiple | Multiple |
| Purpose | Identifies each row | Links to another table | Prevents duplicates |

```sql
CREATE TABLE departments (
    id   INT PRIMARY KEY,
    name VARCHAR(100) UNIQUE
);

CREATE TABLE employees (
    id      INT PRIMARY KEY,
    name    VARCHAR(100),
    dept_id INT,
    email   VARCHAR(100) UNIQUE,
    FOREIGN KEY (dept_id) REFERENCES departments(id)
);
```

## Q6. What are Indexes in SQL? What are their types?

**Answer:**
An **Index** is a database object that improves the speed of data retrieval at the cost of additional storage and slower write operations.

| Type | Description |
|------|-------------|
| **Clustered** | Sorts and stores data rows physically; one per table |
| **Non-Clustered** | Separate structure pointing to data rows; multiple per table |
| **Unique** | Ensures all values in the indexed column are distinct |
| **Composite** | Index on multiple columns |
| **Full-Text** | Optimized for text-search queries |

```sql
-- Create a non-clustered index
CREATE INDEX idx_emp_name ON employees(name);

-- Create a unique index
CREATE UNIQUE INDEX idx_emp_email ON employees(email);

-- Create a composite index
CREATE INDEX idx_emp_dept_salary ON employees(dept_id, salary);

-- Drop an index
DROP INDEX idx_emp_name;
```

> **Tip:** Indexes speed up `SELECT` but slow down `INSERT`, `UPDATE`, and `DELETE`.

## Q7. What are SQL Aggregate Functions?

**Answer:**
Aggregate functions perform calculations on a set of values and return a single value.

| Function | Description |
|----------|-------------|
| `COUNT()` | Number of rows |
| `SUM()` | Total of a numeric column |
| `AVG()` | Average of a numeric column |
| `MIN()` | Minimum value |
| `MAX()` | Maximum value |

```sql
SELECT 
    COUNT(*)          AS total_employees,
    SUM(salary)       AS total_salary,
    AVG(salary)       AS avg_salary,
    MIN(salary)       AS min_salary,
    MAX(salary)       AS max_salary
FROM employees
WHERE dept_id = 2;

-- COUNT with GROUP BY
SELECT dept_id, COUNT(*) AS headcount
FROM employees
GROUP BY dept_id
ORDER BY headcount DESC;
```

## Q8. What are Subqueries? What are the different types?

**Answer:**
A **Subquery** (inner query) is a query nested inside another query.

```sql
-- Single-row subquery: returns one row
SELECT name, salary
FROM employees
WHERE salary = (SELECT MAX(salary) FROM employees);

-- Multi-row subquery: returns multiple rows (use IN, ANY, ALL)
SELECT name FROM employees
WHERE dept_id IN (SELECT id FROM departments WHERE location = 'New York');

-- Correlated subquery: references the outer query
SELECT name, salary
FROM employees e
WHERE salary > (SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id);

-- Subquery in FROM clause (derived table)
SELECT dept_id, avg_sal
FROM (SELECT dept_id, AVG(salary) AS avg_sal FROM employees GROUP BY dept_id) AS dept_avg
WHERE avg_sal > 60000;
```

## Q9. What are Window Functions in SQL?

**Answer:**
Window functions perform calculations across a set of rows related to the current row **without collapsing the result** (unlike `GROUP BY`).

```sql
-- ROW_NUMBER: unique sequential number per partition
SELECT name, dept_id, salary,
       ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rank
FROM employees;

-- RANK: same rank for ties, gaps in sequence
SELECT name, salary,
       RANK() OVER (ORDER BY salary DESC) AS salary_rank
FROM employees;

-- DENSE_RANK: same rank for ties, no gaps
SELECT name, salary,
       DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank
FROM employees;

-- LEAD / LAG: access next/previous row value
SELECT name, salary,
       LAG(salary)  OVER (ORDER BY hire_date) AS prev_salary,
       LEAD(salary) OVER (ORDER BY hire_date) AS next_salary
FROM employees;

-- SUM as running total
SELECT name, salary,
       SUM(salary) OVER (PARTITION BY dept_id ORDER BY hire_date) AS running_total
FROM employees;
```

## Q10. What is Normalization? Explain 1NF, 2NF, and 3NF.

**Answer:**
**Normalization** is the process of organizing a database to reduce redundancy and improve data integrity.

### 1NF (First Normal Form)
- Each column must contain **atomic** (indivisible) values.
- No repeating groups or arrays.

```
BEFORE 1NF:                    AFTER 1NF:
id | phones                    id | phone
1  | 111, 222                  1  | 111
                                1  | 222
```

### 2NF (Second Normal Form)
- Must be in 1NF.
- Every non-key column must depend on the **whole** primary key (no partial dependency).

```
BEFORE 2NF (composite key: order_id + product_id):
order_id | product_id | product_name | quantity
--product_name depends only on product_id, not the full key

AFTER 2NF: Split into Orders(order_id, product_id, quantity)
           and Products(product_id, product_name)
```

### 3NF (Third Normal Form)
- Must be in 2NF.
- No **transitive dependency** (non-key column depending on another non-key column).

```
BEFORE 3NF:
emp_id | dept_id | dept_name   -- dept_name depends on dept_id, not emp_id

AFTER 3NF: Split into Employees(emp_id, dept_id)
           and Departments(dept_id, dept_name)
```

## Q11. What are Transactions and ACID properties?

**Answer:**
A **Transaction** is a sequence of SQL operations executed as a single unit of work.

### ACID Properties

| Property | Description |
|----------|-------------|
| **Atomicity** | All operations succeed or all are rolled back (all or nothing) |
| **Consistency** | Database remains in a valid state before and after transaction |
| **Isolation** | Concurrent transactions do not interfere with each other |
| **Durability** | Committed changes persist even after system failure |

```sql
BEGIN TRANSACTION;

UPDATE accounts SET balance = balance - 1000 WHERE id = 1;  -- debit
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;  -- credit

-- Check for errors
IF @@ERROR <> 0
    ROLLBACK TRANSACTION;  -- undo all changes
ELSE
    COMMIT TRANSACTION;    -- save all changes

-- SAVEPOINT example
SAVEPOINT before_update;
UPDATE employees SET salary = salary * 1.1 WHERE dept_id = 3;
ROLLBACK TO SAVEPOINT before_update;  -- only undo this operation
```

## Q12. What are Views in SQL? What are their advantages and limitations?

**Answer:**
A **View** is a virtual table based on the result of a SQL query. It does not store data itself.

```sql
-- Create a view
CREATE VIEW active_employees AS
SELECT id, name, dept_id, salary
FROM employees
WHERE status = 'ACTIVE';

-- Query the view like a table
SELECT * FROM active_employees WHERE dept_id = 2;

-- Create an updatable view
CREATE VIEW emp_salary_view AS
SELECT id, name, salary FROM employees;

UPDATE emp_salary_view SET salary = 75000 WHERE id = 5;

-- Drop a view
DROP VIEW active_employees;
```

**Advantages:**
- Simplifies complex queries
- Provides security by restricting column/row access
- Logical data independence

**Limitations:**
- Not all views are updatable (e.g., views with `GROUP BY`, `DISTINCT`, aggregates)
- Performance overhead for complex views

## Q13. What is the difference between `UNION` and `UNION ALL`?

**Answer:**

| Feature | UNION | UNION ALL |
|---------|-------|-----------|
| Duplicates | Removes duplicates | Keeps all duplicates |
| Performance | Slower (sorts to remove duplicates) | Faster |
| Use case | When distinct results needed | When all results needed |

```sql
-- UNION: removes duplicate rows
SELECT name FROM employees_us
UNION
SELECT name FROM employees_uk;

-- UNION ALL: keeps all rows including duplicates
SELECT name FROM employees_us
UNION ALL
SELECT name FROM employees_uk;

-- Rules: same number of columns, compatible data types
SELECT id, name, 'US' AS region FROM employees_us
UNION ALL
SELECT id, name, 'UK' AS region FROM employees_uk
ORDER BY name;
```

## Q14. What are Stored Procedures and Functions? How do they differ?

**Answer:**

| Feature | Stored Procedure | Function |
|---------|-----------------|----------|
| Return value | Optional (via OUT params) | Must return a value |
| Return type | None or multiple | Single value or table |
| Usage in SQL | Cannot use in `SELECT` | Can use in `SELECT` |
| Transaction control | Allowed | Not allowed (in most DBs) |
| DML statements | Allowed | Restricted (in most DBs) |

```sql
-- Stored Procedure
CREATE PROCEDURE GetEmployeesByDept(IN dept_id INT)
BEGIN
    SELECT id, name, salary
    FROM employees
    WHERE dept_id = dept_id;
END;

-- Call procedure
CALL GetEmployeesByDept(3);

-- Function
CREATE FUNCTION GetTaxAmount(salary DECIMAL(10,2))
RETURNS DECIMAL(10,2)
DETERMINISTIC
BEGIN
    RETURN salary * 0.20;
END;

-- Use function in SELECT
SELECT name, salary, GetTaxAmount(salary) AS tax
FROM employees;
```

## Q15. What are the different types of SQL Constraints?

**Answer:**
Constraints enforce rules on data in a table to maintain accuracy and integrity.

| Constraint | Description |
|------------|-------------|
| `NOT NULL` | Column cannot have NULL values |
| `UNIQUE` | All values in column must be distinct |
| `PRIMARY KEY` | Uniquely identifies each row; NOT NULL + UNIQUE |
| `FOREIGN KEY` | Links two tables; enforces referential integrity |
| `CHECK` | Ensures values meet a specific condition |
| `DEFAULT` | Sets a default value when none is provided |

```sql
CREATE TABLE employees (
    id         INT           PRIMARY KEY,              -- PRIMARY KEY
    name       VARCHAR(100)  NOT NULL,                 -- NOT NULL
    email      VARCHAR(150)  UNIQUE,                   -- UNIQUE
    salary     DECIMAL(10,2) CHECK (salary > 0),       -- CHECK
    dept_id    INT           REFERENCES departments(id), -- FOREIGN KEY
    created_at TIMESTAMP     DEFAULT CURRENT_TIMESTAMP  -- DEFAULT
);

-- Add constraint to existing table
ALTER TABLE employees
ADD CONSTRAINT chk_salary CHECK (salary BETWEEN 10000 AND 500000);

-- Drop a constraint
ALTER TABLE employees
DROP CONSTRAINT chk_salary;
```
