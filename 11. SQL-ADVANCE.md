## Q1. What are Common Table Expressions (CTEs) and Recursive CTEs?

**Answer:**
A **CTE** (`WITH` clause) defines a named temporary result set scoped to a single query. It improves readability and allows reuse of the same subquery multiple times without repeating code.

### Non-Recursive CTE

```sql
WITH department_totals AS (
    SELECT dept_id, SUM(salary) AS total_salary, COUNT(*) AS headcount
    FROM employees
    GROUP BY dept_id
),
department_avg AS (
    SELECT dept_id, total_salary / headcount AS avg_salary
    FROM department_totals
)
SELECT d.name, da.avg_salary
FROM departments d
JOIN department_avg da ON d.id = da.dept_id
WHERE da.avg_salary > 60000
ORDER BY da.avg_salary DESC;
```

### Recursive CTE — Hierarchical Data (Org Chart)

A recursive CTE has two parts separated by `UNION ALL`:
1. **Anchor member** — the base/starting rows
2. **Recursive member** — references the CTE itself

```sql
-- Build full employee hierarchy from the CEO down
WITH RECURSIVE org_chart AS (

    -- Anchor: start from the top-level manager (no manager)
    SELECT id, name, manager_id, 1 AS level, name::TEXT AS path
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive: join each employee to their manager found so far
    SELECT e.id, e.name, e.manager_id, oc.level + 1, oc.path || ' > ' || e.name
    FROM employees e
    JOIN org_chart oc ON e.manager_id = oc.id
)
SELECT level, LPAD('', (level - 1) * 4, ' ') || name AS employee, path
FROM org_chart
ORDER BY path;
```

**Output example:**
```
level  employee
1      Alice (CEO)
2          Bob (VP)
3              Carol (Manager)
4                  Dave (Engineer)
```

**Common uses:** org charts, category trees, bill of materials, folder structures, graph traversal.

## Q2. What are Window Functions — Advanced Usage?

**Answer:**
Window functions compute a value for each row based on a "window" of related rows, **without** collapsing the result set like `GROUP BY` does.

### Syntax Anatomy

```sql
function_name(args)
  OVER (
    [PARTITION BY col1, col2]    -- divides rows into independent groups
    [ORDER BY col3 DESC]         -- defines row order within each partition
    [ROWS|RANGE BETWEEN ...]     -- defines the frame (sub-window)
  )
```

### Frame Clauses

```sql
-- Running total (default: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
SELECT order_date, amount,
       SUM(amount) OVER (ORDER BY order_date) AS running_total
FROM orders;

-- Sliding 7-day window (rolling average)
SELECT order_date, amount,
       AVG(amount) OVER (
           ORDER BY order_date
           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ) AS rolling_7day_avg
FROM orders;

-- Full partition total in every row (no ORDER BY → all rows in frame)
SELECT name, dept_id, salary,
       SUM(salary) OVER (PARTITION BY dept_id) AS dept_total,
       ROUND(salary * 100.0 / SUM(salary) OVER (PARTITION BY dept_id), 2) AS pct_of_dept
FROM employees;
```

### Ranking Functions

```sql
SELECT name, dept_id, salary,
       ROW_NUMBER()   OVER (PARTITION BY dept_id ORDER BY salary DESC) AS row_num,
       RANK()         OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rank,
       DENSE_RANK()   OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dense_rank,
       NTILE(4)        OVER (ORDER BY salary DESC)                      AS quartile,
       PERCENT_RANK() OVER (ORDER BY salary)                            AS percentile
FROM employees;
```

| Function | Ties | Gaps |
|---|---|---|
| `ROW_NUMBER()` | Unique number always | N/A |
| `RANK()` | Same rank | Yes (skips next) |
| `DENSE_RANK()` | Same rank | No |
| `NTILE(n)` | Distributes evenly | N/A |

### LAG / LEAD — Row-to-Row Comparisons

```sql
SELECT month, revenue,
       LAG(revenue, 1) OVER (ORDER BY month)                          AS prev_month,
       revenue - LAG(revenue, 1) OVER (ORDER BY month)                AS month_diff,
       ROUND((revenue - LAG(revenue, 1) OVER (ORDER BY month))
             * 100.0 / NULLIF(LAG(revenue, 1) OVER (ORDER BY month), 0), 2) AS pct_change
FROM monthly_sales;
```

### FIRST_VALUE / LAST_VALUE / NTH_VALUE

```sql
SELECT name, dept_id, salary,
       FIRST_VALUE(name) OVER (PARTITION BY dept_id ORDER BY salary DESC) AS top_earner,
       LAST_VALUE(salary) OVER (
           PARTITION BY dept_id ORDER BY salary DESC
           ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
       ) AS lowest_salary
FROM employees;
```

## Q3. What are Advanced JOIN Techniques?

**Answer:**

### LATERAL JOIN (PostgreSQL / MySQL 8+)

A `LATERAL` join allows the right-hand subquery to **reference columns from the left table** — like a correlated subquery but returning multiple rows.

```sql
-- Get the 3 most recent orders for each customer
SELECT c.name, recent.order_date, recent.total
FROM customers c
JOIN LATERAL (
    SELECT order_date, total
    FROM orders
    WHERE customer_id = c.id          -- references c — only possible with LATERAL
    ORDER BY order_date DESC
    LIMIT 3
) AS recent ON true;
```

### ANTI JOIN — Rows with No Match (Faster than NOT IN)

```sql
-- Customers who have NEVER placed an order
-- Method 1: LEFT JOIN + IS NULL (classic anti-join)
SELECT c.id, c.name
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.id IS NULL;

-- Method 2: NOT EXISTS (often better for NULLs)
SELECT id, name FROM customers c
WHERE NOT EXISTS (SELECT 1 FROM orders WHERE customer_id = c.id);

-- Avoid: NOT IN — returns no rows if subquery has any NULL
SELECT id, name FROM customers
WHERE id NOT IN (SELECT customer_id FROM orders);  -- ❌ breaks if NULL in subquery
```

### SEMI JOIN — Check Existence Without Duplicates

```sql
-- Customers who placed at least one order in 2025
SELECT DISTINCT c.id, c.name
FROM customers c
JOIN orders o ON c.id = o.customer_id
WHERE EXTRACT(YEAR FROM o.order_date) = 2025;

-- Cleaner with EXISTS (true semi-join — no duplicates, no full join)
SELECT id, name FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders
    WHERE customer_id = c.id
    AND EXTRACT(YEAR FROM order_date) = 2025
);
```

### Non-Equi JOIN

```sql
-- Employee salary bands — join on a range condition
SELECT e.name, e.salary, b.band_name
FROM employees e
JOIN salary_bands b ON e.salary BETWEEN b.min_salary AND b.max_salary;

-- Find all pairs of employees in the same department (avoid self-pairing)
SELECT a.name AS emp1, b.name AS emp2, a.dept_id
FROM employees a
JOIN employees b ON a.dept_id = b.dept_id AND a.id < b.id;
```

## Q4. What are SQL Execution Order and Query Optimization Techniques?

**Answer:**

### SQL Logical Execution Order

SQL clauses are **written** in one order but **executed** in a different order:

```
Execution Order    Clause
───────────────    ──────────────────────────────────────
1.  FROM           Identify source tables
2.  JOIN           Apply joins and produce intermediate set
3.  WHERE          Filter rows (before grouping)
4.  GROUP BY       Group rows
5.  HAVING         Filter groups
6.  SELECT         Evaluate expressions, window functions
7.  DISTINCT       Remove duplicates
8.  ORDER BY       Sort the result
9.  LIMIT/OFFSET   Restrict number of rows returned
```

> This is why column aliases defined in `SELECT` cannot be used in `WHERE` or `HAVING` (they haven't been computed yet), but **can** be used in `ORDER BY`.

### Reading an Execution Plan

```sql
EXPLAIN ANALYZE
SELECT e.name, d.name
FROM employees e
JOIN departments d ON e.dept_id = d.id
WHERE e.salary > 80000;
```

Key operations to look for:

| Plan Node | Meaning | Concern |
|---|---|---|
| `Seq Scan` | Full table scan | Bad on large tables without a filter index |
| `Index Scan` | Uses index to find rows | Good |
| `Index Only Scan` | All data from index (no heap access) | Best |
| `Nested Loop` | For each row in outer, scan inner | Fine for small tables |
| `Hash Join` | Build hash table from smaller set | Good for large sets |
| `Merge Join` | Both sides pre-sorted | Efficient with sorted data |
| `Sort` | Sorting in memory/disk | Expensive — add index |

### Common Optimization Techniques

```sql
-- 1. Use covering index (index contains all SELECTed columns)
CREATE INDEX idx_emp_cover ON employees(dept_id, salary, name);
-- → Index Only Scan: no heap access at all

-- 2. Avoid functions on indexed columns in WHERE
-- ❌ Bad — disables index on created_at
WHERE YEAR(created_at) = 2025

-- ✅ Good — range query uses index
WHERE created_at >= '2025-01-01' AND created_at < '2026-01-01'

-- 3. Avoid leading wildcards
-- ❌ Bad — full scan
WHERE name LIKE '%smith'

-- ✅ Good — uses index (prefix match)
WHERE name LIKE 'smith%'

-- 4. Use EXISTS instead of COUNT(*) > 0
-- ❌ Counts all matching rows
IF (SELECT COUNT(*) FROM orders WHERE customer_id = 1) > 0 ...

-- ✅ Stops at first match
IF EXISTS (SELECT 1 FROM orders WHERE customer_id = 1) ...
```

## Q5. What is Query Execution Plan and How to Use EXPLAIN?

**Answer:**
The query execution plan describes exactly **how the database engine retrieves data** for a query — which indexes it uses, how it joins tables, and estimated vs. actual costs.

```sql
-- Basic plan (estimates only)
EXPLAIN SELECT * FROM employees WHERE dept_id = 3;

-- Actual execution with timing and row counts
EXPLAIN ANALYZE SELECT * FROM employees WHERE dept_id = 3;

-- Verbose output with buffers (PostgreSQL)
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT e.name, d.name
FROM employees e JOIN departments d ON e.dept_id = d.id
WHERE e.salary > 70000;
```

### Reading the Output

```
Hash Join  (cost=15.10..42.30 rows=120 width=64) (actual time=0.5..2.3 rows=115)
  →  cost=startup..total   rows=estimated   width=bytes per row
  →  actual time=startup..total ms   rows=actual

  Hash  (cost=8.00..8.00 rows=10)
    →  Seq Scan on departments  (cost=0..8.00 rows=10)

  Seq Scan on employees  (cost=0..20.00 rows=120)
    Filter: (salary > 70000)
    Rows Removed by Filter: 380
```

### Key Metrics

| Metric | What it means |
|---|---|
| `cost=X..Y` | Estimated startup cost .. total cost (in arbitrary planner units) |
| `rows=N` | Estimated row count (compare with actual to spot bad statistics) |
| `actual time=X..Y` | Real milliseconds for startup .. last row |
| `Buffers: hit=N` | Pages read from cache (good); `read=N` means disk I/O (bad) |
| `Loops=N` | How many times this node was executed |

### Forcing Index Usage (for testing)

```sql
-- PostgreSQL: disable sequential scan to force index test
SET enable_seqscan = OFF;
EXPLAIN ANALYZE SELECT * FROM employees WHERE salary > 70000;
SET enable_seqscan = ON;
```

## Q6. What is Database Partitioning? Types and Use Cases?

**Answer:**
**Partitioning** splits a large table into smaller physical pieces (partitions) while presenting a single logical table to queries. The optimizer **prunes** irrelevant partitions, scanning only relevant data.

### Types of Partitioning

| Type | How | Best For |
|---|---|---|
| **Range** | Rows assigned by value range | Dates, IDs, time-series |
| **List** | Rows assigned by exact value list | Region, status, category |
| **Hash** | Rows assigned by hash of column | Even distribution, no natural range |
| **Composite** | Combination of above | Multi-dimensional filtering |

### Range Partitioning (PostgreSQL)

```sql
-- Create partitioned table
CREATE TABLE orders (
    id          BIGINT,
    order_date  DATE NOT NULL,
    customer_id INT,
    total       DECIMAL(10,2)
) PARTITION BY RANGE (order_date);

-- Create partitions for each year
CREATE TABLE orders_2023 PARTITION OF orders
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

CREATE TABLE orders_2024 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE orders_2025 PARTITION OF orders
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Query uses partition pruning automatically
EXPLAIN SELECT * FROM orders WHERE order_date BETWEEN '2025-03-01' AND '2025-03-31';
-- → only scans orders_2025 partition
```

### List Partitioning

```sql
CREATE TABLE sales (
    id     BIGINT,
    region VARCHAR(20),
    amount DECIMAL(10,2)
) PARTITION BY LIST (region);

CREATE TABLE sales_us  PARTITION OF sales FOR VALUES IN ('US', 'CA', 'MX');
CREATE TABLE sales_eu  PARTITION OF sales FOR VALUES IN ('UK', 'DE', 'FR');
CREATE TABLE sales_apac PARTITION OF sales FOR VALUES IN ('JP', 'AU', 'IN');
```

### Benefits and Tradeoffs

| Benefit | Tradeoff |
|---|---|
| Partition pruning — skip irrelevant data | Complexity in setup and maintenance |
| Faster `DELETE` of old data (drop partition) | Cross-partition queries can be slower |
| Parallel query across partitions | Indexes must be created per partition |
| Better cache utilization | FK constraints across partitions limited |

## Q7. What are Materialized Views and When Should You Use Them?

**Answer:**
A **Materialized View** physically stores the result of a query on disk (unlike a regular view which re-executes the query each time). It trades **storage and staleness** for **read performance**.

```sql
-- Create a materialized view for an expensive aggregation
CREATE MATERIALIZED VIEW monthly_revenue AS
SELECT
    DATE_TRUNC('month', order_date) AS month,
    region,
    SUM(total)   AS revenue,
    COUNT(*)     AS order_count,
    AVG(total)   AS avg_order_value
FROM orders
GROUP BY DATE_TRUNC('month', order_date), region
ORDER BY month, region;

-- Create an index on the materialized view for fast lookups
CREATE INDEX idx_mv_monthly_rev ON monthly_revenue(month, region);

-- Query the materialized view — no computation, just a table scan
SELECT * FROM monthly_revenue WHERE month = '2025-01-01' AND region = 'US';

-- Refresh to sync with base table (non-concurrent — locks the view)
REFRESH MATERIALIZED VIEW monthly_revenue;

-- Refresh without locking reads (requires a UNIQUE index on the view)
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_revenue;
```

### Regular View vs. Materialized View

| Feature | Regular View | Materialized View |
|---|---|---|
| Storage | None (virtual) | Physical (on disk) |
| Data freshness | Always current | Stale until refreshed |
| Query performance | Re-executes query | Reads stored data |
| Index support | No (indexes on base tables) | Yes |
| Use case | Simplify queries, security | Pre-aggregated reports, dashboards |

### When to Use

- Expensive aggregations queried frequently (dashboards, reports)
- Complex multi-table joins needed in many places
- Data warehouse denormalization
- Read replicas feeding BI tools

## Q8. What are SQL Locks, Deadlocks, and How to Prevent Them?

**Answer:**

### Types of Locks

| Lock Type | Compatibility | Acquired by |
|---|---|---|
| **Shared (S)** | Compatible with other Shared | `SELECT` (with some isolation levels) |
| **Exclusive (X)** | Incompatible with all | `INSERT`, `UPDATE`, `DELETE` |
| **Row-level** | Affects one row | `SELECT ... FOR UPDATE` |
| **Table-level** | Affects whole table | `LOCK TABLE`, DDL statements |
| **Advisory** | Application-defined | `pg_advisory_lock()` |

```sql
-- Explicitly lock rows for update (prevents other transactions from updating them)
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;

-- Lock without waiting (fail fast if locked)
SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;

-- Skip locked rows (useful for job queues)
SELECT * FROM tasks WHERE status = 'PENDING'
LIMIT 10
FOR UPDATE SKIP LOCKED;
```

### Deadlock

A deadlock occurs when two transactions each hold a lock the other needs:

```
Transaction A:  LOCK row 1 → waiting for row 2
Transaction B:  LOCK row 2 → waiting for row 1
                              ↑ circular wait → deadlock
```

```sql
-- Transaction A
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- locks row 1
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- waits for row 2
COMMIT;

-- Transaction B (concurrent)
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;   -- locks row 2
UPDATE accounts SET balance = balance + 50 WHERE id = 1;   -- waits for row 1 → DEADLOCK
COMMIT;
```

### Deadlock Prevention Strategies

```sql
-- 1. Always acquire locks in the SAME ORDER across all transactions
-- Both transactions should lock id=1 before id=2

-- 2. Use SELECT ... FOR UPDATE to acquire all locks upfront
BEGIN;
SELECT * FROM accounts WHERE id IN (1, 2) ORDER BY id FOR UPDATE; -- lock both at once
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- 3. Keep transactions short — hold locks for minimum time

-- 4. Set a lock timeout
SET lock_timeout = '5s';  -- fail if lock not acquired in 5 seconds

-- 5. Use SKIP LOCKED for job queues instead of competing for the same rows
```

## Q9. What are Isolation Levels and Concurrency Anomalies?

**Answer:**

### Concurrency Anomalies

| Anomaly | Description |
|---|---|
| **Dirty Read** | Read uncommitted data from another transaction that may roll back |
| **Non-Repeatable Read** | Same row read twice in a transaction returns different values |
| **Phantom Read** | Same query run twice returns different sets of rows |
| **Lost Update** | Two transactions read same data and overwrite each other's changes |

### Isolation Levels vs. Anomalies

| Isolation Level | Dirty Read | Non-Repeatable | Phantom Read | Performance |
|---|---|---|---|---|
| `READ UNCOMMITTED` | Possible | Possible | Possible | Highest |
| `READ COMMITTED` | Prevented | Possible | Possible | High |
| `REPEATABLE READ` | Prevented | Prevented | Possible | Medium |
| `SERIALIZABLE` | Prevented | Prevented | Prevented | Lowest |

```sql
-- Set isolation level for a transaction
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
SELECT balance FROM accounts WHERE id = 1;  -- reads 1000
-- (another transaction updates balance to 900 here)
SELECT balance FROM accounts WHERE id = 1;  -- still reads 1000 (repeatable)
COMMIT;

-- Serializable — prevents phantom reads
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN;
SELECT COUNT(*) FROM orders WHERE status = 'PENDING';  -- 10
-- (another transaction inserts a new PENDING order here)
SELECT COUNT(*) FROM orders WHERE status = 'PENDING';  -- still 10 (no phantom)
COMMIT;
```

### Lost Update Example and Fix

```sql
-- ❌ Lost Update — classic read-modify-write race condition
-- T1: reads balance = 1000, computes 1000 - 100 = 900, writes 900
-- T2: reads balance = 1000, computes 1000 - 200 = 800, writes 800
-- Final: 800 (T1's update lost!)

-- ✅ Fix 1: Atomic UPDATE (no separate SELECT)
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- ✅ Fix 2: Optimistic locking with version column
UPDATE accounts SET balance = 900, version = version + 1
WHERE id = 1 AND version = 5;  -- fails if another transaction already updated it
-- Check rows_affected = 0 → retry
```

## Q10. What is the Difference Between OLTP and OLAP? How Does it Affect Schema Design?

**Answer:**

| Aspect | OLTP | OLAP |
|---|---|---|
| **Purpose** | Day-to-day transactions | Analytics, reporting |
| **Query type** | Short, frequent reads/writes | Long, complex, read-heavy |
| **Data volume** | Thousands of rows per query | Millions/billions of rows |
| **Schema** | Normalized (3NF) | Denormalized (Star/Snowflake) |
| **Indexes** | Narrow, many | Few, but covering |
| **Writes** | Very frequent | Bulk loads (ETL) |
| **Examples** | E-commerce orders, banking | Data warehouse, BI dashboards |

### Star Schema (OLAP)

```sql
-- Fact table: stores measurements/events (large, narrow)
CREATE TABLE fact_sales (
    sale_id       BIGINT PRIMARY KEY,
    date_key      INT REFERENCES dim_date(date_key),
    product_key   INT REFERENCES dim_product(product_key),
    customer_key  INT REFERENCES dim_customer(customer_key),
    quantity      INT,
    revenue       DECIMAL(12,2)
);

-- Dimension tables: descriptive context (small, wide)
CREATE TABLE dim_product (
    product_key   INT PRIMARY KEY,
    product_name  VARCHAR(200),
    category      VARCHAR(100),
    brand         VARCHAR(100),
    unit_price    DECIMAL(10,2)
);

CREATE TABLE dim_date (
    date_key  INT PRIMARY KEY,
    full_date DATE,
    year      INT,
    quarter   INT,
    month     INT,
    week      INT,
    day_name  VARCHAR(10)
);

-- Typical OLAP query
SELECT d.year, d.quarter, p.category, SUM(f.revenue) AS total_revenue
FROM fact_sales f
JOIN dim_date    d ON f.date_key    = d.date_key
JOIN dim_product p ON f.product_key = p.product_key
WHERE d.year = 2025
GROUP BY d.year, d.quarter, p.category
ORDER BY d.quarter, total_revenue DESC;
```

## Q11. What are Advanced Aggregation Techniques — GROUPING SETS, ROLLUP, CUBE?

**Answer:**
These extensions to `GROUP BY` let you compute **multiple levels of aggregation in a single query**, avoiding `UNION ALL` of separate grouped queries.

### GROUPING SETS

Specify exactly which grouping combinations you want:

```sql
-- Equivalent to 3 separate GROUP BY queries combined with UNION ALL
SELECT region, product, SUM(sales) AS total
FROM sales_data
GROUP BY GROUPING SETS (
    (region, product),   -- group by both
    (region),            -- group by region only
    ()                   -- grand total
);
```

### ROLLUP — Hierarchical Subtotals

Produces subtotals that roll up from the most granular level to the grand total:

```sql
SELECT year, quarter, month, SUM(revenue) AS total_revenue
FROM sales
GROUP BY ROLLUP (year, quarter, month);
-- Generates: (year, quarter, month), (year, quarter), (year), ()
```

### CUBE — All Possible Combinations

```sql
SELECT region, product_category, salesperson, SUM(amount)
FROM sales
GROUP BY CUBE (region, product_category, salesperson);
-- Generates all 2^3 = 8 combinations of grouping
```

### GROUPING() Function — Identify Subtotal Rows

```sql
SELECT
    CASE WHEN GROUPING(region) = 1 THEN 'ALL REGIONS' ELSE region END AS region,
    CASE WHEN GROUPING(product) = 1 THEN 'ALL PRODUCTS' ELSE product END AS product,
    SUM(sales) AS total,
    GROUPING(region) AS is_region_subtotal
FROM sales_data
GROUP BY ROLLUP (region, product);
```

## Q12. How to Write Efficient Pagination Queries?

**Answer:**

### Naive OFFSET Pagination — Why It's Slow

```sql
-- Page 1: scans rows 1..20
SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 0;

-- Page 500: scans and DISCARDS rows 1..9980, then returns 20
SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 9980;
-- → O(OFFSET) cost — gets worse as user goes deeper
```

### Keyset (Cursor) Pagination — O(1) Cost

Uses the last seen value to filter instead of skipping rows:

```sql
-- First page
SELECT id, title, created_at
FROM articles
ORDER BY created_at DESC, id DESC
LIMIT 20;

-- Next page — pass last seen (created_at, id) from previous page
SELECT id, title, created_at
FROM articles
WHERE (created_at, id) < ('2025-03-15 10:00:00', 4523)  -- cursor
ORDER BY created_at DESC, id DESC
LIMIT 20;
-- → Index seek directly to cursor position — O(1) regardless of page depth
```

```sql
-- Required composite index for keyset pagination
CREATE INDEX idx_articles_cursor ON articles(created_at DESC, id DESC);
```

### Comparison

| Method | Performance | Supports Random Page? | Supports Inserts Mid-Page? |
|---|---|---|---|
| `OFFSET / LIMIT` | O(OFFSET) — degrades | Yes | Can skip/duplicate rows |
| Keyset / Cursor | O(1) always | No (next/prev only) | Stable — no skips |

### Count Optimization

```sql
-- ❌ Slow — full scan + count
SELECT COUNT(*) FROM orders WHERE status = 'ACTIVE';

-- ✅ Use approximate count for display (PostgreSQL)
SELECT reltuples::BIGINT AS approx_count
FROM pg_class WHERE relname = 'orders';

-- ✅ Cache total count in a separate counter table updated by a trigger
```

## Q13. What are Triggers in SQL? Types and Use Cases?

**Answer:**
A **Trigger** is a stored procedure that automatically executes in response to a DML (`INSERT`, `UPDATE`, `DELETE`) or DDL event on a table.

### Types of Triggers

| Type | Fires | Access |
|---|---|---|
| `BEFORE INSERT/UPDATE/DELETE` | Before the DML changes data | `NEW` (insert/update), `OLD` (update/delete) |
| `AFTER INSERT/UPDATE/DELETE` | After the DML changes data | `NEW`, `OLD` |
| `INSTEAD OF` | Replaces the DML (on views) | `NEW`, `OLD` |
| `FOR EACH ROW` | Once per affected row | Row-level |
| `FOR EACH STATEMENT` | Once per SQL statement | Statement-level |

### Example 1: Audit Log Trigger

```sql
-- Audit table
CREATE TABLE employee_audit (
    audit_id    SERIAL PRIMARY KEY,
    action      VARCHAR(10),
    employee_id INT,
    old_salary  DECIMAL(10,2),
    new_salary  DECIMAL(10,2),
    changed_by  VARCHAR(100),
    changed_at  TIMESTAMP DEFAULT NOW()
);

-- Trigger function (PostgreSQL)
CREATE OR REPLACE FUNCTION log_salary_change()
RETURNS TRIGGER AS $$
BEGIN
    IF OLD.salary <> NEW.salary THEN
        INSERT INTO employee_audit(action, employee_id, old_salary, new_salary, changed_by)
        VALUES ('UPDATE', OLD.id, OLD.salary, NEW.salary, CURRENT_USER);
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Attach trigger
CREATE TRIGGER trg_salary_audit
AFTER UPDATE ON employees
FOR EACH ROW
EXECUTE FUNCTION log_salary_change();
```

### Example 2: BEFORE Trigger — Data Validation

```sql
CREATE OR REPLACE FUNCTION validate_order()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.total <= 0 THEN
        RAISE EXCEPTION 'Order total must be positive, got %', NEW.total;
    END IF;
    NEW.updated_at := NOW();  -- auto-set timestamp
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_validate_order
BEFORE INSERT OR UPDATE ON orders
FOR EACH ROW
EXECUTE FUNCTION validate_order();
```

### Example 3: INSTEAD OF Trigger — Updatable View

```sql
CREATE VIEW active_employees AS
    SELECT id, name, salary FROM employees WHERE status = 'ACTIVE';

CREATE OR REPLACE FUNCTION update_active_emp()
RETURNS TRIGGER AS $$
BEGIN
    UPDATE employees SET name = NEW.name, salary = NEW.salary WHERE id = OLD.id;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_update_active_emp
INSTEAD OF UPDATE ON active_employees
FOR EACH ROW EXECUTE FUNCTION update_active_emp();
```

## Q14. What is Database Sharding, and How Does It Differ from Replication?

**Answer:**

### Replication — Same Data, Multiple Copies

Replication copies the **entire dataset** to one or more replica servers. Used for **high availability** and **read scaling**.

```
Primary DB ──► Replica 1 (read replica)
           └──► Replica 2 (standby / failover)

Writes → Primary only
Reads  → Any replica
```

| Type | Description |
|---|---|
| **Synchronous** | Primary waits for replica to confirm write — no data loss, higher latency |
| **Asynchronous** | Primary commits immediately — faster, small replication lag possible |
| **Logical** | Replicate specific tables/rows (e.g., PostgreSQL logical replication) |

### Sharding — Different Data, Multiple Nodes

Sharding **horizontally partitions** data across multiple independent database instances (shards). Each shard holds a different subset.

```
Shard 1: customer_id 1      – 1,000,000
Shard 2: customer_id 1,000,001 – 2,000,000
Shard 3: customer_id 2,000,001 – 3,000,000
```

**Shard Key Strategies:**

| Strategy | Example | Pros | Cons |
|---|---|---|---|
| **Range** | `customer_id` ranges | Simple, range queries easy | Hot spots on sequential inserts |
| **Hash** | `hash(user_id) % N` | Even distribution | Cross-shard range queries hard |
| **Directory** | Lookup table maps key → shard | Flexible | Lookup table bottleneck |
| **Geographic** | Region-based routing | Data locality | Uneven shard sizes |

### Replication vs. Sharding

| Aspect | Replication | Sharding |
|---|---|---|
| Data per node | Full copy of all data | Subset of data |
| Write throughput | Limited (all writes to primary) | Scales (writes distributed) |
| Read throughput | Scales (many replicas) | Scales (per shard) |
| Complexity | Low | High |
| Cross-node queries | Easy (any node has all data) | Hard (scatter-gather) |
| Use case | HA, disaster recovery, read scaling | Massive data volume, write scaling |

## Q15. What are Advanced SQL Anti-Patterns and How to Fix Them?

**Answer:**

### Anti-Pattern 1: SELECT * in Production Code

```sql
-- ❌ Bad: fetches all columns, breaks when schema changes, prevents index-only scans
SELECT * FROM employees WHERE dept_id = 3;

-- ✅ Good: explicit columns, enables covering index, documents intent
SELECT id, name, salary FROM employees WHERE dept_id = 3;
```

### Anti-Pattern 2: Implicit Type Conversion Disabling Index

```sql
-- ❌ Bad: 'customer_id' is INT but '123' is a string → full table scan
SELECT * FROM orders WHERE customer_id = '123';

-- ✅ Good: matching types
SELECT * FROM orders WHERE customer_id = 123;
```

### Anti-Pattern 3: OR on Different Columns (Index Killer)

```sql
-- ❌ Bad: optimizer can't use a single index for OR across different columns
SELECT * FROM products WHERE category = 'Electronics' OR brand = 'Apple';

-- ✅ Good: UNION of two indexed seeks
SELECT * FROM products WHERE category = 'Electronics'
UNION
SELECT * FROM products WHERE brand = 'Apple';
```

### Anti-Pattern 4: Correlated Subquery in SELECT (N+1 in SQL)

```sql
-- ❌ Bad: runs one subquery per row returned
SELECT name,
    (SELECT SUM(amount) FROM orders WHERE customer_id = c.id) AS total_spent
FROM customers c;

-- ✅ Good: single aggregation join
SELECT c.name, COALESCE(o.total_spent, 0) AS total_spent
FROM customers c
LEFT JOIN (
    SELECT customer_id, SUM(amount) AS total_spent
    FROM orders
    GROUP BY customer_id
) o ON c.id = o.customer_id;
```

### Anti-Pattern 5: Using NOT IN with a Nullable Subquery

```sql
-- ❌ Bad: returns ZERO rows if any manager_id is NULL in the subquery
SELECT name FROM employees
WHERE id NOT IN (SELECT manager_id FROM employees);

-- ✅ Good: filter NULLs explicitly
SELECT name FROM employees
WHERE id NOT IN (SELECT manager_id FROM employees WHERE manager_id IS NOT NULL);

-- ✅ Best: use NOT EXISTS — handles NULLs correctly and often faster
SELECT name FROM employees e
WHERE NOT EXISTS (
    SELECT 1 FROM employees WHERE manager_id = e.id
);
```

### Anti-Pattern 6: DISTINCT to Fix Bad Joins

```sql
-- ❌ Bad: DISTINCT hides the bug — join is producing duplicates
SELECT DISTINCT c.name, c.email
FROM customers c
JOIN orders o ON c.id = o.customer_id;

-- ✅ Good: fix the join logic — use EXISTS if you only need customers who ordered
SELECT name, email FROM customers c
WHERE EXISTS (SELECT 1 FROM orders WHERE customer_id = c.id);
```

### Anti-Pattern 7: Storing Comma-Separated Values in a Column

```sql
-- ❌ Bad: violates 1NF, impossible to index, hard to query
-- tags column: 'java,spring,sql,microservices'
SELECT * FROM articles WHERE tags LIKE '%spring%';  -- full scan always

-- ✅ Good: separate tags table
CREATE TABLE article_tags (
    article_id INT REFERENCES articles(id),
    tag        VARCHAR(50),
    PRIMARY KEY (article_id, tag)
);
CREATE INDEX idx_article_tags_tag ON article_tags(tag);
SELECT a.* FROM articles a JOIN article_tags t ON a.id = t.article_id WHERE t.tag = 'spring';
```

### Summary Reference

| Anti-Pattern | Why It's Bad | Fix |
|---|---|---|
| `SELECT *` | No covering index, fragile | Explicit column list |
| Implicit type cast | Disables index | Match data types |
| `OR` on different columns | Index unusable | `UNION` |
| Correlated subquery in `SELECT` | O(n) queries | Aggregation join |
| `NOT IN` with NULLs | Returns empty set | `NOT EXISTS` |
| `DISTINCT` on bad join | Hides bug | Fix join; use `EXISTS` |
| CSV in column | Breaks 1NF, unsearchable | Normalize into child table |
