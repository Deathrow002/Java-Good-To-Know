# Linux/Unix Interview Questions & Answers for Advanced

A deep-dive reference of 10 advanced Linux/Unix interview questions covering kernel internals, process management, filesystem architecture, networking, and security.

---

## 1. What is the difference between a process and a thread in Linux?

**Topic:** Kernel

**Answer:**  
A process is an independent execution unit with its own virtual address space, file descriptors, and resources. A thread (also called a lightweight process) shares the address space and resources of its parent process. In Linux, both are created with the `clone()` syscall — the difference lies in which resources are shared via flags like `CLONE_VM` (share memory) and `CLONE_FILES` (share file descriptors). The `fork()` call creates a new process; `pthread_create()` creates a thread.

```bash
clone(fn, stack, CLONE_VM|CLONE_FILES|CLONE_SIGHAND, arg)  # thread
clone(fn, stack, SIGCHLD, arg)                              # process (fork)
```

> **Interview tip:** Linux has no separate kernel thread structure — threads and processes are both `task_struct` entries in the kernel.

## 2. Explain the Linux boot process from power-on to login prompt.

**Topic:** Kernel

**Answer:**  
1. BIOS/UEFI runs POST and finds the bootloader.
2. GRUB (bootloader) loads the kernel image and initramfs into memory.
3. The kernel initializes hardware and mounts the initramfs as a temporary root filesystem.
4. The kernel executes `/sbin/init` (or systemd on modern systems).
5. systemd runs targets (equivalent to runlevels), starting services and mounting the real root filesystem.
6. A `getty` process spawns on each TTY, presenting a login prompt.

```
BIOS/UEFI → GRUB → vmlinuz + initramfs → kernel init
→ systemd (PID 1) → multi-user.target → getty → login
```

## 3. What are zombie processes and how do you eliminate them?

**Topic:** Processes

**Answer:**  
A zombie process has finished execution but still has an entry in the process table because its parent has not yet read its exit status via `wait()`. Zombies consume a process table slot but no CPU or memory. You cannot kill a zombie directly — you must signal its parent to call `wait()`. If the parent is unresponsive, killing the parent causes zombies to be re-parented to init (PID 1), which automatically reaps them.

```bash
ps aux | awk '$8 == "Z"'       # find zombie processes
kill -CHLD <parent_pid>        # signal parent to reap
kill -9 <parent_pid>           # last resort: kill parent
```

> **Interview tip:** Knowing that you cannot kill a zombie directly, and why, is a key distinction interviewers test.

## 4. What is the OOM Killer and when does Linux invoke it?

**Topic:** Processes

**Answer:**  
The Out-Of-Memory Killer is a kernel mechanism invoked when the system cannot satisfy a memory allocation request even after exhausting swap. The kernel assigns every process an `oom_score` (0–1000) based on RSS, swap usage, CPU time, and age. The process with the highest score is killed. You can influence this via `/proc/PID/oom_score_adj` (range: -1000 to +1000). Setting it to `-1000` makes a process immune.

```bash
cat /proc/<pid>/oom_score               # view current score
echo -1000 > /proc/<pid>/oom_score_adj  # protect process from OOM killer
dmesg | grep -i 'oom'                   # see past OOM kill events
```

## 5. What is an inode, and what happens when you run out of inodes?

**Topic:** Filesystem

**Answer:**  
An inode is a data structure storing metadata about a file: permissions, owner, timestamps, and pointers to data blocks — but **not** the filename. The filename-to-inode mapping lives in the directory entry. Each filesystem has a fixed inode count set at creation time. When inodes are exhausted, you cannot create new files even if disk space is available. This commonly happens in directories with millions of tiny files.

```bash
df -i                        # check inode usage across filesystems
ls -i file.txt               # show a file's inode number
stat file.txt                # full inode metadata
find / -xdev -printf '%h\n' | sort | uniq -c | sort -k 1 -rn | head
```

> **Interview tip:** A common production issue — always monitor both disk space (`df -h`) and inode usage (`df -i`).

## 6. Explain the difference between a hard link and a symbolic (soft) link.

**Topic:** Filesystem

**Answer:**  
- **Hard link** — a directory entry pointing directly to an inode. Multiple hard links to the same inode mean the file data persists until all links are removed. Hard links cannot span filesystems and cannot link to directories.
- **Symbolic link (symlink)** — a special file that stores a path to another file. Symlinks can span filesystems and link to directories, but break if the target is moved or deleted.

```bash
ln file.txt hardlink.txt        # hard link — shares the same inode
ln -s file.txt symlink.txt      # symlink — stores the target path
ls -li file.txt hardlink.txt    # same inode number shown
readlink symlink.txt            # displays the target path
```

## 7. What is the purpose of the TIME_WAIT state in TCP, and why does it matter on a busy server?

**Topic:** Networking

**Answer:**  
After a TCP connection closes, the side that sends the last FIN enters `TIME_WAIT` for 2×MSL (Maximum Segment Lifetime, typically 60 seconds) to ensure any delayed packets from the old connection are discarded before a new connection reuses the same 4-tuple (src IP, src port, dst IP, dst port). On a high-throughput server this can exhaust ephemeral ports. Mitigation strategies include enabling `SO_REUSEADDR`, tuning `net.ipv4.tcp_tw_reuse`, or reducing `net.ipv4.tcp_fin_timeout`.

```bash
ss -s                               # socket summary including TIME_WAIT count
sysctl net.ipv4.tcp_tw_reuse        # check reuse setting (0 = disabled)
sysctl -w net.ipv4.tcp_tw_reuse=1   # enable socket reuse
sysctl net.ipv4.tcp_fin_timeout     # view FIN timeout value
```

> **Interview tip:** Never blindly set `tcp_tw_recycle=1` — it breaks connections from NAT devices and was removed entirely in Linux 4.12.

## 8. What are Linux capabilities, and how do they improve on the setuid model?

**Topic:** Security

**Answer:**  
Traditionally, privilege was binary: root (all privileges) vs. non-root. Linux capabilities split root's superpowers into ~40 distinct units — for example:

| Capability | Purpose |
|---|---|
| `CAP_NET_BIND_SERVICE` | Bind to ports below 1024 |
| `CAP_SYS_PTRACE` | Trace other processes |
| `CAP_NET_RAW` | Use raw sockets |
| `CAP_SYS_ADMIN` | Broad system administration |

A process or binary can hold only the specific capabilities it needs, reducing the blast radius of a compromise. Capabilities can be granted per-thread via the `prctl()` syscall or per-binary via `setcap`.

```bash
getcap /usr/bin/ping              # view binary capabilities
setcap cap_net_raw+ep ./myapp    # grant raw socket capability
setcap cap_net_bind_service+ep ./server  # bind privileged ports without root
capsh --print                    # view current thread capabilities
```

## 9. How does SELinux differ from standard Unix DAC permissions?

**Topic:** Security

**Answer:**  
- **Standard Unix DAC (Discretionary Access Control):** File owners set permissions and can grant access freely. Root can bypass all restrictions.
- **SELinux MAC (Mandatory Access Control):** A system-wide policy set by an administrator governs all access decisions, overriding DAC. Every process and file has a security context (label). The kernel checks the policy for every operation — even root is constrained.

SELinux operates in three modes:

| Mode | Behaviour |
|---|---|
| Enforcing | Policy is enforced; violations are blocked and logged |
| Permissive | Violations are logged only — nothing is blocked |
| Disabled | SELinux is completely off |

```bash
getenforce                            # check current SELinux mode
setenforce 0                          # switch to permissive (temporary)
ls -Z /etc/passwd                     # show file security context
ps -eZ | grep httpd                   # show process security context
audit2why < /var/log/audit/audit.log  # explain SELinux denials
```

> **Interview tip:** Permissive mode is invaluable for debugging — it logs denials without blocking, letting you build policy iteratively.

## 10. What is the difference between `select`, `poll`, and `epoll` for I/O multiplexing?

**Topic:** Kernel

**Answer:**  
All three mechanisms let a single thread monitor multiple file descriptors for readiness:

| Mechanism | FD Limit | Scan on Call | Complexity | Notes |
|---|---|---|---|---|
| `select()` | 1024 (FD_SETSIZE) | All descriptors | O(n) | Oldest, portable |
| `poll()` | Unlimited | All descriptors | O(n) | No fd limit, still scans all |
| `epoll` | Unlimited | Only ready fds | O(1) | Linux-specific, used by nginx, Redis |

`epoll` maintains a kernel-side event table and only returns file descriptors that are actually ready, making it essential for high-concurrency servers.

```bash
# epoll usage pattern (C pseudocode)
epfd = epoll_create1(0)
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &event)
n = epoll_wait(epfd, events, MAX_EVENTS, timeout)
# Only returns ready fds — no scanning all descriptors
```

> **Interview tip:** Know the distinction between edge-triggered (`EPOLLET`) and level-triggered (default) epoll — edge-triggered only notifies on state change, requiring non-blocking I/O and careful drain loops.
