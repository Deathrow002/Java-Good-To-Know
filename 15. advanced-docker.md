# Advanced Docker Interview Questions & Answers

This document contains 10 advanced interview questions and detailed answers focusing on Docker optimization, security, networking, storage architecture, and production-grade deployments.

---

### **1. How do multi-stage builds optimize Docker images, and how do they work under the hood?**
**Answer:** Multi-stage builds allow you to use multiple `FROM` statements in a single Dockerfile, where each `FROM` instruction begins a new stage of the build using a different base image.

* **Optimization Mechanism:** In traditional builds, all build-time dependencies (compilers, build tools, SDKs) remain in the final image, drastically inflating its size and increasing its attack surface. Multi-stage builds let you compile or build your application in an early, heavy stage and then copy *only* the compiled binaries or production artifacts into a final, minimal runtime image (e.g., Alpine or Distroless).
* **Under the Hood:** Docker's build engine (specifically BuildKit) constructs a directed acyclic graph (DAG) of these stages. If a stage isn't needed for the final target or its layers haven't changed, BuildKit optimizes the execution by running independent stages in parallel or skipping unused stages entirely.

### **2. What is Docker BuildKit, and what advanced features does it bring over the legacy build engine?**
**Answer:** BuildKit is the modern, overhauled backend engine for executing Docker builds. It replaces the legacy builder and introduces several performance and security capabilities:

* **Parallel Execution:** It analyzes the build graph and automatically runs independent build stages concurrently.
* **Secret Mounting:** You can safely pass secrets (like API keys or SSH keys) during build time using `RUN --mount=type=secret`. This prevents secrets from being permanently baked into the image layers or build history.
* **Advanced Caching:** It supports remote cache backends (e.g., pushing/pulling cache directly to/from a container registry) and allows mounting package manager caches (`RUN --mount=type=cache`) so package managers like `npm`, `pip`, or `apt` don't have to re-download dependencies if a layer above them invalidates.

### **3. Explain the difference between Overlay, Macvlan, and Host network drivers. When would you use each?**
**Answer:** 
* **Overlay Network:** Used for multi-host networking, typically within a orchestration engine like Docker Swarm. It creates a distributed network across multiple Docker daemons, encapsulating container traffic so containers on different physical or virtual hosts can communicate securely without host-level routing configuration.
* **Macvlan Network:** Assigns a unique, physical MAC address to a container, making it appear as a distinct physical device directly connected to your local network. The container requests an IP from your actual local network subnet. Use this when migrating legacy applications that expect to reside directly on the physical LAN without network address translation (NAT).
* **Host Network:** Bypasses Docker’s network isolation entirely. The container shares the host's network namespace (e.g., an application running on port 80 inside the container maps directly to port 80 on the host system). Use this for maximum performance because it completely eliminates NAT and bridge routing overhead.

### **4. Why is running a containerized application as `root` a security risk, and how do you mitigate it using Rootless Mode vs the `USER` instruction?**
**Answer:** By default, containers run as the `root` user. If an attacker exploits an application vulnerability and achieves a container breakout, they inherit root privileges on the host kernel, potentially gaining full control over the host system.

**Mitigation Strategies:**
* **`USER` Instruction:** Inside the Dockerfile, you should create a non-privileged system user/group and switch to it using `USER username`. This ensures the application process runs with restricted permissions inside the container's namespaces.
* **Rootless Mode:** An advanced architectural setup where the entire Docker Daemon (`dockerd`) and its containers run inside a user namespace without root privileges on the host. Even if a process executes as root inside the container, it maps to an unprivileged UID on the host, eliminating host-root exposure entirely.

### **5. What are Zombie Processes in Docker, and how does using an init system like `tini` solve this?**
**Answer:** In Linux, when a process terminates, its parent process must reap it to remove it from the kernel's process table. If the parent fails to do this, it becomes a "zombie process". Inside a container, the application process usually runs as PID 1. However, standard applications (like Node.js or Python) are not designed to act as an init system and do not handle reaping orphaned child processes. Over time, zombie processes accumulate, exhausting the host's PID pool.

**Solution:** You can pass the `--init` flag to `docker run`, or explicitly include a lightweight init binary like `tini` as the `ENTRYPOINT` in your Dockerfile. `tini` registers as PID 1, spawns your application as a child process, correctly forwards signals (like `SIGTERM` for graceful shutdowns), and relentlessly reaps zombie processes.

### **6. How does the Docker storage driver architecture work (specifically `overlay2`), and what is the "Copy-on-Write" (CoW) strategy?**
**Answer:** Docker uses storage drivers to manage image layers and the writable container layer. `overlay2` is the modern standard for Linux. It stacks directories on the host filesystem and presents them as a single, unified filesystem view inside the container.

It splits the filesystem into:
* **LowerDir:** The read-only layers (the immutable base image layers).
* **UpperDir:** The read-write layer (unique to the individual running container).
* **MergedDir:** The unified view seen inside the container where UpperDir overlays LowerDir.

**Copy-on-Write (CoW):** If a container needs to modify an existing file belonging to the `LowerDir`, Docker does not modify the original file. Instead, it copies the file up to the `UpperDir` (the container's writable layer) first, and then applies the modifications there. This keeps the base image completely immutable and allows multiple containers to share the same base layers efficiently.

### **7. How do cgroups (Control Groups) and Namespaces differ in how they isolate containers?**
**Answer:** Namespaces and cgroups are the core Linux kernel features that make containerization possible, but they serve entirely different isolation purposes:

| Feature | **Namespaces (The "What you can see")** | **cgroups (The "What you can use")** |
| :--- | :--- | :--- |
| **Purpose** | Resource **isolation**. It limits what a container can see. | Resource **limiting**. It restricts how much a container can consume. |
| **Examples** | **PID:** Isolates process IDs.<br>**NET:** Isolates network interfaces.<br>**MNT:** Isolates filesystem mount points.<br>**IPC:** Isolates inter-process communication. | Limits maximum **CPU** cores or shares.<br>Limits maximum physical **Memory** (RAM).<br>Restricts disk **I/O** bandwidth throughput. |

### **8. What happens when a container hits its memory limit, and how can you troubleshoot an `OOMKilled` error?**
**Answer:** If a container exceeds the hard memory threshold configured via Docker (e.g., `--memory="512m"`), the Linux kernel's **Out-Of-Memory (OOM) Killer** steps in and immediately terminates the container's primary process to protect the host system from crashing. Docker will report an Exit Code `137` (indicating it was killed by `SIGKILL`).

**Troubleshooting Steps:**
1. Run `docker inspect <container_id>` and check the `State` object for `"OOMKilled": true`.
2. Review host system logs using `dmesg -T | grep -i oom` or inspect `/var/log/syslog` to verify that the kernel invoked the OOM killer on that specific process.
3. Profile the application for memory leaks, optimize memory usage, or adjust the container resource allocation using `--memory-reservation` (soft limits) or by increasing the hard limit.

### **9. How can you implement secure, zero-downtime configuration updates in a Docker environment using Docker Compose?**
**Answer:** In a standard Docker Compose setup, running `docker compose up -d` on a modified service will cause a brief period of downtime while the old container stops and the new one boots up. 

To achieve zero-downtime updates with vanilla Compose, you can leverage a rolling upgrade strategy by combining the `deploy` configuration with container health checks:

```yaml
services:
  web:
    image: myapp:v2
    deploy:
      update_config:
        order: start-first
        failure_action: rollback
        delay: 10s
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/health"]
      interval: 5s
      timeout: 3s
      retries: 3
```

> **Key Mechanism:** The `order: start-first` instruction tells Docker to spin up the new container *before* terminating the old one. Traffic is only routed away from the old container once the new container's `healthcheck` passes successfully, ensuring continuous application availability.

### **10. What are dangling images and volumes, and how do you implement safe resource pruning in production cron jobs?**
**Answer:** 
* **Dangling Images:** Layers that are no longer associated with any tagged image (often showing as `<none>:<none>` when running `docker images`). They typically occur when you rebuild an image with an existing tag, causing the older layers to lose their reference name.
* **Dangling Volumes:** Volumes that exist on the host disk but are no longer mapped or attached to any active or stopped container.

**Safe Production Pruning:**
Running a blanket `docker system prune -a --volumes` in production is dangerous because it deletes stopped containers, unused networks, and *all* unattached volumes (potentially wiping out offline database storage). 

A safe automated cron configuration should target specific resources using strict filters:
* `docker image prune -f`: Safely deletes *only* dangling images without touching tagged ones.
* `docker volume prune --filter "label!=keep" -f`: Deletes unattached volumes, while using filters to protect explicitly labeled persistent data volumes.
