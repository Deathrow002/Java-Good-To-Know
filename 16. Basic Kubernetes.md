# Kubernetes Interview Questions & Answers

A comprehensive reference of 10 essential Kubernetes interview questions with clear answers, YAML examples, and real commands.

---

## 1. What are the main components of a Kubernetes cluster?

**Topic:** Architecture

**Answer:**

A Kubernetes cluster is divided into two planes:

**Control Plane** (manages the cluster):

| Component | Description |
|---|---|
| `kube-apiserver` | Front-end REST API; all components communicate through it |
| `etcd` | Distributed key-value store; source of truth for all cluster state |
| `kube-scheduler` | Assigns pods to nodes based on resource requirements and constraints |
| `kube-controller-manager` | Runs controllers (ReplicaSet, Node, Endpoint, etc.) |
| `cloud-controller-manager` | Integrates with cloud provider APIs |

**Data Plane** (runs workloads):

| Component | Description |
|---|---|
| `kubelet` | Agent on each node; ensures containers in pods are running |
| `kube-proxy` | Maintains iptables/IPVS rules for Service networking |
| Container Runtime | (containerd, CRI-O) actually runs the containers |

```bash
kubectl get componentstatuses       # check control plane health
kubectl get nodes                   # list all nodes in the cluster
kubectl describe node <node-name>   # node details and resource usage
```

> **Interview tip:** Knowing that etcd is the single source of truth — and that losing it means losing the cluster state — shows you understand K8s internals deeply.

## 2. What is the difference between a Pod, ReplicaSet, and Deployment?

**Topic:** Workloads

**Answer:**

- **Pod** — the smallest deployable unit in Kubernetes. It wraps one or more containers that share network and storage. Pods are ephemeral; if one dies it is not restarted automatically.

- **ReplicaSet** — ensures a specified number of identical pod replicas are running at all times. If a pod dies, the ReplicaSet creates a new one. However, ReplicaSets do not handle rolling updates.

- **Deployment** — the recommended way to manage stateless workloads. A Deployment owns a ReplicaSet and adds declarative rolling updates, rollback capability, and pause/resume. When you update the image, Kubernetes creates a new ReplicaSet and gradually scales it up while scaling the old one down.

```bash
kubectl create deployment myapp --image=nginx:1.25
kubectl rollout status deployment/myapp
kubectl rollout undo deployment/myapp       # rollback to previous version
kubectl scale deployment myapp --replicas=5
```

> **Interview tip:** You almost never create Pods or ReplicaSets directly — always use a Deployment (or StatefulSet for stateful apps).

## 3. What are the different types of Kubernetes Services and when do you use each?

**Topic:** Networking

**Answer:**

| Type | Accessibility | Use Case |
|---|---|---|
| `ClusterIP` (default) | Internal cluster only | Internal microservice communication |
| `NodePort` | External via `<NodeIP>:<NodePort>` (30000–32767) | Dev/testing environments |
| `LoadBalancer` | External via cloud provider LB | Exposing services to the internet |
| `ExternalName` | Maps to an external DNS name (CNAME) | Connecting to managed external databases |
| `Headless` (`ClusterIP: None`) | Returns individual pod IPs via DNS | StatefulSets with stable pod DNS names |

```bash
kubectl expose deployment myapp --port=80 --type=LoadBalancer
kubectl get svc myapp
kubectl describe svc myapp   # shows endpoints, selectors, ports
```

## 4. What is the difference between a StatefulSet and a Deployment?

**Topic:** Workloads

**Answer:**

**Deployments** are designed for stateless applications — pods are interchangeable, can be replaced in any order, and share storage via a common PVC.

**StatefulSets** are for stateful applications (databases, Kafka, Elasticsearch) and provide:

- Stable, unique pod names (`pod-0`, `pod-1`, `pod-2`) that persist across restarts
- Stable DNS hostnames per pod (`<pod>.<service>.<namespace>.svc.cluster.local`)
- Ordered, graceful pod creation and deletion (`pod-0` before `pod-1`)
- Per-pod PersistentVolumeClaims that are **not** deleted when pods are removed

```bash
# StatefulSet pod DNS pattern
mysql-0.mysql.default.svc.cluster.local
mysql-1.mysql.default.svc.cluster.local

kubectl get statefulsets
kubectl delete pod mysql-2   # pod-2 restarts with same name and PVC
```

> **Interview tip:** StatefulSets are not a silver bullet — managing stateful workloads in K8s is complex. Managed cloud databases are often a better choice.

## 5. What are liveness, readiness, and startup probes, and how do they differ?

**Topic:** Operations

**Answer:**

| Probe | Action on Failure | Purpose |
|---|---|---|
| **Liveness** | Container is killed and restarted | Detect deadlocks or hung processes |
| **Readiness** | Pod removed from Service endpoints (no traffic) | Signal when app is ready to serve |
| **Startup** | Disables liveness/readiness until it succeeds | Handle slow-starting applications |

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 3

startupProbe:
  httpGet:
    path: /healthz
    port: 8080
  failureThreshold: 30
  periodSeconds: 10
```

> **Interview tip:** A misconfigured liveness probe is a common cause of crash loops. Always set `initialDelaySeconds` generously.

## 6. How does Kubernetes handle persistent storage? Explain PV, PVC, and StorageClass.

**Topic:** Storage

**Answer:**

- **PersistentVolume (PV)** — a piece of storage provisioned in the cluster (NFS share, cloud disk, etc.). It exists independently of any pod.

- **PersistentVolumeClaim (PVC)** — a request for storage by a pod. Specifies size and access mode. Kubernetes binds a PVC to a matching PV.

- **StorageClass** — defines a "class" of storage (SSD, HDD, regional disk). Enables dynamic provisioning: when a PVC references a StorageClass, Kubernetes automatically creates a PV, eliminating manual PV creation.

**Access Modes:**

| Mode | Description |
|---|---|
| `ReadWriteOnce` (RWO) | Mounted by one node at a time |
| `ReadOnlyMany` (ROX) | Mounted read-only by many nodes |
| `ReadWriteMany` (RWX) | Mounted read-write by many nodes (e.g., NFS, EFS) |

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: gp2
  resources:
    requests:
      storage: 10Gi
```

```bash
kubectl get pv,pvc
kubectl describe pvc my-pvc   # check binding status
```

## 7. How does Kubernetes schedule pods, and what are taints, tolerations, and node affinity?

**Topic:** Operations

**Answer:**

The `kube-scheduler` selects a node for each pod in two phases:
1. **Filtering** — eliminates nodes that don't meet requirements
2. **Scoring** — ranks remaining nodes

**Taints and Tolerations** — taints repel pods from nodes; tolerations allow pods to be scheduled on tainted nodes. Used to dedicate nodes for specific workloads (e.g., GPU nodes).

**Node Affinity** — allows pods to express preferences or hard requirements for which nodes they run on, based on node labels:
- `requiredDuringScheduling` — hard rule; pod will not be scheduled if unmet
- `preferredDuringScheduling` — soft rule; scheduler tries but may skip

**Pod Anti-Affinity** — commonly used to spread replicas across availability zones.

```bash
# Taint a node (only GPU pods should run here)
kubectl taint nodes gpu-node-1 gpu=true:NoSchedule
```

```yaml
# Toleration in pod spec
tolerations:
- key: "gpu"
  operator: "Equal"
  value: "true"
  effect: "NoSchedule"

# Node affinity
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values: [us-east-1a, us-east-1b]
```

> **Interview tip:** Taints/tolerations control which pods **can** run on a node; node affinity controls which nodes a pod **wants** to run on — they work in opposite directions.

## 8. What is an Ingress and how does it differ from a LoadBalancer Service?

**Topic:** Networking

**Answer:**

A **LoadBalancer Service** provisions one external load balancer per service — this becomes expensive at scale (one cloud LB per microservice).

An **Ingress** is an API object that provides HTTP/HTTPS routing for multiple services through a single entry point. An **Ingress Controller** (nginx, Traefik, AWS ALB Ingress Controller) watches Ingress resources and configures the load balancer.

**Ingress features:**

- Host-based routing (`api.example.com` → `api-service`)
- Path-based routing (`/admin` → `admin-service`, `/app` → `app-service`)
- TLS termination (single certificate for multiple services)
- Rewrite rules, rate limiting, auth middleware (depends on controller)

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /v1
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80
```

## 9. What are resource requests and limits in Kubernetes, and why do they matter?

**Topic:** Operations

**Answer:**

- **Requests** — the amount of CPU/memory a container is guaranteed. The scheduler uses requests to decide which node can fit the pod.

- **Limits** — the maximum CPU/memory a container can use. If a container exceeds its memory limit, it is OOMKilled. If it exceeds its CPU limit, it is throttled (not killed).

**QoS Classes (assigned automatically):**

| Class | Condition | Eviction Priority |
|---|---|---|
| `Guaranteed` | requests == limits for all containers | Last to be evicted |
| `Burstable` | limits > requests | Middle priority |
| `BestEffort` | No requests or limits set | First to be evicted |

```yaml
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"       # 250 millicores = 0.25 CPU
  limits:
    memory: "512Mi"
    cpu: "500m"
```

```bash
kubectl top pods                        # actual resource usage
kubectl describe node | grep -A5 Allocated
```

> **Interview tip:** Always set requests for production workloads. Pods without requests are `BestEffort` and will be evicted first during node memory pressure.

## 10. What is etcd and what happens to the cluster if etcd becomes unavailable?

**Topic:** Architecture

**Answer:**

`etcd` is a distributed, consistent key-value store using the **Raft consensus algorithm**. It stores all Kubernetes cluster state: pod specs, service definitions, secrets, ConfigMaps, RBAC policies, and node information. It is typically deployed as a 3 or 5-node cluster for high availability.

**If etcd becomes unavailable:**

| Component | Impact |
|---|---|
| `kube-apiserver` | Cannot read/write state — `kubectl` commands fail |
| `kube-scheduler` | Cannot assign new pods to nodes |
| Controllers | Cannot reconcile desired vs actual state |
| Running pods | **Continue running** — kubelet keeps containers alive independently |

The cluster effectively becomes read-only at the node level — existing workloads survive but cannot be modified or self-healed.

```bash
# Check etcd cluster health
kubectl -n kube-system exec -it etcd-controlplane -- \
  etcdctl --cacert /etc/kubernetes/pki/etcd/ca.crt \
          --cert   /etc/kubernetes/pki/etcd/server.crt \
          --key    /etc/kubernetes/pki/etcd/server.key \
          endpoint health

# Backup etcd (critical for disaster recovery)
etcdctl snapshot save /backup/etcd-snapshot.db
```

> **Interview tip:** etcd backup and restore is a core CKA exam topic. A 3-node etcd cluster can tolerate 1 failure; a 5-node cluster can tolerate 2 (quorum = ⌊n/2⌋ + 1).
