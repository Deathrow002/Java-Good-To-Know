# Advanced Kubernetes Interview Questions & Answers

A deep-dive reference of 10 advanced Kubernetes interview questions covering RBAC, autoscaling, network policy, operators, service mesh, GitOps, admission webhooks, and zero-downtime deployments.

---

## 1. Explain Kubernetes RBAC — how do Role, ClusterRole, RoleBinding, and ClusterRoleBinding work together?

**Topic:** Security

**Answer:**

RBAC (Role-Based Access Control) controls who can do what on which resources. It has four objects:

| Object | Scope | Purpose |
|---|---|---|
| `Role` | Namespace | Grants permissions within a single namespace |
| `ClusterRole` | Cluster-wide | Grants permissions on cluster-scoped or all-namespace resources |
| `RoleBinding` | Namespace | Binds a Role or ClusterRole to subjects within one namespace |
| `ClusterRoleBinding` | Cluster-wide | Binds a ClusterRole to subjects across the entire cluster |

Common pattern: create a `ClusterRole` with read-only access, then use `RoleBinding`s per namespace — giving different teams access to their own namespaces without duplicating role definitions.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
kind: RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: read-pods
  namespace: production
subjects:
- kind: ServiceAccount
  name: ci-runner
  namespace: production
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

> **Interview tip:** Always apply least-privilege. A ServiceAccount with a wildcard verb (`*`) on all resources is a critical security misconfiguration.

## 2. How does the Horizontal Pod Autoscaler (HPA) work, and what is the difference between HPA and VPA?

**Topic:** Scaling

**Answer:**

**HPA (Horizontal Pod Autoscaler)** automatically adjusts the replica count of a Deployment or StatefulSet based on observed metrics (CPU, memory, or custom metrics from Prometheus via the custom metrics API). The control loop runs every 15 seconds:

```
desiredReplicas = ceil(currentReplicas × (currentMetric / targetMetric))
```

**VPA (Vertical Pod Autoscaler)** adjusts the CPU and memory requests/limits of individual pods based on historical usage — it scales up or down a single pod rather than adding more replicas.

| | HPA | VPA |
|---|---|---|
| Strategy | More pods (scale-out) | Bigger pods (scale-up) |
| Pod restart required | No | Yes |
| Best for | Stateless workloads, RPS spikes | Right-sizing memory, stateful apps |
| Can combine with each other? | Avoid combining on same metric (CPU) | Use on different metrics |

**KEDA** (Kubernetes Event-Driven Autoscaler) extends HPA to scale on external metrics like Kafka consumer lag, queue depth, or database connections.

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
```

> **Interview tip:** HPA needs `metrics-server` installed. For production, combine HPA (reactive scaling) with Cluster Autoscaler (node provisioning) for end-to-end elasticity.

## 3. What is a NetworkPolicy and how does it enforce pod-level firewall rules?

**Topic:** Networking

**Answer:**

By default, all pods in a Kubernetes cluster can communicate with each other freely. `NetworkPolicy` is a namespaced resource that restricts ingress and/or egress traffic for a selected set of pods.

**Key concepts:**

| Field | Description |
|---|---|
| `podSelector` | Selects which pods the policy applies to |
| `ingress` | Controls inbound traffic (from which pods/namespaces/CIDRs, on which ports) |
| `egress` | Controls outbound traffic |
| `policyTypes` | Which directions are controlled: `Ingress`, `Egress`, or both |

**Important rules:**
- NetworkPolicy is enforced by the CNI plugin (Calico, Cilium, Weave) — not by kube-proxy. Flannel does **not** support NetworkPolicy.
- Once a NetworkPolicy selects a pod, all traffic **not explicitly allowed** is denied (implicit default-deny for that direction).
- Multiple policies on the same pod are **additive** (unioned) — there is no explicit `deny` rule.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-api
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

> **Interview tip:** A common pattern is a `default-deny-all` policy per namespace, then explicit allow rules for each service-to-service path.

## 4. What are Custom Resource Definitions (CRDs) and what is the Operator pattern?

**Topic:** Extensibility

**Answer:**

**CRDs** extend the Kubernetes API with your own resource types. Once registered, custom objects can be created, listed, updated, and deleted using `kubectl` just like built-in resources. The API server stores them in etcd automatically.

**The Operator pattern** combines a CRD with a custom controller that watches those resources and reconciles actual state toward desired state. It encodes domain-specific operational knowledge — how to deploy, scale, backup, and recover a stateful application — into code.

**Operator lifecycle:**

1. User creates a custom resource (e.g., `MySQLCluster`)
2. The controller's reconcile loop detects the new object
3. The controller creates necessary K8s objects (StatefulSet, Services, Secrets)
4. On changes or failures, the controller self-heals the cluster

Popular operators: Prometheus Operator, Cert-Manager, Strimzi (Kafka), CloudNativePG (PostgreSQL).

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: mysqlclusters.db.example.com
spec:
  group: db.example.com
  versions:
  - name: v1alpha1
    served: true
    storage: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            properties:
              replicas:
                type: integer
  scope: Namespaced
  names:
    plural: mysqlclusters
    singular: mysqlcluster
    kind: MySQLCluster
```

> **Interview tip:** Use `controller-runtime` (Go) or `Kopf` (Python) to build operators. The reconcile loop must be **idempotent** — it can be called any number of times and produce the same result.

## 5. What is a service mesh (Istio/Linkerd) and what problems does it solve that Kubernetes alone cannot?

**Topic:** Networking

**Answer:**

Kubernetes Services provide basic load balancing and discovery, but lack observability, fine-grained traffic control, and mutual TLS between services. A service mesh solves these by injecting a **sidecar proxy** (Envoy in Istio, linkerd-proxy in Linkerd) into every pod. All inter-pod traffic flows through these proxies, which are centrally configured by the mesh control plane.

**Capabilities added by a service mesh:**

| Feature | Description |
|---|---|
| mTLS everywhere | Automatic certificate rotation, zero-trust pod-to-pod encryption |
| Traffic management | Canary deployments, A/B testing, traffic mirroring, circuit breaking, retries |
| Observability | Automatic golden signal metrics and distributed tracing without code changes |
| Fault injection | Inject delays or errors for chaos testing |

The trade-off is operational complexity and ~1ms latency overhead from the sidecar hop. Cilium's eBPF-based mesh avoids sidecars by enforcing policy in the kernel.

```yaml
# Istio traffic split — 90/10 canary deployment
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts: [myapp]
  http:
  - route:
    - destination:
        host: myapp
        subset: v1
      weight: 90
    - destination:
        host: myapp
        subset: v2
      weight: 10
```

> **Interview tip:** If asked "how do you do a canary deployment in K8s?", a plain Deployment answer is basic. Mentioning Istio VirtualService or Argo Rollouts shows senior-level thinking.

## 6. What is GitOps and how do Argo CD and Flux implement it in Kubernetes?

**Topic:** Operations

**Answer:**

GitOps is a deployment model where **Git is the single source of truth** for all cluster state. Desired infrastructure and application manifests live in a Git repository. An automated agent continuously reconciles the live cluster toward the state declared in Git.

**Core GitOps principles:**

1. **Declarative** — all system state is described declaratively (YAML/Helm/Kustomize)
2. **Versioned** — Git history provides full audit trail and rollback via `git revert`
3. **Automatically applied** — approved changes are applied without manual `kubectl`
4. **Continuously reconciled** — any drift from Git state is automatically corrected

| | Argo CD | Flux |
|---|---|---|
| UI | Full web UI with SSO, RBAC | CLI-first, no built-in UI |
| Model | Application CRD per app | Modular controllers (source, kustomize, helm) |
| Multi-cluster | Native | Supported |
| OCI registries | Supported | Native, first-class |
| CNCF status | Graduated | Graduated |

```yaml
# Argo CD Application manifest
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/myapp
    targetRevision: main
    path: k8s/overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

> **Interview tip:** The key GitOps security advantage — cluster credentials never leave the cluster. The agent pulls from Git, rather than CI/CD pipelines pushing with cluster-admin credentials.

## 7. How does Kubernetes handle rolling updates and what strategies exist for zero-downtime deployments?

**Topic:** Operations

**Answer:**

A Deployment's default strategy is `RollingUpdate`, controlled by two parameters:

| Parameter | Default | Meaning |
|---|---|---|
| `maxUnavailable` | 25% | Max pods that can be unavailable during the update |
| `maxSurge` | 25% | Max extra pods allowed above the desired count |

Setting `maxUnavailable: 0` and `maxSurge: 1` gives a strict one-at-a-time rollout with zero downtime.

**Advanced deployment strategies:**

- **Blue/Green** — run two identical environments; switch traffic at the Service or Ingress level instantly. Full rollback by switching back.
- **Canary** — route a small percentage of traffic to the new version (via Istio, Nginx ingress weights, or Argo Rollouts) and gradually increase as metrics are validated.
- **Argo Rollouts** — extends Deployments with native canary and blue/green support, automated analysis (Prometheus metric gates), and pause/promote/abort controls.

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0    # never reduce below desired count
      maxSurge: 1          # allow one extra pod during rollout
```

```bash
kubectl rollout status deployment/myapp
kubectl rollout history deployment/myapp
kubectl rollout undo deployment/myapp --to-revision=3
```

> **Interview tip:** "Zero-downtime" requires readiness probes, PodDisruptionBudgets, AND a `preStop` hook (`sleep 5`) to handle connection draining — setting `maxUnavailable: 0` alone is not sufficient.

## 8. What is Pod Security Admission (PSA) and how does it replace PodSecurityPolicy?

**Topic:** Security

**Answer:**

`PodSecurityPolicy` (PSP) was deprecated in K8s 1.21 and removed in 1.25. It was complex to configure, easy to misconfigure, and a blank PSP silently granted full permissions.

**Pod Security Admission (PSA)** is the built-in replacement, enforcing one of three security profiles at the **namespace level** via labels:

| Profile | Description | Use case |
|---|---|---|
| `privileged` | No restrictions | System namespaces (`kube-system`) |
| `baseline` | Prevents known privilege escalations | Most workloads |
| `restricted` | Heavily restricted, follows hardening best practices | Sensitive/regulated workloads |

Each profile runs in three modes:

| Mode | Behavior |
|---|---|
| `enforce` | Reject non-compliant pods |
| `audit` | Allow but log a violation |
| `warn` | Allow but show a user-facing warning |

```bash
# Apply restricted policy to a namespace
kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/warn=restricted
```

```yaml
# Pod spec required for restricted profile
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  allowPrivilegeEscalation: false
  capabilities:
    drop: ["ALL"]
  seccompProfile:
    type: RuntimeDefault
```

> **Interview tip:** PSA is profile-based — it cannot enforce "allow image X but not image Y". For image allowlists, mutation, or per-pod OPA rules, use **Kyverno** or **OPA Gatekeeper** alongside PSA.

## 9. What is the Cluster Autoscaler and how does it interact with HPA?

**Topic:** Scaling

**Answer:**

The **Cluster Autoscaler (CA)** automatically adjusts the number of nodes in a node group based on pod scheduling pressure and node utilization.

**Scale-up trigger:** When pods are in `Pending` state because no node has sufficient resources, CA provisions new nodes from the cloud provider (AWS ASG, GKE node pool, AKS VMSS).

**Scale-down trigger:** When a node's utilization is consistently below a threshold (default 50%) and all its pods can be safely rescheduled elsewhere, CA cordons, drains, and deletes the node.

**CA respects:**
- `PodDisruptionBudgets` — will not drain a node if doing so violates a PDB
- Pod annotation `cluster-autoscaler.kubernetes.io/safe-to-evict: "false"` — protects individual pods
- Pods with local PVs or unmanaged pods — block scale-down indefinitely

**HPA + CA together:** HPA scales pods when CPU spikes → CA scales nodes when pods are `Pending`. This gives full elastic auto-scaling from zero to thousands of pods.

```yaml
# PodDisruptionBudget — protect availability during CA scale-down
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: myapp
```

```yaml
# Prevent CA from evicting a critical pod
metadata:
  annotations:
    cluster-autoscaler.kubernetes.io/safe-to-evict: "false"
```

> **Interview tip:** CA will not scale down a node that holds an unmanaged pod (not owned by a controller), a pod with a local PersistentVolume, or a pod with `safe-to-evict: false`. These can permanently block scale-down.

## 10. What are admission webhooks and what is the difference between validating and mutating webhooks?

**Topic:** Extensibility

**Answer:**

Admission webhooks intercept API server requests **after authentication and authorization** but **before the object is persisted to etcd**. They allow custom logic to validate or mutate any Kubernetes resource.

| | Mutating Webhook | Validating Webhook |
|---|---|---|
| Runs | First | After mutation |
| Can modify object | Yes (returns JSON Patch) | No |
| Can reject request | Yes | Yes |
| Use case | Inject sidecars, add defaults, set limits | Enforce policies, block non-compliant resources |

**Common use cases:**

- **Istio sidecar injection** (mutating) — automatically adds the Envoy proxy container to every pod
- **Kyverno / OPA Gatekeeper** (validating) — enforce policies like "all images must come from our registry"
- **cert-manager** (mutating) — injects CA bundles into CRD fields

The `failurePolicy` field controls behavior when the webhook is unreachable:

| Value | Behavior |
|---|---|
| `Fail` | Block the request (safe default for security policies) |
| `Ignore` | Allow the request through (safe default for non-critical webhooks) |

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: sidecar-injector
webhooks:
- name: inject.sidecar.example.com
  admissionReviewVersions: ["v1"]
  clientConfig:
    service:
      name: sidecar-injector
      namespace: kube-system
      path: /inject
  rules:
  - operations: ["CREATE"]
    apiGroups: [""]
    resources: ["pods"]
  failurePolicy: Ignore
  sideEffects: None
```

> **Interview tip:** Webhooks with `failurePolicy: Fail` that become unavailable can prevent all pod creation in the cluster. Always run webhook servers with high availability and set appropriate timeouts (default 10s).
