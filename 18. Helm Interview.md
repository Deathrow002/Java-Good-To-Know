# Helm Interview Questions & Answers

A comprehensive reference of 10 Helm interview questions covering chart structure, templating, lifecycle hooks, repositories, values management, subcharts, testing, and debugging.

---

## 1. What is Helm and what problems does it solve in Kubernetes?

**Topic:** Concepts

**Answer:**

Helm is the package manager for Kubernetes. It bundles related Kubernetes manifests (Deployments, Services, ConfigMaps, etc.) into a single unit called a **chart**, making it easy to install, upgrade, and uninstall complex applications as one atomic operation.

**Problems Helm solves:**

| Problem | Without Helm | With Helm |
|---|---|---|
| Repetition | Copy-paste YAML across environments | Go templates with parameterised values |
| Versioning | No history of what was deployed | Every install/upgrade is a numbered revision |
| Dependencies | Manual ordering of manifests | Declare deps in `Chart.yaml`; Helm resolves them |
| Environments | Separate manifest copies per env | One chart, multiple `values-*.yaml` overlays |

```bash
helm install myapp ./mychart              # install from local chart
helm install myapp bitnami/nginx          # install from repository
helm upgrade myapp ./mychart              # upgrade existing release
helm rollback myapp 1                     # rollback to revision 1
helm uninstall myapp                      # delete release and all resources
```

> **Interview tip:** Helm v3 (current) removed the server-side Tiller component from Helm v2. In v3, Helm runs entirely client-side and stores release state as Secrets in the target namespace.

## 2. What is the structure of a Helm chart and what does each file/directory do?

**Topic:** Concepts

**Answer:**

A Helm chart is a directory with a specific layout:

```
mychart/
├── Chart.yaml        — chart metadata (name, version, description, dependencies)
├── values.yaml       — default configuration values for the chart
├── charts/           — directory for dependency (sub)charts
├── templates/        — Go template files that render into Kubernetes manifests
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── _helpers.tpl  — named template definitions (not rendered directly)
│   ├── NOTES.txt     — post-install usage instructions printed to the user
│   └── tests/        — test pods triggered only by 'helm test'
└── .helmignore       — files to exclude when packaging (like .gitignore)
```

**`Chart.yaml` required fields:** `apiVersion` (v2 for Helm 3), `name`, and `version` (SemVer).

```yaml
# Chart.yaml example
apiVersion: v2
name: myapp
description: A Helm chart for MyApp
type: application
version: 1.3.0          # chart version (SemVer)
appVersion: "2.5.1"     # version of the application being packaged

dependencies:
- name: postgresql
  version: "12.x.x"
  repository: https://charts.bitnami.com/bitnami
  condition: postgresql.enabled
```

> **Interview tip:** `version` and `appVersion` are separate — `version` tracks changes to the chart itself; `appVersion` tracks the application it deploys. They often differ.

## 3. How does Helm templating work? Explain the Values object and common template functions.

**Topic:** Templating

**Answer:**

Helm uses Go's `text/template` engine extended with the **Sprig** function library. Templates in `templates/` are rendered against a context object:

| Object | Contents |
|---|---|
| `.Values` | Merged values from `values.yaml` and `--set` / `-f` overrides |
| `.Release` | Release metadata: `.Release.Name`, `.Release.Namespace`, `.Release.Revision` |
| `.Chart` | `Chart.yaml` fields: `.Chart.Name`, `.Chart.Version`, `.Chart.AppVersion` |
| `.Files` | Access to non-template files bundled in the chart |
| `.Capabilities` | Cluster info: `.Capabilities.KubeVersion.Minor` |

**Common template directives:**

```yaml
{{ .Values.image.tag }}              # output a value
{{- ... -}}                          # trim surrounding whitespace
{{ if .Values.ingress.enabled }}     # conditional block
{{ range .Values.env }}              # iterate over a list
{{ include "mychart.labels" . }}     # include a named template
{{ required "host is required" .Values.host }}  # fail if missing
{{ default "latest" .Values.tag }}   # fallback value
```

```yaml
# _helpers.tpl — reusable named template
{{- define "myapp.labels" -}}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion }}
{{- end }}

# deployment.yaml — using the named template
metadata:
  name: {{ .Release.Name }}-deployment
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
```

> **Interview tip:** `nindent N` is one of the most-used functions — it adds a newline then indents by N spaces, which is essential for embedding named templates inside YAML blocks.

## 4. What are Helm hooks and what are common use cases?

**Topic:** Lifecycle

**Answer:**

Helm hooks let you execute Kubernetes Jobs (or other resources) at specific points in a release lifecycle, using the `helm.sh/hook` annotation.

**Hook events:**

| Hook | When it runs |
|---|---|
| `pre-install` | Before resources are created on first install |
| `post-install` | After all resources are ready on first install |
| `pre-upgrade` | Before upgrade manifests are applied |
| `post-upgrade` | After upgrade completes successfully |
| `pre-rollback` | Before a rollback is applied |
| `post-rollback` | After a rollback completes |
| `pre-delete` | Before `helm uninstall` removes resources |
| `post-delete` | After all resources are removed |
| `test` | Triggered only by `helm test` |

**Common use cases:**
- `pre-upgrade` — run database migrations before the new app version starts
- `post-install` — send a Slack notification or register the service in a registry
- `pre-delete` — back up data before the release is destroyed
- `test` — smoke-test the deployed application with a `curl` Job

```yaml
# Database migration Job as a pre-upgrade hook
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ .Release.Name }}-migrate
  annotations:
    "helm.sh/hook": pre-upgrade
    "helm.sh/hook-weight": "-5"           # lower value runs first
    "helm.sh/hook-delete-policy": hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: migrate
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        command: ["python", "manage.py", "migrate"]
```

> **Interview tip:** `hook-delete-policy: hook-succeeded` cleans up the Job after success. Without it, old hook Jobs accumulate and can block future runs since Kubernetes Job names must be unique.

## 5. How do you manage values in Helm and what is the order of precedence?

**Topic:** Operations

**Answer:**

Helm merges values from multiple sources. The order of precedence from **lowest to highest**:

| Priority | Source |
|---|---|
| 1 (lowest) | `values.yaml` — chart defaults |
| 2 | `values.yaml` of parent chart (if this is a subchart) |
| 3 | User-supplied file: `helm install -f custom-values.yaml` |
| 4 (highest) | `--set key=value` flags |

Multiple `-f` files are merged left to right (later files win). Multiple `--set` flags are applied in order (last one wins).

**Best practices:**
- Keep `values.yaml` as a full reference with comments and sensible defaults
- Use separate overlay files per environment: `values-dev.yaml`, `values-prod.yaml`
- Use `--set` sparingly in CI for dynamic values (image tags, commit SHAs)
- Never commit secrets in `values.yaml` — use External Secrets Operator or Vault

```bash
# Install with environment overlay
helm install myapp ./mychart -f values-prod.yaml

# Override single values from CI pipeline
helm upgrade myapp ./mychart \
  -f values-prod.yaml \
  --set image.tag=abc1234 \
  --set replicaCount=5

# Inspect the final merged values of a live release
helm get values myapp
helm get values myapp --all    # include chart defaults
```

> **Interview tip:** `--set` uses dot notation for nested keys (`image.tag`) and comma separation for multiple values (`a=1,b=2`). For complex values (lists, multiline strings), `-f` is always cleaner.

## 6. What are Helm repositories and how do OCI registries change chart distribution?

**Topic:** Repository

**Answer:**

A Helm repository is an HTTP server hosting an `index.yaml` file and chart tarballs (`.tgz`). Helm downloads the index to discover available charts and versions, then fetches the tarball on install.

**Traditional HTTP repo workflow:**

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm search repo bitnami/postgresql
helm install pg bitnami/postgresql --version 12.5.6
```

**OCI registries** — since Helm 3.8 (stable), charts can be stored in any OCI-compliant registry (Docker Hub, AWS ECR, GCR, Harbor, Artifactory) as OCI artifacts alongside container images. No `index.yaml` required.

**Benefits of OCI over traditional repos:**

| Benefit | Description |
|---|---|
| Unified registry | One registry for both images and charts |
| Access control | Fine-grained IAM/RBAC per chart |
| Immutability | Content-addressable digests; tags cannot be silently overwritten |
| Supply chain | Native support in Cosign and Sigstore for chart signing |

```bash
# Push and install via OCI registry
helm push mychart-1.3.0.tgz oci://registry.example.com/charts
helm install myapp oci://registry.example.com/charts/myapp --version 1.3.0
helm pull oci://registry.example.com/charts/myapp --version 1.3.0

# Authenticate to an OCI registry
helm registry login registry.example.com \
  --username myuser --password mypass
```

> **Interview tip:** OCI registries are the future of chart distribution. Artifact Hub (artifacthub.io) is the public discovery layer — charts hosted on OCI registries can still be listed and discovered there.

## 7. How does Helm handle release upgrades and rollbacks? What is stored in a release Secret?

**Topic:** Lifecycle

**Answer:**

Every `helm install` or `helm upgrade` creates a new release revision stored as a Kubernetes Secret in the release namespace, named `sh.helm.release.v1.<release>.<revision>`. The Secret contains the complete chart, values, and rendered manifests for that revision — this is how rollback works without a separate database.

**Upgrade process:**

1. Helm renders the new manifests from updated chart and values
2. Performs a **3-way strategic merge patch** between: old manifest, live cluster state, and new manifest
3. Applies the diff to the cluster
4. On success, stores the new revision Secret
5. On failure (if `--atomic` is set), automatically rolls back to the previous revision

**Rollback:**

`helm rollback <release> <revision>` re-applies the manifests stored in that revision's Secret.

```bash
helm history myapp                    # list all revisions and status
helm get manifest myapp               # rendered YAML of current release
helm get manifest myapp --revision 3  # rendered YAML of a past revision
helm rollback myapp 3                 # roll back to revision 3

# Upgrade with automatic rollback on failure
helm upgrade myapp ./mychart \
  --atomic \              # rollback automatically if any resource fails
  --timeout 5m \
  --cleanup-on-fail
```

> **Interview tip:** `--atomic` is essential for production upgrades. Without it, a failed upgrade leaves the release in a `failed` state with partial changes applied — which can be difficult to recover from manually.

## 8. What are subcharts and library charts, and when would you use each?

**Topic:** Concepts

**Answer:**

**Subcharts (dependency charts)** are full charts packaged inside another chart's `charts/` directory. The parent chart passes values to them via `values.yaml` using the subchart name as the key. They render their own templates and create their own Kubernetes resources.

Use subcharts when you want to bundle a dependency (PostgreSQL, Redis, Kafka) alongside your app chart and deploy them together as one release.

**Library charts** (`type: library` in `Chart.yaml`) contain only named templates — they produce no Kubernetes manifests and cannot be installed directly. They must be imported as a dependency by application charts.

Use library charts when you want to share common template logic (standard labels, `securityContext` defaults, affinity rules) across multiple application charts without copy-pasting.

```yaml
# Chart.yaml — declare dependencies
dependencies:
- name: postgresql           # subchart: creates real resources
  version: "12.5.6"
  repository: https://charts.bitnami.com/bitnami
  condition: postgresql.enabled

- name: common               # library chart: templates only
  version: "2.x.x"
  repository: https://charts.bitnami.com/bitnami
```

```yaml
# values.yaml — pass values to subchart using its name as key
postgresql:
  enabled: true
  auth:
    database: myapp
    username: myuser
```

```bash
helm dependency update ./mychart   # download deps, generate Chart.lock
helm dependency list ./mychart     # show resolved dependency versions
```

> **Interview tip:** `Chart.lock` pins exact resolved versions — commit it to Git so builds are reproducible. Re-run `helm dependency update` only when you intentionally want to update dependency versions.

## 9. What is `helm test` and how do you write a Helm test?

**Topic:** Lifecycle

**Answer:**

`helm test` runs test hooks against a deployed release to verify it is working correctly. A Helm test is a Kubernetes Pod or Job annotated with `helm.sh/hook: test`. After the pod exits, Helm reports pass (exit code 0) or fail (non-zero).

Tests live in `templates/tests/` and are only created when `helm test` is invoked — they are not part of a normal install or upgrade.

**Common test patterns:**

| Pattern | What it checks |
|---|---|
| HTTP smoke test | `curl` the Service and assert a 200 response |
| DNS resolution | Verify the Service DNS name resolves correctly |
| DB connectivity | Connect to PostgreSQL and run a `SELECT 1` |
| Auth test | Attempt login and verify a valid JWT is returned |

```yaml
# templates/tests/test-connection.yaml
apiVersion: v1
kind: Pod
metadata:
  name: {{ .Release.Name }}-test-connection
  annotations:
    "helm.sh/hook": test
    "helm.sh/hook-delete-policy": hook-succeeded
spec:
  restartPolicy: Never
  containers:
  - name: curl
    image: curlimages/curl:latest
    command:
    - curl
    - --fail
    - --silent
    - "http://{{ .Release.Name }}-svc:{{ .Values.service.port }}/healthz"
```

```bash
helm test myapp            # run tests for a release
helm test myapp --logs     # print container logs on completion
```

> **Interview tip:** `helm test` is often overlooked, but it is a clean way to add post-deploy smoke tests that are version-controlled alongside the chart and require no external CI tooling.

## 10. How do you debug a failing Helm chart and what commands are most useful?

**Topic:** Operations

**Answer:**

Helm provides several layers of debugging tools that should be applied in order:

| Step | Command | What it catches |
|---|---|---|
| 1 | `helm template` | Template syntax errors, missing values, malformed YAML |
| 2 | `helm install --dry-run --debug` | API server schema validation without applying resources |
| 3 | `helm lint` | Chart structure issues and common mistakes |
| 4 | `helm get all` | Full picture of a live release (values, manifests, hooks) |
| 5 | `kubectl describe / logs` | Pod-level failures Helm cannot see (OOMKill, image pull) |

**Common failure modes:**

| Failure | Diagnosed by |
|---|---|
| Template rendering error | `helm template` |
| Invalid resource schema | `helm install --dry-run` |
| Image pull failure | `kubectl describe pod` |
| Application crash loop | `kubectl logs` |
| Wrong values applied | `helm get values myapp --all` |

```bash
# Step 1 — render and inspect templates locally (no cluster needed)
helm template myapp ./mychart -f values-prod.yaml | less

# Step 2 — dry-run with API server validation
helm install myapp ./mychart --dry-run --debug \
  -f values-prod.yaml 2>&1 | less

# Step 3 — lint the chart
helm lint ./mychart -f values-prod.yaml

# Step 4 — inspect a live release
helm get all myapp
helm get values myapp --all
helm history myapp

# Step 5 — check Kubernetes events after deployment
kubectl get events -n production --sort-by='.lastTimestamp'
kubectl describe pod -l app.kubernetes.io/instance=myapp
```

> **Interview tip:** Always run `helm template` first — it is instant, requires no cluster access, and catches 90% of issues. Pipe its output through `kubeval` or `kubeconform` for schema validation without a live cluster.

## Quick Reference Cheat Sheet

| Command | Description |
|---|---|
| `helm create <name>` | Scaffold a new chart directory |
| `helm install <release> <chart>` | Install a chart as a new release |
| `helm upgrade <release> <chart>` | Upgrade an existing release |
| `helm upgrade --install` | Install if not present, upgrade if it is |
| `helm rollback <release> <rev>` | Roll back to a previous revision |
| `helm uninstall <release>` | Delete a release and all its resources |
| `helm history <release>` | Show all revisions for a release |
| `helm get values <release>` | Show user-supplied values of a release |
| `helm get manifest <release>` | Show rendered manifests of a release |
| `helm template <release> <chart>` | Render templates locally (no cluster) |
| `helm lint <chart>` | Static analysis of chart structure |
| `helm test <release>` | Run test hooks against a deployed release |
| `helm dependency update <chart>` | Download and lock chart dependencies |
| `helm repo add <name> <url>` | Add a chart repository |
| `helm search repo <keyword>` | Search repositories for charts |
| `helm pull <chart> --untar` | Download and extract a chart locally |
