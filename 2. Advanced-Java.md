# Java Advanced Concepts — Questions & Answers

## 1. What is the difference between checked and unchecked exceptions?

**Answer:**

Java exceptions fall into two categories based on how the compiler enforces handling.

#### Exception Hierarchy

```text
Throwable
-- Error (unchecked) -- JVM errors, not recoverable
|   +-- OutOfMemoryError, StackOverflowError
-- Exception
    +-- RuntimeException (unchecked) -- programming errors
    |   +-- NullPointerException, IllegalArgumentException
    +-- Other Exceptions (checked) -- recoverable conditions
        +-- IOException, SQLException
```

#### Best Practices

- Prefer checked exceptions for recoverable conditions; use runtime exceptions for programming errors.

## 2. Describe Java Memory Structure?

**Answer:**

The JVM divides process memory into several logical regions. Understanding these regions helps diagnose performance and memory issues and tune the runtime.

Overview of regions

- Java Heap: where object instances are allocated. The heap is usually split into generations:
    - Young (Eden + two Survivor spaces S0/S1): short-lived objects are allocated here; Minor GCs reclaim most of these.
    - Old / Tenured: holds longer-lived objects promoted from the Young generation.
- Metaspace: stores class metadata, method metadata, and related native structures (replaced PermGen). Controlled by `-XX:MaxMetaspaceSize`.
- Java Stacks: per-thread stacks that contain local variables, operand stacks, and frame data. Stack overflow occurs when stacks grow beyond limits.
- Native / Off-heap: memory used by native code and direct buffers (e.g., `ByteBuffer.allocateDirect`). Not part of the Java heap.
- Code Cache: JIT-compiled native code and related structures (tunable via `-XX:ReservedCodeCacheSize`).

Logical layout (simplified):

```
                                                                        JVM MEMORY LAYOUT
        +---------------------------------------------------+
        |                  Native / Off-heap                |
        +---------------------------------------------------+
        |                    Java Heap                       |
        |  +-------------+  +------------+   +------------+  |
        |  |   Young     |  |   Old/     |   |  Metaspace |  |
        |  |  (Eden/S0/1)|  |  Tenured   |   |  (class    |  |
        |  |             |  |            |   |  metadata) |  |
        |  +-------------+  +------------+   +------------+  |
        +---------------------------------------------------+
        |                    Java Stacks                     |
        +---------------------------------------------------+
```

GC roots & reachability

- GC roots include active thread stacks (local variables), static fields, JNI references, and certain internal JVM references.
- Objects are reachable (strong/soft/weak/phantom) and only unreachable objects are eligible for collection.

Common tuning knobs and inspection

- Heap sizing: `-Xms` (initial) and `-Xmx` (maximum).
- Metaspace sizing: `-XX:MaxMetaspaceSize`.
- Collector selection: `-XX:+UseG1GC`, `-XX:+UseZGC`, `-XX:+UseParallelGC`.
- GC logging / diagnostics: `-Xlog:gc*:file=gc.log:time,uptime,level`, `jcmd <pid> GC.heap_info`, `jmap -heap <pid>`, `jstat -gc <pid> [interval]`.

Practical notes

- Prefer modern, region-based collectors (G1, ZGC, Shenandoah) for large heaps and low pause requirements.
- Keep stacks reasonably sized per thread; use virtual threads for very high concurrency workloads when supported.
- Use off-heap/direct buffers for large binary data to reduce GC pressure, but monitor native memory usage.

This section focuses on the logical memory model; details of GC lifecycle and algorithms are covered in the next question.

## 3. Describe Java garbage collection process lifecycle and How does it work?

**Answer:**

Overview

Garbage Collection (GC) is the JVM's automatic memory management mechanism that reclaims memory occupied by objects that are no longer reachable from the application. The JVM uses a generational heap and several GC algorithms to balance throughput, latency, and footprint.

Generational Heap

- Young Generation: where new objects are allocated (Eden + two Survivor spaces S0/S1). Most objects die here.
- Old/Tenured Generation: holds long-lived objects promoted from the young generation.
- (Optional) Permanent/Metaspace: stores class metadata and related structures (Metaspace replaces PermGen).

GC Roots & Reachability

- GC roots: stack/local variables, static fields, JNI references, and certain internal JVM references.
- Reachability determines object lifetime: strongly reachable, softly reachable, weakly reachable, phantom reachable. Only unreachable objects are eligible for collection.

Typical GC Phases (high level)

1. Allocation: objects are allocated in Eden (fast path). When Eden fills, a Minor GC is triggered.
2. Minor GC (Young collection): copies live objects from Eden and one Survivor to the other Survivor space; survivors may be promoted to the Old generation based on age.
3. Major/Full GC: collects Old generation (can include Young). May be stop-the-world or largely concurrent depending on the collector.
4. Compaction/Relocation: some collectors compact memory to remove fragmentation; others (e.g., G1, ZGC) relocate regions concurrently.
5. Reference Processing & Finalization: JVM processes Reference objects (Soft/Weak/Phantom) and enqueues them; finalize() is deprecated and unreliable.

Collectors (selected)

- Serial GC: single-threaded, good for small heaps and simple environments.
- Parallel (Throughput) GC: multi-threaded young/old collection focused on throughput.
- Concurrent Mark-Sweep (CMS): concurrent marking to reduce pause times (older, deprecated in favor of G1/ZGC).
- G1 (Garbage First): region-based, concurrent marking and evacuation, balances latency and throughput for large heaps.
- ZGC / Shenandoah: low-latency, concurrent compaction/relocation designed for very large heaps with minimal pauses.

Pause Behaviour

- Minor GC typically causes short pauses (collects Young gen).
- Full/Major GC can cause longer pauses, though modern collectors aim to make most work concurrent to reduce pauses.
- Safe-points: certain GC operations require threads to reach safepoints; collectors minimize full safepoint work.

Example object lifecycle (short)

1. New object allocated in Eden.
2. Survives a Minor GC -> moved to Survivor and age incremented.
3. After reaching promotion threshold -> promoted to Old gen.
4. When no live references from GC roots exist -> eligible and reclaimed in a future GC.

Monitoring & Tuning

- Enable GC logging: `-Xlog:gc*:file=gc.log:time,uptime,level` (Java 9+) or `-XX:+PrintGCDetails -Xloggc:gc.log` (older).
- Heap sizing: `-Xms` and `-Xmx` to set initial and max heap.
- Choose collector by workload: `-XX:+UseG1GC`, `-XX:+UseZGC`, `-XX:+UseParallelGC`.
- Tune `-XX:MaxGCPauseMillis`, `-XX:InitiatingHeapOccupancyPercent`, `-XX:NewRatio`, and survivor/promotion thresholds as needed.

Best Practices

- Prefer modern collectors (G1, ZGC/Shenandoah) for large heaps and low-latency requirements.
- Avoid creating unnecessary long-lived objects; short-lived objects are cheap with generational GC.
- Use appropriate caching strategies (Weak/Soft references) and external caches (Caffeine) for controlled memory use.
- Profile with heap dumps (jmap, jcmd) and GC logs before applying tuning changes.

When to Force GC

- `System.gc()` is only a hint; don't rely on it in production. Use monitoring and proper collector/tuning instead.

Further reading

- JVM options and GC ergonomics: official OpenJDK documentation and `jcmd`/`jstat` tools for diagnostics.


## 4. What are Java annotations, and how are they used?

**Answer:**

- **Definition:** Annotations are Java metadata (since Java 5) that you attach to types, fields, methods, parameters, etc., to convey declarative information to the compiler, build tools, or runtime frameworks.

- **Declaration example:**

```java
@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.TYPE, ElementType.METHOD})
public @interface Loggable {
        String value() default "INFO";
        boolean includeArgs() default false;
}
```

- **Retention & targets:** Use `@Retention` (`SOURCE` | `CLASS` | `RUNTIME`) to control availability, and `@Target` (e.g., `TYPE`, `METHOD`, `FIELD`) to limit where the annotation can be applied. Other meta-annotations include `@Inherited`, `@Repeatable`, and `@Documented`.

- **How they're used:**
    - Compile-time: annotation processors (`javax.annotation.processing`) generate code, validation, or resources (e.g., AutoValue, Dagger).
    - Runtime: frameworks (Spring, JPA, Jackson) read `RUNTIME` annotations via reflection to drive behavior (DI, mapping, AOP, serialization).
    - Source-only: used for static analysis, linting, or IDE hints and discarded at compile time.

- **Access & processing:**
    - Reflection: `MyClass.class.getAnnotation(MyAnno.class)` or `method.getAnnotations()`.
    - Annotation processor: implement `AbstractProcessor` to inspect annotated elements and generate artifacts.

- **Best practices:** Keep annotations declarative and small, choose correct retention, provide sensible defaults, use `@Repeatable` when appropriate, and avoid embedding heavy logic inside annotations.

## 5. Explain the concept of reflection in Java

```java
// Get Class object
Class<?> clazz = Class.forName("com.example.User");

// Create instance
Object obj = clazz.getDeclaredConstructor().newInstance();

// Access private field

```

Use cases: frameworks, DI containers, serialization, testing, tooling.

## 6. What is the difference between ArrayList and LinkedList?
**Answer:**

- **ArrayList**
    - Backed by a resizable array.
    - Random access: O(1) for `get(index)`.
    - Insert/remove at end: amortized O(1); insert/remove in middle: O(n) (shifts elements).
    - Better memory locality and cache friendliness.

- **LinkedList**
    - Doubly-linked list of nodes.
    - Random access: O(n) for `get(index)`.
    - Insert/remove at known position (iterator) or head/tail: O(1) (no shifting).
    - Higher per-element memory overhead (node objects with pointers).

- **When to use which**
    - Use `ArrayList` for most cases where fast random access and compact memory are desired (e.g., read-mostly lists, iteration-heavy workloads).
    - Use `LinkedList` when your workload frequently inserts/removes at both ends or when you manipulate the list via iterators and need constant-time insertions/removals at those points.

- **Example**

```java
List<String> a = new ArrayList<>();
List<String> l = new LinkedList<>();

// random access is fast on ArrayList
String s = a.get(100);

// inserting in middle is cheaper with LinkedList if you already have the ListIterator
ListIterator<String> it = l.listIterator(50);
it.add("value");
```

- **Practical tip:** In modern applications `ArrayList` is usually the better default due to CPU/cache behavior; prefer `ArrayDeque` for queue-like usage and `LinkedList` only when its specific characteristics are required.

## 7. What is the difference between HashMap, TreeMap, and LinkedHashMap?

**Answer:**

- **HashMap**
    - Unordered map backed by a hash table; average O(1) for `get`/`put`.
    - Null keys and values allowed (one null key, multiple null values).
    - Iteration order is not guaranteed and can change over time.

- **TreeMap**
    - Sorted map implemented as a Red-Black tree; O(log n) for `get`/`put`/`remove`.
    - Keys are ordered either by natural ordering or by a provided `Comparator`.
    - Does not allow null keys (throws `NullPointerException` when comparing null keys).

- **LinkedHashMap**
    - Hash table + doubly-linked list that preserves insertion-order or access-order.
    - O(1) for `get`/`put` on average with predictable iteration order.
    - Useful for LRU caches when created with access-order and overriding `removeEldestEntry`.

- **Concurrency & thread-safety**
    - None of these are thread-safe for concurrent writes; for concurrent access prefer `ConcurrentHashMap`.
    - For read-mostly maps you can use `Collections.synchronizedMap(...)` or wrap appropriately.

- **When to use which**
    - Use `HashMap` for general-purpose, fastest average performance when order doesn't matter.
    - Use `TreeMap` when you need sorted keys or range views (`subMap`, `headMap`, `tailMap`).
    - Use `LinkedHashMap` when you need predictable iteration order (insertion or LRU access-order).

- **Example**

```java
Map<String,Integer> hash = new HashMap<>();
Map<String,Integer> tree = new TreeMap<>();
Map<String,Integer> linked = new LinkedHashMap<>();

hash.put("b",1); hash.put("a",2);
tree.put("b",1); tree.put("a",2); // tree iteration: a, b
linked.put("b",1); linked.put("a",2); // linked iteration: b, a
```

## 9. What is the difference between synchronized methods and blocks?

- Synchronized method: locks entire method for the object/class.
- Synchronized block: locks only critical section; preferred for finer-grained locking.

Best practice: use a private lock object rather than `this`.

```java
private final Object lock = new Object();
public void safeMethod() {
    synchronized (lock) {
        // critical section
    }
}
```

## 10. What are Java Streams, and how do they differ from Collections?

Streams are pipelines of computations: source -> zero or more intermediate operations (lazy) -> terminal operation (triggers execution).

```java
List<String> result = users.stream()
    .filter(u -> u.isActive())
    .map(User::getName)
    .sorted()
    .collect(Collectors.toList());
```

Streams are not data structures; they don't store elements and can be single-use.

## 13. Explain the concept of multithreading and concurrency in Java

**Answer:**

- **Definitions:**
    - *Multithreading* — running multiple threads within a single JVM process to perform tasks concurrently.
    - *Concurrency* — structuring a program to handle multiple tasks that can make progress independently; *parallelism* is when those tasks run at the same time on multiple CPU cores.

- **Creating and running threads:**
    - Low-level: extend `Thread` or implement `Runnable`.
    - Higher-level (recommended): use `ExecutorService` / thread pools, `ForkJoinPool` for divide-and-conquer, `CompletableFuture` for async composition.
    - Virtual threads (Project Loom): lightweight threads for massive concurrency in I/O-bound workloads.

- **Synchronization primitives & patterns:**
    - `synchronized` keyword and monitor locks for mutual exclusion.
    - `volatile` for visibility of simple shared variables.
    - `java.util.concurrent.locks` (e.g., `ReentrantLock`) for advanced locking.
    - Atomic classes (`AtomicInteger`, `AtomicReference`) for lock-free updates.
    - High-level synchronizers: `CountDownLatch`, `Semaphore`, `CyclicBarrier`, `Phaser`.

- **Concurrent collections & utilities:**
    - `ConcurrentHashMap`, `ConcurrentLinkedQueue`, `CopyOnWriteArrayList` provide thread-safe data structures.
    - Use `BlockingQueue` implementations for producer-consumer patterns.

- **Asynchronous composition:**
    - `CompletableFuture.supplyAsync(...)` and combinators (`thenApply`, `thenCompose`, `exceptionally`) for non-blocking workflows.

- **Best practices:**
    - Prefer higher-level concurrency APIs over raw threads.
    - Avoid shared mutable state when possible; favor immutability and thread confinement.
    - Use appropriate thread pool sizes (CPU-bound ≈ cores, I/O-bound larger) and tune for workload.
    - Add timeouts and cancellation support; handle interrupted status correctly.
    - Use concurrent collections and atomic types instead of coarse-grained locking when appropriate.
    - Profile and test concurrency under load using tools and thread profilers.

- **Small examples:**

```java
// ExecutorService example
ExecutorService pool = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());
pool.submit(() -> doWork());
pool.shutdown();

// CompletableFuture example
CompletableFuture.supplyAsync(() -> fetchData())
        .thenApply(data -> process(data))
        .exceptionally(ex -> handle(ex));
```

This answer focuses on concepts and practical guidance; use the `java.util.concurrent` package for production-grade concurrency and consider virtual threads for highly concurrent I/O workloads.

## 14. What is a Thread in Java and explain the Thread Lifecycle?

**Answer:**

A thread is a single sequence of execution within a JVM process. A Java program can run multiple threads concurrently to perform tasks in parallel or to structure I/O-bound and background work.

Thread lifecycle (official `java.lang.Thread.State`):

- `NEW`: Thread object created but `start()` has not been called.
- `RUNNABLE`: Thread is eligible to run and either running on a CPU or waiting for a CPU (the JVM and OS schedule it). Conceptually covers both "ready" and "running" states.
- `BLOCKED`: Waiting for a monitor lock (e.g., attempting to enter a `synchronized` block while another thread holds the lock).
- `WAITING`: Waiting indefinitely for another thread to perform a particular action (e.g., `Object.wait()` without timeout, `Thread.join()` without timeout, `LockSupport.park()`).
- `TIMED_WAITING`: Waiting with a timeout (e.g., `Thread.sleep()`, `wait(timeout)`, `join(timeout)`, `LockSupport.parkNanos()`).
- `TERMINATED`: Thread has completed execution (the `run()` method returned or an uncaught exception terminated it).

Typical state transitions

- `NEW` -> `start()` -> `RUNNABLE`
- `RUNNABLE` -> acquires CPU -> (running) -> may `sleep()`/`wait()` -> `TIMED_WAITING`/`WAITING`
- `RUNNABLE` -> attempts to enter synchronized block -> `BLOCKED` (until lock is available)
- `RUNNABLE` -> `run()` returns -> `TERMINATED`

Key thread operations and methods

- `start()` — schedules the thread and causes `run()` to be invoked on a new call stack.
- `run()` — contains the thread's logic; do not call `run()` directly to create a new thread.
- `join()` — waits for another thread to terminate.
- `sleep(long millis)` — places current thread into `TIMED_WAITING` for the duration; throws `InterruptedException` when interrupted.
- `interrupt()` — sets the interrupt flag; causes blocking operations to throw `InterruptedException` or can be checked with `Thread.interrupted()`.
- `setDaemon(true)` — marks thread as a daemon; JVM exits when only daemon threads remain.

Ways to create threads (examples)

```java
// 1. Subclass Thread
class MyThread extends Thread {
    public void run() { System.out.println("running"); }
}
new MyThread().start();

// 2. Runnable
Runnable r = () -> System.out.println("runnable");
new Thread(r).start();

// 3. Callable + Future (with ExecutorService)
Callable<String> task = () -> {
    Thread.sleep(100);
    return "result";
};
ExecutorService pool = Executors.newFixedThreadPool(4);
Future<String> f = pool.submit(task);
String result = f.get();
pool.shutdown();
```

Important concurrency notes and best practices

- Prefer higher-level APIs (`ExecutorService`, `CompletableFuture`, `ForkJoinPool`, or virtual-thread executors) over manual `Thread` management.
- Always handle `InterruptedException` correctly: restore the interrupt status (`Thread.currentThread().interrupt()`) if you cannot rethrow.
- Avoid `Thread.stop()` and other deprecated forceful termination methods.
- Use `synchronized`, `ReentrantLock`, atomic types, and concurrent collections (`ConcurrentHashMap`, `BlockingQueue`) to protect shared state.
- Use `daemon` threads for background tasks that should not prevent JVM shutdown; avoid using daemon threads for important cleanup work.
- Tune thread pool sizes to workload: CPU-bound ≈ number of cores; I/O-bound can be larger.

Common pitfalls

- Deadlock: ensure consistent lock ordering and prefer timed waits to help detect issues.
- Starvation: avoid unfair locks and unbounded priorities.
- Busy-waiting: prefer blocking constructs to reduce CPU use.

This gives a compact but practical overview of what a thread is, the JVM thread states/lifecycle, how to create and manage threads, and best practices for concurrency in Java.


## 15. What are Virtual Threads in Java (Project Loom)?

**Answer:**

Virtual threads (Project Loom) are lightweight threads managed by the JVM scheduler that make it practical to create millions of concurrent, mostly-blocking tasks without the heavy resource cost of platform threads.

- **Key characteristics:**
    - Extremely cheap to create and suspend; each virtual thread uses far less memory than a platform thread.
    - Designed to map many virtual threads onto a much smaller number of carrier (platform) threads.
    - Integrates with existing blocking APIs: blocking a virtual thread does not block a carrier thread.

- **Creation examples:**

```java
// Single virtual thread
Thread.startVirtualThread(() -> doBlockingIO());

// Virtual-thread-per-task executor (try-with-resources)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
        executor.submit(() -> handleRequest());
}
```

- **Benefits:**
    - Simplifies asynchronous programming by allowing straightforward blocking-style code at scale.
    - Greatly improves scalability for I/O-bound workloads (servers, crawlers, stream processors).

- **Caveats & migration notes:**
    - Some low-level libraries or native code expecting platform-thread semantics may need review.
    - Use structured concurrency (`StructuredTaskScope`) where available to manage lifecycles and failures.
    - Tune executor and be mindful of per-thread stack footprints when creating extremely large numbers of virtual threads.

- **Best practices:**
    - Prefer virtual threads for I/O-bound workloads; use platform threads for APIs that require thread-local platform behavior.
    - Adopt structured concurrency patterns to keep task lifecycles predictable.
    - Test libraries and integrations under realistic load to find edge cases.

## 16. What is the `java.util.concurrent` package?

Provides executors & thread pools, concurrent collections, synchronizers (CountDownLatch, Semaphore, CyclicBarrier), atomic variables, `Future` and `CompletableFuture`.

Examples:

```java
ExecutorService fixedPool = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());
ExecutorService cachedPool = Executors.newCachedThreadPool();
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);
ExecutorService virtualPool = Executors.newVirtualThreadPerTaskExecutor();

// CountDownLatch
CountDownLatch latch = new CountDownLatch(3);
latch.countDown();
latch.await();

// CompletableFuture example
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> fetchUser(userId))
    .thenApply(user -> user.getEmail())
    .exceptionally(ex -> "fallback@email.com");
```

Best practices: prefer high-level constructs (`ExecutorService`, `CompletableFuture`), avoid manual thread management when possible.
