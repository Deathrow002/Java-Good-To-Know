# Java Advanced Concepts — Questions & Answers

## 1. What is the difference between checked and unchecked exceptions?

**Answer:**

Java exceptions are divided into checked and unchecked exceptions based on how the compiler enforces handling.

- **Checked exceptions** (subclasses of `Exception` but not `RuntimeException`):
    - Must be declared in a method's `throws` clause or handled with a `try/catch` block.
    - Represent recoverable conditions that the caller should be forced to consider (e.g., `IOException`, `SQLException`).
    - Compiler enforces handling at compile time.

- **Unchecked exceptions** (subclasses of `RuntimeException` and `Error`):
    - Not required to be declared or caught. Examples: `NullPointerException`, `IllegalArgumentException`, `IndexOutOfBoundsException`.
    - Indicate programming errors or unrecoverable conditions; often reflect bugs that should be fixed rather than handled at runtime.

Examples

```java
// Checked: must be declared or caught
public void readFile(String path) throws IOException {
        Files.readAllLines(Path.of(path));
}

// Unchecked: no declaration required
public int divide(int a, int b) {
        return a / b; // may throw ArithmeticException (unchecked) if b == 0
}
```

When to use which

- Use checked exceptions for recoverable errors where the API consumer can meaningfully handle the condition (e.g., retrying I/O, prompting for credentials).
- Use unchecked exceptions for programming errors (invalid arguments, illegal state) that callers should fix rather than catch.

Best practices

- Prefer clear exception types with useful messages and causal chaining (`new MyException("msg", cause)`).
- Don't overuse checked exceptions—excessive checked exceptions can make APIs verbose and hard to use.
- For library APIs, consider converting low-level checked exceptions into a small set of meaningful checked exceptions or documented unchecked exceptions depending on the expected caller behavior.
- Always clean up resources in a `finally` block or use try-with-resources to avoid leaks, regardless of exception type.

Notes

- `Error` subclasses (e.g., `OutOfMemoryError`) are unchecked and indicate JVM-level failures — they are usually not handled.
- You can wrap checked exceptions in runtime exceptions (e.g., `UncheckedIOException`) when you need to propagate them through interfaces that don't allow checked exceptions.

This distinction helps API designers and callers decide which conditions should be handled explicitly and which indicate bugs or unrecoverable states.

## 2. Describe Java Memory Structure?

**Answer:**

The JVM divides process memory into several logical regions. Understanding these regions helps diagnose performance and memory issues and tune the runtime.

Overview of regions

- Java Heap: where object instances are allocated. The heap is usually split into generations:
    - Young (Eden + two Survivor spaces S0/S1): short-lived objects are allocated here; Minor GCs reclaim most of these.
    - Old / Tenured: holds longer-lived objects promoted from the Young generation.
- Metaspace: stores class metadata, method metadata, and related native structures (replaced PermGen). Controlled by `-XX:MaxMetaspaceSize`.
- Java Stacks: per-thread stacks that contain local variables, operand stacks, and frame data. Stack overflow occurs when stacks grow beyond limits.
- Native / Off-heap: memory used by native code and direct buffers (e.g., `ByteBuffer.allocateDirect`). Not part of the Java heap.
- Code Cache: JIT-compiled native code and related structures (tunable via `-XX:ReservedCodeCacheSize`).

Logical layout (simplified):

```
                                                                        JVM MEMORY LAYOUT
        +---------------------------------------------------+
        |                  Native / Off-heap                |
        +---------------------------------------------------+
        |                    Java Heap                       |
        |  +-------------+  +------------+   +------------+  |
        |  |   Young     |  |   Old/     |   |  Metaspace |  |
        |  |  (Eden/S0/1)|  |  Tenured   |   |  (class    |  |
        |  |             |  |            |   |  metadata) |  |
        |  +-------------+  +------------+   +------------+  |
        +---------------------------------------------------+
        |                    Java Stacks                     |
        +---------------------------------------------------+
```

GC roots & reachability

- GC roots include active thread stacks (local variables), static fields, JNI references, and certain internal JVM references.
- Objects are reachable (strong/soft/weak/phantom) and only unreachable objects are eligible for collection.

Common tuning knobs and inspection

- Heap sizing: `-Xms` (initial) and `-Xmx` (maximum).
- Metaspace sizing: `-XX:MaxMetaspaceSize`.
- Collector selection: `-XX:+UseG1GC`, `-XX:+UseZGC`, `-XX:+UseParallelGC`.
- GC logging / diagnostics: `-Xlog:gc*:file=gc.log:time,uptime,level`, `jcmd <pid> GC.heap_info`, `jmap -heap <pid>`, `jstat -gc <pid> [interval]`.

Practical notes

- Prefer modern, region-based collectors (G1, ZGC, Shenandoah) for large heaps and low pause requirements.
- Keep stacks reasonably sized per thread; use virtual threads for very high concurrency workloads when supported.
- Use off-heap/direct buffers for large binary data to reduce GC pressure, but monitor native memory usage.

This section focuses on the logical memory model; details of GC lifecycle and algorithms are covered in the next question.

## 3. Describe Java garbage collection process lifecycle and How does it work?

**Answer:**

Overview

Garbage Collection (GC) is the JVM's automatic memory management mechanism that reclaims memory occupied by objects that are no longer reachable from the application. The JVM uses a generational heap and several GC algorithms to balance throughput, latency, and footprint.

Generational Heap

- Young Generation: where new objects are allocated (Eden + two Survivor spaces S0/S1). Most objects die here.
- Old/Tenured Generation: holds long-lived objects promoted from the young generation.
- (Optional) Permanent/Metaspace: stores class metadata and related structures (Metaspace replaces PermGen).

GC Roots & Reachability

- GC roots: stack/local variables, static fields, JNI references, and certain internal JVM references.
- Reachability determines object lifetime: strongly reachable, softly reachable, weakly reachable, phantom reachable. Only unreachable objects are eligible for collection.

Typical GC Phases (high level)

1. Allocation: objects are allocated in Eden (fast path). When Eden fills, a Minor GC is triggered.
2. Minor GC (Young collection): copies live objects from Eden and one Survivor to the other Survivor space; survivors may be promoted to the Old generation based on age.
3. Major/Full GC: collects Old generation (can include Young). May be stop-the-world or largely concurrent depending on the collector.
4. Compaction/Relocation: some collectors compact memory to remove fragmentation; others (e.g., G1, ZGC) relocate regions concurrently.
5. Reference Processing & Finalization: JVM processes Reference objects (Soft/Weak/Phantom) and enqueues them; finalize() is deprecated and unreliable.

Collectors (selected)

- Serial GC: single-threaded, good for small heaps and simple environments.
- Parallel (Throughput) GC: multi-threaded young/old collection focused on throughput.
- Concurrent Mark-Sweep (CMS): concurrent marking to reduce pause times (older, deprecated in favor of G1/ZGC).
- G1 (Garbage First): region-based, concurrent marking and evacuation, balances latency and throughput for large heaps.
- ZGC / Shenandoah: low-latency, concurrent compaction/relocation designed for very large heaps with minimal pauses.

Pause Behaviour

- Minor GC typically causes short pauses (collects Young gen).
- Full/Major GC can cause longer pauses, though modern collectors aim to make most work concurrent to reduce pauses.
- Safe-points: certain GC operations require threads to reach safepoints; collectors minimize full safepoint work.

Example object lifecycle (short)

1. New object allocated in Eden.
2. Survives a Minor GC -> moved to Survivor and age incremented.
3. After reaching promotion threshold -> promoted to Old gen.
4. When no live references from GC roots exist -> eligible and reclaimed in a future GC.

Monitoring & Tuning

- Enable GC logging: `-Xlog:gc*:file=gc.log:time,uptime,level` (Java 9+) or `-XX:+PrintGCDetails -Xloggc:gc.log` (older).
- Heap sizing: `-Xms` and `-Xmx` to set initial and max heap.
- Choose collector by workload: `-XX:+UseG1GC`, `-XX:+UseZGC`, `-XX:+UseParallelGC`.
- Tune `-XX:MaxGCPauseMillis`, `-XX:InitiatingHeapOccupancyPercent`, `-XX:NewRatio`, and survivor/promotion thresholds as needed.

Best Practices

- Prefer modern collectors (G1, ZGC/Shenandoah) for large heaps and low-latency requirements.
- Avoid creating unnecessary long-lived objects; short-lived objects are cheap with generational GC.
- Use appropriate caching strategies (Weak/Soft references) and external caches (Caffeine) for controlled memory use.
- Profile with heap dumps (jmap, jcmd) and GC logs before applying tuning changes.

When to Force GC

- `System.gc()` is only a hint; don't rely on it in production. Use monitoring and proper collector/tuning instead.

Further reading

- JVM options and GC ergonomics: official OpenJDK documentation and `jcmd`/`jstat` tools for diagnostics.


## 4. What are Java annotations, and how are they used?

**Answer:**

- **Definition:** Annotations are Java metadata (since Java 5) that you attach to types, fields, methods, parameters, etc., to convey declarative information to the compiler, build tools, or runtime frameworks.

- **Declaration example:**

```java
@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.TYPE, ElementType.METHOD})
public @interface Loggable {
        String value() default "INFO";
        boolean includeArgs() default false;
}
```

- **Retention & targets:** Use `@Retention` (`SOURCE` | `CLASS` | `RUNTIME`) to control availability, and `@Target` (e.g., `TYPE`, `METHOD`, `FIELD`) to limit where the annotation can be applied. Other meta-annotations include `@Inherited`, `@Repeatable`, and `@Documented`.

- **How they're used:**
    - Compile-time: annotation processors (`javax.annotation.processing`) generate code, validation, or resources (e.g., AutoValue, Dagger).
    - Runtime: frameworks (Spring, JPA, Jackson) read `RUNTIME` annotations via reflection to drive behavior (DI, mapping, AOP, serialization).
    - Source-only: used for static analysis, linting, or IDE hints and discarded at compile time.

- **Access & processing:**
    - Reflection: `MyClass.class.getAnnotation(MyAnno.class)` or `method.getAnnotations()`.
    - Annotation processor: implement `AbstractProcessor` to inspect annotated elements and generate artifacts.

- **Best practices:** Keep annotations declarative and small, choose correct retention, provide sensible defaults, use `@Repeatable` when appropriate, and avoid embedding heavy logic inside annotations.

## 6. What is the difference between ArrayList and LinkedList?
**Answer:**

- **ArrayList**
    - Backed by a resizable array.
    - Random access: O(1) for `get(index)`.
    - Insert/remove at end: amortized O(1); insert/remove in middle: O(n) (shifts elements).
    - Better memory locality and cache friendliness.

- **LinkedList**
    - Doubly-linked list of nodes.
    - Random access: O(n) for `get(index)`.
    - Insert/remove at known position (iterator) or head/tail: O(1) (no shifting).
    - Higher per-element memory overhead (node objects with pointers).

- **When to use which**
    - Use `ArrayList` for most cases where fast random access and compact memory are desired (e.g., read-mostly lists, iteration-heavy workloads).
    - Use `LinkedList` when your workload frequently inserts/removes at both ends or when you manipulate the list via iterators and need constant-time insertions/removals at those points.

- **Example**

```java
List<String> a = new ArrayList<>();
List<String> l = new LinkedList<>();

// random access is fast on ArrayList
String s = a.get(100);

// inserting in middle is cheaper with LinkedList if you already have the ListIterator
ListIterator<String> it = l.listIterator(50);
it.add("value");
```

- **Practical tip:** In modern applications `ArrayList` is usually the better default due to CPU/cache behavior; prefer `ArrayDeque` for queue-like usage and `LinkedList` only when its specific characteristics are required.

## 7. What is the difference between HashMap, TreeMap, and LinkedHashMap?

**Answer:**

The Java Collections Framework provides three primary `Map` implementations. Choosing the right one depends on your requirements for **ordering**, **performance**, and **null handling**.

---

### a. Comparison Matrix

| Feature | HashMap | LinkedHashMap | TreeMap |
|---|---|---|---|
| **Data Structure** | Hash Table | Hash Table + Doubly-linked list | Red-Black Tree (Self-balancing) |
| **Iteration Order** | **Unordered** (Random) | **Insertion Order** (or Access Order) | **Sorted Order** (Natural or Comparator) |
| **Get / Put / Remove** | **O(1)** (Amortized) | **O(1)** (Amortized) | **O(log n)** |
| **Null Keys/Values** | One Null Key, Multiple Values | One Null Key, Multiple Values | **No Null Keys**; Multiple Values |
| **Best Used For** | General-purpose high performance. | Caches (LRU) and maintain order. | Range queries and sorted data. |

---

### b. Implementation Details

#### b-1. HashMap (The Default Choice)
- **Bucket to Tree Conversion**: Since Java 8, if a bucket (linked list) grows beyond a certain threshold (**`TREEIFY_THRESHOLD = 8`**), it is converted into a balanced tree. This prevents "Hash Collision Attacks" by keeping worst-case performance at **O(log n)** instead of O(n).
- **Initial Capacity**: Default is 16 with a load factor of 0.75.

#### b-2. LinkedHashMap (Predictable Iteration)
- **LRU Cache**: By passing `true` to the `accessOrder` constructor parameter and overriding `removeEldestEntry()`, you can easily build an **LRU (Least Recently Used) cache**.
- **Performance**: Slightly slower than `HashMap` due to the overhead of maintaining the doubly-linked list for each entry.

#### b-3. TreeMap (Navigable & Sorted)
- **Navigation**: Implements `NavigableMap`, providing powerful methods like `firstKey()`, `lastKey()`, `ceilingKey()` (closest match >=), and `floorKey()` (closest match <=).
- **Ordering**: Requires keys to implement `Comparable` or a `Comparator` must be provided at construction.

---

### c. Thread-Safety & Concurrency
None of these implementations are thread-safe.
- **Legacy**: `Hashtable` (thread-safe but slow/obsolete).
- **Synchronized Wrapper**: `Collections.synchronizedMap(map)` (coarse-grained locking).
- **Modern Standard**: **`ConcurrentHashMap`** (uses "Lock Striping" or CAS for high concurrency/performance).

---

### d. Illustrative Example
```java
public void compareMaps() {
    // 1. HashMap: Output is unpredictable
    Map<String, String> hash = new HashMap<>();
    
    // 2. LinkedHashMap: Output matches insertion order
    Map<String, String> linked = new LinkedHashMap<>();
    
    // 3. TreeMap: Output follows alphabetical order (Natural Ordering)
    Map<String, String> tree = new TreeMap<>();

    for (String key : Arrays.asList("Z", "A", "M")) {
        hash.put(key, key);
        linked.put(key, key);
        tree.put(key, key);
    }

    System.out.println("HashMap: " + hash.keySet());        // [A, Z, M] (or random)
    System.out.println("LinkedHashMap: " + linked.keySet()); // [Z, A, M] (Strictly insertion)
    System.out.println("TreeMap: " + tree.keySet());       // [A, M, Z] (Sorted)
}
```

## 8. What are generics in Java, and why are they useful?

**Answer:**

**Generics** (introduced in Java 5) allow you to parameterize types (classes, interfaces, and methods). They enable you to write code that can work with different types while providing **compile-time type safety**.

---

### a. Why use Generics?
1.  **Type Safety**: The compiler ensures that you only insert objects of the correct type into a collection.
2.  **Elimination of Casts**: You don't need to manually cast objects when retrieving them from a collection.
3.  **Code Reuse**: You can write a single class or method that works with any type (e.g., `ArrayList<T>`).

---

### b. Core Concepts

#### b-1. Type Erasure
Java implements generics using **Type Erasure**. The compiler replaces all generic type parameters with their bounds (or `Object` if unbounded) and inserts the necessary casts. This ensures backward compatibility with older Java versions but means that generic type information is **not available at runtime**.
- *Example*: `List<String>` and `List<Integer>` both become `List` at runtime.

#### b-2. Bounded Wildcards (PECS)
To handle the "variance" of generic types, Java uses wildcards (`?`).
- **Upper Bounded (`<? extends T>`)**: The type must be `T` or a subclass of `T`.
- **Lower Bounded (`<? super T>`)**: The type must be `T` or a superclass of `T`.

**The PECS Principle (Producer Extends, Consumer Super):**
- Use **`extends`** when you are **producing** (reading) items from a collection.
- Use **`super`** when you are **consuming** (writing/adding) items to a collection.

---

### c. Illustrative Example (PECS)
```java
public void processElements(List<? extends Number> producer, List<? super Integer> consumer) {
    // 1. PRODUCER: We can read Numbers or their subclasses
    for (Number n : producer) {
        System.out.println(n.doubleValue());
    }

    // 2. CONSUMER: We can write Integers into a list that accepts Integer or its supertypes
    consumer.add(10);
    consumer.add(20);
}
```

---

### d. Limitations of Generics
1.  **No Primitive Types**: You must use wrappers (e.g., `List<Integer>`, not `List<int>`).
2.  **Cannot Create Instances**: You cannot do `new T()`.
3.  **No Static Fields**: You cannot declare static fields of the type parameter `T`.
4.  **No `instanceof` with specifics**: `if (list instanceof ArrayList<String>)` is illegal due to type erasure.

**Bottom Line:** Generics make your code more robust by shifting type-related errors from runtime (ClassCastException) to compile-time.

## 9. What is the difference between synchronized methods and blocks?

**Answer:**

**Synchronization** in Java ensures that only one thread can execute a critical section of code at a time (Mutual Exclusion) and ensuring that changes made by one thread are visible to others (Memory Visibility).

---

### a. Comparison of Methods vs. Blocks

| Feature | Synchronized Method | Synchronized Block |
|---|---|---|
| **Scope** | Locks the **entire method**. | Locks only a **specific section** of code. |
| **Lock Object** | Uses `this` (instance) or `Class` (static). | Can use **any object** as a lock. |
| **Performance** | Coarse-grained; can lead to unnecessary waiting. | **Fine-grained**; minimizes the blocked time. |
| **Flexibility** | Lowest; hard to use different locks for different fields. | **Highest**; allows multiple locks for better concurrency. |

---

### b. The "Private Lock" Pattern
Using `synchronized(this)` is generally discouraged because the lock is public. Any other code (outside your control) could also lock on your object, leading to potential deadlocks or performance issues.

**Best Practice**: Use a private, final object as a lock.
```java
public class Counter {
    private int count = 0;
    private final Object lock = new Object(); // Private lock

    public void increment() {
        // Only the critical update is synchronized
        synchronized (lock) {
            count++;
        }
    }
}
```

---

### c. Instance vs. Static Synchronization
1.  **Instance Methods**: Synchronized on `this`. Two different instances can run the method concurrently.
2.  **Static Methods**: Synchronized on the `Class` object (`MyClass.class`). Only one thread can run the method across all instances of that class.

---

### d. Key Concepts: Monitor & Reentrancy
- **Intrinsic Lock (Monitor)**: Every Java object has an internal lock (monitor). The `synchronized` keyword is built upon this mechanism.
- **Reentrancy**: Java locks are **reentrant**. This means if a thread already holds a lock, it can enter other `synchronized` blocks that use the same lock without getting blocked. This prevents a thread from deadlocking itself.

---

### e. The Modern Alternative: `java.util.concurrent.locks`
For more advanced scenarios (like try-locks, timed-waits, or multiple condition variables), prefer **`ReentrantLock`** over the `synchronized` keyword.

**Bottom Line:** Favor **synchronized blocks** over methods to reduce the "critical section" size and improve application throughput.

## 10. What are Java Streams, and how do they differ from Collections?

**Answer:**

A **Java Stream** (introduced in Java 8) is a sequence of elements supporting sequential and parallel aggregate operations. Unlike a Collection, a Stream is a **functional pipeline** that processes data but does not store it.

---

### a. Key Differences: Collections vs. Streams

| Feature | Collections | Streams |
|---|---|---|
| **Data Storage** | Stores data physically in memory. | Does **not store** data; just a view/pipeline. |
| **Modification** | You can add/remove elements. | Elements cannot be added or removed. |
| **Iteration** | **External**: Manual `for-each` or `Iterator`. | **Internal**: The library handles the iteration. |
| **Execution** | **Eager**: Computed immediately. | **Lazy**: Computed only when a terminal op is called. |
| **Reusable?** | **Yes**: Can be traversed multiple times. | **No**: Single-use; once consumed, it's gone. |

---

### b. Three Phases of a Stream Pipeline
A stream is not processed until a terminal operation is called.

1.  **The Source**: Where the data comes from (e.g., `list.stream()`, `Arrays.stream()`).
2.  **Intermediate Operations**: Transform the data (e.g., `filter`, `map`, `sorted`). These are **lazy**; they just build the "recipe" for processing.
3.  **Terminal Operation**: Triggers the execution and produces a result (e.g., `collect`, `forEach`, `reduce`, `count`).

---

### c. Core Benefits
- **Lazy Evaluation**: If you have a `.filter().findFirst()` chain, the stream only processes elements until it finds a match, skipping the rest of the collection (**Short-circuiting**).
- **Declarative Code**: Focus on *what* to do (filter, map) rather than *how* to do it (loops, temporary variables).
- **Parallelism**: Simply calling `.parallelStream()` allows the JVM to use multiple cores for the computation with zero manual thread management.

---

### d. Illustrative Example
```java
List<String> result = users.stream()           // 1. Source
    .filter(u -> u.getAge() > 18)             // 2. Intermediate (Lazy)
    .map(User::getName)                       // 2. Intermediate (Lazy)
    .limit(5)                                 // 2. Short-circuiting
    .collect(Collectors.toList());            // 3. Terminal (Triggers execution)
```

**Bottom Line:** Use **Collections** to manage and store your data; use **Streams** to compute and transform that data in a clean, functional way.

## 11. What is the Java Collections Framework?

**Answer:**

The **Java Collections Framework (JCF)** is a unified architecture for representing and manipulating collections. It provides high-performance implementations of data structures, reducing programming effort and increasing code reuse.

---

### a. The Hierarchy at a Glance
The JCF consists of two primary hierarchies:

1.  **`Collection` Interface**: The root of the hierarchy for `List`, `Set`, and `Queue`.
2.  **`Map` Interface**: A separate hierarchy for key-value pairs (though still part of the framework).

---

### b. Core Interfaces & Implementations

| Interface | Characteristics | Popular Implementations |
|---|---|---|
| **`List`** | Ordered, allows duplicates, positional access. | `ArrayList`, `LinkedList`, `Vector`. |
| **`Set`** | Unordered, no duplicates, unique elements. | `HashSet`, `LinkedHashSet`, `TreeSet` (Sorted). |
| **`Queue`** | Holds elements for processing (FIFO/LIFO). | `PriorityQueue`, `ArrayDeque`. |
| **`Map`** | Key-Value pairs, unique keys. | `HashMap`, `LinkedHashMap`, `TreeMap`. |

---

### c. Modern Enhancements: Java 9+ Factory Methods
Since Java 9, creating unmodifiable collections has become much simpler using the `.of()` methods.

```java
List<String> list = List.of("A", "B", "C"); // Immutable
Set<Integer> set = Set.of(1, 2, 3);          // Immutable
Map<String, Integer> map = Map.of("K1", 1, "K2", 2);
```

---

### d. The `Collections` Utility Class
The `java.util.Collections` class provides static methods that operate on or return collections:
- **Algorithms**: `sort()`, `binarySearch()`, `reverse()`, `shuffle()`.
- **Wrappers**: `unmodifiableList()`, `synchronizedMap()`, `singletonList()`.

---

### e. Best Practices
1.  **Program to Interface**: Always declare variables using the interface type (e.g., `List<String> list = new ArrayList<>();`). This allows you to swap implementations without breaking client code.
2.  **Choose Wisely**: Use `ArrayList` for random access, `LinkedList` for frequent insertions/removals at ends, and `HashSet` for high-performance uniqueness checks.
3.  **Favor Immutability**: Use `List.of()` or `Collections.unmodifiableList()` when the collection should not be modified after creation to avoid side effects.

**Bottom Line:** The JCF is much more than just a set of lists; it's a toolbox of specialized data structures designed to optimize performance and safety.

## 12. What are common Stream API operations and best practices?

**Answer:**

The Stream API provides a functional approach to processing collections. Operations are divided into two categories: **Intermediate** (transformative and lazy) and **Terminal** (triggers execution and produces a result).

---

### a. Common Operations

#### a-1. Intermediate Operations (Lazy)
- **`filter(Predicate)`**: Filters elements based on a condition.
- **`map(Function)`**: Transforms each element (1-to-1).
- **`flatMap(Function)`**: Flattens nested structures (e.g., `Stream<List<String>>` to `Stream<String>`).
- **`sorted()` / `distinct()`**: State-based operations that require seeing all elements before proceeding.
- **`limit(n)` / `skip(n)`**: Short-circuiting operations to truncate the stream.

#### a- 2. Terminal Operations (Eager)
- **`collect(Collector)`**: Converts the stream into a Collection, Map, or String.
- **`reduce(BinaryOperator)`**: Combines all elements into a single result (e.g., sum, min/max).
- **`anyMatch` / `allMatch`**: Returns a boolean based on predicates.
- **`findFirst` / `findAny`**: Returns an `Optional` of an element.

---

### b. Advanced Collectors
The `Collectors` utility class provides powerful reduction operations:
```java
// 1. Grouping: Map<Category, List<Product>>
Map<Category, List<Product>> byCategory = products.stream()
    .collect(Collectors.groupingBy(Product::getCategory));

// 2. Joining: "Alice, Bob, Charlie"
String names = users.stream()
    .map(User::getName)
    .collect(Collectors.joining(", "));

// 3. Partitioning: Map<Boolean, List<User>> (Adults vs Minors)
Map<Boolean, List<User>> adults = users.stream()
    .collect(Collectors.partitioningBy(u -> u.getAge() >= 18));
```

---

### c. Performance & Best Practices
1.  **Use Primitive Streams**: Avoid performance hits from boxing/unboxing. Use `IntStream`, `LongStream`, or `DoubleStream` instead of `Stream<Integer>`.
    ```java
    int sum = menu.stream().mapToInt(Dish::getCalories).sum();
    ```
2.  **Stateless Lambdas**: Ensure your lambdas do not rely on or modify external state. This is especially critical for **Parallel Streams** to avoid race conditions.
3.  **Short-Circuiting**: Use operations like `limit()` or `findFirst()` early in the pipeline if you don't need to process the entire dataset.
4.  **Readability**: If a stream pipeline exceeds 3-4 operations, consider breaking it up or using descriptive method references (`User::getName`) instead of complex lambdas.
5.  **Resource Management**: When using streams with I/O (like `Files.lines()`), always use a **try-with-resources** block to ensure the stream is closed.

**Bottom Line:** Streams are for processing, not storing. Keep them pure, stateless, and leverage specialized primitive streams for maximum performance.

## 13. Explain the concept of multithreading and concurrency in Java

**Answer:**

Concurrency in Java has evolved from low-level thread management to high-level asynchronous pipelines and lightweight virtual threads. Understanding the difference between **Concurrency** (handling multiple tasks) and **Parallelism** (executing tasks simultaneously) is key to building scalable systems.

---

### a. Evolution of Java Concurrency

| Era | Primary API | Key Characteristics |
|---|---|---|
| **Legacy (1.0)** | `Thread`, `Runnable` | Manual thread management; heavy OS threads. |
| **Foundational (5.0)** | `java.util.concurrent` | **Executors**, **Locks**, **Atomics**, and **BlockingQueues**. |
| **Functional (8.0)** | `CompletableFuture` | Asynchronous, non-blocking callback chains. |
| **Modern (21+)** | **Virtual Threads** | Extremely lightweight; allows "thread-per-request" at scale. |

---

### b. The Java Memory Model (JMM)
To write correct concurrent code, you must manage three things:
1.  **Visibility**: Ensuring that a change made by one thread is visible to others. Use the **`volatile`** keyword or explicit locks.
2.  **Atomicity**: Ensuring an operation is uninterruptible (e.g., `count++` is NOT atomic). Use **`AtomicInteger`** or `synchronized`.
3.  **Ordering**: Preventing the compiler/CPU from reordering instructions in a way that breaks logic.

---

### c. Core Synchronization Primitives
- **`synchronized`**: The "intrinsic lock" for mutual exclusion.
- **`ReentrantLock`**: A more flexible explicit lock with features like `tryLock()` and fairness.
- **`CountDownLatch` / `CyclicBarrier`**: For coordinating multiple threads.
- **`Semaphore`**: For limiting access to a restricted resource (e.g., DB connections).
- **`CompletableFuture`**: For building complex async workflows without "Callback Hell."

---

### d. Modern Asynchronous Example
Instead of raw threads, modern Java uses `ExecutorService` and `CompletableFuture`.

```java
public void processAsync() {
    // 1. Create a thread pool
    try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
        
        // 2. Chain async operations
        CompletableFuture.supplyAsync(() -> fetchData(), executor)
            .thenApply(data -> transform(data))
            .thenAccept(result -> save(result))
            .exceptionally(ex -> {
                log.error("Failed: ", ex);
                return null;
            });
    } // Executor auto-closes in Java 19+
}
```

---

### e. Best Practices
1.  **Prefer High-Level APIs**: Never use `new Thread().start()`. Use `Executors` or `Virtual Threads`.
2.  **Minimize Shared Mutable State**: Use **Records** or immutable classes to reduce the need for synchronization.
3.  **Use Thread-Safe Collections**: Favor `ConcurrentHashMap` over `Collections.synchronizedMap()`.
4.  **Avoid Deadlocks**: Always acquire locks in a consistent order and use timed waits (`tryLock(timeout)`) to detect hangs.
5.  **Virtual Threads for I/O**: If your app is I/O-bound (REST calls, DB access), use Virtual Threads to handle millions of concurrent requests.

**Bottom Line:** Java concurrency is shifting from "managing locks" to "orchestrating data flows" using functional and asynchronous patterns.

## 14. What is a Thread in Java and explain the Thread Lifecycle?

**Answer:**

A **Thread** is the smallest unit of execution within a process. In Java, every thread is an instance of `java.lang.Thread`. Understanding the lifecycle of a thread is critical for debugging race conditions, deadlocks, and performance bottlenecks.

---

### a. The Six Thread States (`Thread.State`)

| State | Description | Transition Trigger |
|---|---|---|
| **`NEW`** | Created but not yet started. | `new Thread()` |
| **`RUNNABLE`** | Executing in the JVM (may be waiting for OS resources like CPU). | `thread.start()` |
| **`BLOCKED`** | Waiting for a monitor lock to enter a `synchronized` block/method. | Entry into a locked `synchronized` section. |
| **`WAITING`** | Waiting indefinitely for another thread to perform an action. | `object.wait()`, `thread.join()` |
| **`TIMED_WAITING`** | Waiting for a specified period. | `thread.sleep(ms)`, `wait(ms)`, `join(ms)` |
| **`TERMINATED`** | Finished execution or died due to an exception. | `run()` method completes. |

---

### b. State Transition Diagram (Conceptual)
```
       [ NEW ]
          |
       (start)
          |
    [  RUNNABLE  ] <---- (Lock Acquired) ---- [ BLOCKED ]
     /     |      \                          ^
 (sleep) (wait) (sync)                       |
   /       |       \____ (Lock Needed) ______/
  /        |        
[TIMED_WAIT] [WAITING]
  \        /
 (notify/time ok)
    \    /
  [RUNNABLE]
      |
 (completion)
      |
 [ TERMINATED ]
```

---

### c. Core Operations & Mechanisms

1.  **`start()` vs `run()`**: `start()` creates a new call stack and triggers the JVM to call `run()`. Calling `run()` directly executed the code in the *caller's* thread, no new thread is created.
2.  **`interrupt()`**: A **cooperative** mechanism. It sets an "interrupt flag." Blocking methods like `sleep()` or `wait()` will throw `InterruptedException` if the flag is set. Code should check `Thread.interrupted()` periodically to exit gracefully.
3.  **`join()`**: Allows one thread to wait for the completion of another.
4.  **Daemon Threads**: Background threads (e.g., GC). The JVM exits when the only remaining threads are daemon threads. They are created via `setDaemon(true)`.

---

### d. Modern Best Practices
- **Never use `stop()`, `suspend()`, or `resume()`**: These are deprecated because they are inherently unsafe (e.g., `stop()` releases all locks, leaving objects in inconsistent states).
- **Handle `InterruptedException` Properly**: Either rethrow it or restore the status using `Thread.currentThread().interrupt()`. Never swallow it.
- **Prefer Virtual Threads (Java 21+)**: For I/O-heavy workloads, virtual threads have much simpler lifecycles and significantly lower memory overhead compared to platform threads.

**Bottom Line:** A thread is a state machine. Most concurrency bugs occur when a thread is inadvertently stuck in `BLOCKED` or `WAITING` states due to improper lock management.

## 15. What are Virtual Threads in Java (Project Loom)?

**Answer:**

**Virtual Threads** (introduced in Java 21) are lightweight, JVM-managed threads designed to handle thousands—or even millions—of concurrent tasks. They replace the heavy "Platform Thread" (mapped 1:1 to OS threads) with a **many-to-many** model, where many virtual threads are scheduled onto a small pool of OS threads.

---

### a. Why Virtual Threads?
Traditionally, high-concurrency servers used "Reactive" or "Asynchronous" programming to avoid blocking limited OS threads. Virtual threads allow you to go back to simple, **blocking "Thread-per-request"** code without sacrificing scalability.

| Feature | Platform Threads (OS) | Virtual Threads (JVM) |
|---|---|---|
| **Memory** | ~2MB (Fixed stack) | ~KB (Dynamic stack) |
| **Context Switch** | Heavy (OS involved) | Lightweight (JVM task) |
| **Scalability** | Limited (thousands) | Massive (millions) |
| **I/O Handling** | Blocks the core thread. | JVM "unmounts" the thread. |

---

### b. How it Works: Mounting & Unmounting
The JVM uses a pool of **Carrier Threads** (usually a `ForkJoinPool`). 
1.  **Mounting**: When a virtual thread starts, the JVM "mounts" it onto a carrier thread.
2.  **Unmounting**: When the virtual thread hits a blocking I/O operation (e.g., waiting for a DB query), the JVM "unmounts" it, saves its state to the heap, and releases the carrier thread to do other work.
3.  **Resuming**: Once the I/O is done, the JVM schedules the virtual thread again, mounting it back onto *any* available carrier thread.

---

### c. The "Pinning" Problem (Critical Caveat)
A virtual thread is **pinned** to its carrier thread if it performs a blocking operation while:
- Inside a **`synchronized`** block or method.
- Executing a **native method** (JNI).

**Fix**: Replace `synchronized` with `ReentrantLock` in code that expects high concurrency on virtual threads.

---

### d. Creation Examples
```java
// 1. One-off virtual thread
Thread.startVirtualThread(() -> {
    System.out.println("Processing in: " + Thread.currentThread());
});

// 2. Executor for task-based work (Recommended)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> doDBWork());
    executor.submit(() -> callExternalAPI());
} // Auto-closes and waits for tasks
```

---

### e. Structured Concurrency
Introduced alongside virtual threads, **Structured Concurrency** (`StructuredTaskScope`) treats groups of related tasks as a single unit, ensuring that if one sub-task fails, all others are cancelled, preventing "leaked" background work.

**Bottom Line:** Virtual threads are the bridge between "Simple Code" and "Extreme Scalable Performance." They are perfect for I/O-bound microservices but provide no benefit for CPU-bound computations.

## 16. What is the `java.util.concurrent` package?

**Answer:**

The **`java.util.concurrent` (JUC)** package is the heartbeat of modern Java concurrency. Introduced in Java 5 and significantly expanded in Java 8 and 21, it provides a high-level toolkit that makes writing thread-safe, high-performance code easier and less error-prone than using low-level `synchronized` keywords.

---

### a. The Five Pillars of JUC

| Category | Key Classes | Primary Goal |
|---|---|---|
| **1. Executors** | `ExecutorService`, `ScheduledThreadPoolExecutor` | decoupling task **submission** from task **execution**. |
| **2. Collections** | `ConcurrentHashMap`, `BlockingQueue`, `CopyOnWriteArrayList` | High-performance, **thread-safe data structures** without coarse-grained locking. |
| **3. Synchronizers** | `CountDownLatch`, `Semaphore`, `CyclicBarrier`, `Phaser` | Coordinating the **flow and timing** of multiple threads. |
| **4. Atomics** | `AtomicInteger`, `AtomicReference`, `LongAdder` | **Lock-free** operations using low-level CPU instructions (CAS). |
| **5. Locks** | `ReentrantLock`, `ReadWriteLock`, `StampedLock` | Advanced locking with **try-locks**, **timed waits**, and fairness. |

---

### b. Why JUC is Better than Legacy Sync
1.  **Granularity**: JUC collections (like `ConcurrentHashMap`) use "Lock Striping" or CAS, allowing multiple threads to read and write different parts of the map simultaneously.
2.  **No Deadlock risk with `tryLock`**: Explicit locks allow you to attempt a lock and back off if it's unavailable, preventing deadlocks.
3.  **Throughput**: Classes like `LongAdder` are specifically designed for high-contention scenarios, outperforming `AtomicLong` by reducing CPU cache-line contention.

---

### c. The Modern Async Pipeline
JUC's **`CompletableFuture`** is the standard for non-blocking asynchronous programming.

```java
public void executeWorkflow() {
    ExecutorService executor = Executors.newFixedThreadPool(10);

    CompletableFuture.supplyAsync(() -> fetchOrder(123), executor)
        .thenApplyAsync(order -> processPayment(order), executor)
        .thenAccept(receipt -> emailCustomer(receipt))
        .exceptionally(ex -> {
            System.err.println("Workflow failed: " + ex.getMessage());
            return null;
        });
}
```

---

### d. Best Practices
- **Use `ConcurrentHashMap`**: It's almost always better than wrapping a `HashMap` in `Collections.synchronizedMap()`.
- **Prefer `LongAdder` for counters**: If you have a high-traffic counter, `LongAdder` is significantly faster than `AtomicLong`.
- **Bound your Queues**: When using `ExecutorService`, use a bounded `BlockingQueue` (like `ArrayBlockingQueue`) to prevent `OutOfMemoryError` if the system is overloaded.

**Bottom Line:** The `java.util.concurrent` package is an essential toolbox. It shifts the burden of complex thread coordination from the developer to a highly-optimized library.
