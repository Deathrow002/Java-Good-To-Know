Interview Questions and Answers

Spring Boot

1. What is Spring Boot, and how does it differ from Spring Framework?

Answer:

Spring Boot is an opinionated framework built on top of Spring Framework that dramatically simplifies application development and deployment.

Key Features of Spring Boot

  • Auto-configuration: Automatically configures beans based on classpath dependencies and existing beans
  • Embedded Servers: Includes Tomcat, Jetty, or Undertow — no need to deploy WAR files
  • Starter Dependencies: Pre-configured dependency bundles for common use cases
  • Production-Ready Features: Health checks, metrics, and externalized configuration out of the box
  • No XML Configuration: Java-based configuration with annotations

Spring Framework vs Spring Boot

Aspect Spring Framework Spring Boot
Configuration Manual XML or Java config Auto-configuration
Server External deployment required Embedded server included
Dependencies Manual version management Managed via starters
Setup Time Hours to days Minutes

Bottom Line: Spring Framework provides the core modules (IoC, AOP, MVC, Data), while Spring Boot provides the tooling to use them with minimal configuration.

2. What are Spring Boot starters?

Answer:

Spring Boot Starters are pre-packaged dependency bundles that provide all the libraries needed for a specific functionality with compatible versions.

How Starters Work

Instead of manually adding multiple dependencies with version conflicts, you add one starter and get everything needed:

<!-- One starter replaces many individual dependencies -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>

Common Starters

Starter Purpose Includes
spring-boot-starter-web REST APIs & Web MVC Spring MVC, Jackson, Tomcat
spring-boot-starter-data-jpa Database access Hibernate, Spring Data JPA, HikariCP
spring-boot-starter-security Authentication & Authorization Spring Security
spring-boot-starter-test Testing JUnit 5, Mockito, AssertJ
spring-boot-starter-actuator Production monitoring Health checks, metrics, endpoints

Benefits

  • No version conflicts: All dependencies are tested together
  • Faster setup: One dependency instead of many
  • Auto-configuration: Starters trigger automatic bean configuration

3. What is Spring Boot auto-configuration?

Answer:

Auto-configuration is Spring Boot's mechanism to automatically configure beans based on the classpath, existing beans, and properties — eliminating most manual configuration.

How It Works

  1. Classpath Scanning: Spring Boot detects libraries on the classpath (e.g., H2 database, Jackson)
  2. Conditional Evaluation: Uses @Conditional annotations to decide what to configure
  3. Bean Registration: Automatically creates and configures required beans

Example: DataSource Auto-Configuration

// Just add H2 dependency + these properties:
spring.datasource.url=jdbc:h2:mem:testdb
spring.datasource.driver-class-name=org.h2.Driver

// Spring Boot automatically:
// - Creates DataSource bean
// - Configures HikariCP connection pool
// - Sets up JdbcTemplate (if on classpath)

Key Conditional Annotations

  • @ConditionalOnClass — Configure only if a class exists on classpath
  • @ConditionalOnMissingBean — Configure only if no user-defined bean exists
  • @ConditionalOnProperty — Configure based on property value

Customization Options

  • Override: Define your own bean to replace auto-configured one
  • Exclude: Use @SpringBootApplication(exclude = DataSourceAutoConfiguration.class)
  • Debug: Set debug=true in properties to see auto-configuration report

Key Point: Auto-configuration is "opinionated but flexible" — sensible defaults that you can always override.

4. How do you create a REST API using Spring Boot?

Answer:

Creating a REST API in Spring Boot follows a layered architecture pattern:

Step 1: Add Dependency

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>

Step 2: Create REST Controller

@RestController
@RequestMapping("/api/users")
public class UserController {
    
    private final UserService userService;
    
    public UserController(UserService userService) {
        this.userService = userService;
    }
    
    @GetMapping
    public List<User> getAllUsers() {
        return userService.findAll();
    }
    
    @GetMapping("/{id}")
    public ResponseEntity<User> getUser(@PathVariable Long id) {
        return userService.findById(id)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
    }
    
    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public User createUser(@Valid @RequestBody UserDTO dto) {
        return userService.create(dto);
    }
    
    @PutMapping("/{id}")
    public User updateUser(@PathVariable Long id, @Valid @RequestBody UserDTO dto) {
        return userService.update(id, dto);
    }
    
    @DeleteMapping("/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void deleteUser(@PathVariable Long id) {
        userService.delete(id);
    }
}

Key Annotations

Annotation Purpose
@RestController Combines @Controller + @ResponseBody
@RequestMapping Base path for all endpoints
@GetMapping, @PostMapping... HTTP method shortcuts
@PathVariable Extract value from URL path
@RequestBody Deserialize JSON to object
@Valid Trigger bean validation

Step 3: Run Application

@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

Result: API available at http://localhost:8080/api/users with automatic JSON serialization.

5. What is the role of application.properties or application.yml?

Answer:

Configuration files externalize application settings, making apps configurable without code changes.

Properties vs YAML Format

📄 application.properties 📋 application.yml 
server.port=8081
spring.datasource.url=jdbc:mysql://localhost/db
spring.datasource.username=root
spring.datasource.password=${DB_PASSWORD}
 
server:
  port: 8081
spring:
  datasource:
    url: jdbc:mysql://localhost/db
    username: root
    password: ${DB_PASSWORD}
Simple key=value pairs
Easy to read for flat configs
s️ Repetitive prefixes 		Hierarchical structure
Less repetition
Supports lists natively

Common Configuration Categories

  • Server: server.port, server.servlet.context-path
  • Database: spring.datasource.*, spring.jpa.*
  • Logging: logging.level.*, logging.file.name
  • Security: spring.security.*
  • Custom: Your own properties accessible via @Value or @ConfigurationProperties

Profile-Specific Configuration

# application-dev.yml (activated with spring.profiles.active=dev)
spring:
  datasource:
    url: jdbc:h2:mem:devdb

# application-prod.yml
spring:
  datasource:
    url: jdbc:mysql://prod-server/db

Property Resolution Order (Highest Priority First)

  1. Command-line arguments (--server.port=9000)
  2. Environment variables (SERVER_PORT=9000)
  3. application-{profile}.yml outside JAR
  4. application-{profile}.yml inside JAR
  5. application.yml outside JAR
  6. application.yml inside JAR

Accessing Properties in Code

// Option 1: @Value annotation
@Value("${app.feature.enabled:false}")
private boolean featureEnabled;

// Option 2: @ConfigurationProperties (recommended for groups)
@ConfigurationProperties(prefix = "app.mail")
public record MailProperties(String host, int port, String from) {}

6. What is the difference between @SpringBootApplication and @EnableAutoConfiguration?

Answer:

@SpringBootApplication is a convenience annotation that combines three annotations into one:

What @SpringBootApplication Includes

@SpringBootApplication
// Is equivalent to:
@SpringBootConfiguration  // Marks this as a configuration class
@EnableAutoConfiguration  // Enables auto-configuration
@ComponentScan            // Scans for @Component classes in current package & below

Comparison

Annotation What It Does When to Use
@SpringBootApplication Configuration + Auto-config + Component scanning 99% of applications (standard choice)
@EnableAutoConfiguration Only enables auto-configuration When you need fine-grained control

Customization Examples

// Exclude specific auto-configurations
@SpringBootApplication(exclude = {
    DataSourceAutoConfiguration.class,
    SecurityAutoConfiguration.class
})
public class Application { }

// Custom component scan base packages
@SpringBootApplication(scanBasePackages = "com.example")
public class Application { }

Best Practice: Always use @SpringBootApplication on your main class. Only use individual annotations when you need explicit control over specific features.

7. How do you secure a Spring Boot application?

Answer:

Spring Security provides comprehensive protection through multiple layers:

Step 1: Add Dependency

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>

Just adding this dependency secures ALL endpoints with basic auth (random password in logs).

Step 2: Configure Security (Spring Security 6.x)

@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable()) // Disable for REST APIs
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .sessionManagement(session -> 
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .httpBasic(Customizer.withDefaults())
            .build();
    }
    
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

Authentication Options

Method Use Case
Form Login Traditional web apps with sessions
HTTP Basic Simple APIs, testing
JWT Tokens Stateless REST APIs, microservices
OAuth2/OIDC Social login, enterprise SSO

Method-Level Security

@EnableMethodSecurity
@Service
public class UserService {
    
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteUser(Long id) { ... }
    
    @PreAuthorize("#userId == authentication.principal.id")
    public User getUser(Long userId) { ... }
}

Security Checklist

  • Use BCrypt for password hashing
  • Enable HTTPS in production
  • Secure actuator endpoints
  • Store secrets in environment variables or vault
  • Implement rate limiting
  • Validate all inputs

8. What is Spring Boot Actuator, and why is it useful?

Answer:

Spring Boot Actuator provides production-ready monitoring and management features through HTTP endpoints or JMX.

Add Dependency

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>

Key Endpoints

Endpoint URL Purpose
/health /actuator/health Application health status (UP/DOWN)
/metrics /actuator/metrics JVM, HTTP, custom metrics
/info /actuator/info Build info, Git commit, custom info
/env /actuator/env Environment properties
/loggers /actuator/loggers View/change log levels at runtime
/threaddump /actuator/threaddump Thread dump for debugging

Configuration

# application.yml
management:
  endpoints:
    web:
      exposure:
        include: health, metrics, info, prometheus
  endpoint:
    health:
      show-details: when_authorized  # never, always, when_authorized
  metrics:
    export:
      prometheus:
        enabled: true  # For Prometheus/Grafana integration

Custom Health Indicator

@Component
public class DatabaseHealthIndicator implements HealthIndicator {
    
    @Override
    public Health health() {
        if (isDatabaseReachable()) {
            return Health.up()
                .withDetail("database", "PostgreSQL")
                .withDetail("status", "Connected")
                .build();
        }
        return Health.down()
            .withDetail("error", "Cannot connect to database")
            .build();
    }
}

Use Cases

  • Kubernetes: Liveness/readiness probes via /health/liveness and /health/readiness
  • Monitoring: Export metrics to Prometheus, Datadog, New Relic
  • Debugging: View beans, environment, thread dumps
  • Operations: Change log levels without restart

Security Note: Always secure sensitive endpoints in production — expose only health and info publicly.

9. How do you handle exceptions in Spring Boot REST APIs?

Answer:

Spring Boot provides multiple ways to handle exceptions, with @RestControllerAdvice being the recommended approach for REST APIs.

1. Global Exception Handler (Recommended)

@RestControllerAdvice
public class GlobalExceptionHandler {
    
    // Handle specific exception
    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(ResourceNotFoundException ex) {
        return new ErrorResponse(
            HttpStatus.NOT_FOUND.value(),
            "NOT_FOUND",
            ex.getMessage(),
            LocalDateTime.now()
        );
    }
    
    // Handle validation errors
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(e -> e.getField() + ": " + e.getDefaultMessage())
            .toList();
        return new ErrorResponse(400, "VALIDATION_ERROR", errors.toString(), LocalDateTime.now());
    }
    
    // Catch-all for unexpected errors
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleAll(Exception ex) {
        log.error("Unexpected error", ex);
        return new ErrorResponse(500, "INTERNAL_ERROR", "An unexpected error occurred", LocalDateTime.now());
    }
}

2. Error Response DTO (RFC 7807 Problem Details)

public record ErrorResponse(
    int status,
    String code,
    String message,
    LocalDateTime timestamp,
    String path,           // Optional: request path
    List<String> details   // Optional: validation errors
) {
    public ErrorResponse(int status, String code, String message, LocalDateTime timestamp) {
        this(status, code, message, timestamp, null, null);
    }
}

3. Custom Business Exceptions

@ResponseStatus(HttpStatus.NOT_FOUND)
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String resource, Long id) {
        super(resource + " not found with id: " + id);
    }
}

// Usage in service
public User findById(Long id) {
    return userRepository.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException("User", id));
}

Exception Handling Approaches Compared

Approach Scope Best For
@RestControllerAdvice Global Centralized handling (recommended)
@ExceptionHandler in controller Single controller Controller-specific errors
@ResponseStatus on exception Per exception class Simple status mapping
ResponseEntityExceptionHandler Extend for Spring MVC exceptions Customize built-in exception handling

Best Practices

  • Use consistent error response format across all APIs
  • Log full stack traces server-side, return safe messages to clients
  • Use appropriate HTTP status codes (4xx for client errors, 5xx for server errors)
  • Include correlation IDs for request tracing
  • Never expose internal details (stack traces, SQL) in production
