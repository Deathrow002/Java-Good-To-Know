## 1. What is Spring Boot, and how does it differ from Spring Framework?

**Answer:**

**Spring Framework** is a massive ecosystem that provides the core infrastructure for Java applications (IoC, AOP, Data Access, Web MVC, etc.). However, configuring a raw Spring application requires a lot of manual setup (XML or Java Config, DispatcherServlet registration, library version management).

**Spring Boot** is an extension of the Spring Framework that follows the **"Convention over Configuration"** philosophy. It is designed to get an application up and running as quickly as possible with minimal manual configuration.

---

### a. Key Pillars of Spring Boot

1.  **Auto-Configuration:** This is the "magic" of Spring Boot. It automatically configures Spring beans based on the dependencies found on your classpath. For example, if `h2.jar` is present, it automatically sets up an in-memory database.
2.  **Starter Dependencies:** These are curated sets of dependency descriptors. Instead of hunting for compatible versions of 10 different libraries, you just add `spring-boot-starter-web`, and Spring Boot pulls in everything needed for a REST API.
3.  **Embedded Servers:** Spring Boot applications come with an embedded server (Tomcat by default). You don't need to install a standalone web server or package your app as a `.war` file; you simply run it as a standard `.jar`.
4.  **Production-Ready Features (Actuator):** Out of the box, you get endpoints for health checks, metrics, and environment info, which are essential for modern cloud-native monitoring.

---

### b. Technical Comparison

| Feature | Spring Framework | Spring Boot |
|---|---|---|
| **Objective** | Provides core features (DI, AOP). | Simplifies development and deployment. |
| **Configuration** | **Manual**: Requires explicit bean definitions. | **Automatic**: Based on classpath and properties. |
| **Server** | Needs an external server (Tomcat/JBoss). | Includes **Embedded Server** (Run as JAR). |
| **Deployment** | Usually packaged as a **WAR** file. | Usually packaged as an executable **JAR**. |
| **Decision Making** | You choose every library and version. | **Opinionated**: Provides sensible defaults. |

---

### c. The "Bottom Line"
Think of **Spring Framework** as a box of high-quality engine parts (pistons, gears, valves). You have total control, but you have to build the engine yourself. 

**Spring Boot** is a pre-assembled, high-performance car built using those parts. You can start driving immediately, but you still have the freedom to open the hood and swap out parts (override defaults) whenever you need to.

## 2. What are Spring Boot starters?

**Answer:**

**Spring Boot Starters** are a set of convenient dependency descriptors that you can include in your application. They act as "curated bundles" of all the relevant transitive dependencies you need to get a specific feature working, ensuring that all library versions are compatible with each other.

---

### a. Why use Starters?
Before starters, developers had to manually add dozens of individual dependencies (e.g., for a Web app: Spring MVC, Jackson, Tomcat, Logback, etc.) and ensure their versions didn't conflict. 

**Key Benefits:**
1.  **Dependency Consolidation:** One starter replaces many individual library declarations in your `pom.xml` or `build.gradle`.
2.  **Version Management:** You don't need to specify versions for starters if you use the Spring Boot Parent POM; the framework ensures all libraries in the bundle work perfectly together.
3.  **Auto-Configuration Trigger:** The presence of a starter on the classpath is the "signal" that triggers Spring Boot's **Auto-Configuration** (e.g., adding `spring-boot-starter-data-jpa` tells Spring to look for a DataSource and set up Hibernate).

---

### b. Naming Conventions
- **Official Starters:** Always follow the pattern `spring-boot-starter-{name}` (e.g., `spring-boot-starter-web`).
- **Third-Party Starters:** Should follow the pattern `{name}-spring-boot-starter` (e.g., `mybatis-spring-boot-starter`). This helps distinguish official Spring modules from community-driven ones.

---

### c. Common Starters and Their "Payload"

| Starter | Primary Purpose | What's Inside? |
|---|---|---|
| **`spring-boot-starter-web`** | REST APIs / Web Apps | Spring MVC, Jackson (JSON), Embedded Tomcat, Validation. |
| **`spring-boot-starter-data-jpa`** | Database / ORM | Hibernate, Spring Data JPA, HikariCP (Connection Pool). |
| **`spring-boot-starter-security`** | Authentication / AuthZ | Spring Security Core, Web Security, BCrypt. |
| **`spring-boot-starter-test`** | Testing | JUnit 5, Mockito, AssertJ, Hamcrest, JsonPath. |
| **`spring-boot-starter-actuator`** | Monitoring | Health checks, metrics, info endpoints. |

---

### d. Implementation Example (Maven)
Instead of 10+ lines of XML, you just need this:
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
```

**Bottom Line:** Starters turn complex dependency management into a "feature-based" selection process. You don't ask for "Jackson 2.15"; you ask for "Web support," and Spring Boot handles the rest.

## 3. What is Spring Boot auto-configuration?

**Answer:**

**Auto-configuration** is arguably the most powerful feature of Spring Boot. It uses a series of "intelligent guesses" to automatically configure Spring beans in the ApplicationContext based on the JAR dependencies you've added to the classpath.

---

### a. Internal Mechanism: How it Works
When you annotate your main class with `@SpringBootApplication`, it triggers `@EnableAutoConfiguration`.

1.  **Scanning**: Spring Boot uses the `AutoConfigurationImportSelector` to look for a specific file in your projectâ€™s libraries: `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (in Spring Boot 3.x) or `spring.factories` (in legacy versions).
2.  **Candidates**: This file lists all possible configuration classes (e.g., `DataSourceAutoConfiguration`, `RabbitAutoConfiguration`).
3.  **Filtration**: Spring Boot doesn't load everything. It runs each candidate through a series of **Conditional Checks** to decide if that configuration is relevant to your current project.

---

### b. The "Condition" Family
Auto-configuration relies on `@Conditional` annotations to make its decisions:

- **`@ConditionalOnClass`**: Only configures the bean if a specific class is found on the classpath (e.g., "Only set up Mongo if `MongoClient.class` exists").
- **`@ConditionalOnMissingBean`**: This is a crucial "safety valve." It only configures the default Spring Boot bean if you haven't already defined your own bean of that type.
- **`@ConditionalOnProperty`**: Only enables the configuration if a specific property is set in your `application.yml` or `application.properties`.

---

### c. Example: DataSource Auto-Configuration
If you add the `h2` database to your classpath and set these properties:

```properties
spring.datasource.url=jdbc:h2:mem:testdb
spring.datasource.driver-class-name=org.h2.Driver
```

**Spring Boot will automatically:**
1.  Detect the H2 driver.
2.  Instantiate a **HikariCP** connection pool (`@ConditionalOnClass`).
3.  Expose it as a `DataSource` bean (`@ConditionalOnMissingBean`).
4.  Configure a `JdbcTemplate` for you to use immediately.

---

### d. Customization & Debugging
- **Overriding**: To override an auto-configuration, simply define your own `@Bean`. Since auto-configuration runs *after* user configuration, your bean will be detected first, and Spring's bean will be skipped via `@ConditionalOnMissingBean`.
- **Exclusion**: If a specific configuration is causing issues, you can disable it:
  `@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})`
- **Debugging**: If you're unsure why a bean was or wasn't created, run your app with the `--debug` flag. Spring Boot will print a **"Conditions Evaluation Report"** showing exactly which auto-configurations were matched and why.

**Bottom Line:** Auto-configuration provides the "sane defaults" that make Spring Boot so productive, while ensuring the developer always has the final say through bean overriding.

## 4. How do you create a REST API using Spring Boot?

**Answer (summary):**

1. Add `spring-boot-starter-web` dependency.
2. Create a `@RestController` with `@RequestMapping` and handler methods (`@GetMapping`, `@PostMapping`, ...).
3. Run application with `@SpringBootApplication` and `SpringApplication.run(...)`.

Example controller:

```java
@RestController
@RequestMapping("/api/users")
public class UserController {
    private final UserService userService;

    public UserController(UserService userService) { this.userService = userService; }

    @GetMapping
    public List<User> getAllUsers() { return userService.findAll(); }

    @GetMapping("/{id}")
    public ResponseEntity<User> getUser(@PathVariable Long id) { /* ... */ }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public User createUser(@Valid @RequestBody UserDTO dto) { return userService.create(dto); }

    @PutMapping("/{id}")
    public User updateUser(@PathVariable Long id, @Valid @RequestBody UserDTO dto) { return userService.update(id, dto); }

    @DeleteMapping("/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void deleteUser(@PathVariable Long id) { userService.delete(id); }
}
```

Result: API available at `http://localhost:8080/api/users` with automatic JSON serialization.

## 5. What is the role of `application.properties` or `application.yml`?

**Answer:**

In Spring Boot, configuration files are used to **externalize configuration**â€”allowing the same application code to behave differently in different environments (dev, test, prod) without needing a recompile.

---

### a. Properties vs. YAML

Spring Boot supports both flat `.properties` and hierarchical `.yml` files.

1.  **`application.properties`**: Uses a key-value format. It's simple and legacy-friendly but can become repetitive for deeply nested settings.
2.  **`application.yml`**: Uses an indentation-based hierarchy. It's much more readable for complex configurations and reduces key repetition.

---

### b. Spring Profiles
Profiles allow you to group your configuration and only activate it when needed.
- **Naming Convention**: `application-{profile}.yml` (e.g., `application-dev.yml`, `application-prod.yml`).
- **Activation**: Set the `spring.profiles.active` property in your main file or as an environment variable.

---

### c. Binding Properties to Java Code

Spring Boot provides two primary ways to access these settings:

#### 1. `@Value` (Direct Injection)
Good for simple, single-value injections. It supports default values if the property is missing.
```java
@Value("${app.feature.enabled:false}")
private boolean featureEnabled;
```

#### 2. `@ConfigurationProperties` (Type-Safe Binding)
**The Recommended Approach.** It binds entire sections of the configuration file to a Java object (POJO or Record). It supports nested objects, lists, and validation via JSR-303.

```java
@ConfigurationProperties(prefix = "app.mail")
public record MailProperties(
    String host,
    int port,
    String from,
    List<String> recipients
) {}
```

---

### d. External Configuration Hierarchy
Spring Boot is designed to be "cloud-native," meaning it looks for configuration in multiple places, with a specific priority. **Environment Variables** and **Command Line Arguments** will override values defined in `application.yml`.

**Hierarchy (Highest to Lowest):**
1.  Command-line arguments (`--server.port=9000`)
2.  JVM System properties
3.  OS Environment Variables (`SPRING_APPLICATION_JSON`, etc.)
4.  Profile-specific files (`application-prod.yml`)
5.  Default application files (`application.yml`)

**Bottom Line:** Configuration files are the "knobs and dials" of your application. Use **`@ConfigurationProperties`** for type safety and **Profiles** to manage environment-specific behavior.

## 6. What is the difference between `@SpringBootApplication` and `@EnableAutoConfiguration`?

**Answer:**

`@SpringBootApplication` is a **meta-annotation** (a convenience annotation) that serves as the entry point of most Spring Boot applications. It is actually a composition of three primary annotations.

---

### a. The Three Pillars of `@SpringBootApplication`

1.  **`@EnableAutoConfiguration`**: This is the "magic" of Spring Boot. It tells the framework to automatically configure Spring beans based on the dependencies found in your `pom.xml` or `build.gradle` (the classpath). For example, if `spring-boot-starter-web` is present, it automatically sets up an embedded Tomcat and `DispatcherServlet`.
2.  **`@ComponentScan`**: This tells Spring to scan the current package and all sub-packages for annotated components (`@Component`, `@Service`, `@Repository`, `@RestController`, etc.) and register them as beans in the Application Context.
3.  **`@SpringBootConfiguration`**: A specialized version of the standard `@Configuration` annotation. It identifies the class as a source of bean definitions and is used to help Spring find your configuration during unit tests (test slices).

---

### b. Comparison & Usage

-   **`@EnableAutoConfiguration`** is just **one part** of the equation. It only handles the "automatic" part of the configuration.
-   **`@SpringBootApplication`** is the **entire package**. It handles automatic configuration, component scanning, and identifying the class as a configuration source.

---

### c. Advanced Features: Excluding Auto-Configuration
Sometimes you want Spring Boot's magic, but not *all* of it. You can disable specific auto-configurations directly through the `@SpringBootApplication` annotation.

```java
@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})
public class MySimpleApplication {
    public static void main(String[] args) {
        SpringApplication.run(MySimpleApplication.class, args);
    }
}
```

---

### d. Why the Location Matters
By default, `@SpringBootApplication` (via `@ComponentScan`) only looks for beans in its **current package and sub-packages**.
- **Best Practice**: Always place your main application class in a **root package** (e.g., `com.example.myapp`) to ensure all your components are discovered without manual configuration.

**Bottom Line:** Use `@SpringBootApplication` for essentially all your main entry points. Use `@EnableAutoConfiguration` individually only if you are building a modular library or need extremely fine-grained, manual control over component scanning and configuration.

## 7. How do you secure a Spring Boot application?

**Answer:**

Spring Boot uses **Spring Security** as the industry-standard framework for Authentication (AuthN) and Authorization (AuthZ). Modern security in Spring Boot 3.x/Spring Security 6.x focuses on functional, bean-based configuration.

---

### a. Core Implementation Steps

1.  **Dependency**: Add `spring-boot-starter-security`.
2.  **Modern Configuration**: Define a `SecurityFilterChain` bean.
3.  **Password Management**: Use **`BCryptPasswordEncoder`** for secure one-way hashing of passwords.

---

### b. Modern Security Configuration (Spring Security 6+)
The modern way to configure security is to avoid extending legacy classes and instead use the **Lambda DSL**.

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Disable for stateless REST APIs
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll() // Public endpoints
                .requestMatchers("/api/admin/**").hasRole("ADMIN") // Role-based access
                .anyRequest().authenticated() // All others require authentication
            )
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS) // No sessions for JWT
            )
            .httpBasic(Customizer.withDefaults()); // Basic Auth for simplicity

        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

---

### c. Method-Level Security
For more granular control, you can secure individual service methods using annotations.

```java
@EnableMethodSecurity
@Service
public class OrderService {

    @PreAuthorize("hasRole('MANAGER') or #order.author == authentication.name")
    public void cancelOrder(Order order) {
        // Business logic...
    }
}
```

---

### d. Critical Security Checklist
1.  **Defense in Depth**: Use HTTPS (TLS) everywhere. Never transmit passwords or tokens over plain HTTP.
2.  **Stateless APIs**: For REST, prefer **JWT (JSON Web Tokens)** or **OAuth2** over Session ID cookies to avoid CSRF and improve scalability.
3.  **Secure Actuator**: Ensure management endpoints (`/actuator/**`) are restricted to admins or only accessible on an internal port.
4.  **No Plaintext Passwords**: Never store passwords in the database even with "reversable" encryption. Always use BCrypt or Argon2.
5.  **CORS**: Configure **CORS (Cross-Origin Resource Sharing)** strictly; never use `*` as an allowed origin in production.

**Bottom Line:** Security is a layered process. Spring Security handles the heavy lifting, but developers must ensure the application follows the principle of **Least Privilege**.

## 8. What is Spring Boot Actuator, and why is it useful?

**Answer:**

**Spring Boot Actuator** is a sub-project that provides "production-ready" features to your application. It allows you to monitor and interact with your application via HTTP endpoints or JMX, giving you deep insights into how the application is performing in a live environment.

---

### a. Why is it Useful?
In a modern microservices or cloud-native architecture (like Kubernetes), you cannot manually check logs or restart servers to see what's happening. Actuator provides the "eyes and ears" for your monitoring tools.

**Core Use Cases:**
1.  **Health Monitoring**: Probes to tell orchestrators like Kubernetes if the app is alive (`liveness`) and ready to serve traffic (`readiness`).
2.  **Metrics Gathering**: Exports data (CPU, memory, request counts) to external tools like **Prometheus**, **Grafana**, or **New Relic**.
3.  **Dynamic Log Management**: Change the log level (e.g., from `INFO` to `DEBUG`) of a specific package at runtime without restarting the application.
4.  **Troubleshooting**: Generate thread dumps or heap dumps directly from an endpoint to analyze performance bottlenecks or memory leaks.

---

### b. Essential Actuator Endpoints

| Endpoint | Purpose |
|---|---|
| `/actuator/health` | Shows application health status (UP/DOWN) and component details (DB, Disk, RabbitMQ). |
| `/actuator/metrics` | Shows OS and JVM metrics, plus custom application counters. |
| `/actuator/loggers` | Allows viewing and modifying the log levels of specific classes/packages. |
| `/actuator/info` | Displays arbitrary application information (e.g., git commit ID, build version). |
| `/actuator/beans` | Displays a complete list of all Spring beans in the `ApplicationContext`. |

---

### c. Configuration & Security
By default, most endpoints are disabled for security reasons. You must explicitly expose the ones you need in your `application.yml`.

```yaml
management:
  endpoints:
    web:
      exposure:
        include: "health,info,metrics,loggers" # Expose specific endpoints
  endpoint:
    health:
      show-details: always # Show DB/Disk status instead of just "UP"
```

> [!CAUTION]
> **Security Warning:** Endpoints like `/heapdump`, `/env`, and `/beans` can leak sensitive data (database credentials, IP addresses). Always protect these endpoints behind **Spring Security** or keep them strictly within an internal network (VPC).

---

### d. Modern Monitoring with Micrometer
Actuator uses **Micrometer** (the "SLF4J for metrics") as its underlying engine. This allows you to write custom metrics once and export them to dozens of different monitoring systems with zero code changes.

**Bottom Line:** Actuator turns your "black box" application into a transparent system that can be easily managed and monitored in professional production environments.

## 9. How do you handle exceptions in Spring Boot REST APIs?

**Answer:**

Effective exception handling is crucial for building a professional API. Spring Boot provides several ways to manage errors, but the **Global Exception Handler** pattern using `@RestControllerAdvice` is the industry standard for ensuring consistency across all controllers.

---

### a. The Three Levels of Exception Handling

1.  **`@ResponseStatus` (Entity level):** Easiest for custom business exceptions. You annotate the exception class itself.
2.  **`@ExceptionHandler` (Controller level):** Best for logic specific to a single controller. No impact on other controllers.
3.  **`@RestControllerAdvice` (Global level):** The "Intercepting" approach. It catches exceptions thrown by *any* controller in the application, allowing for a unified error response format.

---

### b. Modern Standard: Problem Detail (RFC 7807)
Introduced in **Spring Boot 3.x**, the framework now natively supports the **Problem Detail for HTTP APIs** (RFC 7807). This standardizes the JSON structure of errors so that API clients can easily parse them.

#### Global Handler Example:
```java
@RestControllerAdvice
public class GlobalExceptionHandler extends ResponseEntityExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ProblemDetail handleUserNotFound(UserNotFoundException ex) {
        ProblemDetail problemDetail = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
        problemDetail.setTitle("User Not Found");
        problemDetail.setProperty("timestamp", Instant.now());
        return problemDetail;
    }

    @ExceptionHandler(ValidationException.class)
    public ResponseEntity<ErrorResponse> handleValidation(ValidationException ex) {
        ErrorResponse error = new ErrorResponse(
                HttpStatus.BAD_REQUEST.value(),
                "VALIDATION_ERROR",
                ex.getMessage(),
                LocalDateTime.now()
        );
        return new ResponseEntity<>(error, HttpStatus.BAD_REQUEST);
    }

    // Catch-all for unexpected internal errors
    @ExceptionHandler(Exception.class)
    public ProblemDetail handleSecurityException(Exception ex) {
        return ProblemDetail.forStatusAndDetail(HttpStatus.INTERNAL_SERVER_ERROR, "An unexpected error occurred.");
    }
}
```

---

### c. The Unified Error Record
Using a Java **Record** for your error DTO is a clean and modern approach.

```java
public record ErrorResponse(
    int status,
    String errorCode,
    String message,
    LocalDateTime timestamp
) {}
```

---

### d. Security & Technical Best Practices
1.  **Never Expose Stack Traces:** Leaking internal code paths or library versions is a security risk. In production, always mask the internal details.
2.  **Specific Status Codes:** Use `400 Bad Request` for validation, `401/403` for security, `404` for missing resources, and `409 Conflict` for business rule violations.
3.  **Logging:** Log the full exception stack trace and a **Correlation ID** on the server side so developers can trace the error, while returning only a safe message to the client.
4.  **Consistency:** Ensure every errorâ€”no matter where it occursâ€”returns the same JSON structure. This makes life easier for your frontend/mobile developers.

**Bottom Line:** A well-handled exception shouldn't be a crash; it should be a meaningful conversation with the client.
