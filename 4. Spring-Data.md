# Spring Data, JPA, Hibernate, Redis & Transactions — Questions & Answers

---

## 1. What is JPA (Java Persistence API)?

**Answer:**

**JPA (Java Persistence API)** is a Java specification (part of Jakarta EE) that defines a standard way to **map Java objects to relational database tables** — a technique known as **ORM (Object-Relational Mapping)**. It is not a library itself; it defines the rules that ORM providers like Hibernate must implement.

---

### a. Core Concepts

| Concept | Description |
|---|---|
| **Entity** | A Java class annotated with `@Entity` that maps to a database table |
| **EntityManager** | The main API for CRUD operations and JPQL queries |
| **Persistence Context** | A first-level cache of managed entity instances within a transaction |
| **JPQL** | Java Persistence Query Language — SQL-like but operates on entity objects, not tables |
| **Persistence Unit** | Configuration unit grouping entities and provider settings |

---

### b. JPA Annotations — Quick Reference

```java
@Entity                              // Marks class as a persistent entity
@Table(name = "users")              // Maps to 'users' table
public class User {

    @Id                              // Primary key
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "first_name", nullable = false, length = 100)
    private String firstName;

    @Column(unique = true)
    private String email;

    @Enumerated(EnumType.STRING)     // Store enum as String, not ordinal
    private UserStatus status;

    @CreatedDate
    private LocalDateTime createdAt;

    @Transient                       // Not persisted to the database
    private String temporaryToken;
}
```

---

### c. EntityManager API

```java
@Repository
public class UserDao {
    @PersistenceContext
    private EntityManager em;

    public void save(User user)   { em.persist(user); }               // INSERT
    public User find(Long id)     { return em.find(User.class, id); }  // SELECT by PK
    public void update(User user) { em.merge(user); }                  // UPDATE
    public void delete(Long id)   { em.remove(em.find(User.class, id)); } // DELETE

    public List<User> findByStatus(String status) {
        return em.createQuery(
            "SELECT u FROM User u WHERE u.status = :status", User.class)
            .setParameter("status", status)
            .getResultList();
    }
}
```

---

### d. JPA vs. JDBC

| Aspect | JDBC | JPA |
|---|---|---|
| Mapping | Manual `ResultSet` → Object | Automatic via annotations |
| Queries | Raw SQL | JPQL or Criteria API |
| Transactions | Manual `commit`/`rollback` | Declarative `@Transactional` |
| Caching | None | First-level + optional second-level |
| Portability | Database-specific SQL | Database-agnostic JPQL |

## 2. What is Hibernate, and how does it relate to JPA?

**Answer:**

**Hibernate** is the most popular **implementation** of the JPA specification. JPA defines the standard API; Hibernate provides the actual engine that performs SQL generation, caching, lazy loading, and dirty checking.

---

### a. Layered Architecture

```
┌────────────────────────────────────────────────────────┐
│  Spring Data JPA (Convenience Layer)                   │  ← Repositories, derived queries
├────────────────────────────────────────────────────────┤
│  JPA API  (jakarta.persistence.*)                      │  ← @Entity, EntityManager spec
├────────────────────────────────────────────────────────┤
│  Hibernate (JPA Provider / Engine)                     │  ← Implements JPA + extras
├────────────────────────────────────────────────────────┤
│  JDBC / Database Driver                                │  ← Raw SQL to the database
└────────────────────────────────────────────────────────┘
```

---

### b. Hibernate-Specific Features (Beyond Standard JPA)

| Feature | Description |
|---|---|
| **HQL** | Hibernate Query Language — superset of JPQL |
| **Second-Level Cache** | Shared cache across sessions (EhCache, Caffeine, Redis) |
| **Dirty Checking** | Auto-detects changed fields and generates `UPDATE` at flush |
| **Batch Processing** | `hibernate.jdbc.batch_size` — bulk INSERT/UPDATE |
| **Envers** | Automatic entity auditing/versioning (full change history) |
| **Multi-tenancy** | Schema or database-level multi-tenancy |

---

### c. Key Hibernate Configuration

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate        # validate | update | create | create-drop | none
    show-sql: true
    properties:
      hibernate:
        format_sql: true
        dialect: org.hibernate.dialect.PostgreSQLDialect
        jdbc.batch_size: 25
        cache.use_second_level_cache: true
```

---

### d. `ddl-auto` Values

| Value | Behaviour | When to Use |
|---|---|---|
| `none` | No schema changes | Production (use Flyway/Liquibase) |
| `validate` | Fails if schema doesn't match entities | Production |
| `update` | Alters schema to match entities | Development only |
| `create` | Drops & recreates schema on startup | Testing |
| `create-drop` | Creates on startup, drops on shutdown | Integration tests |

---

### e. Dirty Checking in Practice

Hibernate snapshots every managed entity on load. At flush time it compares current vs. snapshot — any differences generate `UPDATE` automatically.

```java
@Transactional
public void updateEmail(Long id, String newEmail) {
    User user = userRepository.findById(id).orElseThrow(); // MANAGED
    user.setEmail(newEmail); // NO explicit save() needed — dirty check handles it
}
```

## 3. What are JPA Entity Lifecycle States?

**Answer:**

Every JPA entity passes through four distinct **lifecycle states** managed by the **Persistence Context**. Knowing these states determines when SQL is generated.

---

### a. The Four States

| State | Description | SQL Generated? |
|---|---|---|
| **Transient** | Created with `new`, unknown to JPA | No |
| **Managed** | Tracked by Persistence Context; changes auto-synced | INSERT on persist; UPDATE on dirty check |
| **Detached** | Was managed; context closed or `em.detach()` called | No (until re-merged) |
| **Removed** | Marked for deletion via `em.remove()` | DELETE on flush/commit |

---

### b. State Transitions

```java
User user = new User("Alice");   // TRANSIENT

em.persist(user);                // TRANSIENT  → MANAGED  (INSERT queued)
em.detach(user);                 // MANAGED    → DETACHED
em.merge(user);                  // DETACHED   → MANAGED  (UPDATE queued)
em.remove(user);                 // MANAGED    → REMOVED  (DELETE queued)
// tx.commit() / em.flush()      → SQL statements executed
```

---

### c. Lifecycle Callback Annotations

```java
@Entity
public class Order {

    @PrePersist
    void onPrePersist() { this.createdAt = LocalDateTime.now(); }

    @PreUpdate
    void onPreUpdate()  { this.updatedAt = LocalDateTime.now(); }

    @PostLoad
    void onPostLoad()   { /* called after entity loaded from DB */ }

    @PreRemove
    void onPreRemove()  { /* called before DELETE — useful for audit */ }
}
```

## 4. What are JPA Relationship Mappings?

**Answer:**

JPA provides four relationship annotations to map associations between entities, mirroring database foreign key relationships.

---

### a. @OneToOne

```java
@Entity
public class User {
    @OneToOne(cascade = CascadeType.ALL, fetch = FetchType.LAZY)
    @JoinColumn(name = "profile_id")
    private UserProfile profile;
}
```

---

### b. @OneToMany / @ManyToOne (most common)

```java
@Entity
public class Order {
    // Inverse side — mappedBy avoids a join table
    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderItem> items = new ArrayList<>();
}

@Entity
public class OrderItem {
    // Owning side — FK is written from here
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    private Order order;
}
```

---

### c. @ManyToMany

```java
@Entity
public class Student {
    @ManyToMany(cascade = {CascadeType.PERSIST, CascadeType.MERGE})
    @JoinTable(
        name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private Set<Course> courses = new HashSet<>();
}

@Entity
public class Course {
    @ManyToMany(mappedBy = "courses")
    private Set<Student> students = new HashSet<>();
}
```

---

### d. CascadeType & orphanRemoval

| CascadeType | Effect |
|---|---|
| `PERSIST` | Save children when parent is saved |
| `MERGE` | Merge children when parent is merged |
| `REMOVE` | Delete children when parent is deleted |
| `ALL` | All of the above |

**`orphanRemoval = true`** — Deletes a child entity when removed from the parent's collection, even without `CascadeType.REMOVE`.

---

### e. Owning Side vs. Inverse Side Rule

JPA only writes the FK from the **owning side** (the side with `@JoinColumn`). Always set both sides in code:

```java
// CORRECT: set both sides for in-memory consistency + FK persistence
item.setOrder(order);          // owning side → FK written to DB
order.getItems().add(item);    // inverse side → in-memory list updated
```

## 5. What is JPQL, and how does it compare to the Criteria API and Native Queries?

**Answer:**

Spring Data JPA supports three query strategies, each suited to different scenarios.

---

### a. JPQL (Java Persistence Query Language)

Operates on **entity classes and fields**, not tables and columns. Database-agnostic — Hibernate translates to the target SQL dialect.

```java
@Query("SELECT u FROM User u WHERE u.status = :status AND u.age > :minAge")
List<User> findActiveAdults(@Param("status") String status, @Param("minAge") int minAge);

// JOIN
@Query("SELECT o FROM Order o JOIN o.items i WHERE i.product.sku = :sku")
List<Order> findOrdersByProductSku(@Param("sku") String sku);

// Modifying (UPDATE/DELETE)
@Modifying
@Query("UPDATE User u SET u.status = 'INACTIVE' WHERE u.lastLogin < :cutoff")
int deactivateInactiveUsers(@Param("cutoff") LocalDate cutoff);
```

---

### b. Criteria API

Type-safe, programmatic query builder — ideal for **dynamic queries** whose conditions are unknown at compile time.

```java
public List<User> search(String name, String status) {
    CriteriaBuilder cb = em.getCriteriaBuilder();
    CriteriaQuery<User> cq = cb.createQuery(User.class);
    Root<User> root = cq.from(User.class);

    List<Predicate> predicates = new ArrayList<>();
    if (name != null)   predicates.add(cb.like(cb.lower(root.get("name")), "%" + name.toLowerCase() + "%"));
    if (status != null) predicates.add(cb.equal(root.get("status"), status));

    cq.where(predicates.toArray(new Predicate[0]));
    return em.createQuery(cq).getResultList();
}
```

---

### c. Native Queries

Execute **raw SQL** — used for database-specific features (window functions, CTEs, `RETURNING`).

```java
@Query(value = "SELECT * FROM users WHERE created_at > NOW() - INTERVAL '30 days'", nativeQuery = true)
List<User> findRecentUsers();
```

---

### d. Comparison Table

| Aspect | JPQL | Criteria API | Native SQL |
|---|---|---|---|
| **Syntax** | String-based, SQL-like | Java fluent API | Raw SQL |
| **Type Safety** | Partial | Full (with Metamodel) | None |
| **Dynamic Queries** | Limited | Excellent | Possible |
| **DB Portability** | Yes | Yes | No |
| **Complex Aggregations** | Limited | Limited | Full |
| **Readability** | High | Low | Medium |

---


## 6. What is Spring Data JPA, and how does it simplify data access?

**Answer:**

**Spring Data JPA** provides a high-level abstraction on top of JPA/Hibernate. You define a **repository interface** and Spring provides the implementation at runtime — no boilerplate `EntityManager` code required.

---

### a. Core Benefits

1. **Repository Abstraction** — Define an interface; Spring generates the implementation.
2. **Derived Query Methods** — Method names like `findByLastNameAndStatus` auto-generate SQL.
3. **Pagination & Sorting** — Via `Pageable` and `Sort` objects.
4. **Audit Support** — `@CreatedDate`, `@LastModifiedBy` populated automatically.
5. **Custom Queries** — `@Query` for JPQL or native SQL when derived methods fall short.

---

### b. Entity + Repository Example

```java
@Entity
@Table(name = "products")
public class Product {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false) private String name;
    private BigDecimal price;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id")
    private Category category;
}

@Repository
public interface ProductRepository extends JpaRepository<Product, Long> {

    // Derived query — auto-generated SQL
    List<Product> findByNameContainingIgnoreCase(String name);

    // Paginated
    Page<Product> findByCategoryName(String categoryName, Pageable pageable);

    // Custom JPQL
    @Query("SELECT p FROM Product p WHERE p.category.active = true")
    List<Product> findFromActiveCategories();

    // Native SQL
    @Query(value = "SELECT * FROM products WHERE stock < :limit", nativeQuery = true)
    List<Product> findLowStock(@Param("limit") int limit);
}
```

---

### c. JPA vs. Hibernate vs. Spring Data JPA

| Layer | Role |
|---|---|
| **JPA** | The specification ("the rules") |
| **Hibernate** | The implementation ("the engine") |
| **Spring Data JPA** | The convenience layer ("the steering wheel") |

---


## 7. What are the Repository Interfaces in Spring Data?

**Answer:**

Spring Data provides a hierarchy of repository interfaces — choose the level of abstraction appropriate to your needs.

---

### a. Repository Hierarchy

```
Repository<T, ID>                        ← Marker interface (no methods)
  └── CrudRepository<T, ID>             ← save, findById, findAll, delete, count
        └── PagingAndSortingRepository  ← findAll(Pageable), findAll(Sort)
              └── JpaRepository<T, ID>  ← JPA-specific: flush, saveAndFlush, batch ops
```

---

### b. Interface Descriptions

| Interface | Key Methods | Use When |
|---|---|---|
| `CrudRepository` | `save`, `findById`, `existsById`, `deleteById` | Basic CRUD |
| `PagingAndSortingRepository` | `findAll(Pageable)`, `findAll(Sort)` | Paginated lists |
| `JpaRepository` | Inherits all + `flush()`, `saveAndFlush()`, `deleteAllInBatch()` | JPA projects (most common) |
| `JpaSpecificationExecutor` | `findAll(Specification)` | Complex dynamic criteria queries |
| `ReactiveCrudRepository` | Returns `Mono<T>` / `Flux<T>` | WebFlux / reactive stack |

---

### c. Example with JpaSpecificationExecutor

```java
public interface UserRepository extends JpaRepository<User, Long>,
                                        JpaSpecificationExecutor<User> {
    List<User> findByStatus(String status);
}

// Dynamic specification
Specification<User> spec = (root, query, cb) ->
    cb.and(
        cb.equal(root.get("status"), "ACTIVE"),
        cb.greaterThan(root.get("age"), 18)
    );

List<User> users = userRepository.findAll(spec);
```

## 8. What are Projections and DTOs in Spring Data JPA?

**Answer:**

Projections allow you to retrieve **only specific columns** rather than full entities, reducing memory usage and improving query performance.

---

### a. Interface-Based Projections

**Closed projection** — fetches only specified fields; Spring generates a SELECT for exactly those columns:

```java
public interface UserSummary {
    String getFirstName();
    String getLastName();
}

List<UserSummary> findByStatus(String status);
```

**Open projection** — uses SpEL; less optimized (full entity loaded):

```java
public interface UserFullName {
    @Value("#{target.firstName + ' ' + target.lastName}")
    String getFullName();
}
```

---

### b. Class-Based Projections (DTO / Record)

```java
public record UserDto(String email, String phoneNumber) {}

List<UserDto> findByEmail(String email);
```

**Benefits of Java Records**: Immutable, no boilerplate (getters, equals, hashCode), no proxy overhead.

---

### c. Dynamic Projections

```java
<T> List<T> findByStatus(String status, Class<T> type);

// Usage
List<UserSummary> summaries = repo.findByStatus("ACTIVE", UserSummary.class);
List<UserDto>     dtos      = repo.findByStatus("ACTIVE", UserDto.class);
```

---

### d. When to Use Projections

- **Read-only operations** — search results, table views where you won't modify the data.
- **Performance** — entities with large `BLOB`/`TEXT` columns when you only need a name or ID.
- **API Design** — prevent accidentally exposing sensitive fields like `passwordHash`.

---


## 9. What is Lazy vs. Eager Loading in JPA?

**Answer:**

---

### a. Definitions

| Strategy | Behaviour | Default for |
|---|---|---|
| **Lazy** | Association loaded only when first accessed in code | `@OneToMany`, `@ManyToMany` |
| **Eager** | Association loaded immediately with the parent (JOIN) | `@ManyToOne`, `@OneToOne` |

---

### b. When to Use Each

- **Lazy** — Keeps initial queries lightweight. Prefer for large collections or rarely-used associations.
- **Eager** — Use when you *always* need the related data and want to avoid extra queries. Keep it rare — it can cause heavy joins and load unnecessary data.

---

### c. Common Pitfalls

**`LazyInitializationException`** — Accessing a lazy association outside a transaction (after the session is closed):

```java
// WRONG — session closed after findById returns; getting items throws exception
User user = userRepository.findById(id).orElseThrow();
user.getOrders().size(); // LazyInitializationException if no active transaction
```

Fix: Access lazy fields **inside a `@Transactional` method**, or use `JOIN FETCH` / `@EntityGraph`.

---

### d. JOIN FETCH Solution

```java
@Query("SELECT u FROM User u JOIN FETCH u.orders WHERE u.id = :id")
Optional<User> findByIdWithOrders(@Param("id") Long id);
```

---

### e. @EntityGraph Solution (pagination-safe)

```java
@EntityGraph(attributePaths = {"orders", "orders.items"})
Page<User> findByStatus(String status, Pageable pageable);
```

## 10. What is the `@Transactional` annotation, and how does it work?

**Answer:**

`@Transactional` is the core of Spring's **Declarative Transaction Management**. It wraps a method in a database transaction using an **AOP proxy** — ensuring all operations either fully commit or fully roll back.

---

### a. How it Works Under the Hood

```
Method call
    ↓
AOP Proxy intercepts
    ↓
TransactionManager.begin()
    ↓
Your method executes
    ↓
Success → commit()   |   RuntimeException → rollback()
```

---

### b. Key Attributes

#### Propagation — how transactions relate when one method calls another

| Value | Behaviour |
|---|---|
| `REQUIRED` (default) | Join existing transaction; create new if none |
| `REQUIRES_NEW` | Always create a new transaction; suspend existing |
| `NESTED` | Create a savepoint; can rollback independently |
| `MANDATORY` | Require existing transaction; throw if none |
| `NEVER` | Throw if a transaction is active |
| `SUPPORTS` | Use transaction if exists; otherwise run non-transactionally |

#### Isolation — degree of isolation from concurrent transactions

| Value | Protection | Performance |
|---|---|---|
| `DEFAULT` | Database default (usually `READ_COMMITTED`) | High |
| `READ_COMMITTED` | Prevents dirty reads | High |
| `REPEATABLE_READ` | Prevents non-repeatable reads | Medium |
| `SERIALIZABLE` | Prevents phantom reads | Low |

#### Rollback

```java
// Default: rolls back on RuntimeException and Error only
@Transactional(rollbackFor = Exception.class)       // also roll back checked exceptions
@Transactional(noRollbackFor = BusinessException.class) // don't roll back this one
```

---

### c. The Self-Invocation Pitfall

```java
@Service
public class OrderService {
    public void placeOrder(Order order) {
        processPayment(order); // ❌ Bypasses the proxy — @Transactional is IGNORED
    }

    @Transactional
    public void processPayment(Order order) { ... }
}
```

**Fix**: Inject `OrderService` into itself (via `@Autowired`), or refactor into a separate bean.

---

### d. Read-Only Optimization

```java
@Transactional(readOnly = true)
public List<Product> getAllProducts() {
    return productRepository.findAll();
    // Hibernate: skips dirty checking, disables flush
    // DB: may route to replica node
}
```

---

### e. Full Example

```java
@Service
public class OrderService {

    @Transactional(rollbackFor = Exception.class)
    public void placeOrder(Order order) {
        inventoryRepository.decreaseStock(order.getItems());
        orderRepository.save(order);
        paymentService.charge(order); // If this throws → everything rolls back
    }
}
```

## 11. Explain Transaction Propagation with a Real Example

**Answer:**

Transaction propagation defines **what happens to the current transaction** when a `@Transactional` method calls another `@Transactional` method.

---

### a. REQUIRED vs. REQUIRES_NEW

```java
@Service
public class OrderService {

    private final InventoryService inventoryService;
    private final AuditService auditService;

    @Transactional // REQUIRED (default) — outer transaction
    public void placeOrder(Order order) {
        inventoryService.decreaseStock(order);  // joins outer transaction
        auditService.logAction("ORDER_PLACED"); // always runs in its own transaction
    }
}

@Service
public class InventoryService {
    @Transactional(propagation = Propagation.REQUIRED)
    public void decreaseStock(Order order) {
        // Runs inside placeOrder's transaction
        // If this fails → everything in placeOrder rolls back
    }
}

@Service
public class AuditService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logAction(String action) {
        // NEW transaction — even if placeOrder rolls back, this audit log is committed
        auditRepository.save(new AuditLog(action));
    }
}
```

---

### b. NESTED — Savepoints

```java
@Transactional
public void processOrders(List<Order> orders) {
    for (Order order : orders) {
        try {
            processingleOrder(order); // Has NESTED propagation
        } catch (Exception e) {
            // Only this order's savepoint is rolled back; others proceed
            log.warn("Failed order {}: {}", order.getId(), e.getMessage());
        }
    }
}

@Transactional(propagation = Propagation.NESTED)
public void processingleOrder(Order order) {
    // Creates a savepoint; rolling back here does NOT roll back the outer transaction
}
```

---

### c. Propagation Reference

| Outer TX exists? | `REQUIRED` | `REQUIRES_NEW` | `NESTED` | `MANDATORY` | `NEVER` |
|---|---|---|---|---|---|
| **Yes** | Joins it | Creates new (suspends outer) | Creates savepoint | Joins it | Throws |
| **No** | Creates new | Creates new | Creates new | Throws | Runs without TX |

---


## 12. What is Spring Boot Caching, and how does it work?

**Answer:**

Spring Boot provides a **caching abstraction** (`@EnableCaching`) that lets you cache method results declaratively. The abstraction delegates to a pluggable `CacheManager` — the same annotations work with any provider (ConcurrentMap, EhCache, Caffeine, Redis).

---

### a. Core Annotations

| Annotation | Purpose |
|---|---|
| `@Cacheable` | Cache the return value; skip method on cache hit |
| `@CacheEvict` | Remove one or all entries from the cache |
| `@CachePut` | Always execute method AND update the cache |
| `@Caching` | Group multiple cache annotations on one method |

---

### b. Example

```java
@Configuration
@EnableCaching
public class CacheConfig { }

@Service
public class ProductService {

    @Cacheable(value = "products", key = "#id")
    public Product getById(Long id) {
        return productRepository.findById(id).orElseThrow(); // skipped on cache hit
    }

    @CachePut(value = "products", key = "#product.id")
    public Product update(Product product) {
        return productRepository.save(product); // always runs, updates cache
    }

    @CacheEvict(value = "products", key = "#id")
    public void delete(Long id) {
        productRepository.deleteById(id); // removes from cache
    }

    @CacheEvict(value = "products", allEntries = true)
    public void clearAll() { } // removes all entries from "products" cache
}
```

---

### c. Cache Providers Comparison

| Provider | Type | Best For |
|---|---|---|
| `ConcurrentMapCache` | In-JVM | Development / simple apps |
| **Caffeine** | In-JVM | High-performance local cache; supports TTL/eviction |
| **EhCache** | In-JVM / disk | Java-native, JSR-107 compliant |
| **Redis** | Distributed | Multi-instance apps, shared state, TTL, persistence |

---


## 13. What is Redis, and how is it used as a cache in Spring Boot?

**Answer:**

**Redis** is an open-source, in-memory data store that supports rich data structures (strings, hashes, lists, sets, sorted sets) with optional persistence. It is widely used as a **distributed cache**, session store, and message broker.

---

### a. Spring Boot Redis Setup

**`pom.xml` dependency:**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
```

**`application.yml`:**
```yaml
spring:
  data:
    redis:
      host: localhost
      port: 6379
  cache:
    type: redis
    redis:
      time-to-live: 600000   # 10 minutes in ms
      cache-null-values: false
```

---

### b. Redis Cache Configuration (Custom TTL per Cache)

```java
@Configuration
@EnableCaching
public class RedisConfig {

    @Bean
    public LettuceConnectionFactory redisConnectionFactory() {
        return new LettuceConnectionFactory("localhost", 6379);
    }

    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration defaultConfig = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))
            .serializeValuesWith(
                RedisSerializationContext.SerializationPair.fromSerializer(
                    new GenericJackson2JsonRedisSerializer())); // store as JSON

        Map<String, RedisCacheConfiguration> cacheConfigs = Map.of(
            "products",      defaultConfig.entryTtl(Duration.ofMinutes(30)),
            "user-sessions", defaultConfig.entryTtl(Duration.ofHours(1))
        );

        return RedisCacheManager.builder(factory)
            .cacheDefaults(defaultConfig)
            .withInitialCacheConfigurations(cacheConfigs)
            .build();
    }
}
```

---

### c. Direct Redis Operations with `RedisTemplate`

```java
@Service
public class RateLimitService {

    private final StringRedisTemplate redisTemplate;

    public boolean isAllowed(String userId) {
        String key = "rate:" + userId;
        Long count = redisTemplate.opsForValue().increment(key);
        if (count == 1) {
            redisTemplate.expire(key, Duration.ofMinutes(1)); // reset window
        }
        return count <= 100; // allow 100 requests/minute
    }
}
```

**Common `RedisTemplate` operations:**
```java
// String
redisTemplate.opsForValue().set("key", "value", Duration.ofMinutes(5));
redisTemplate.opsForValue().get("key");

// Hash
redisTemplate.opsForHash().put("user:1", "name", "Alice");

// List
redisTemplate.opsForList().rightPush("queue", "task1");

// Set & Sorted Set
redisTemplate.opsForSet().add("tags", "java", "spring");
redisTemplate.opsForZSet().add("leaderboard", "alice", 980.0);
```

## 14. What is the difference between Spring Boot Caching and Redis Caching?

**Answer:**

| Aspect | Spring Caching Abstraction | Redis as Cache Provider |
|---|---|---|
| **What it is** | API + programming model (`@Cacheable`, `CacheManager`) | An external, networked in-memory data store |
| **Location** | In-process API that delegates to a provider | Out-of-process; shared across all app instances |
| **Scope** | Single JVM unless backed by a distributed store | Distributed — visible to all instances immediately |
| **Consistency** | Local caches can be stale across instances | Single source of truth |
| **Persistence** | No (in-memory only for local providers) | Optional (RDB snapshotting, AOF logging) |
| **Extra Features** | None beyond caching | TTL, eviction policies, pub/sub, atomic ops, data structures |
| **Latency** | Nanoseconds (in-JVM) | Microseconds (network round-trip) |
| **Scalability** | Single instance only | Scales horizontally; Redis Cluster supported |

---

### How They Work Together

```
@Cacheable("products")          ← Spring abstraction (the "what")
       ↓
RedisCacheManager               ← Spring delegates to this
       ↓
Redis Server (distributed)      ← Actual storage (the "where")
```

**Recommendation**: Always code against the Spring abstraction (`@Cacheable`) — swap providers via configuration without changing application code. Use Redis in production for multi-instance deployments; use Caffeine for high-performance single-instance or local caching.

## 15. What is the N+1 Select Problem, and how do you solve it?

**Answer:**

The **N+1 problem** is one of the most common and impactful performance issues in JPA/Hibernate. It occurs when loading a parent entity triggers **N additional queries** to fetch lazy-loaded child collections — one per parent row.

---

### a. The Problem

```java
List<Order> orders = orderRepository.findAll();
// Query 1: SELECT * FROM orders  → 100 rows

for (Order order : orders) {
    order.getItems().size(); // lazy load triggers here
    // Queries 2..101: SELECT * FROM order_items WHERE order_id = ?  (×100!)
}
// Total: 101 queries instead of 1–2
```

---

### b. Solution 1 — JOIN FETCH (JPQL)

```java
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.status = :status")
List<Order> findWithItems(@Param("status") OrderStatus status);
// → 1 SQL query with a JOIN
```

**Caution**: Avoid `JOIN FETCH` with `Pageable` — Hibernate loads all rows into memory then paginates (use `@EntityGraph` instead).

---

### c. Solution 2 — @EntityGraph (pagination-safe)

```java
@EntityGraph(attributePaths = {"items", "items.product"})
Page<Order> findByCustomerId(Long customerId, Pageable pageable);
```

```java
// Or on a named graph defined on the entity
@Entity
@NamedEntityGraph(name = "Order.withItems",
    attributeNodes = @NamedAttributeNode("items"))
public class Order { ... }

@EntityGraph(value = "Order.withItems", type = EntityGraph.EntityGraphType.FETCH)
List<Order> findByStatus(OrderStatus status);
```

---

### d. Solution 3 — @BatchSize (minimal code change)

```java
@Entity
public class Order {
    @OneToMany(mappedBy = "order")
    @BatchSize(size = 25)          // 100 orders → ceil(100/25) = 4 queries
    private List<OrderItem> items;
}
```

Or globally:
```yaml
spring.jpa.properties.hibernate.default_batch_fetch_size: 25
```

---

### e. Solution 4 — DTO Projection (read-only, best performance)

```java
@Query("SELECT new com.example.dto.OrderSummary(o.id, o.status, COUNT(i)) " +
       "FROM Order o LEFT JOIN o.items i GROUP BY o.id, o.status")
List<OrderSummary> findOrderSummaries();
// → 1 query, no entity graph, no dirty checking
```

---

### f. Detecting N+1

```yaml
spring:
  jpa:
    show-sql: true
    properties:
      hibernate:
        generate_statistics: true
logging:
  level:
    org.hibernate.stat: DEBUG
```

Use **p6spy** or **Hypersistence Optimizer** to catch N+1 automatically in CI.

---

### g. Solution Comparison

| Solution | Best For | Pagination Safe? | Code Change |
|---|---|---|---|
| `JOIN FETCH` | Simple, non-paginated queries | No | Low |
| `@EntityGraph` | Paginated queries | Yes | Low |
| `@BatchSize` | Minimal refactor on existing code | Yes | Minimal |
| DTO Projection | Read-only / reporting / API responses | Yes | Medium |

**Rule of thumb**: Use `@EntityGraph` for paginated queries, `JOIN FETCH` for single non-paginated result sets, and DTO projections for read-only API responses.
