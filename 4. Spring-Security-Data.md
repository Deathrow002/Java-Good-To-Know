# Spring Security & Data — Questions & Answers

## 1. What is Spring Security, and how does it work?

**Answer:**

Spring Security is a framework that provides authentication, authorization, and other security features for Java applications. It works by applying a chain of servlet filters (`SecurityFilterChain`) that handle security concerns (authentication, CSRF, headers, authorization, exceptions) before requests reach controllers.

Example filter chain responsibilities (simplified):

- SecurityContext persistence
- HeaderWriter (security headers)
- CsrfFilter (CSRF token validation)
- Authentication filters (Basic, Form, Bearer, OAuth2)
- Authorization filters (URL-based access control)
- Exception translation (401/403 handling)

Basic configuration outline:

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        // configure http security
        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

## 2. How do Authentication and Authorization differ?

**Answer:**

- Authentication: verify identity (who you are). Implemented via `UserDetailsService`, tokens, credentials.
- Authorization: verify permissions (what you're allowed to do). Implemented via URL rules, method-level security (`@PreAuthorize`, `@PostAuthorize`).

Method-level security example:

```java
@EnableMethodSecurity
@Service
public class DocumentService {
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteDocument(Long id) { ... }
}
```

## 3. Explain JWT and OAuth2 in Spring Security

**Answer:**

JWT (JSON Web Token) is a compact token with header, payload, and signature used for stateless authentication. Spring apps use a token provider to create/validate tokens and a filter to extract authentication from requests.

Minimal JWT provider outline:

```java
public class JwtTokenProvider {
    @Value("${jwt.secret}")
    private String secretKey;

    @PostConstruct
    protected void init() { /* initialize key */ }

    public String createToken(String username, List<String> roles) { /* ... */ }
    public boolean validateToken(String token) { /* ... */ }
    public String getUsername(String token) { /* ... */ }
}
```

OAuth2 (Authorization Code flow) delegates login to an Authorization Server (Google, GitHub), exchanging a code for tokens to access protected resources.

Spring Security: configure OAuth2 clients in `application.yml` and wire into `SecurityFilterChain`.

## 4. How do you secure REST APIs in Spring Boot?

**Answer (layers):**

1. Transport: HTTPS/TLS
2. Rate limiting / DDoS protection (Bucket4j, API Gateway)
3. Authentication: JWT, OAuth2, API keys
4. Authorization: RBAC, method-level security
5. Input validation: `@Valid`, parameterized queries

JWT authentication filter outline:

```java
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    private final JwtTokenProvider tokenProvider;

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain filterChain)
            throws ServletException, IOException {
        String token = resolveToken(request);
        if (token != null && tokenProvider.validateToken(token)) {
            Authentication auth = tokenProvider.getAuthentication(token);
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        filterChain.doFilter(request, response);
    }
}
```

Best practices: secure CORS, protect secrets, rotate keys, validate tokens, use short expirations and refresh tokens.

## 5. What is CSRF, and how does Spring Security protect against it?

**Answer:**

CSRF (Cross-Site Request Forgery) tricks authenticated browsers into performing unwanted actions. Spring Security issues a CSRF token per session which must be submitted with state-changing requests (POST/PUT/DELETE). Templates and Thymeleaf auto-include tokens in forms; for AJAX, include the token in headers.

Example form (Thymeleaf):

```html
<form th:action="@{/transfer}" method="post">
  <input type="hidden" name="_csrf" th:value="${_csrf.token}" />
</form>
```

CSRF is usually enabled for browser-based clients; APIs using stateless JWT often disable CSRF and rely on tokens.

## 6. What is Spring Data JPA, and how does it simplify data access?

**Answer:**

Spring Data JPA provides repository abstractions so you can define interfaces and get implementations automatically. It integrates with JPA/Hibernate and `DataSource`/connection pools.

Entity example:

```java
@Entity
@Table(name = "products")
public class Product {
    @Id @GeneratedValue
    private Long id;
    @Column(nullable = false)
    private String name;
    @ManyToOne(fetch = FetchType.LAZY)
    private Supplier supplier;
}
```

Repository example:

```java
public interface ProductRepository extends JpaRepository<Product, Long> {
    List<Product> findByCategory(Category category);
    Page<Product> findByCategory(Category category, Pageable pageable);
    @Modifying
    @Query("UPDATE Product p SET p.price = p.price * :multiplier WHERE p.category = :category")
    int updatePricesByCategory(@Param("multiplier") BigDecimal multiplier, @Param("category") Category category);
}
```

## 7. What are the different types of repository interfaces in Spring Data?
**Answer:**

Spring Data provides several repository interfaces that you can extend to get CRUD, paging, sorting, and more advanced capabilities without writing boilerplate. The most common interfaces:

- `Repository<T, ID>`: Marker interface that establishes the repository abstraction. It doesn't declare methods itself but is the parent of more specific interfaces.
- `CrudRepository<T, ID>`: Adds CRUD operations: `save`, `findById`, `existsById`, `findAll`, `count`, `deleteById`, etc. Good for basic data access.
- `PagingAndSortingRepository<T, ID>`: Extends `CrudRepository` and adds paging and sorting support (`findAll(Pageable)`, `findAll(Sort)`). Use when you need pagination and sortable queries.
- `JpaRepository<T, ID>`: JPA-specific extension that adds JPA-related methods such as `flush()`, `saveAndFlush()`, and batch operations. Common choice for Spring Data JPA projects.

Specialized and additional interfaces

- `MongoRepository`, `CassandraRepository`, `ElasticsearchRepository`, etc.: Data-store-specific repositories that extend the core interfaces and add store-specific operations.
- `ReactiveCrudRepository<T, ID>` / `ReactiveMongoRepository`: Reactive (Project Reactor) variants for non-blocking access.
- `JpaSpecificationExecutor<T>`: Provides support for JPA `Specification` queries (dynamic predicates) when you need complex, type-safe criteria queries.

How to use (example):

```java
public interface ProductRepository extends JpaRepository<Product, Long>, JpaSpecificationExecutor<Product> {
    List<Product> findByCategory(Category category);
}
```

Notes & best practices:

- Prefer `JpaRepository` for JPA/Hibernate projects; it gives useful convenience methods and integrates well with Spring Data JPA features.
- Use `PagingAndSortingRepository` (or `Pageable` overloads on `JpaRepository`) when implementing pageable APIs.
- For reactive stacks (WebFlux), use the reactive repository interfaces to avoid blocking calls.
- Keep repository interfaces focused: use method naming conventions or `@Query` for custom queries; use `@Modifying` for update/delete JPQL operations.
- Use `JpaSpecificationExecutor` or QueryDSL for complex dynamic queries rather than large numbers of custom methods.

This set of interfaces lets you choose the abstraction level appropriate to your application (simple CRUD, pagination, JPA-specific features, or reactive access).

## 8. What is the difference between Lazy and Eager loading in JPA?

**Answer:**

- **Lazy loading:** Associated entities or collections are not fetched from the database until they are accessed in code. This is the default for collections (`@OneToMany`, `@ManyToMany`) and helps reduce initial query cost and memory usage. However, it can cause the N+1 select problem or `LazyInitializationException` if accessed outside a transactional context (e.g., after the session is closed).

- **Eager loading:** Associated entities are fetched immediately with the parent entity (typically via a join). This reduces the risk of `LazyInitializationException` but can produce heavy joins and load more data than required, possibly hurting performance.

Key trade-offs

- Use lazy loading to keep initial queries lightweight and avoid unnecessary data retrieval; prefer it for large collections or rarely-used relationships.
- Use eager loading when you always need the related data and want to avoid additional queries (but be mindful of join size and duplicated rows).

How to manage and avoid pitfalls

- Prevent N+1 queries using `JOIN FETCH` in queries, `@EntityGraph`, or batch fetching (`@BatchSize`).
- Access lazy relations within a transactional service method (Open Session in View is an anti-pattern for APIs).
- Use DTO projections to fetch exactly the fields you need instead of relying on entity graphs when building API responses.

Example `JOIN FETCH`:

```java
@Query("SELECT u FROM User u JOIN FETCH u.orders WHERE u.id = :id")
Optional<User> findByIdWithOrders(@Param("id") Long id);
```

Best practice: favor lazy associations by default, use targeted fetch strategies (fetch joins or entity graphs) in repository queries when you need related data, and prefer DTO projections for API-layer queries to control payload size and avoid entity-backing pitfalls.

## 9. Explain transaction management in Spring

**Answer:**

Spring manages transactions via `PlatformTransactionManager` and `@Transactional`. On success, transactions commit; on exceptions they roll back (by default on `RuntimeException`).

Propagation examples: `REQUIRED`, `REQUIRES_NEW`, etc. Use `readOnly=true` for read-only optimization. Beware of self-invocation which bypasses proxies.

Example:

```java
@Service
public class OrderService {
    @Transactional
    public Order createOrder(OrderRequest request) { ... }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logAudit(String action, Long orderId) { ... }
}
```

## 10. What is the difference between Lazy and Eager loading in JPA?

**Answer:**

- Lazy: associated entities are loaded when accessed (default for collections). Can cause N+1 queries if not handled.
- Eager: associated entities are loaded immediately with the parent (can produce heavy joins).

Solutions for N+1 problem: `JOIN FETCH`, `@EntityGraph`, `@BatchSize`, or DTO projections.

Example `JOIN FETCH`:

```java
@Query("SELECT u FROM User u JOIN FETCH u.orders")
List<User> findAllWithOrders();
```

## 11. What is Spring Boot Caching?
**Answer:**

Spring Boot caching uses Spring's caching abstraction to declaratively cache method results and other values. Enable it with `@EnableCaching` and annotate methods with `@Cacheable`, `@CacheEvict`, or `@CachePut`. The abstraction delegates to a `CacheManager` which can be backed by different providers (in-memory, Redis, Ehcache, Caffeine, etc.).

Basic example:

```java
@Configuration
@EnableCaching
public class CacheConfig { }

@Service
public class ProductService {
    @Cacheable(value = "products", key = "#id")
    public Product getById(Long id) {
        return productRepository.findById(id).orElse(null);
    }
}
```

**Use cases:**

- Cache expensive database lookups (read-heavy endpoints) to reduce DB load.
- Cache results of CPU-intensive computations.
- Store reference or lookup data (country lists, config) that changes rarely.
- Caching session or user profile data to speed repeated reads (when appropriate).
- Implement request-level caching for idempotent service calls.

**Notes:** choose cache provider based on requirements (local JVM cache for ultra-low latency; distributed cache like Redis for shared caches across instances, high availability, and larger working sets).

## 12. What is Redis Caching?

**Answer:**

Redis caching refers to using Redis — an open-source, in-memory data store — as the cache provider. Redis stores key-value data (and richer data structures) in memory, supports TTLs, eviction policies, persistence (RDB/AOF), replication, pub/sub, and atomic operations, making it suitable as a distributed cache for high-performance systems.

**Spring Boot integration (quick example):**

application.yml

```yaml
spring:
    redis:
        host: localhost
        port: 6379
    cache:
        type: redis
```

Java (auto-configured with `spring-boot-starter-data-redis`):

```java
@Configuration
@EnableCaching
public class RedisConfig {
        @Bean
        public LettuceConnectionFactory redisConnectionFactory() {
                return new LettuceConnectionFactory("localhost", 6379);
        }
}
```

**Use cases:**

- Distributed caching across multiple application instances (shared cache).
- Session storage for stateless apps (HTTP session in Redis).
- Rate limiting and request counters (using atomic increments).
- Leaderboards, sorted sets, and time-series short-term storage.
- Distributed locks (e.g., RedLock) and coordination primitives.

**Notes:**

- Redis is networked (adds latency vs in-process caches) but scales and persists; combine with a near-cache if you need ultra-low latency.
- Use appropriate serialization (JSON, Avro, or native) and set TTLs to avoid stale data and memory pressure.


## 13. What is the difference between Spring Boot Caching and Redis Caching?

**Answer:**

- **Scope & role:** "Spring Boot caching" refers to Spring's caching abstraction (`@EnableCaching`, `@Cacheable`, `@CacheEvict`, `CacheManager`). It's an API and programming model that lets you cache method results without being tied to a specific provider. Redis caching refers to using Redis (an external in-memory data store) as the underlying cache provider implementation.

- **Location:** Spring's abstraction runs in-process (the API) and delegates to a `CacheManager`. Redis is an out-of-process, networked cache that stores data centrally and can be shared across services and application instances.

- **Use cases:** Use Spring Cache abstraction to add caching behavior to methods. Use Redis when you need a distributed, durable, or shared cache (e.g., multiple app instances, cross-service caching, session store, leaderboards).

- **Consistency & visibility:** In-process caches (e.g., `ConcurrentMapCache`) are fast but local to a single JVM and may lead to stale caches across instances. Redis provides a single source of truth for cached entries; changes are immediately visible to all clients.

- **Features & capabilities:** Redis supports eviction policies, persistence (RDB/AOF), TTLs, pub/sub, atomic operations, and data structures (lists, sets, sorted sets). A Spring `CacheManager` backed by Redis leverages these features for advanced caching scenarios.

- **Performance & latency:** Local caches have lower latency (in-memory access), while Redis adds network latency but can handle larger datasets and distribute load. Choose local caches for ultra-low-latency single-instance apps and Redis for scale and sharing.

- **Configuration in Spring Boot:** Configure `spring.cache.type` (e.g., `redis`) and add a `RedisCacheManager` (auto-configured when `spring-boot-starter-data-redis` is present). You can also use hybrid approaches (near-cache + Redis).

**Recommendation:** Prefer Spring's caching abstraction for clean code and portability; choose the cache implementation according to needs — `ConcurrentMapCache` for simple local caching during development, Redis (or another distributed cache) for production scenarios requiring distribution, durability, or advanced features.
