# Spring Security & Data — Questions & Answers

## 1. What is Spring Security, and how does it work?

**Answer:**

**Spring Security** is a powerful, highly customizable authentication and access-control framework. It is the de-facto standard for securing Spring-based applications.

At its core, **Spring Security works through a chain of Servlet Filters**. When a request enters your application, it must pass through this chain before reaching your `@Controller`.

---

### a. High-Level Architecture
The entry point is a standard Servlet filter called the **`DelegatingFilterProxy`**. It bridges the gap between the Servlet container (Tomcat, Jetty) and the Spring application context, delegating the work to a Spring-managed bean called **`FilterChainProxy`**.

```text
Request ──▶ DelegatingFilterProxy ──▶ FilterChainProxy (Virtual Filter Chain) ──▶ Controller
```

The `FilterChainProxy` contains one or more **`SecurityFilterChain`** beans. Each filter in this chain has a specific responsibility:
- **`SecurityContextPersistenceFilter`**: Loads/saves the `SecurityContext` across requests.
- **`CsrfFilter`**: Protects against Cross-Site Request Forgery.
- **`UsernamePasswordAuthenticationFilter`**: Handles form-based logins.
- **`BearerTokenAuthenticationFilter`**: Handles JWT/OAuth2 tokens.
- **`AuthorizationFilter`**: Replaces the legacy `FilterSecurityInterceptor`. It checks if the current user has the required roles for the requested URL.

---

### b. How Authentication Works
1.  **Extraction**: A filter (e.g., `UsernamePasswordAuthenticationFilter`) extracts credentials from the request.
2.  **Delegation**: It passes an unauthenticated `Authentication` object to the **`AuthenticationManager`**.
3.  **Verification**: The manager iterates through **`AuthenticationProvider`**s (like `DaoAuthenticationProvider` which uses **`UserDetailsService`**) to verify credentials.
4.  **Storage**: On success, the fully populated `Authentication` object is stored in the **`SecurityContextHolder`**, making it available via `SecurityContextHolder.getContext().getAuthentication()`.

---

### c. Modern Configuration (Spring Security 6+)
Spring Security 6 moved away from the legacy `WebSecurityConfigurerAdapter` in favor of a component-based configuration using the **Lambda DSL**.

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity // Enables @PreAuthorize/@PostAuthorize
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Disable for stateless APIs
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll() // Public endpoints
                .requestMatchers("/api/admin/**").hasRole("ADMIN") // Role-based access
                .anyRequest().authenticated() // All others require login
            )
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS) // REST API best practice
            )
            .httpBasic(Customizer.withDefaults()); // Enable Basic Auth

        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder(); // Industry standard for hashing
    }
}
```

---

### d. Why use Spring Security?
1.  **Decoupling**: Security logic is kept out of your business controllers.
2.  **Comprehensive Protection**: Includes built-in protection against CSRF, Clickjacking, Session Fixation, and XSS.
3.  **Flexibility**: Easily switch between Form Login, Basic Auth, OAuth2, SAML, or custom SSO providers.
4.  **Integration**: Seamlessly integrates with Spring MVC, WebFlux, and Spring Data.

---


## 2. How do Authentication and Authorization differ?

**Answer:**

While both are essential for security, they address two fundamentally different questions:

| Feature | **Authentication (AuthN)** | **Authorization (AuthZ)** |
|:---:|---|---|
| **Core Question** | Who are you? | What are you allowed to do? |
| **Input** | Credentials (User/Pass, Token, OTP) | Roles, Permissions, Attributes |
| **Failed Result** | **401 Unauthorized** (No identity) | **403 Forbidden** (Denied permission) |
| **Object** | **`UserDetails`** / **`Principal`** | **`GrantedAuthority`** |

---

### a. Authentication (AuthN)
The process of verifying that an entity is who it claims to be. In Spring Security, this is typically handled by an `AuthenticationProvider` that checks credentials against a source (`Database`, `LDAP`, `OIDC`).

```java
@Bean
public UserDetailsService userDetailsService() {
    // Defining users for authentication
    return new InMemoryUserDetailsManager(
        User.withUsername("alice").password("{noop}pass123").roles("USER").build(),
        User.withUsername("bob").password("{noop}admin123").roles("ADMIN").build()
    );
}
```

---

### b. Authorization (AuthZ)
The process of determining whether an authenticated user has the necessary rights to access a resource. Spring Security provides two levels of control:

#### b-1. URL-Based Authorization
Controls access to specific endpoints in the `SecurityFilterChain`.

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/user/**").hasAnyRole("USER", "ADMIN")
    .anyRequest().authenticated()
);
```

#### b-2. Method-Level Authorization
Uses AOP proxies to secure individual service methods. Requires `@EnableMethodSecurity`.

```java
@Service
public class DocumentService {

    @PreAuthorize("hasRole('ADMIN')")
    public void deleteDocument(Long id) {
        // Only Bob (ADMIN) can call this
    }

    @PreAuthorize("hasAuthority('SCOPE_read') or hasRole('USER')")
    public Document getDocument(Long id) {
        // Alice or a user with specific 'read' authority
        return repository.findById(id).orElseThrow();
    }
}
```

---

### c. Roles vs. Authorities
- **Authorities**: Granular permissions (e.g., `READ_PRIVILEGE`, `DELETE_DOCUMENT`).
- **Roles**: Logical groupings of authorities (e.g., `ROLE_ADMIN`, `ROLE_EDITOR`).
- **Convention**: In Spring Security, `hasRole("ADMIN")` is a shorthand for `hasAuthority("ROLE_ADMIN")`. The `ROLE_` prefix is automatically added by the framework when using the "Role" version of the check.

## 3. Explain JWT and OAuth2 in Spring Security

**Answer:**

**JWT** and **OAuth2** are often used together but serve different purposes in the modern security landscape.

---

### a. JWT (JSON Web Token)
JWT is an open standard (**RFC 7519**) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object.

- **Structure**: Consists of three parts separated by dots (`.`): **`Header.Payload.Signature`**.
- **Statelessness**: The server doesn't need to store session data in memory. All user information is stored directly in the token.
- **Security**: The payload is base64 encoded (not encrypted!) but the signature ensures that the data hasn't been tampered with.

#### Minimal JWT Service (using `jjwt` library):
```java
@Service
public class JwtService {
    private static final String SECRET_KEY = "your_secret_key_must_be_very_long...";

    public String generateToken(UserDetails userDetails) {
        return Jwts.builder()
                .subject(userDetails.getUsername())
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + 1000 * 60 * 60)) // 1 hour
                .signWith(getSignInKey())
                .compact();
    }

    public boolean isTokenValid(String token, UserDetails userDetails) {
        final String username = extractUsername(token);
        return (username.equals(userDetails.getUsername())) && !isTokenExpired(token);
    }
}
```

---

### b. OAuth2 (The Protocol)
OAuth2 is an **authorization framework** that enables applications to obtain limited access to user accounts on an HTTP service (e.g., Google, GitHub, or your own Auth Server).

#### b-1. Core Roles:
1.  **Resource Owner**: The User.
2.  **Client**: The application trying to access the user's data.
3.  **Authorization Server**: The server that authenticates the user and issues tokens.
4.  **Resource Server**: The API that stores the user's data (protected by Spring Security).

#### b-2. Common Flow: Authorization Code
The most secure flow for web/mobile apps. The user logs in to the Auth Server, which returns a **Code**. The Client then exchanges this code for an **Access Token**.

---

### c. Spring Security Implementation
In a modern microservices architecture, you typically configure your API as an **OAuth2 Resource Server** that validates JWTs issued by an external provider (like Keycloak, Auth0, or Okta).

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf(AbstractHttpConfigurer::disable)
        .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/public/**").permitAll()
            .anyRequest().authenticated()
        )
        // Configure app as an OAuth2 Resource Server
        .oauth2ResourceServer(oauth2 -> oauth2
            .jwt(jwt -> jwt.decoder(jwtDecoder()))
        );

    return http.build();
}
```

---

### Relationship: JWT in OAuth2
OAuth2 is the **protocol** describing how to get the key. JWT is the **format** of the key itself. Most modern OAuth2 implementations (like OpenID Connect) use JWT as the format for Access and Identity tokens.

---


## 4. How do you secure REST APIs in Spring Boot?

**Answer:**

Securing a REST API requires a **Defense in Depth** approach, applying multiple security layers to ensure that if one layer fails, others remain to protect the system.

---

### a. The Layered Security Model

#### a-1. Transport Layer (Privacy & Integrity)
- **HTTPS/TLS**: Always use SSL/TLS to encrypt data in transit. In Spring Security, you can force HTTPS:
```java
http.requiresChannel(chan -> chan.anyRequest().requiresSecure());
```

#### a-2. Network & Perimeter (Rate Limiting & CORS)
- **CORS (Cross-Origin Resource Sharing)**: Restrict which domains can call your API.
- **Rate Limiting**: Prevent DDoS and brute force attacks using **Bucket4j** or a Cloud Gateway layer.

#### a-3. Authentication Layer (Identity)
- **Statelessness**: REST APIs should be stateless (no sessions). Use **JWT (JSON Web Tokens)** or **OAuth2** Bearer tokens.
- **API Keys**: For machine-to-machine (M2M) communication where full OAuth2 might be overkill.

#### a-4. Authorization Layer (Access Control)
- **RBAC (Role-Based Access Control)**: Enforce roles like `ADMIN` or `USER` using `requestMatchers`.
- **ABAC (Attribute-Based Access Control)**: More granular control based on user attributes or resource ownership.

#### a-5. Application Layer (Integrity & Logging)
- **Input Validation**: Use `@Valid` and `@NotNull` (JSR-303) to prevent malicious payloads.
- **Error Handling**: Mask internal stack traces. Don't reveal "User not found"; use generic "Invalid credentials".

---

### b. Typical JWT Authentication Filter
The heart of a stateless REST API is the `OncePerRequestFilter`, which intercepts every request to check for a valid Bearer token.

```java
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest request, 
                                    HttpServletResponse response, 
                                    FilterChain filterChain) throws ServletException, IOException {
        final String authHeader = request.getHeader("Authorization");
        
        // 1. Check for Bearer token
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        final String jwt = authHeader.substring(7);
        final String username = jwtService.extractUsername(jwt);

        // 2. Validate and set SecurityContext
        if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            UserDetails userDetails = this.userDetailsService.loadUserByUsername(username);
            if (jwtService.isTokenValid(jwt, userDetails)) {
                UsernamePasswordAuthenticationToken authToken = new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());
                authToken.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }
        filterChain.doFilter(request, response);
    }
}
```

---

### c. Advanced Security Checkpoints
- **CORS Config Bean**: Define allowed origins, methods, and headers specifically.
- **Sensitive Data Masking**: Use `@JsonIgnore` or DTOs to avoid leaking PII or internal IDs.
- **Secrets Management**: Never commit `jwt.secret` to Git. Use **Spring Cloud Vault** or Environment Variables.
- **Token Rotation**: Implement **Refresh Tokens** so that short-lived Access Tokens (e.g., 15 mins) don't force users to re-login frequently.

## 5. What is CSRF, and how does Spring Security protect against it?

**Answer:**

**CSRF (Cross-Site Request Forgery)** is an attack that tricks an authenticated user's browser into sending an unwanted request to a web application. Because the browser automatically includes cookies (like session IDs) with every request to the target domain, the server incorrectly assumes the request is intentional.

---

### a. How Spring Security Protects: The Synchronizer Token Pattern
Spring Security uses the **Synchronizer Token Pattern** to defend against CSRF:
1.  The server generates a unique, cryptographically strong **CSRF Token** for the user's session.
2.  For every "state-changing" request (POST, PUT, DELETE, PATCH), the application requires the token to be present in the request (as a hidden form field or a header).
3.  The server compares the token in the request with the one stored in the session/cookie. If they don't match, the request is rejected with a **403 Forbidden**.

---

### b. Implementation by Architecture

#### b-1. Server-Side Rendering (Thymeleaf, JSP)
Spring Security automatically includes the CSRF token in any form generated by Thymeleaf's `th:action`.

```html
<!-- Thymeleaf automatically adds this hidden field -->
<form th:action="@{/transfer}" method="post">
  <input type="hidden" name="_csrf" th:value="f1e2-d3c4-b5a6..." />
  <button type="submit">Transfer Money</button>
</form>
```

#### b-2. Single Page Applications (SPA: React, Angular, Vue)
For SPAs, the server typically sends the token as a cookie, which the JavaScript client reads and sends back in a custom header (e.g., `X-XSRF-TOKEN`).

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf
            .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()) // Allows JS to read the cookie
            .csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())
        );
    return http.build();
}
```

---

### c. When to Enable vs. Disable CSRF

| Scenarios | Status | Reasoning |
|---|---|---|
| **Web App (Browsers/Cookies)** | **Enable** | Essential if you use session-based authentication (cookies). |
| **Stateless API (JWT)** | **Disable** | Stateless APIs (which use `Bearer` tokens in headers) are inherently immune to CSRF because headers are not automatically sent by the browser. |
| **Non-Browser Clients** | **Disable** | Mobile apps and CLI tools don't have the "automatic cookie" behavior of browsers. |

---

### d. Additional Defense: SameSite Cookie Attribute
Modern browsers support the **`SameSite`** attribute for cookies (`Strict` or `Lax`), which instructs the browser not to send cookies with cross-site requests. Spring Security 6+ configures `SameSite=Lax` by default, providing an extra layer of protection even if CSRF tokens aren't used correctly.

## 6. What is Spring Data JPA, and how does it simplify data access?

**Answer:**

**Spring Data JPA** is a library that provides a high-level abstraction on top of the **JPA (Java Persistence API)** and its providers (like **Hibernate**). Its primary goal is to significantly reduce the boilerplate code required to implement data access layers.

---

### a. Why use Spring Data JPA?

Before Spring Data JPA, developers had to interact directly with the `EntityManager`, managing transactions and writing native or JPQL queries manually.

**Core Benefits:**
1.  **Repository Abstraction**: You define an interface, and Spring automatically provides the implementation at runtime.
2.  **Boilerplate Reduction**: Common CRUD operations (`save`, `delete`, `findById`) are inherited from base interfaces.
3.  **Dynamic Query Generation**: Spring "parses" your method names (e.g., `findByLastNameAndStatus`) to generate SQL queries automatically.
4.  **Pagination & Sorting**: Built-in support for handling large datasets using `Pageable` and `Sort` objects.
5.  **Audit Support**: Automatically tracks entity creation and modification dates/users via `@CreatedDate` and `@LastModifiedBy`.

---

### b. Typical Implementation

#### b-1. The Entity
An entity represents a table in the database.

```java
@Entity
@Table(name = "products")
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true)
    private String sku;

    private String name;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id")
    private Category category;
}
```

#### b-2. The Repository
The repository provides the data access logic.

```java
@Repository
public interface ProductRepository extends JpaRepository<Product, Long> {

    // Derived Query: Find by name containing a string
    List<Product> findByNameContainingIgnoreCase(String name);

    // Pagination: Find by category with paging
    Page<Product> findByCategoryName(String categoryName, Pageable pageable);

    // Custom JPQL: Using @Query for complex logic
    @Query("SELECT p FROM Product p WHERE p.category.active = true")
    List<Product> findProductsFromActiveCategories();

    // Native Query: Directly using SQL
    @Query(value = "SELECT * FROM products WHERE stock_count < :limit", nativeQuery = true)
    List<Product> findLowStockItems(@Param("limit") int limit);
}
```

---

### c. JPA vs. Hibernate vs. Spring Data JPA
- **JPA**: The Java specification for ORM (The "Rules").
- **Hibernate**: The implementation of the JPA specification (The "Engine").
- **Spring Data JPA**: The convenience layer that hides the complexity of Hibernate (The "Steering Wheel").

---


## 7. What are the different types of repository interfaces in Spring Data?
**Answer:**

Spring Data provides several repository interfaces that you can extend to get CRUD, paging, sorting, and more advanced capabilities without writing boilerplate. The most common interfaces:

- `Repository<T, ID>`: Marker interface that establishes the repository abstraction. It doesn't declare methods itself but is the parent of more specific interfaces.
- `CrudRepository<T, ID>`: Adds CRUD operations: `save`, `findById`, `existsById`, `findAll`, `count`, `deleteById`, etc. Good for basic data access.
- `PagingAndSortingRepository<T, ID>`: Extends `CrudRepository` and adds paging and sorting support (`findAll(Pageable)`, `findAll(Sort)`). Use when you need pagination and sortable queries.
- `JpaRepository<T, ID>`: JPA-specific extension that adds JPA-related methods such as `flush()`, `saveAndFlush()`, and batch operations. Common choice for Spring Data JPA projects.

Specialized and additional interfaces

- `MongoRepository`, `CassandraRepository`, `ElasticsearchRepository`, etc.: Data-store-specific repositories that extend the core interfaces and add store-specific operations.
- `ReactiveCrudRepository<T, ID>` / `ReactiveMongoRepository`: Reactive (Project Reactor) variants for non-blocking access.
- `JpaSpecificationExecutor<T>`: Provides support for JPA `Specification` queries (dynamic predicates) when you need complex, type-safe criteria queries.

How to use (example):

```java
public interface ProductRepository extends JpaRepository<Product, Long>, JpaSpecificationExecutor<Product> {
    List<Product> findByCategory(Category category);
}
```

Notes & best practices:

- Prefer `JpaRepository` for JPA/Hibernate projects; it gives useful convenience methods and integrates well with Spring Data JPA features.
- Use `PagingAndSortingRepository` (or `Pageable` overloads on `JpaRepository`) when implementing pageable APIs.
- For reactive stacks (WebFlux), use the reactive repository interfaces to avoid blocking calls.
- Keep repository interfaces focused: use method naming conventions or `@Query` for custom queries; use `@Modifying` for update/delete JPQL operations.
- Use `JpaSpecificationExecutor` or QueryDSL for complex dynamic queries rather than large numbers of custom methods.

This set of interfaces lets you choose the abstraction level appropriate to your application (simple CRUD, pagination, JPA-specific features, or reactive access).

## 8. What are Projections and DTOs in Spring Data JPA?

**Answer:**

In high-performance applications, you often don't need to fetch the entire entity. **Projections** allow you to retrieve only specific columns from the database, reducing memory usage and network overhead.

---

### a. Interface-based Projections
This is the simplest form. You define an interface with "getter" methods that match the entity's field names.

#### a-1. Closed Projections
Only includes specific fields. Highly optimized because Spring JPA generates a query picking only those columns.
```java
public interface UserSummary {
    String getFirstName();
    String getLastName();
}

// In Repository
List<UserSummary> findByLastName(String lastName);
```

#### a-2. Open Projections
Uses SpEL (Spring Expression Language) to compute values. Less optimized because Spring must fetch the whole entity to evaluate the expression.
```java
public interface UserFullNames {
    @Value("#{target.firstName + ' ' + target.lastName}")
    String getFullName();
}
```

---

### b. Class-based Projections (DTOs)
Uses a standard Java class (or **Record** in Java 16+). This approach is more robust for complex data transfers.

```java
public record UserDto(String email, String phoneNumber) {}

// In Repository
List<UserDto> findByEmail(String email);
```
**Benefits of Records:**
- **Immutable**: Data cannot be changed after fetching.
- **Clean**: No boilerplate (getters, equals, hashCode).
- **Native**: No proxying overhead.

---

### c. Dynamic Projections
You can make a repository method generic to support multiple projection types.

```java
public interface UserRepository extends JpaRepository<User, Long> {
    <T> List<T> findByStatus(String status, Class<T> type);
}

// Usage
List<UserSummary> summaries = userRepository.findByStatus("ACTIVE", UserSummary.class);
List<UserDto> dtos = userRepository.findByStatus("ACTIVE", UserDto.class);
```

---

### d. When to use Projections?
1.  **Read-Only Operations**: Use projections for search results and tables where you don't intend to modify and save the entities.
2.  **Performance**: When an entity has many large columns (like `BLOB` or long `TEXT`) and you only need IDs or Names.
3.  **API Design**: To avoid exposing internal entity secrets (like `passwordHash`) by mistake in your REST responses.

---

## 9. What is the difference between Lazy and Eager loading in JPA?

**Answer:**

- **Lazy loading:** Associated entities or collections are not fetched from the database until they are accessed in code. This is the default for collections (`@OneToMany`, `@ManyToMany`) and helps reduce initial query cost and memory usage. However, it can cause the N+1 select problem or `LazyInitializationException` if accessed outside a transactional context (e.g., after the session is closed).

- **Eager loading:** Associated entities are fetched immediately with the parent entity (typically via a join). This reduces the risk of `LazyInitializationException` but can produce heavy joins and load more data than required, possibly hurting performance.

Key trade-offs

- Use lazy loading to keep initial queries lightweight and avoid unnecessary data retrieval; prefer it for large collections or rarely-used relationships.
- Use eager loading when you always need the related data and want to avoid additional queries (but be mindful of join size and duplicated rows).

How to manage and avoid pitfalls

- Prevent N+1 queries using `JOIN FETCH` in queries, `@EntityGraph`, or batch fetching (`@BatchSize`).
- Access lazy relations within a transactional service method (Open Session in View is an anti-pattern for APIs).
- Use DTO projections to fetch exactly the fields you need instead of relying on entity graphs when building API responses.

Example `JOIN FETCH`:

```java
@Query("SELECT u FROM User u JOIN FETCH u.orders WHERE u.id = :id")
Optional<User> findByIdWithOrders(@Param("id") Long id);
```

Best practice: favor lazy associations by default, use targeted fetch strategies (fetch joins or entity graphs) in repository queries when you need related data, and prefer DTO projections for API-layer queries to control payload size and avoid entity-backing pitfalls.

## 10. What is the `@Transactional` annotation in Spring?

**Answer:**

The **`@Transactional`** annotation is the heart of Spring's **Declarative Transaction Management**. It allows developers to manage database transactions via AOP (Aspect-Oriented Programming) proxies, ensuring that a series of database operations either all succeed or all fail (**ACID properties**).

---

### a. How it Works (Under the Hood)
Spring creates a **Proxy** around the bean. When a `@Transactional` method is called:
1.  The proxy starts a transaction using the `TransactionManager`.
2.  The target method executes.
3.  If it returns successfully, the proxy **commits** the transaction.
4.  If an unhandled exception occurs, the proxy **rolls back** the transaction.

---

### b. Key Attributes

#### b-1. Propagation
Defines how transactions relate to each other when one transactional method calls another.
- **`REQUIRED` (Default)**: Use the existing transaction or create a new one if none exists.
- **`REQUIRES_NEW`**: Always suspend the current transaction and create a fresh one.
- **`MANDATORY`**: Must have an existing transaction, otherwise throw an exception.
- **`SUPPORTS`**: Use a transaction if it exists, otherwise run non-transactionally.

#### b-2. Isolation Levels
Defines the degree to which one transaction is isolated from others (Consistency).
- **`DEFAULT`**: Uses the underlying database's default level (usually `READ_COMMITTED`).
- **`SERIALIZABLE`**: Highest isolation; slowest performance.

#### b-3. Rollback Rules
- By default, Spring only rolls back for **Unchecked Exceptions** (`RuntimeException` and `Error`).
- **Checked Exceptions** do NOT trigger a rollback unless explicitly configured:
  `@Transactional(rollbackFor = Exception.class)`

---

### c. The "Self-Invocation" Pitfall
A common mistake is calling a `@Transactional` method from another method within the **same class**.
```java
public void parentMethod() {
    childTransactionalMethod(); // TRANSACTION WILL NOT START
}

@Transactional
public void childTransactionalMethod() { ... }
```
**Why?** Because the call bypasses the Spring Proxy. The proxy only intercepts calls coming from *outside* the bean.

---

### d. Illustrative Example
```java
@Service
public class OrderService {

    @Transactional(propagation = Propagation.REQUIRED, rollbackFor = BusinessException.class)
    public void placeOrder(Order order) {
        inventoryService.decreaseStock(order.getItems()); // Part of same transaction
        repository.save(order);
        
        if (paymentFailed()) {
            throw new BusinessException("Payment failed!"); // Triggers rollback
        }
    }
}
```

**Bottom Line:** `@Transactional` simplifies code by removing boilerplate `begin/commit/rollback` logic, but requires a deep understanding of AOP proxies and database propagation to avoid data integrity issues.

## 11. Explain transaction management in Spring

**Answer:**

Spring provides a consistent abstraction for transaction management called **Declarative Transaction Management**. It allows developers to manage transactions using the **`@Transactional`** annotation without writing low-level code for committing or rolling back.

---

### a. How it Works: The AOP Proxy
When you annotate a method with `@Transactional`, Spring creates a **CGLIB** or **JDK Dynamic Proxy** around your bean.
1.  **Proxy Entry**: The proxy intercepts the method call.
2.  **Transaction Start**: The `PlatformTransactionManager` starts a new transaction.
3.  **Method Execution**: Your actual business logic executes.
4.  **Completion**:
    - If successful, the proxy tells the manager to **commit**.
    - If an exception occurs, the proxy tells the manager to **rollback**.

---

### b. Transaction Propagation
Propagation defines how transactions behave when one transactional method calls another.

| Propagation | Behavior |
|---|---|
| **`REQUIRED`** (Default) | Joins an existing transaction or creates a new one if none exists. |
| **`REQUIRES_NEW`** | Always creates a new transaction, suspending the current one if it exists. |
| **`NESTED`** | Creates a "savepoint" within the current transaction; can rollback independently to that point. |
| **`MANDATORY`** | Requires an existing transaction; throws an exception if none is present. |
| **`NEVER`** | Throws an exception if a transaction is currently active. |

---

### c. Rollback Rules
By default, Spring only rolls back for **Unchecked Exceptions** (`RuntimeException` and `Error`).
- **Checked Exceptions**: Will NOT trigger a rollback unless explicitly configured.
- **Customization**: Use `@Transactional(rollbackFor = Exception.class)` to include checked exceptions.

---

### d. Performance Optimization: Read-Only
Setting `@Transactional(readOnly = true)` provides a hint to the underlying persistence provider (like Hibernate) to optimize performance:
- Flushes are disabled.
- Dirty checking is skipped.
- Some databases can route read-only transactions to replica nodes.

---

### e. Typical Pitfalls
1.  **Self-Invocation**: Calling a `@Transactional` method from another method within the *same* class bypasses the proxy, meaning no transaction is started.
2.  **Private Methods**: `@Transactional` only works on `public` methods because proxies can only intercept public calls.
3.  **Database Support**: `NESTED` and certain isolation levels require specific database support (e.g., MySQL vs. Oracle).

---

## 12. What is Spring Boot Caching?
**Answer:**

Spring Boot caching uses Spring's caching abstraction to declaratively cache method results and other values. Enable it with `@EnableCaching` and annotate methods with `@Cacheable`, `@CacheEvict`, or `@CachePut`. The abstraction delegates to a `CacheManager` which can be backed by different providers (in-memory, Redis, Ehcache, Caffeine, etc.).

Basic example:

```java
@Configuration
@EnableCaching
public class CacheConfig { }

@Service
public class ProductService {
    @Cacheable(value = "products", key = "#id")
    public Product getById(Long id) {
        return productRepository.findById(id).orElse(null);
    }
}
```

**Use cases:**

- Cache expensive database lookups (read-heavy endpoints) to reduce DB load.
- Cache results of CPU-intensive computations.
- Store reference or lookup data (country lists, config) that changes rarely.
- Caching session or user profile data to speed repeated reads (when appropriate).
- Implement request-level caching for idempotent service calls.

**Notes:** choose cache provider based on requirements (local JVM cache for ultra-low latency; distributed cache like Redis for shared caches across instances, high availability, and larger working sets).

## 13. What is Redis Caching?

**Answer:**

Redis caching refers to using Redis — an open-source, in-memory data store — as the cache provider. Redis stores key-value data (and richer data structures) in memory, supports TTLs, eviction policies, persistence (RDB/AOF), replication, pub/sub, and atomic operations, making it suitable as a distributed cache for high-performance systems.

**Spring Boot integration (quick example):**

application.yml

```yaml
spring:
    redis:
        host: localhost
        port: 6379
    cache:
        type: redis
```

Java (auto-configured with `spring-boot-starter-data-redis`):

```java
@Configuration
@EnableCaching
public class RedisConfig {
        @Bean
        public LettuceConnectionFactory redisConnectionFactory() {
                return new LettuceConnectionFactory("localhost", 6379);
        }
}
```

**Use cases:**

- Distributed caching across multiple application instances (shared cache).
- Session storage for stateless apps (HTTP session in Redis).
- Rate limiting and request counters (using atomic increments).
- Leaderboards, sorted sets, and time-series short-term storage.
- Distributed locks (e.g., RedLock) and coordination primitives.

**Notes:**

- Redis is networked (adds latency vs in-process caches) but scales and persists; combine with a near-cache if you need ultra-low latency.
- Use appropriate serialization (JSON, Avro, or native) and set TTLs to avoid stale data and memory pressure.

## 14. What is the difference between Spring Boot Caching and Redis Caching?

**Answer:**

- **Scope & role:** "Spring Boot caching" refers to Spring's caching abstraction (`@EnableCaching`, `@Cacheable`, `@CacheEvict`, `CacheManager`). It's an API and programming model that lets you cache method results without being tied to a specific provider. Redis caching refers to using Redis (an external in-memory data store) as the underlying cache provider implementation.

- **Location:** Spring's abstraction runs in-process (the API) and delegates to a `CacheManager`. Redis is an out-of-process, networked cache that stores data centrally and can be shared across services and application instances.

- **Use cases:** Use Spring Cache abstraction to add caching behavior to methods. Use Redis when you need a distributed, durable, or shared cache (e.g., multiple app instances, cross-service caching, session store, leaderboards).

- **Consistency & visibility:** In-process caches (e.g., `ConcurrentMapCache`) are fast but local to a single JVM and may lead to stale caches across instances. Redis provides a single source of truth for cached entries; changes are immediately visible to all clients.

- **Features & capabilities:** Redis supports eviction policies, persistence (RDB/AOF), TTLs, pub/sub, atomic operations, and data structures (lists, sets, sorted sets). A Spring `CacheManager` backed by Redis leverages these features for advanced caching scenarios.

- **Performance & latency:** Local caches have lower latency (in-memory access), while Redis adds network latency but can handle larger datasets and distribute load. Choose local caches for ultra-low-latency single-instance apps and Redis for scale and sharing.

- **Configuration in Spring Boot:** Configure `spring.cache.type` (e.g., `redis`) and add a `RedisCacheManager` (auto-configured when `spring-boot-starter-data-redis` is present). You can also use hybrid approaches (near-cache + Redis).

**Recommendation:** Prefer Spring's caching abstraction for clean code and portability; choose the cache implementation according to needs — `ConcurrentMapCache` for simple local caching during development, Redis (or another distributed cache) for production scenarios requiring distribution, durability, or advanced features.
