# Spring Security & Data — Questions & Answers

## 1. What is Spring Security, and how does it work?

**Answer:**

**Spring Security** is a powerful, highly customizable authentication and access-control framework. It is the de-facto standard for securing Spring-based applications.

At its core, **Spring Security works through a chain of Servlet Filters**. When a request enters your application, it must pass through this chain before reaching your `@Controller`.

---

### a. High-Level Architecture
The entry point is a standard Servlet filter called the **`DelegatingFilterProxy`**. It bridges the gap between the Servlet container (Tomcat, Jetty) and the Spring application context, delegating the work to a Spring-managed bean called **`FilterChainProxy`**.

```text
Request ──▶ DelegatingFilterProxy ──▶ FilterChainProxy (Virtual Filter Chain) ──▶ Controller
```

The `FilterChainProxy` contains one or more **`SecurityFilterChain`** beans. Each filter in this chain has a specific responsibility:
- **`SecurityContextPersistenceFilter`**: Loads/saves the `SecurityContext` across requests.
- **`CsrfFilter`**: Protects against Cross-Site Request Forgery.
- **`UsernamePasswordAuthenticationFilter`**: Handles form-based logins.
- **`BearerTokenAuthenticationFilter`**: Handles JWT/OAuth2 tokens.
- **`AuthorizationFilter`**: Replaces the legacy `FilterSecurityInterceptor`. It checks if the current user has the required roles for the requested URL.

---

### b. How Authentication Works
1.  **Extraction**: A filter (e.g., `UsernamePasswordAuthenticationFilter`) extracts credentials from the request.
2.  **Delegation**: It passes an unauthenticated `Authentication` object to the **`AuthenticationManager`**.
3.  **Verification**: The manager iterates through **`AuthenticationProvider`**s (like `DaoAuthenticationProvider` which uses **`UserDetailsService`**) to verify credentials.
4.  **Storage**: On success, the fully populated `Authentication` object is stored in the **`SecurityContextHolder`**, making it available via `SecurityContextHolder.getContext().getAuthentication()`.

---

### c. Modern Configuration (Spring Security 6+)
Spring Security 6 moved away from the legacy `WebSecurityConfigurerAdapter` in favor of a component-based configuration using the **Lambda DSL**.

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity // Enables @PreAuthorize/@PostAuthorize
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Disable for stateless APIs
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll() // Public endpoints
                .requestMatchers("/api/admin/**").hasRole("ADMIN") // Role-based access
                .anyRequest().authenticated() // All others require login
            )
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS) // REST API best practice
            )
            .httpBasic(Customizer.withDefaults()); // Enable Basic Auth

        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder(); // Industry standard for hashing
    }
}
```

---

### d. Why use Spring Security?
1.  **Decoupling**: Security logic is kept out of your business controllers.
2.  **Comprehensive Protection**: Includes built-in protection against CSRF, Clickjacking, Session Fixation, and XSS.
3.  **Flexibility**: Easily switch between Form Login, Basic Auth, OAuth2, SAML, or custom SSO providers.
4.  **Integration**: Seamlessly integrates with Spring MVC, WebFlux, and Spring Data.

---


## 2. How do Authentication and 
 differ?

**Answer:**

While both are essential for security, they address two fundamentally different questions:

| Feature | **Authentication (AuthN)** | **Authorization (AuthZ)** |
|:---:|---|---|
| **Core Question** | Who are you? | What are you allowed to do? |
| **Input** | Credentials (User/Pass, Token, OTP) | Roles, Permissions, Attributes |
| **Failed Result** | **401 Unauthorized** (No identity) | **403 Forbidden** (Denied permission) |
| **Object** | **`UserDetails`** / **`Principal`** | **`GrantedAuthority`** |

---

### a. Authentication (AuthN)
The process of verifying that an entity is who it claims to be. In Spring Security, this is typically handled by an `AuthenticationProvider` that checks credentials against a source (`Database`, `LDAP`, `OIDC`).

```java
@Bean
public UserDetailsService userDetailsService() {
    // Defining users for authentication
    return new InMemoryUserDetailsManager(
        User.withUsername("alice").password("{noop}pass123").roles("USER").build(),
        User.withUsername("bob").password("{noop}admin123").roles("ADMIN").build()
    );
}
```

---

### b. Authorization (AuthZ)
The process of determining whether an authenticated user has the necessary rights to access a resource. Spring Security provides two levels of control:

#### b-1. URL-Based Authorization
Controls access to specific endpoints in the `SecurityFilterChain`.

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/user/**").hasAnyRole("USER", "ADMIN")
    .anyRequest().authenticated()
);
```

#### b-2. Method-Level Authorization
Uses AOP proxies to secure individual service methods. Requires `@EnableMethodSecurity`.

```java
@Service
public class DocumentService {

    @PreAuthorize("hasRole('ADMIN')")
    public void deleteDocument(Long id) {
        // Only Bob (ADMIN) can call this
    }

    @PreAuthorize("hasAuthority('SCOPE_read') or hasRole('USER')")
    public Document getDocument(Long id) {
        // Alice or a user with specific 'read' authority
        return repository.findById(id).orElseThrow();
    }
}
```

---

### c. Roles vs. Authorities
- **Authorities**: Granular permissions (e.g., `READ_PRIVILEGE`, `DELETE_DOCUMENT`).
- **Roles**: Logical groupings of authorities (e.g., `ROLE_ADMIN`, `ROLE_EDITOR`).
- **Convention**: In Spring Security, `hasRole("ADMIN")` is a shorthand for `hasAuthority("ROLE_ADMIN")`. The `ROLE_` prefix is automatically added by the framework when using the "Role" version of the check.

---


## 3. Explain JWT and OAuth2 in Spring Security

**Answer:**

**JWT** and **OAuth2** are often used together but serve different purposes in the modern security landscape.

---

### a. JWT (JSON Web Token)
JWT is an open standard (**RFC 7519**) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object.

- **Structure**: Consists of three parts separated by dots (`.`): **`Header.Payload.Signature`**.
- **Statelessness**: The server doesn't need to store session data in memory. All user information is stored directly in the token.
- **Security**: The payload is base64 encoded (not encrypted!) but the signature ensures that the data hasn't been tampered with.

#### Minimal JWT Service (using `jjwt` library):
```java
@Service
public class JwtService {
    private static final String SECRET_KEY = "your_secret_key_must_be_very_long...";

    public String generateToken(UserDetails userDetails) {
        return Jwts.builder()
                .subject(userDetails.getUsername())
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + 1000 * 60 * 60)) // 1 hour
                .signWith(getSignInKey())
                .compact();
    }

    public boolean isTokenValid(String token, UserDetails userDetails) {
        final String username = extractUsername(token);
        return (username.equals(userDetails.getUsername())) && !isTokenExpired(token);
    }
}
```

---

### b. OAuth2 (The Protocol)
OAuth2 is an **authorization framework** that enables applications to obtain limited access to user accounts on an HTTP service (e.g., Google, GitHub, or your own Auth Server).

#### b-1. Core Roles:
1.  **Resource Owner**: The User.
2.  **Client**: The application trying to access the user's data.
3.  **Authorization Server**: The server that authenticates the user and issues tokens.
4.  **Resource Server**: The API that stores the user's data (protected by Spring Security).

#### b-2. Common Flow: Authorization Code
The most secure flow for web/mobile apps. The user logs in to the Auth Server, which returns a **Code**. The Client then exchanges this code for an **Access Token**.

---

### c. Spring Security Implementation
In a modern microservices architecture, you typically configure your API as an **OAuth2 Resource Server** that validates JWTs issued by an external provider (like Keycloak, Auth0, or Okta).

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf(AbstractHttpConfigurer::disable)
        .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/public/**").permitAll()
            .anyRequest().authenticated()
        )
        // Configure app as an OAuth2 Resource Server
        .oauth2ResourceServer(oauth2 -> oauth2
            .jwt(jwt -> jwt.decoder(jwtDecoder()))
        );

    return http.build();
}
```

---

### Relationship: JWT in OAuth2
OAuth2 is the **protocol** describing how to get the key. JWT is the **format** of the key itself. Most modern OAuth2 implementations (like OpenID Connect) use JWT as the format for Access and Identity tokens.

---


## 4. How do you secure REST APIs in Spring Boot?

**Answer:**

Securing a REST API requires a **Defense in Depth** approach, applying multiple security layers to ensure that if one layer fails, others remain to protect the system.

---

### a. The Layered Security Model

#### a-1. Transport Layer (Privacy & Integrity)
- **HTTPS/TLS**: Always use SSL/TLS to encrypt data in transit. In Spring Security, you can force HTTPS:
```java
http.requiresChannel(chan -> chan.anyRequest().requiresSecure());
```

#### a-2. Network & Perimeter (Rate Limiting & CORS)
- **CORS (Cross-Origin Resource Sharing)**: Restrict which domains can call your API.
- **Rate Limiting**: Prevent DDoS and brute force attacks using **Bucket4j** or a Cloud Gateway layer.

#### a-3. Authentication Layer (Identity)
- **Statelessness**: REST APIs should be stateless (no sessions). Use **JWT (JSON Web Tokens)** or **OAuth2** Bearer tokens.
- **API Keys**: For machine-to-machine (M2M) communication where full OAuth2 might be overkill.

#### a-4. Authorization Layer (Access Control)
- **RBAC (Role-Based Access Control)**: Enforce roles like `ADMIN` or `USER` using `requestMatchers`.
- **ABAC (Attribute-Based Access Control)**: More granular control based on user attributes or resource ownership.

#### a-5. Application Layer (Integrity & Logging)
- **Input Validation**: Use `@Valid` and `@NotNull` (JSR-303) to prevent malicious payloads.
- **Error Handling**: Mask internal stack traces. Don't reveal "User not found"; use generic "Invalid credentials".

---

### b. Typical JWT Authentication Filter
The heart of a stateless REST API is the `OncePerRequestFilter`, which intercepts every request to check for a valid Bearer token.

```java
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest request, 
                                    HttpServletResponse response, 
                                    FilterChain filterChain) throws ServletException, IOException {
        final String authHeader = request.getHeader("Authorization");
        
        // 1. Check for Bearer token
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        final String jwt = authHeader.substring(7);
        final String username = jwtService.extractUsername(jwt);

        // 2. Validate and set SecurityContext
        if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            UserDetails userDetails = this.userDetailsService.loadUserByUsername(username);
            if (jwtService.isTokenValid(jwt, userDetails)) {
                UsernamePasswordAuthenticationToken authToken = new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());
                authToken.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }
        filterChain.doFilter(request, response);
    }
}
```

---

### c. Advanced Security Checkpoints
- **CORS Config Bean**: Define allowed origins, methods, and headers specifically.
- **Sensitive Data Masking**: Use `@JsonIgnore` or DTOs to avoid leaking PII or internal IDs.
- **Secrets Management**: Never commit `jwt.secret` to Git. Use **Spring Cloud Vault** or Environment Variables.
- **Token Rotation**: Implement **Refresh Tokens** so that short-lived Access Tokens (e.g., 15 mins) don't force users to re-login frequently.

## 5. What is CSRF, and how does Spring Security protect against it?

**Answer:**

**CSRF (Cross-Site Request Forgery)** is an attack that tricks an authenticated user's browser into sending an unwanted request to a web application. Because the browser automatically includes cookies (like session IDs) with every request to the target domain, the server incorrectly assumes the request is intentional.

---

### a. How Spring Security Protects: The Synchronizer Token Pattern
Spring Security uses the **Synchronizer Token Pattern** to defend against CSRF:
1.  The server generates a unique, cryptographically strong **CSRF Token** for the user's session.
2.  For every "state-changing" request (POST, PUT, DELETE, PATCH), the application requires the token to be present in the request (as a hidden form field or a header).
3.  The server compares the token in the request with the one stored in the session/cookie. If they don't match, the request is rejected with a **403 Forbidden**.

---

### b. Implementation by Architecture

#### b-1. Server-Side Rendering (Thymeleaf, JSP)
Spring Security automatically includes the CSRF token in any form generated by Thymeleaf's `th:action`.

```html
<!-- Thymeleaf automatically adds this hidden field -->
<form th:action="@{/transfer}" method="post">
  <input type="hidden" name="_csrf" th:value="f1e2-d3c4-b5a6..." />
  <button type="submit">Transfer Money</button>
</form>
```

#### b-2. Single Page Applications (SPA: React, Angular, Vue)
For SPAs, the server typically sends the token as a cookie, which the JavaScript client reads and sends back in a custom header (e.g., `X-XSRF-TOKEN`).

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf
            .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()) // Allows JS to read the cookie
            .csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())
        );
    return http.build();
}
```

---

### c. When to Enable vs. Disable CSRF

| Scenarios | Status | Reasoning |
|---|---|---|
| **Web App (Browsers/Cookies)** | **Enable** | Essential if you use session-based authentication (cookies). |
| **Stateless API (JWT)** | **Disable** | Stateless APIs (which use `Bearer` tokens in headers) are inherently immune to CSRF because headers are not automatically sent by the browser. |
| **Non-Browser Clients** | **Disable** | Mobile apps and CLI tools don't have the "automatic cookie" behavior of browsers. |

---

### d. Additional Defense: SameSite Cookie Attribute
Modern browsers support the **`SameSite`** attribute for cookies (`Strict` or `Lax`), which instructs the browser not to send cookies with cross-site requests. Spring Security 6+ configures `SameSite=Lax` by default, providing an extra layer of protection even if CSRF tokens aren't used correctly.

## 6. What is the JWT structure, and what are Access Tokens vs. Refresh Tokens?

**Answer:**

---

### a. JWT Structure
A JWT consists of three Base64URL-encoded parts separated by dots:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9   ← Header
.eyJzdWIiOiJhbGljZSIsInJvbGUiOiJVU0VSIiwiaWF0IjoxNzEwMDAwMDAwLCJleHAiOjE3MTAwMDM2MDB9  ← Payload
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c  ← Signature
```

| Part | Content | Purpose |
|---|---|---|
| **Header** | Algorithm (`alg`: HS256, RS256) + type | Defines signing algorithm |
| **Payload** | Claims: `sub`, `iat`, `exp`, `roles`, custom fields | Carries user information |
| **Signature** | `HMAC(base64(header) + "." + base64(payload), secret)` | Ensures data integrity |

**Important**: The payload is **Base64-encoded, not encrypted**. Never put passwords or sensitive PII in a JWT payload.

---

### b. Standard Claims (Registered)
| Claim | Name | Description |
|---|---|---|
| `sub` | Subject | User identifier |
| `iat` | Issued At | Unix timestamp when issued |
| `exp` | Expiration | Unix timestamp when it expires |
| `iss` | Issuer | Who issued the token (e.g., `auth.myapp.com`) |
| `aud` | Audience | Intended recipient (e.g., `api.myapp.com`) |
| `jti` | JWT ID | Unique token ID (used for revocation) |

---

### c. Access Token vs. Refresh Token

| Property | Access Token | Refresh Token |
|---|---|---|
| **Purpose** | Authenticate API requests | Obtain a new Access Token |
| **Lifetime** | Short (5–15 minutes) | Long (7–30 days) |
| **Sent to** | Every protected API endpoint | Only to `/auth/refresh` endpoint |
| **Storage** | Memory (in-app) or short-lived cookie | `HttpOnly` secure cookie or secure storage |
| **Revocable** | Hard (stateless) | Yes (stored server-side) |

**Token rotation flow:**
```
1. User logs in  →  Server issues: Access Token (15 min) + Refresh Token (7 days)
2. Client uses Access Token for API calls until it expires
3. On 401 response  →  Client sends Refresh Token to /auth/refresh
4. Server validates Refresh Token  →  Issues new Access Token + rotates Refresh Token
5. Old Refresh Token is invalidated (rotation prevents reuse attacks)
```

**Spring implementation — refresh endpoint:**
```java
@PostMapping("/auth/refresh")
public ResponseEntity<TokenResponse> refresh(@CookieValue("refresh_token") String refreshToken) {
    return refreshTokenService.findByToken(refreshToken)
        .filter(rt -> !rt.isExpired() && !rt.isRevoked())
        .map(rt -> {
            refreshTokenService.rotate(rt); // Invalidate old, issue new
            String newAccessToken = jwtService.generateToken(rt.getUser());
            return ResponseEntity.ok(new TokenResponse(newAccessToken));
        })
        .orElseThrow(() -> new InvalidTokenException("Refresh token invalid or expired"));
}
```

## 7. How do Cookies work in web security, and what are their security attributes?

**Answer:**

A **cookie** is a small piece of data the server sends to the browser via the `Set-Cookie` response header. The browser stores it and automatically sends it back with every subsequent request to the same domain.

---

### a. Cookie Security Attributes

| Attribute | Purpose | Example |
|---|---|---|
| `HttpOnly` | Prevents JavaScript (`document.cookie`) from reading the cookie — blocks XSS theft | `HttpOnly` |
| `Secure` | Cookie is only sent over HTTPS connections | `Secure` |
| `SameSite` | Controls cross-site sending behaviour — mitigates CSRF | `SameSite=Strict` |
| `Domain` | Which domain the cookie is sent to | `Domain=myapp.com` |
| `Path` | Which URL paths the cookie is sent for | `Path=/api` |
| `Max-Age` / `Expires` | Cookie lifetime; omitting makes it a session cookie | `Max-Age=3600` |

**Ideal secure cookie header:**
```
Set-Cookie: refresh_token=<value>; HttpOnly; Secure; SameSite=Strict; Path=/auth/refresh; Max-Age=604800
```

---

### b. SameSite Values Explained

| Value | Behaviour | Use Case |
|---|---|---|
| `Strict` | Cookie never sent on cross-site requests | Highest security; may break OAuth redirects |
| `Lax` | Cookie sent on top-level navigations (GET links), not POST | Good balance; Spring Security 6+ default |
| `None` | Cookie sent on all cross-site requests | Requires `Secure`; needed for embedded iframes/SPAs on different domains |

---

### c. Cookie-Based Session Auth vs. Token-Based Auth

| Aspect | Cookie + Session | JWT Bearer Token |
|---|---|---|
| **State** | Server stores session | Stateless (client holds token) |
| **CSRF risk** | Yes (mitigated with CSRF tokens) | No (header not auto-sent by browser) |
| **XSS risk** | Low (`HttpOnly` cookie) | High if stored in `localStorage` |
| **Scalability** | Harder (shared session store) | Easier (no server state) |
| **Revocation** | Immediate (delete session) | Hard (wait for expiry) |

**Spring Security — setting a secure cookie for refresh token:**
```java
@PostMapping("/auth/login")
public ResponseEntity<AccessTokenResponse> login(@RequestBody LoginRequest request,
                                                  HttpServletResponse response) {
    // ... authenticate user ...
    String refreshToken = refreshTokenService.create(user);

    ResponseCookie cookie = ResponseCookie.from("refresh_token", refreshToken)
        .httpOnly(true)
        .secure(true)
        .sameSite("Strict")
        .path("/auth/refresh")
        .maxAge(Duration.ofDays(7))
        .build();

    response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());
    return ResponseEntity.ok(new AccessTokenResponse(accessToken));
}
```

---

### d. Session Fixation Attack & Spring's Defense
**Session Fixation**: An attacker tricks a user into using a known session ID. After the user logs in, the attacker uses that same ID to hijack the session.

Spring Security automatically mitigates this by **issuing a new session ID** after authentication:
```java
http.sessionManagement(session -> session
    .sessionFixation(fix -> fix.newSession()) // Default: migrateSession
);
```

## 8. What are the main Authentication Design Patterns used in Spring Security?

**Answer:**

Modern applications use a variety of authentication patterns depending on the use case (web apps, APIs, microservices, M2M).

---

### Pattern 1: Form-Based Authentication (Session + Cookie)
**Use case**: Traditional server-rendered web applications (Thymeleaf, JSP).

**How it works**: User submits a login form → server validates credentials → creates a session → sends `JSESSIONID` cookie.

```java
http
    .formLogin(form -> form
        .loginPage("/login")
        .defaultSuccessUrl("/dashboard", true)
        .failureUrl("/login?error")
        .permitAll()
    )
    .logout(logout -> logout
        .logoutUrl("/logout")
        .logoutSuccessUrl("/login?logout")
        .deleteCookies("JSESSIONID")
    );
```

**Pros**: Simple, browser-native, built-in CSRF protection.  
**Cons**: Not suitable for APIs, stateful (session store needed for scale).

---

### Pattern 2: HTTP Basic Authentication
**Use case**: Machine-to-machine (M2M), internal APIs, development/testing.

**How it works**: Credentials are Base64-encoded and sent in every request header: `Authorization: Basic dXNlcjpwYXNz`.

```java
http.httpBasic(Customizer.withDefaults());
```

**Pros**: Simple, no session needed.  
**Cons**: Credentials sent with every request (requires HTTPS), no logout mechanism.

---

### Pattern 3: JWT Bearer Token Authentication
**Use case**: Stateless REST APIs, microservices, mobile backends.

**How it works**: User logs in once → receives a signed JWT → sends it in every request as `Authorization: Bearer <token>`.

```java
http.oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()));
```

**Pros**: Stateless, scalable, cross-domain friendly.  
**Cons**: Token revocation is hard; payload is visible (not encrypted).

---

### Pattern 4: OAuth2 / OpenID Connect (OIDC)
**Use case**: Social login (Google, GitHub), enterprise SSO, delegated access.

**How it works**: User authenticates with an **Authorization Server** (Keycloak, Auth0). The app receives an Access Token and (for OIDC) an ID Token.

```java
http.oauth2Login(oauth2 -> oauth2
    .loginPage("/oauth2/authorization/google")
    .defaultSuccessUrl("/dashboard")
);
```

**Pros**: Delegates identity management, supports SSO, standardized.  
**Cons**: Complex setup, external dependency.

---

### Pattern 5: API Key Authentication
**Use case**: Third-party integrations, public APIs, developer portals.

**How it works**: Client includes a pre-shared key in every request header (e.g., `X-API-Key: abc123`). A custom filter validates it.

```java
public class ApiKeyFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        String apiKey = request.getHeader("X-API-Key");
        if (apiKeyService.isValid(apiKey)) {
            SecurityContextHolder.getContext().setAuthentication(
                apiKeyService.getAuthentication(apiKey));
        }
        chain.doFilter(request, response);
    }
}
```

**Pros**: Simple, easy for programmatic clients.  
**Cons**: No user identity, key rotation is manual, no expiry by default.

---

### Comparison Summary

| Pattern | Stateless | User Identity | Best For |
|---|---|---|---|
| Form-Based | No | Yes | Web apps (UI) |
| HTTP Basic | Yes | Yes | Internal tools / dev |
| JWT Bearer | Yes | Yes | REST APIs |
| OAuth2 / OIDC | Yes | Yes (OIDC) | SSO / Social Login |
| API Key | Yes | No (app-level) | M2M integrations |

---


## 9. What are the main Authorization Design Patterns, and how are they implemented?

**Answer:**

Authorization determines **what an authenticated user is allowed to do**. Several design patterns exist, each with different levels of granularity and complexity.

---

### Pattern 1: RBAC — Role-Based Access Control
**Concept**: Permissions are grouped into **Roles**. Users are assigned Roles, not individual permissions.

```
User → Role(s) → Permissions
Alice → ROLE_EDITOR → [READ, WRITE]
Bob   → ROLE_ADMIN  → [READ, WRITE, DELETE]
```

**Spring implementation:**
```java
// URL-level RBAC
http.authorizeHttpRequests(auth -> auth
    .requestMatchers(HttpMethod.DELETE, "/articles/**").hasRole("ADMIN")
    .requestMatchers(HttpMethod.POST, "/articles/**").hasAnyRole("ADMIN", "EDITOR")
    .requestMatchers(HttpMethod.GET, "/articles/**").permitAll()
);

// Method-level RBAC
@PreAuthorize("hasRole('ADMIN')")
public void deleteArticle(Long id) { ... }
```

**Pros**: Simple to manage, well-understood.  
**Cons**: Role explosion for complex systems; not fine-grained enough when resource ownership matters.

---

### Pattern 2: ABAC — Attribute-Based Access Control
**Concept**: Access is controlled based on **attributes** of the user, resource, and environment (time, location, IP).

```
ALLOW if: user.department == resource.department AND environment.time < 18:00
```

**Spring implementation using SpEL in `@PreAuthorize`:**
```java
@PreAuthorize("#document.owner == authentication.name or hasRole('ADMIN')")
public Document getDocument(Document document) {
    return document;
}

// Custom security evaluator
@PreAuthorize("@documentSecurity.canEdit(authentication, #documentId)")
public void updateDocument(Long documentId, DocumentDto dto) { ... }
```

```java
@Component("documentSecurity")
public class DocumentSecurityEvaluator {
    public boolean canEdit(Authentication auth, Long documentId) {
        Document doc = documentRepository.findById(documentId).orElseThrow();
        return doc.getOwner().equals(auth.getName()) || 
               auth.getAuthorities().stream()
                   .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"));
    }
}
```

**Pros**: Highly flexible, fine-grained.  
**Cons**: Complex to design and maintain.

---

### Pattern 3: PBAC — Policy-Based Access Control
**Concept**: Access decisions are driven by explicit **policies** expressed in a policy language (e.g., OPA — Open Policy Agent). Policies are externalized from the application code.

```rego
# OPA Policy
allow {
    input.method == "DELETE"
    input.user.role == "admin"
    input.resource.type == "article"
}
```

**Integration with Spring:**
```java
@Component
public class OpaAuthorizationManager<T> implements AuthorizationManager<T> {
    private final WebClient opaClient;

    @Override
    public AuthorizationDecision check(Supplier<Authentication> authentication, T object) {
        OpaRequest opaRequest = buildRequest(authentication.get(), object);
        Boolean allowed = opaClient.post()
            .uri("/v1/data/authz/allow")
            .bodyValue(opaRequest)
            .retrieve()
            .bodyToMono(OpaResponse.class)
            .map(OpaResponse::isResult)
            .block();
        return new AuthorizationDecision(Boolean.TRUE.equals(allowed));
    }
}
```

**Pros**: Decoupled, auditable, consistent across services.  
**Cons**: External dependency, network latency per request.

---

### Pattern 4: ReBAC — Relationship-Based Access Control
**Concept**: Access is determined by the **relationship** between the user and the resource (inspired by Google Zanzibar). Used by Google Drive, GitHub.

```
Alice → owner   → Document#123  → can READ, WRITE, DELETE
Bob   → viewer  → Document#123  → can READ only
Carol → editor  → Document#123  → can READ, WRITE
```

**Conceptual Spring implementation:**
```java
@PreAuthorize("@relationshipService.hasRelationship(authentication.name, #docId, 'editor')")
public void editDocument(Long docId, DocumentDto dto) { ... }
```

**Pros**: Intuitive for collaborative/hierarchical resources.  
**Cons**: Complex to implement; requires a specialized store (e.g., SpiceDB, Ory Keto).

---

### Pattern 5: ACL — Access Control List (Spring Security ACL)
**Concept**: Each individual resource (domain object) has an explicit list of which users/roles can perform which operations on it.

```
Document#42: [alice: READ+WRITE, bob: READ, ROLE_ADMIN: READ+WRITE+DELETE]
```

**Spring Security ACL setup:**
```java
// Grant permission programmatically
MutableAcl acl = aclService.createAcl(new ObjectIdentityImpl(Document.class, documentId));
acl.insertAce(acl.getEntries().size(), BasePermission.READ,
    new PrincipalSid("alice"), true);
acl.insertAce(acl.getEntries().size(), BasePermission.WRITE,
    new PrincipalSid("alice"), true);
aclService.updateAcl(acl);

// Method-level enforcement
@PreAuthorize("hasPermission(#document, 'READ')")
public Document getDocument(Document document) {
    return document;
}
```

**Pros**: Very fine-grained per-object control.  
**Cons**: Database overhead per resource; complex to manage at scale.

---

### Authorization Patterns Comparison

| Pattern | Granularity | Complexity | Best For |
|---|---|---|---|
| **RBAC** | Role level | Low | Most enterprise apps |
| **ABAC** | Attribute level | Medium | Context-aware access |
| **PBAC** | Policy level | High | Multi-service / audit-heavy |
| **ReBAC** | Relationship level | High | Collaborative platforms |
| **ACL** | Per-object | Medium | Document/file systems |

## 10. How does Spring Security handle Security Context propagation in async and reactive applications?

**Answer:**

The **`SecurityContextHolder`** stores the currently authenticated user's `Authentication` in a `ThreadLocal` variable by default. This works perfectly for synchronous, blocking code, but breaks in async contexts where code runs on different threads.

---

### a. The Problem: ThreadLocal and Async

```java
@GetMapping("/profile")
public CompletableFuture<String> getProfile() {
    // ❌ This runs on a different thread — SecurityContext is NULL here
    return CompletableFuture.supplyAsync(() -> {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        return auth.getName(); // NullPointerException!
    });
}
```

---

### b. Solutions for Async (Spring MVC)

#### b-1. Use `DelegatingSecurityContextExecutor`
Wraps an `Executor` to propagate the `SecurityContext` to child threads:

```java
@Bean
public Executor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.initialize();
    return new DelegatingSecurityContextExecutorService(executor.getThreadPoolExecutor());
}
```

#### b-2. `@Async` with `SecurityContextHolder` strategy
Configure `SecurityContextHolder` to use the inheritable thread local strategy:

```java
SecurityContextHolder.setStrategyName(SecurityContextHolder.MODE_INHERITABLETHREADLOCAL);
```

**Caution**: `MODE_INHERITABLETHREADLOCAL` can leak the context to thread-pool threads that outlive the request.

#### b-3. Explicit context propagation (recommended):
```java
@GetMapping("/profile")
public CompletableFuture<String> getProfile() {
    SecurityContext context = SecurityContextHolder.getContext();
    return CompletableFuture.supplyAsync(
        SecurityContextCallable.create(context, () -> {
            return context.getAuthentication().getName();
        })
    );
}
```

---

### c. Security Context in Spring WebFlux (Reactive)

In WebFlux, the `SecurityContext` is stored in the **Reactor Context** (not `ThreadLocal`), because reactive pipelines may jump across threads.

**Accessing the current user in WebFlux:**
```java
@GetMapping("/profile")
public Mono<String> getProfile() {
    return ReactiveSecurityContextHolder.getContext()
        .map(ctx -> ctx.getAuthentication().getName());
}
```

**In a method:**
```java
@GetMapping("/orders")
public Flux<Order> getMyOrders() {
    return ReactiveSecurityContextHolder.getContext()
        .map(ctx -> ctx.getAuthentication().getName())
        .flatMapMany(username -> orderRepository.findByUsername(username));
}
```

---

### d. Security Context in WebFlux Security Config

```java
@Bean
public SecurityWebFilterChain springSecurityFilterChain(ServerHttpSecurity http) {
    return http
        .csrf(ServerHttpSecurity.CsrfSpec::disable)
        .authorizeExchange(auth -> auth
            .pathMatchers("/api/public/**").permitAll()
            .anyExchange().authenticated()
        )
        .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()))
        .build();
}
```

---

### e. Summary: Context Propagation Strategies

| Context | Storage | Access |
|---|---|---|
| **Sync (Spring MVC)** | `ThreadLocal` | `SecurityContextHolder.getContext()` |
| **Async (`@Async`, `CompletableFuture`)** | Delegating executor / inheritable TL | `DelegatingSecurityContextExecutor` |
| **Reactive (WebFlux)** | Reactor Context | `ReactiveSecurityContextHolder.getContext()` |
