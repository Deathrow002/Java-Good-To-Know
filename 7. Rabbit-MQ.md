# RabbitMQ Interview Questions & Answers

---

## Q1. What is RabbitMQ and what problem does it solve?

**Answer:**
RabbitMQ is an open-source message broker that implements the **AMQP (Advanced Message Queuing Protocol)**. It acts as an intermediary for messaging between producers (applications that send messages) and consumers (applications that receive messages).

**Problems it solves:**
- **Decoupling** — producers and consumers don't need to know about each other.
- **Asynchronous processing** — producers can continue working without waiting for consumers to process the message.
- **Load leveling** — message queues absorb spikes in traffic, preventing consumer overload.
- **Reliability** — messages can be persisted and redelivered if a consumer fails.

---

## Q2. Explain the core components of RabbitMQ.

**Answer:**

| Component      | Description |
|----------------|-------------|
| **Producer**   | Application that publishes messages to RabbitMQ. |
| **Consumer**   | Application that subscribes to a queue and processes messages. |
| **Queue**      | A buffer that stores messages until a consumer picks them up. |
| **Exchange**   | Receives messages from producers and routes them to queues based on routing rules. |
| **Binding**    | A link between an exchange and a queue, optionally with a routing key. |
| **Routing Key**| A message attribute the exchange uses to decide how to route the message. |
| **Virtual Host (vhost)** | A logical grouping providing isolation between applications. |
| **Channel**    | A lightweight connection inside a TCP connection used for communication. |

## Q3. What are the different types of exchanges in RabbitMQ?

**Answer:**
RabbitMQ provides four built-in exchange types:

1. **Direct Exchange** — Routes messages to queues whose binding key **exactly matches** the routing key of the message.
   - Use case: Task distribution to specific workers.

2. **Fanout Exchange** — Routes messages to **all** bound queues, ignoring the routing key.
   - Use case: Broadcasting events (e.g., notifications, pub/sub).

3. **Topic Exchange** — Routes messages based on **wildcard pattern** matching on the routing key.
   - `*` matches exactly one word; `#` matches zero or more words.
   - Use case: Categorized logging (`logs.info`, `logs.error.*`).

4. **Headers Exchange** — Routes messages based on **message header attributes** instead of the routing key.
   - Use case: Complex routing scenarios based on multiple criteria.

## Q4. What is the difference between a durable queue and a transient queue?

**Answer:**

| Feature         | Durable Queue                         | Transient Queue                     |
|-----------------|---------------------------------------|-------------------------------------|
| **Persistence** | Survives broker restart               | Lost when broker restarts           |
| **Declaration** | `durable = true`                      | `durable = false`                   |
| **Use case**    | Critical tasks that must not be lost  | Temporary or ephemeral workloads    |

> **Note:** A durable queue alone does **not** guarantee message persistence. Messages must also be published with `deliveryMode = 2` (persistent) to survive a restart.

## Q5. What is message acknowledgement (ACK) in RabbitMQ and why is it important?

**Answer:**
**Message acknowledgement** is a mechanism by which a consumer notifies RabbitMQ that a message has been successfully processed.

- **`basicAck`** — Confirms the message was processed. RabbitMQ removes it from the queue.
- **`basicNack` / `basicReject`** — Signals failure. The message can be requeued or sent to a Dead Letter Exchange (DLX).

**Why it matters:**
- Without ACKs, if a consumer crashes after receiving but before processing a message, that message is **lost**.
- With ACKs, RabbitMQ keeps the message in an *unacknowledged* state and redelivers it if the consumer dies.

**Auto-ACK vs Manual ACK:**
- `autoAck = true` — Message is removed immediately on delivery (risk of data loss).
- `autoAck = false` — Application explicitly calls `basicAck` after processing (recommended for reliability).

## Q6. What is a Dead Letter Exchange (DLX) in RabbitMQ?

**Answer:**
A **Dead Letter Exchange (DLX)** is a special exchange to which messages are routed when they cannot be processed normally.

**A message becomes a dead letter when:**
1. It is **rejected** (`basicNack` / `basicReject` with `requeue = false`).
2. Its **TTL (Time-To-Live)** expires.
3. The **queue length limit** is exceeded.

**Configuration (Spring Boot example):**
```java
@Bean
Queue mainQueue() {
    return QueueBuilder.durable("main.queue")
        .withArgument("x-dead-letter-exchange", "dlx.exchange")
        .withArgument("x-message-ttl", 60000)
        .build();
}

@Bean
Queue deadLetterQueue() {
    return QueueBuilder.durable("dead.letter.queue").build();
}
```

**Use case:** Auditing failed messages, alerting on processing errors, retry logic.

## Q7. How does RabbitMQ handle message routing with Topic Exchange? Give an example.

**Answer:**
A **Topic Exchange** routes messages using routing keys with dot-separated words and wildcard patterns:
- `*` — matches **exactly one** word.
- `#` — matches **zero or more** words.

**Example:**

| Binding Key       | Matches Routing Key Examples                     |
|-------------------|--------------------------------------------------|
| `order.*`         | `order.created`, `order.updated` (not `order.item.deleted`) |
| `order.#`         | `order.created`, `order.item.deleted`, `order` |
| `*.error`         | `app.error`, `db.error` (not `app.db.error`)    |
| `#`               | Any routing key                                  |

```java
// Producer
rabbitTemplate.convertAndSend("topic.exchange", "order.created", message);

// Consumer binding
@RabbitListener(bindings = @QueueBinding(
    value = @Queue("order.queue"),
    exchange = @Exchange(value = "topic.exchange", type = ExchangeTypes.TOPIC),
    key = "order.*"
))
public void handleOrder(String message) { ... }
```

## Q8. What is the prefetch count (QoS) in RabbitMQ and how does it affect performance?

**Answer:**
**Prefetch count** (Quality of Service) controls **how many unacknowledged messages** RabbitMQ delivers to a consumer at a time before waiting for ACKs.

```java
// Spring Boot configuration
@Bean
public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory(
        ConnectionFactory connectionFactory) {
    SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    factory.setPrefetchCount(10); // consumer gets max 10 unacked messages at once
    return factory;
}
```

**Impact:**
- **prefetchCount = 1** — Fair dispatch; each consumer gets one message at a time. Best for tasks with variable processing time but lowest throughput.
- **prefetchCount = high** — Higher throughput but risk of one consumer hoarding messages while others are idle.
- **prefetchCount = 0** — Unlimited (no QoS), consumer receives all messages at once — not recommended.

## Q9. How do you integrate RabbitMQ with Spring Boot?

**Answer:**

**1. Add dependency (`pom.xml`):**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-amqp</artifactId>
</dependency>
```

**2. Configure properties (`application.yml`):**
```yaml
spring:
  rabbitmq:
    host: localhost
    port: 5672
    username: guest
    password: guest
    virtual-host: /
```

**3. Declare exchange, queue, and binding:**
```java
@Configuration
public class RabbitMQConfig {

    @Bean
    public Queue orderQueue() {
        return QueueBuilder.durable("order.queue").build();
    }

    @Bean
    public DirectExchange orderExchange() {
        return new DirectExchange("order.exchange");
    }

    @Bean
    public Binding binding(Queue orderQueue, DirectExchange orderExchange) {
        return BindingBuilder.bind(orderQueue).to(orderExchange).with("order.routing.key");
    }
}
```

**4. Produce a message:**
```java
@Service
public class OrderProducer {
    @Autowired
    private RabbitTemplate rabbitTemplate;

    public void sendOrder(Order order) {
        rabbitTemplate.convertAndSend("order.exchange", "order.routing.key", order);
    }
}
```

**5. Consume a message:**
```java
@Service
public class OrderConsumer {

    @RabbitListener(queues = "order.queue")
    public void processOrder(Order order) {
        System.out.println("Processing order: " + order.getId());
    }
}
```

## Q10. What strategies exist for implementing retry logic in RabbitMQ?

**Answer:**
RabbitMQ does not have built-in retry scheduling, but several strategies are commonly used:

### Strategy 1: Spring Retry with `@RabbitListener`
```java
@Bean
public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory(
        ConnectionFactory connectionFactory) {
    SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    factory.setAdviceChain(RetryInterceptorBuilder.stateless()
        .maxAttempts(3)
        .backOffOptions(1000, 2.0, 10000) // initial, multiplier, max
        .build());
    return factory;
}
```

### Strategy 2: Dead Letter Exchange + TTL (Delayed Retry Queue)
Route failed messages to a DLX with a TTL, then re-route them back to the original queue after the delay:

```
Main Queue --> (failure) --> DLX --> Retry Queue (TTL=5s) --> (TTL expired) --> Main Queue
```

```java
@Bean
Queue retryQueue() {
    return QueueBuilder.durable("retry.queue")
        .withArgument("x-dead-letter-exchange", "order.exchange")
        .withArgument("x-dead-letter-routing-key", "order.routing.key")
        .withArgument("x-message-ttl", 5000) // retry after 5 seconds
        .build();
}
```

### Strategy 3: RabbitMQ Delayed Message Plugin
Use the `x-delayed-message` exchange type from the **rabbitmq_delayed_message_exchange** plugin for fine-grained scheduling.

**Best practice:** Combine retries with a DLQ — after N retries, route to the Dead Letter Queue for manual inspection or alerting.
