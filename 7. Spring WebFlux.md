# Spring WebFlux — Interview Questions & Answers

---

### Q1. What is Spring WebFlux, and how does it differ from Spring MVC?

**Answer:**
Spring WebFlux is a reactive, non-blocking web framework introduced in Spring 5. It is built on **Project Reactor** and supports reactive streams (using `Mono` and `Flux`). It is designed for high-concurrency scenarios with fewer threads.

| Feature | Spring MVC | Spring WebFlux |
|---|---|---|
| Programming model | Imperative (blocking) | Reactive (non-blocking) |
| Thread model | One thread per request | Event loop (fewer threads) |
| Core types | `HttpServletRequest` | `ServerRequest` / `ServerResponse` |
| Reactive types | None | `Mono<T>`, `Flux<T>` |
| Server support | Tomcat, Jetty | Netty, Undertow, Tomcat |

WebFlux is ideal when building systems that need to handle a large number of concurrent connections efficiently (e.g., streaming, real-time APIs).

### Q2. What are `Mono` and `Flux` in Spring WebFlux?

**Answer:**
Both `Mono` and `Flux` are **Publisher** types from **Project Reactor**, and they represent asynchronous sequences of data.

- **`Mono<T>`** — Emits **0 or 1** item, then completes or errors. Used for single-value responses.
- **`Flux<T>`** — Emits **0 to N** items, then completes or errors. Used for multiple-value streams.

```java
// Mono example
Mono<String> mono = Mono.just("Hello WebFlux");

// Flux example
Flux<Integer> flux = Flux.just(1, 2, 3, 4, 5);
flux.subscribe(System.out::println);
```

**Key operators:**
- `map()` — transform each element
- `flatMap()` — async transform returning a Publisher
- `filter()` — filter elements
- `zip()` — combine multiple Publishers
- `subscribe()` — trigger execution (Mono/Flux are lazy)

### Q3. What is the Reactive Streams specification, and how does Spring WebFlux relate to it?

**Answer:**
**Reactive Streams** is a specification (JVM standard since Java 9 via `java.util.concurrent.Flow`) that defines a standard for asynchronous stream processing with **non-blocking back-pressure**.

It defines four interfaces:
1. `Publisher<T>` — produces data
2. `Subscriber<T>` — consumes data
3. `Subscription` — link between publisher and subscriber (request/cancel)
4. `Processor<T, R>` — both publisher and subscriber

**Spring WebFlux** is built on top of **Project Reactor**, which is a fully compliant implementation of the Reactive Streams specification. `Mono` and `Flux` both implement `Publisher<T>`.

Back-pressure allows the **subscriber to control the rate** at which it receives data from the publisher, preventing it from being overwhelmed.

### Q4. What is back-pressure, and how is it handled in Spring WebFlux?

**Answer:**
**Back-pressure** is a mechanism that allows a **downstream subscriber** to signal to an **upstream publisher** how many items it can handle, preventing the subscriber from being overwhelmed.

In Spring WebFlux (Project Reactor):
- The `Subscriber` requests a specific number of items via `Subscription.request(n)`.
- The `Publisher` only emits as many items as requested.

**Handling strategies in Reactor:**
```java
// Buffer: buffer overflow items
flux.onBackpressureBuffer(100);

// Drop: drop items that can't be processed
flux.onBackpressureDrop(dropped -> log.warn("Dropped: {}", dropped));

// Latest: keep only the latest item
flux.onBackpressureLatest();

// Error: signal error when overwhelmed
flux.onBackpressureError();
```

Back-pressure is critical in streaming scenarios like reading from a database or consuming a message queue.

### Q5. What are the two programming models available in Spring WebFlux?

**Answer:**
Spring WebFlux supports two distinct programming models:

**1. Annotated Controllers (same as Spring MVC)**
Uses the familiar `@RestController`, `@GetMapping`, etc., but returns reactive types:
```java
@RestController
@RequestMapping("/users")
public class UserController {

    @GetMapping("/{id}")
    public Mono<User> getUser(@PathVariable Long id) {
        return userService.findById(id);
    }

    @GetMapping
    public Flux<User> getAllUsers() {
        return userService.findAll();
    }
}
```

**2. Functional Endpoints (Router Functions)**
Uses `RouterFunction` and `HandlerFunction` for a more functional style:
```java
@Bean
public RouterFunction<ServerResponse> routes(UserHandler handler) {
    return RouterFunctions.route()
        .GET("/users/{id}", handler::getUser)
        .GET("/users", handler::getAllUsers)
        .POST("/users", handler::createUser)
        .build();
}
```

The functional model gives finer control over routing and is fully type-safe.

### Q6. How do you handle errors in Spring WebFlux?

**Answer:**
Spring WebFlux provides several ways to handle errors in reactive pipelines:

**1. `onErrorReturn` — return a fallback value:**
```java
Mono<User> user = userService.findById(id)
    .onErrorReturn(new User("default"));
```

**2. `onErrorResume` — switch to a fallback Publisher:**
```java
Mono<User> user = userService.findById(id)
    .onErrorResume(NotFoundException.class, ex -> Mono.just(new User("guest")));
```

**3. `onErrorMap` — transform exceptions:**
```java
Mono<User> user = userService.findById(id)
    .onErrorMap(DataAccessException.class, ex -> new ServiceException("DB error", ex));
```

**4. `@ExceptionHandler` in a `@ControllerAdvice`:**
```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(NotFoundException.class)
    public ResponseEntity<String> handleNotFound(NotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(ex.getMessage());
    }
}
```

**5. `WebExceptionHandler` (lowest level, global):**
Implement `WebExceptionHandler` bean for infrastructure-level error handling.

### Q7. What is `WebClient`, and how does it compare to `RestTemplate`?

**Answer:**
`WebClient` is the **reactive, non-blocking HTTP client** introduced in Spring 5 to replace the older `RestTemplate`.

| Feature | `RestTemplate` | `WebClient` |
|---|---|---|
| Blocking | Yes | No (non-blocking) |
| Reactive support | No | Yes (`Mono`/`Flux`) |
| Streaming | No | Yes |
| Status | Maintenance mode | Actively supported |

**WebClient example:**
```java
WebClient client = WebClient.create("https://api.example.com");

// GET single resource
Mono<User> user = client.get()
    .uri("/users/{id}", 1)
    .retrieve()
    .bodyToMono(User.class);

// POST resource
Mono<User> created = client.post()
    .uri("/users")
    .bodyValue(new User("John"))
    .retrieve()
    .bodyToMono(User.class);

// Handle errors
Mono<User> result = client.get()
    .uri("/users/{id}", 99)
    .retrieve()
    .onStatus(HttpStatusCode::is4xxClientError, response ->
        Mono.error(new NotFoundException("User not found")))
    .bodyToMono(User.class);
```

`WebClient` should be used in all new Spring applications, especially reactive ones.

### Q8. How does Spring WebFlux integrate with reactive databases?

**Answer:**
Spring WebFlux works with **R2DBC** (Reactive Relational Database Connectivity) for relational databases and **Spring Data Reactive** repositories for NoSQL databases.

**Supported reactive drivers:**
- **R2DBC** — MySQL, PostgreSQL, H2, MS SQL Server (relational)
- **Spring Data MongoDB Reactive** — MongoDB
- **Spring Data Redis Reactive** — Redis
- **Spring Data Cassandra Reactive** — Cassandra

**Example with R2DBC (Spring Data):**
```java
// Repository
public interface UserRepository extends ReactiveCrudRepository<User, Long> {
    Flux<User> findByLastName(String lastName);
}

// Service
@Service
public class UserService {
    private final UserRepository userRepository;

    public Mono<User> findById(Long id) {
        return userRepository.findById(id);
    }

    public Flux<User> findAll() {
        return userRepository.findAll();
    }

    public Mono<User> save(User user) {
        return userRepository.save(user);
    }
}
```

Using blocking JDBC drivers inside WebFlux is strongly discouraged as it blocks the event loop. For blocking calls, wrap with `Schedulers.boundedElastic()`.

### Q9. What are `RouterFunction` and `HandlerFunction` in Spring WebFlux?

**Answer:**
These are the building blocks of the **functional web framework** in Spring WebFlux.

- **`HandlerFunction<T>`** — Takes a `ServerRequest` and returns `Mono<ServerResponse>`. It is the functional equivalent of a controller method.
- **`RouterFunction<T>`** — Maps incoming requests to a `HandlerFunction` based on predicates (path, method, headers, etc.).

**HandlerFunction example:**
```java
@Component
public class UserHandler {
    private final UserService userService;

    public Mono<ServerResponse> getUser(ServerRequest request) {
        Long id = Long.parseLong(request.pathVariable("id"));
        return userService.findById(id)
            .flatMap(user -> ServerResponse.ok().bodyValue(user))
            .switchIfEmpty(ServerResponse.notFound().build());
    }

    public Mono<ServerResponse> createUser(ServerRequest request) {
        return request.bodyToMono(User.class)
            .flatMap(userService::save)
            .flatMap(saved -> ServerResponse.status(HttpStatus.CREATED).bodyValue(saved));
    }
}
```

**RouterFunction example:**
```java
@Configuration
public class UserRouter {

    @Bean
    public RouterFunction<ServerResponse> userRoutes(UserHandler handler) {
        return RouterFunctions.route()
            .GET("/users/{id}", handler::getUser)
            .POST("/users", handler::createUser)
            .build();
    }
}
```

Router functions can be composed and nested, making them suitable for modular route definitions.

### Q10. How do you perform testing in Spring WebFlux using `WebTestClient`?

**Answer:**
`WebTestClient` is the test-oriented counterpart of `WebClient`, designed specifically for testing Spring WebFlux applications without starting a real server (or optionally against a live server).

**Setup options:**

```java
// Bind to a controller (no server needed)
WebTestClient client = WebTestClient
    .bindToController(new UserController())
    .build();

// Bind to the full ApplicationContext
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class UserControllerTest {

    @Autowired
    private WebTestClient webTestClient;
}
```

**Example tests:**
```java
@Test
void shouldReturnUser() {
    webTestClient.get()
        .uri("/users/1")
        .exchange()
        .expectStatus().isOk()
        .expectBody(User.class)
        .value(user -> assertThat(user.getName()).isEqualTo("John"));
}

@Test
void shouldReturnUserList() {
    webTestClient.get()
        .uri("/users")
        .exchange()
        .expectStatus().isOk()
        .expectBodyList(User.class)
        .hasSize(3);
}

@Test
void shouldCreateUser() {
    webTestClient.post()
        .uri("/users")
        .bodyValue(new User("Jane"))
        .exchange()
        .expectStatus().isCreated()
        .expectBody(User.class)
        .value(user -> assertThat(user.getId()).isNotNull());
}
```

`WebTestClient` supports both real HTTP connections and mock server interactions, making it the standard testing tool for Spring WebFlux applications.
