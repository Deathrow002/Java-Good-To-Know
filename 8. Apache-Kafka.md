# Apache Kafka

## 1. What is Apache Kafka?

**Answer:**

Apache Kafka is a distributed, fault-tolerant, high-throughput event streaming platform designed for building real-time data pipelines and streaming applications. It provides a durable, ordered, and partitioned commit log (topics) that allows producers to publish events and consumers to read them at their own pace. Kafka decouples data producers and consumers, supports horizontal scalability via partitions and brokers, and ensures durability and replication for resilience.

Key characteristics:

- **Publish–subscribe + storage:** Topics retain messages for a configurable retention period; consumers track offsets independently.
- **Partitioned & distributed:** Topics are split into partitions for parallelism and scalability; partitions are replicated for fault tolerance.
- **High throughput & low latency:** Optimized for sequential disk I/O and batching to achieve very high message rates.
- **Durability & ordering:** Messages are durably stored on disk and ordering is guaranteed within a partition.
- **Ecosystem & integrations:** Rich client libraries, Kafka Connect for integrations, Kafka Streams and ksqlDB for stream processing, and Schema Registry for schema management.

Common use cases

- **Event sourcing / domain events:** Persist domain events as the source of truth for rebuilding state or auditing.
- **Real-time stream processing:** Transform, aggregate, enrich, and analyze event streams with Kafka Streams, ksqlDB, or external consumers.
- **Change Data Capture (CDC):** Capture DB changes (Debezium) and stream them to downstream systems for replication, analytics, or cache updates.
- **Log aggregation & centralized telemetry:** Collect logs, metrics, and observability data from many services into a central pipeline.
- **Buffering and decoupling:** Smooth bursty traffic between producers and consumers and decouple system components for resilience.
- **Event-driven microservices:** Use Kafka as the backbone for integrating microservices with event-driven communication patterns.
- **Realtime analytics & monitoring:** Feed dashboards, alerts, and ML pipelines with low-latency streaming data.

Example one-line summary: Apache Kafka = durable, distributed commit-log for high-throughput event streaming and real-time data pipelines.

## 2. What are the core components of Kafka architecture?
**Answer:**

Kafka is composed of several tightly related components that together provide a distributed, durable, and scalable event streaming platform:

- **Broker:** A single Kafka server process that stores topic partitions and handles producer/consumer requests. A Kafka cluster is a set of brokers.
- **Topic:** A named log that stores messages of a particular category. Topics are the primary abstraction for organizing streams of data.
- **Partition:** A topic is divided into partitions. Each partition is an ordered, immutable sequence of messages and is the unit of parallelism and scalability.
- **Replica / Leader / Follower:** Each partition is replicated across multiple brokers. One replica is the partition **leader** (handles reads/writes); others are **followers** that replicate the leader. Replication provides fault tolerance.
- **In-Sync Replicas (ISR):** Set of replicas that are fully caught up with the leader. Only ISR members can be elected leader to avoid data loss.
- **Producer:** Client that publishes messages to topics. Producers choose partition placement (key-based, round-robin, or custom partitioner) and can enable idempotence and transactions.
- **Consumer:** Client that reads messages from topics. Consumers belong to **consumer groups** which distribute partitions among group members for parallel consumption.
- **Consumer Group:** Logical group of consumers that coordinate to consume partitions with one consumer reading each partition in the group.
- **Controller / Cluster Metadata Manager:** A broker elected as controller (or controller quorum in KRaft) manages partition leadership, assignment, and administrative operations.
- **ZooKeeper / KRaft (controller quorum):** Historically Kafka used ZooKeeper for cluster metadata and leader election; newer Kafka versions use KRaft (Kafka Raft) to remove the ZooKeeper dependency.
- **Log Segment & Storage:** Partitions are stored as a sequence of segment files on disk. Kafka uses sequential disk writes for high throughput; retention and compaction policies control how long data is kept.
- **Offset:** Numeric position of a message in a partition used by consumers to track progress.
- **Kafka Connect:** Framework for connecting Kafka with external systems (databases, object stores, message systems) via source and sink connectors.
- **Kafka Streams / ksqlDB:** Client libraries for building stream processing applications that consume, process, and produce Kafka records with local state stores and windowing.
- **Schema Registry (optional):** Centralized schema store (e.g., for Avro/Protobuf/JSON Schema) that helps manage schema evolution and compatibility.

How these pieces work together (brief): producers write records to a topic partition on the leader broker; followers asynchronously replicate the records; consumers in a group read partitions assigned to them and commit offsets to track progress. The controller handles leadership and reassignments when brokers fail.

Best practices / notes:

- Choose partition count to balance throughput and consumer parallelism; partitions are hard to change without impact.
- Set appropriate replication factor (typical = 3) and `min.insync.replicas` to ensure durability.
- Use log compaction for changelog-style topics (de-duplicated keys) and retention for time-based buffering.
- Prefer idempotent producers and transactions for exactly-once semantics when required.

Short diagram (logical):

Producer -> [Topic: partition0, partition1, ...] -> Broker cluster (leaders/followers) -> Consumers (consumer groups)

This covers the essential core components and how they fit into Kafka's architecture.

## 3. Describe Kafka Process Lifecycle?

**Answer:**

The lifecycle of a Kafka record involves a journey through three main stages: the **Producer**, the **Broker Cluster**, and the **Consumer**. Each stage is optimized for durability, throughput, and decoupled processing.

---

### Step 1: The Producer (Data Ingestion)
When a producer sends a record, several internal operations occur before the data leaves the client:
1.  **Serialization:** The record's key and value (objects) are converted into byte arrays using the configured serializers (e.g., String, JSON, Avro).
2.  **Partitioning:** If a key is provided, the **Partitioner** hashes it to determine the destination partition. If the key is null, Kafka uses a **Sticky Partitioner** to batch records to the same partition for better throughput.
3.  **Record Accumulator:** Records are not sent one by one. Instead, they are placed in a memory buffer (Accumulator) where they are grouped into **Batches** (by partition).
4.  **Sender Thread:** A dedicated background thread (I/O thread) polls the accumulator, gathers batches for the same broker, and sends them over the network.
5.  **Acknowledgments (acks):** The producer waits for a response based on the `acks` configuration (`0`, `1`, or `all`) before marking the send as successful or triggering a retry.

---

### Step 2: The Broker (Persistence & Replication)
Once the leader broker receives the request, it handles storage and synchronization:
1.  **Append to Log:** The leader appends the byte-level records to the **Active Segment** file of the partition's log on disk. This is a sequential write, making it extremely fast.
2.  **Replication:** Followers in the **In-Sync Replicas (ISR)** set poll the leader for new data and append it to their own local logs.
3.  **Commitment:** Once the message is successfully replicated by all replicas in the ISR, it is considered **Committed**.
4.  **High Watermark (HW):** The leader advances the HW index. Only messages below the HW are visible to consumers, ensuring that consumers only read data that is guaranteed to be durable across replicas.

---

### Step 3: The Consumer (Data Consumption)
The consumer's lifecycle is driven by the pull-based model:
1.  **Polling:** Consumers call `poll()` to request data. The client library handles the low-level fetching from the leader broker (or a local follower in some configurations).
2.  **Unpacking:** The consumer receives a batch of records, which are then de-serialized back into Java objects.
3.  **Processing:** The application processes the records (e.g., business logic, database updates).
4.  **Offset Commitment:** To track progress, the consumer commits the **Offset** (the position of the last processed record) back to Kafka's internal `__consumer_offsets` topic. This ensures that if the consumer restarts, it can resume from where it left off.

## 4. How does Kafka ensure message ordering and partitioning?

**Answer:**

Kafka enforces ordering at the partition level and gives you control over which records map to which partitions. Proper partitioning is the primary mechanism to ensure ordering for related records.

Message ordering (practical details)

- Ordering is guaranteed only *within a single partition*. Each message in a partition has a monotonically increasing offset.
- If you need ordering for a set of related events (e.g., all events for a user or an order), produce them with the same key so they land in the same partition.
- Global ordering across all messages requires a single partition (costs parallelism and throughput).

Producer-side partitioning strategies

- **Key-based hashing (default):** The default partitioner hashes the record key (Murmur2 on serialized bytes) to choose a partition — same key -> same partition.
- **Sticky / Round-robin:** When key is null, Kafka uses a sticky partitioner (keeps one partition for a batch for better throughput) or round-robin in older versions.
- **Custom Partitioner:** Implement `org.apache.kafka.clients.producer.Partitioner` to route records using application logic.

Example custom partitioner (simplified)

```java
public class MyPartitioner implements Partitioner {
  @Override
  public int partition(String topic, Object keyObj, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster) {
    String key = (String) keyObj;
    // custom routing: users starting with A-M -> partition 0, others -> partition 1
    return key.charAt(0) <= 'M' ? 0 : 1;
  }
  // configure/close methods omitted
}
```

Producer configuration notes that affect ordering

- `enable.idempotence=true`: recommended — prevents duplicates on retries and works with transactional producers to improve ordering/semantics.
- `max.in.flight.requests.per.connection`: set to 1 if you want strict ordering on retries without idempotence; with idempotence enabled, higher values are safe.
- `retries`: allow sensible retries but combine with idempotence to avoid duplicates and reordering.

Consumer-side considerations

- Consumers read partitions sequentially; processing must preserve order if required (single-threaded processing per partition or use partition-aware handlers).
- Commit offsets only after processing to avoid losing ordering guarantees on restart.

Trade-offs and best practices

- Use meaningful keys to preserve ordering for related records.
- Keep partition counts stable; increasing partitions changes mapping and may affect ordering semantics for key-less routing.
- For high throughput while preserving per-key ordering, use many partitions and ensure keys partition evenly.
- Use `enable.idempotence=true` and transactional producers when you need strong delivery semantics (exactly-once) together with ordering guarantees.

Short example (produce with key):

```java
ProducerRecord<String,String> r = new ProducerRecord<>("orders", orderId, orderJson);
producer.send(r);
```

Summary: ordering = guaranteed per-partition; partitioning (via key or custom partitioner) is the tool you use to group related messages into the same partition so they are processed in order.

## 5. What is the difference between at-most-once, at-least-once, and exactly-once delivery?

**Answer:**

Message delivery semantics define the reliability and consistency of a data pipeline. Choosing the right one is a trade-off between **durability (correctness)** and **latency (performance)**.

| Semantic | Data Loss? | Duplicates? | Typical Use Case |
|---|:---:|:---:|---|
| **At-Most-Once** | Likely | No | Non-critical logs, metrics, or lossy sensors |
| **At-Least-Once** (Default) | No | Likely | Most processing pipelines (requires idempotency) |
| **Exactly-Once** (EOS) | No | No | Financial transactions, exactly-once counting |

---

### a. At-Most-Once (Best Effort)
Messages may be lost, but they are never redelivered. This is the fastest mode because it minimizes acknowledgments and safety checks.
- **Producer Configuration:** `acks=0`. The producer sends the record and immediately forgets it. If the broker crashes before writing to disk, the message is gone.
- **Consumer Configuration:** `enable.auto.commit=true` or manual commit **before** the record is fully processed. If the consumer crashes during processing, it resumes from the next offset, effectively skipping the partially processed record.

---

### b. At-Least-Once (The Standard)
Messages are never lost, but duplicates may occur if a failure happens during the acknowledgment phase. This is the default and most common behavior in Kafka.
- **Producer Configuration:** `acks=1` (leader only) or `acks=all` (all ISR). If the producer doesn't receive an ACK, it will retry. This ensures the message lands in Kafka, but if the ACK was lost on the wire, the retry creates a duplicate.
- **Consumer Configuration:** Manual offset commit **after** the record is successfully processed. If the consumer crashes after processing but before committing, it will re-read and re-process the message upon restart.
- **Handling Duplicates:** Downstream systems should be designed for **idempotency** (processing the same message twice has the same result as once).

---

### c. Exactly-Once Semantics (EOS)
Messages are processed exactly once, with zero loss and zero duplicates, across the entire pipeline. This is the most complex level and requires a combination of features.

**The "Magic" of Exactly-Once:**
1.  **Idempotent Producer (`enable.idempotence=true`):** Each batch of records is given a sequence number. If a producer retries due to a lost ACK, the broker detects the duplicate sequence number and discards it.
2.  **Transactions (`transactional.id`):** Allows a producer to send a batch of messages to multiple topics and partitions atomically. Either all are committed, or none are.
3.  **Isolation Level (`isolation.level=read_committed`):** Consumers are configured to only read messages that have been part of a successfully committed transaction. They skip over "aborted" or "ongoing" transactional records.

**Standard EOS Setup:**
```java
// Producer Config
props.put("enable.idempotence", "true");
props.put("transactional.id", "order-processing-tx-###");

// Consumer Config
props.put("isolation.level", "read_committed");
props.put("enable.auto.commit", "false"); // Use manual transactional commits
```

---

**Summary Rule:**
Use **At-Least-Once** with **idempotent logic** whenever possible for the best balance of speed and reliability. Save **Exactly-Once** for critical paths where duplicate processing is unacceptable. Avoid **At-Most-Once** unless losing data is truly irrelevant.

## 6. How does Kafka handle fault tolerance and replication?

**Answer:**

Kafka’s fault tolerance is built on **partition-level replication**. By copying data across multiple brokers, Kafka ensures that even if one or more servers fail, data remains available and consistent.

---

### a. Replication Architecture (Leader-Follower)
Every topic partition is replicated based on its **Replication Factor** (typical value = 3).
- **Partition Leader:** One broker is elected as the leader for each partition. All incoming writes and reads go through the leader to ensure strong consistency.
- **Partition Followers:** The other brokers act as followers. They do not serve client requests; their only job is to continuously pull data from the leader to keep their local logs synchronized.

---

### b. In-Sync Replicas (ISR)
The **ISR** set is the most critical concept in Kafka durability. It is a subset of all replicas that are "catching up" to the leader's data.
- **Criteria for ISR:** A follower is in the ISR if it has fetched data from the leader within the last `replica.lag.time.max.ms` (default: 30s).
- **The "Commit" Rule:** A message is considered **committed** only after it has been written to the local logs of *all* current ISR members.
- **Consumer Safety:** Consumers can ONLY read messages that are committed (below the **High Watermark**). This prevents consumers from reading data that might disappear if the leader fails before replication.

---

### c. Handling Failures (Leader Election)
- **Leader Failure:** If the leader broker crashes, the **Cluster Controller** (or KRaft quorum) detects the failure and elects a new leader from the existing **ISR** set. Since ISR members are guaranteed to have all committed data, there is 0 data loss.
- **Follower Failure:** If a follower crashes, it is simply removed from the ISR. When it restarts, it catches up with the leader and joins the ISR again.
- **Unclean Leader Election:** If all ISR members fail, Kafka can be configured to allow a non-sync follower to become leader (`unclean.leader.election.enable=true`). This prioritizes **availability** over **consistency** (but may result in data loss).

---

### d. Durability Controls (Producer & Cluster)

Durability is a "team effort" between the Producer's configuration and the Cluster's settings:

| Setting | Type | Role |
|---|---|---|
| **`acks=all`** | Producer | Producer waits for acknowledgments from all ISR members before continuing. |
| **`min.insync.replicas`** | Cluster/Topic | The minimum number of ISR members required to accept a write when `acks=all`. |
| **`acks=1`** | Producer | Producer only waits for the leader to ACK (faster, but risky if leader fails immediately). |
| **`acks=0`** | Producer | Fire-and-forget; no confirmation is waited for (highest throughput, no durability). |

> **Pro Tip:** For the highest durability, always use `acks=all` combined with `min.insync.replicas=2` (in a 3-replica cluster). If the ISR count drops below 2, the producer will receive a `NotEnoughReplicasException`, preventing it from writing data that wouldn't be adequately replicated.

---

### e. The Cluster Controller (KRaft)
The coordination of leaders and the detection of broker failures are managed by the **Controller**. 
- In modern Kafka versions (3.3+), the **KRaft (Kafka Raft)** protocol is used to manage this metadata within Kafka itself, removing the legacy dependency on Apache Zookeeper.

## 7. How do you integrate Kafka with Spring Boot?

**Answer:**

**Dependencies**

Add the `spring-kafka` starter to your project:

```xml
<dependency>
  <groupId>org.springframework.kafka</groupId>
  <artifactId>spring-kafka</artifactId>
</dependency>
```

**Configuration (application.yml)**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
    consumer:
      group-id: my-group
      auto-offset-reset: earliest
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
```

**Producer (example)**

```java
@Service
public class KafkaProducer {
    private final KafkaTemplate<String, Object> kafkaTemplate;

    public void send(String topic, String key, Object message) {
        kafkaTemplate.send(topic, key, message);
    }
}
```

**Consumer (example)**

```java
@Service
public class KafkaConsumer {
    @KafkaListener(topics = "my-topic", groupId = "my-group")
    public void consume(ConsumerRecord<String, MyEvent> record) {
        // Process message
    }
}
```

## 8. What are Consumer Groups, and how do they enable scalability?

**Answer:**

A **Consumer Group** is a collective of consumers (sharing a common `group.id`) that work together to consume and process data from one or more topics. This mechanism is Kafka’s primary way of scaling out message processing and providing fault tolerance.

---

### Core Principles of Consumer Groups

#### a. Partition-to-Consumer Mapping
The fundamental rule of a consumer group is that **each partition is assigned to exactly one consumer** within that group.
- **Parallelism:** If a topic has 10 partitions and a consumer group has 10 consumers, each consumer processes one partition.
- **Idle Consumers:** If you have more consumers than partitions, the extra consumers will remain idle (waiting to take over if others fail).
- **Multiple Partitions per Consumer:** If you have 10 partitions and 2 consumers, each consumer handles 5 partitions.

#### b. Scalability Mechanism
To scale your processing capacity:
1.  **Increase Partitions:** Increase the number of partitions for the topic to allow for higher parallelism.
2.  **Add Consumers:** Spin up more instances of your consumer application using the same `group.id`.
3.  **Horizontal Scale-out:** This architecture allows you to scale from processing thousands to millions of records per second simply by adding more compute resources.

#### c. Load Balancing and Fault Tolerance
- **Rebalancing:** When a consumer joins or leaves the group (e.g., due to a crash or a deployment), Kafka triggers a **Rebalance**. It reassigns partitions across the remaining healthy members.
- **High Availability:** If one consumer in a group fails, its workload is automatically redistributed to other members, ensuring no data is "stuck."

---

### Internal Workings: The Coordinator and Offsets

- **Group Coordinator:** One of the Kafka brokers acts as the coordinator for the group. It handles heartbeats from consumers to monitor their health.
- **Group Leader:** One consumer in the group is elected as the leader. It receives partition metadata from the coordinator and decides which consumer gets which partition (using an **Assignor**).
- **Offset Storage:** Committed offsets (where the consumer currently is in the log) are stored in a special internal Kafka topic called `__consumer_offsets`. This allows a reallocated consumer to resume processing from the exact point where its predecessor stopped.

---

### Subscription Patterns

| Pattern | Setup | Result |
|---|---|---|
| **Point-to-Point (Queue)** | All consumers use the **same** `group.id` | Messages are distributed across consumers; each message is processed only once by the group. |
| **Pub/Sub (Topic)** | Each consumer use a **unique** `group.id` | Each consumer group receives its own independent copy of every message in the topic. |

---

### Modern Rebalance Strategies
- **Eager Rebalance:** The "Stop-the-World" approach where all consumers stop processing and rejoin the group to get new assignments.
- **Cooperative Rebalancing (Sticky):** Consumers only give up the specific partitions that need to be moved, while continuing to process their other partitions. This significantly reduces downtime during deployments.

## 9. What is Kafka Streams, and how does it differ from Kafka Consumer API?

**Answer:**

While the **Consumer API** is a low-level tool for simple "poll-and-process" logic, **Kafka Streams** is a high-level, JVM-native library designed for complex, stateful stream processing. It builds *on top* of the Producer and Consumer APIs, providing abstractions for transformations, aggregations, and joins.

---

### a. Technical Comparison

| Feature | Consumer API | Kafka Streams |
|---|---|---|
| **Abstractions** | `ConsumerRecord` (poll-based) | `KStream`, `KTable`, `GlobalKTable` (graph-based) |
| **State** | **Stateless**: You must manage any state (e.g., in a DB) yourself. | **Stateful**: Built-in local state stores (RocksDB) to handle aggregations and joins. |
| **Transformations** | Manual implementation. | Rich DSL: `map`, `filter`, `join`, `aggregate`, `windowedBy`. |
| **Exactly-Once** | Difficult: Requires manual transaction management. | **Built-in**: Simply set `processing.guarantee=exactly_once_v2`. |
| **Reparitioning** | Manual: You must write back to a new topic and re-consume. | **Automatic**: Operations like `groupBy` trigger internal repartitioning topic management. |
| **Ease of Use** | Simple for basic one-off reads. | Powerful for complex event-driven microservices. |

---

### b. The "Dual Nature" of Streams and Tables
Kafka Streams treats streams and tables as two sides of the same coin, known as **Stream-Table Duality**.

- **KStream (Event Stream):** A stream where each record represents an independent event (e.g., a "click" or a "purchase"). It is an unbounded log of history.
- **KTable (Changelog):** A stream representing the latest state of a key. Each record is an update to the previous one (e.g., a "user profile" or "current stock price").
- **GlobalKTable:** Similar to KTable, but replicated to every instance of the app, making it useful for broadcasting lookups (e.g., tax rates across all processing nodes).

---

### c. Local State and Durability
Kafka Streams manages state locally in **RocksDB** (an embedded, persistent key-value store). 
- **Efficiency:** Since data is local, lookups for joins and counts happen at memory/SSD speed without network round-trips to an external database.
- **Durability:** Every update to a local state store is automatically backed up to an internal **Changelog Topic** in Kafka. If a node fails, its state can be perfectly reconstructed on another node by replaying that topic.

---

### d. DSL vs. Processor API
Kafka Streams provides two ways to build your logic:
1.  **Kafka Streams DSL (High-level):** Most common. Uses functional operators like `.map()`, `.filter()`, and `.groupByKey()`. It's declarative and easy to read.
2.  **Processor API (Low-level):** Allows you to define custom processors and interact with state stores directly. Use this when the DSL is not expressive enough for your complex logic.

---

### e. Windowing Strategies
Streams allows you to group data over time for aggregations:
- **Tumbling Windows:** Fixed-size, non-overlapping (e.g., "count events every 5 minutes").
- **Hopping Windows:** Fixed-size, overlapping (e.g., "count events for the last 5 minutes, updated every 1 minute").
- **Sliding Windows:** Triggered by events (e.g., "find all events occurring within 1 minute of each other").
- **Session Windows:** Group messages by periods of activity followed by a "gap" of inactivity (e.g., "user session on a website").

---

### When to choose which?

- **Use Consumer API if:** You are building a simple "source to sink" connector, or if you need to integrate Kafka with a non-JVM environment.
- **Use Kafka Streams if:** You need to correlate events, maintain local state, perform windowed averages, or join multiple streams together in a fault-tolerant way.

## 10. How do you handle schema evolution in Kafka?

**Answer:**

Schema evolution is managed using a **Schema Registry** (e.g., Confluent Schema Registry) with serialization formats like Avro, Protobuf, or JSON Schema.

**Schema Registry**

- Centralized repository for schemas
- Schemas are versioned and associated with topics
- Producers/consumers register and retrieve schemas via REST API

**Compatibility Modes**

- **BACKWARD:** New schema can read data written by old schema (add optional fields, remove fields with defaults)
- **FORWARD:** Old schema can read data written by new schema
- **FULL:** Both backward and forward compatible
- **NONE:** No compatibility checking

**Best Practices**

- Always provide default values for new fields
- Never remove required fields; mark them as deprecated
- Use `BACKWARD` compatibility for most use cases
- Test schema changes in non-production environments first
- Use Avro or Protobuf for efficient binary serialization with schema support

## 11. What are common Kafka performance tuning strategies?

**Answer:**

**Producer Tuning**

- **`batch.size`:** Increase batch size for higher throughput (default 16KB)
- **`linger.ms`:** Add small delay to allow batching (e.g., 5-10ms)
- **`compression.type`:** Use lz4 or snappy for compression (reduces network I/O)
- **`buffer.memory`:** Increase buffer for high-throughput producers
- **`acks`:** Trade durability for latency (`acks=1` vs `acks=all`)

**Consumer Tuning**

- **`fetch.min.bytes`:** Minimum data to fetch (reduces requests)
- **`fetch.max.wait.ms`:** Maximum wait time for fetch.min.bytes
- **`max.poll.records`:** Control records per poll (balance throughput vs processing time)
- **`max.partition.fetch.bytes`:** Maximum bytes per partition per fetch

**Broker Tuning**

- **`num.io.threads`:** I/O threads for disk operations
- **`num.network.threads`:** Threads for network requests
- **`log.segment.bytes`:** Segment size affects retention and compaction
- **Disk:** Use SSDs and separate disks for logs

**Cluster Design**

- Partition count: More partitions = more parallelism (but more overhead)
- Replication factor: Balance durability vs. resource usage (typically 3)
- Monitor: Use JMX metrics, Kafka Manager, or Confluent Control Center
