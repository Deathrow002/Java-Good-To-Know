# ActiveMQ Interview Questions & Answers

---

## Q1. What is Apache ActiveMQ and what are its key features?

**Answer:**
**Apache ActiveMQ** is an open-source, multi-protocol message broker written in Java. It implements the **JMS (Java Message Service)** specification and supports a wide range of messaging protocols and enterprise integration patterns.

**Key Features:**
- Supports **JMS 1.1 and 2.0** APIs.
- Multiple protocol support: **AMQP, STOMP, MQTT, OpenWire, WebSockets**.
- **Message persistence** via KahaDB or JDBC.
- **Clustering and high availability** (Master-Slave, Network of Brokers).
- **Message filtering** using selectors.
- **Transaction support** (local and XA/distributed transactions).
- **ActiveMQ Artemis** is the next-generation broker (used as the default in ActiveMQ 6+).

## Q2. What is JMS and how does ActiveMQ relate to it?

**Answer:**
**JMS (Java Message Service)** is a Java API (part of Jakarta EE) that defines a standard way for Java applications to create, send, receive, and read messages from a message broker.

**ActiveMQ's relationship with JMS:**
- ActiveMQ is a **JMS provider** — it implements the JMS specification.
- Applications use the JMS API (interfaces like `ConnectionFactory`, `Queue`, `Topic`, `MessageProducer`, `MessageConsumer`) to communicate with ActiveMQ without being tightly coupled to it.

**JMS Core Interfaces:**

| Interface            | Description                                  |
|----------------------|----------------------------------------------|
| `ConnectionFactory`  | Creates connections to the broker            |
| `Connection`         | Represents a TCP connection to the broker    |
| `Session`            | Single-threaded context for producing/consuming messages |
| `Destination`        | Queue or Topic                               |
| `MessageProducer`    | Sends messages to a destination              |
| `MessageConsumer`    | Receives messages from a destination         |

## Q3. What is the difference between a Queue and a Topic in ActiveMQ?

**Answer:**

| Feature              | Queue (Point-to-Point)                         | Topic (Publish-Subscribe)                     |
|----------------------|------------------------------------------------|-----------------------------------------------|
| **Model**            | One producer → One consumer                    | One producer → Many consumers                 |
| **Message delivery** | Each message delivered to **exactly one** consumer | Each message delivered to **all** active subscribers |
| **Message retention**| Message stays until consumed or expires        | Message delivered only to **currently active** subscribers (unless durable) |
| **Use case**         | Task queues, order processing                  | Event broadcasting, notifications             |
| **JMS type**         | `javax.jms.Queue`                              | `javax.jms.Topic`                             |

```java
// Queue destination
Destination queue = session.createQueue("ORDER.QUEUE");

// Topic destination
Destination topic = session.createTopic("ORDER.EVENTS");
```

## Q4. What are Durable Subscriptions in ActiveMQ?

**Answer:**
By default, a **Topic** subscriber only receives messages published **while it is actively connected**. If the subscriber goes offline, it misses messages.

A **Durable Subscription** allows a subscriber to receive messages that were published **while it was offline** — the broker retains the messages until the subscriber reconnects.

**How to create a durable subscription:**
```java
// Requires a unique clientID on the connection
connection.setClientID("client-app-1");
connection.start();

Session session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE);
Topic topic = session.createTopic("ORDER.EVENTS");

// createDurableSubscriber instead of createConsumer
MessageConsumer consumer = session.createDurableSubscriber(topic, "durable-sub-1");
```

**To unsubscribe permanently:**
```java
session.unsubscribe("durable-sub-1");
```

> **Note:** Durable subscriptions require a persistent message store (e.g., KahaDB).

## Q5. What message types does JMS/ActiveMQ support?

**Answer:**
JMS defines five standard message types, all extending `javax.jms.Message`:

| Message Type       | Interface             | Content                              |
|--------------------|-----------------------|--------------------------------------|
| **TextMessage**    | `TextMessage`         | Plain text / JSON / XML string       |
| **BytesMessage**   | `BytesMessage`        | Raw byte stream                      |
| **ObjectMessage**  | `ObjectMessage`       | Serialized Java object               |
| **MapMessage**     | `MapMessage`          | Key-value pairs (String keys)        |
| **StreamMessage**  | `StreamMessage`       | Sequential stream of Java primitives |

```java
// TextMessage example
TextMessage message = session.createTextMessage();
message.setText("{\"orderId\": 101}");
producer.send(message);

// MapMessage example
MapMessage mapMessage = session.createMapMessage();
mapMessage.setString("orderId", "101");
mapMessage.setDouble("amount", 250.00);
producer.send(mapMessage);
```

## Q6. How does ActiveMQ handle message persistence?

**Answer:**
ActiveMQ supports **persistent** and **non-persistent** message delivery:

- **Persistent (default):** Messages are stored to disk before acknowledgement. Survives broker restarts. Slower due to disk I/O.
- **Non-persistent:** Messages are stored in memory only. Lost on broker restart. Faster.

```java
// Set delivery mode on the producer
producer.setDeliveryMode(DeliveryMode.PERSISTENT);       // default
producer.setDeliveryMode(DeliveryMode.NON_PERSISTENT);   // faster, not durable
```

**Persistence stores in ActiveMQ:**

| Store        | Description                                               |
|--------------|-----------------------------------------------------------|
| **KahaDB**   | Default embedded file-based store; high performance       |
| **JDBC**     | Stores messages in a relational database (MySQL, PostgreSQL, etc.) |
| **LevelDB**  | Deprecated in favour of KahaDB                            |
| **Artemis Journal** | Used by ActiveMQ Artemis; high-throughput append-only journal |

## Q7. What is the Dead Letter Queue (DLQ) in ActiveMQ?

**Answer:**
A **Dead Letter Queue (DLQ)** is a special queue where ActiveMQ moves messages that **could not be delivered** after a configured number of redelivery attempts.

**Default behaviour:**
- ActiveMQ moves failed messages to `ActiveMQ.DLQ` by default.
- This applies to persistent messages that are redelivered more than `maximumRedeliveries` times.

**Custom DLQ configuration (`activemq.xml`):**
```xml
<destinationPolicy>
  <policyMap>
    <policyEntries>
      <policyEntry queue="ORDER.QUEUE">
        <deadLetterStrategy>
          <individualDeadLetterStrategy
            queuePrefix="DLQ."
            useQueueForQueueMessages="true"/>
        </deadLetterStrategy>
      </policyEntry>
    </policyEntries>
  </policyMap>
</destinationPolicy>
```
This sends dead letters for `ORDER.QUEUE` to `DLQ.ORDER.QUEUE` instead of the shared `ActiveMQ.DLQ`.

**Redelivery policy (Spring Boot):**
```java
@Bean
public RedeliveryPolicy redeliveryPolicy() {
    RedeliveryPolicy policy = new RedeliveryPolicy();
    policy.setMaximumRedeliveries(3);
    policy.setInitialRedeliveryDelay(1000L);
    policy.setBackOffMultiplier(2.0);
    policy.setUseExponentialBackOff(true);
    return policy;
}
```

## Q8. How do you integrate ActiveMQ with Spring Boot?

**Answer:**

**1. Add dependency (`pom.xml`):**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-activemq</artifactId>
</dependency>
```

**2. Configure properties (`application.yml`):**
```yaml
spring:
  activemq:
    broker-url: tcp://localhost:61616
    user: admin
    password: admin
    in-memory: false
  jms:
    pub-sub-domain: false   # false = Queue, true = Topic
```

**3. Produce a message:**
```java
@Service
public class OrderProducer {

    @Autowired
    private JmsTemplate jmsTemplate;

    public void sendOrder(Order order) {
        jmsTemplate.convertAndSend("ORDER.QUEUE", order);
    }
}
```

**4. Consume a message:**
```java
@Service
public class OrderConsumer {

    @JmsListener(destination = "ORDER.QUEUE")
    public void processOrder(Order order) {
        System.out.println("Processing: " + order.getId());
    }
}
```

**5. Enable JMS (main class):**
```java
@SpringBootApplication
@EnableJms
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

## Q9. What is the difference between ActiveMQ Classic and ActiveMQ Artemis?

**Answer:**

| Feature                  | ActiveMQ Classic (5.x)              | ActiveMQ Artemis (6.x / next-gen)          |
|--------------------------|-------------------------------------|--------------------------------------------|
| **Architecture**         | Older broker codebase               | Rewritten from HornetQ; modern architecture |
| **Performance**          | Moderate                            | Higher throughput, low latency             |
| **Persistence**          | KahaDB, JDBC                        | Journal-based (NIO/AIO), JDBC              |
| **Protocol support**     | OpenWire, STOMP, AMQP, MQTT         | OpenWire, STOMP, AMQP 1.0, MQTT, Core     |
| **Thread model**         | Thread-per-connection               | Non-blocking, Netty-based                  |
| **Configuration**        | `activemq.xml`                      | `broker.xml`                               |
| **Spring Boot default**  | `spring-boot-starter-activemq`      | `spring-boot-starter-artemis`              |
| **JMS 2.0**              | Partial                             | Full JMS 2.0 support                       |

> **Recommendation:** For new projects, prefer **ActiveMQ Artemis** for better performance and full protocol support.

## Q10. What is Message Selector in ActiveMQ and how is it used?

**Answer:**
A **Message Selector** is a filter expression (SQL92 syntax) that allows a consumer to receive **only messages that match certain header or property criteria**.

- Evaluated against **message headers** and **user-defined properties**.
- Cannot filter on message body content.

**Producer — set message properties:**
```java
TextMessage message = session.createTextMessage("Order payload");
message.setStringProperty("region", "APAC");
message.setIntProperty("priority", 1);
producer.send(message);
```

**Consumer — apply selector:**
```java
String selector = "region = 'APAC' AND priority > 0";
MessageConsumer consumer = session.createConsumer(queue, selector);
```

**Spring Boot with `@JmsListener`:**
```java
@JmsListener(destination = "ORDER.QUEUE", selector = "region = 'APAC'")
public void handleApacOrders(TextMessage message) throws JMSException {
    System.out.println("APAC order: " + message.getText());
}
```

**Common selector expressions:**

| Expression                    | Description                          |
|-------------------------------|--------------------------------------|
| `priority > 5`                | Numeric comparison                   |
| `region = 'APAC'`             | String equality                       |
| `region IN ('APAC', 'EMEA')`  | Set membership                        |
| `JMSPriority >= 7`            | Filter on JMS standard header         |
| `discount IS NOT NULL`        | Property existence check              |

## Q11. Conclusion — RabbitMQ vs Apache Kafka vs ActiveMQ: Advantages, Disadvantages & Best Use Cases

**Answer:**

### Overview Comparison

| Criteria                  | RabbitMQ                            | Apache Kafka                              | ActiveMQ                                  |
|---------------------------|--------------------------------------|-------------------------------------------|-------------------------------------------|
| **Type**                  | Message Broker (AMQP)               | Distributed Event Streaming Platform      | Message Broker (JMS)                      |
| **Protocol**              | AMQP, STOMP, MQTT, WebSocket        | Custom binary (TCP), Kafka protocol       | JMS, AMQP, STOMP, MQTT, OpenWire          |
| **Message Model**         | Queue + Exchange routing            | Partitioned commit log (topic/partition)  | Queue + Topic (JMS)                       |
| **Message Ordering**      | Per-queue FIFO                      | Per-partition ordering                    | Per-queue/destination FIFO                |
| **Throughput**            | Moderate (tens of thousands/sec)    | Very high (millions/sec)                  | Moderate                                  |
| **Persistence**           | Optional (durable queues)           | Always persistent (configurable retention)| Optional (KahaDB / JDBC)                 |
| **Consumer Model**        | Push-based                          | Pull-based                                | Push-based (JMS)                          |
| **Message Replay**        | No (consumed messages are deleted)  | Yes (retain and re-consume via offset)    | No (consumed messages are deleted)        |
| **Ecosystem**             | Spring AMQP, Shovel, Federation     | Kafka Streams, Kafka Connect, ksqlDB      | Spring JMS, Camel, JEE                    |
| **Operational Complexity**| Low–Medium                          | High (cluster, ZooKeeper/KRaft, tuning)   | Low–Medium                                |

---

### RabbitMQ

#### Advantages
- **Simple and flexible routing** via exchanges (Direct, Fanout, Topic, Headers) without custom code.
- **Low latency** for individual message delivery — excellent for request/reply and task queues.
- **Rich management UI** and HTTP API for monitoring out of the box.
- **Message TTL, DLX, priority queues** — broad feature set for enterprise messaging.
- **Easy to set up** — minimal infrastructure overhead compared to Kafka.
- **Supports multiple protocols** (AMQP, MQTT, STOMP) making it polyglot-friendly.

#### Disadvantages
- **No native message replay** — once a message is consumed and acknowledged, it is gone.
- **Lower throughput** than Kafka for massive event streams.
- **Not designed for long-term event storage** or event sourcing.
- **Horizontal scaling is harder** — clustering has limitations compared to Kafka's partition model.
- **State stored in broker** — heavy queues can put memory and disk pressure on the broker.

#### Best Use Cases
- **Task/work queues** — distribute background jobs to worker pools (e.g., image processing, email sending).
- **Request/reply patterns** — synchronous-style RPC over messaging.
- **Complex routing logic** — message routing based on content, headers, or wildcard keys.
- **Microservice integration** — lightweight async communication between services.
- **IoT messaging** — MQTT support for device-to-backend communication.

---

### Apache Kafka

#### Advantages
- **Extremely high throughput** — handles millions of messages per second with sequential disk I/O.
- **Message replay and retention** — consumers can re-read any historical message within the retention period.
- **Horizontal scalability** — add partitions and brokers linearly to scale.
- **Fault tolerance** — replication across brokers ensures no data loss even on node failure.
- **Exactly-once semantics (EOS)** — idempotent producers and transactional APIs.
- **Rich ecosystem** — Kafka Streams, Kafka Connect, ksqlDB for stream processing and integrations.
- **Event sourcing backbone** — topics act as the durable source of truth.

#### Disadvantages
- **High operational complexity** — requires tuning brokers, partitions, replication factors, consumer groups, and ZooKeeper/KRaft.
- **No traditional routing** — no exchange/binding concept; routing must be handled at application or Kafka Streams level.
- **Ordering only within a partition** — global ordering across all partitions is not guaranteed.
- **Overkill for simple task queues** — heavy infrastructure for small-scale messaging needs.
- **Message TTL is time/size-based not consumption-based** — deleted after retention period regardless of whether all consumers have read them.
- **Higher learning curve** — consumer groups, offsets, partition assignment, and rebalancing require deep understanding.

#### Best Use Cases
- **Event streaming & real-time analytics** — clickstream analysis, fraud detection, live dashboards.
- **Event sourcing** — store domain events durably as the source of truth.
- **Change Data Capture (CDC)** — stream database changes to downstream systems via Debezium.
- **Log aggregation** — collect and centralize logs and metrics from many services.
- **Data pipeline / ETL** — reliably move large volumes of data between systems.
- **Microservice choreography** — backbone for event-driven microservice architectures at scale.

---

### ActiveMQ

#### Advantages
- **JMS compliance** — first-class JMS 1.1/2.0 support, ideal for Java/Jakarta EE enterprise applications.
- **Wide protocol support** — AMQP, STOMP, MQTT, OpenWire, WebSocket in one broker.
- **Message selectors** — fine-grained SQL92 filtering without extra routing infrastructure.
- **Transaction support** — local and XA (distributed) transactions for guaranteed consistency.
- **Spring Boot integration** — seamless `spring-boot-starter-activemq` / `spring-boot-starter-artemis`.
- **Mature and battle-tested** — decades of use in enterprise Java environments.
- **ActiveMQ Artemis** — modern, high-performance next-gen broker with Netty-based non-blocking I/O.

#### Disadvantages
- **Lower throughput** than Kafka for high-volume streaming scenarios.
- **No message replay** — consumed messages are gone (similar to RabbitMQ).
- **Older Classic architecture** (5.x) can struggle under heavy load compared to modern alternatives.
- **Less cloud-native** — not as well-suited for containerized, cloud-native environments as Kafka or RabbitMQ.
- **Smaller ecosystem** for stream processing compared to Kafka.
- **KahaDB** can become a bottleneck under very high message volumes.

#### Best Use Cases
- **Enterprise Java applications** — systems deeply integrated with JMS/Jakarta EE standards.
- **Legacy system integration** — interoperating with existing JMS-based infrastructure.
- **Financial / transactional systems** — XA distributed transaction support for multi-resource consistency.
- **Multi-protocol environments** — where different consumers use different protocols (AMQP, MQTT, STOMP).
- **Request/reply and workflow orchestration** — JMS-based synchronous and asynchronous workflows.

---

### Decision Guide — When to Choose Which?

```
Need high-throughput event streaming or replay?
  → Apache Kafka

Need simple, flexible message routing (Direct, Fanout, Topic)?
  → RabbitMQ

Need JMS compliance or enterprise Java / XA transactions?
  → ActiveMQ (Artemis)

Need IoT / MQTT device messaging at scale?
  → RabbitMQ or ActiveMQ

Need event sourcing or CDC pipelines?
  → Apache Kafka

Need easy setup with low operational overhead?
  → RabbitMQ or ActiveMQ

Need to handle millions of events/sec with durable retention?
  → Apache Kafka
```

- Use **ActiveMQ** when you are in a Java/Jakarta EE environment requiring JMS compliance, XA transactions, or legacy system integration.
