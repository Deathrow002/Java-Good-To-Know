Java Advanced Concepts

1. What is the difference between checked and unchecked exceptions?

Answer:

Java exceptions fall into two categories based on how the compiler enforces handling:

Exception Hierarchy

Throwable
+-- Error (unchecked) -- JVM errors, not recoverable
|   +-- OutOfMemoryError, StackOverflowError
+-- Exception
    +-- RuntimeException (unchecked) -- programming errors
    |   +-- NullPointerException, IllegalArgumentException
    +-- Other Exceptions (checked) -- recoverable conditions
        +-- IOException, SQLException

Comparison

Aspect Checked Exceptions Unchecked Exceptions
Compile-time check Must declare or handle [OK] No requirement
Extends Exception (not RuntimeException) RuntimeException or Error
Use case Recoverable external conditions Programming bugs, logic errors
Examples IOException, SQLException, FileNotFoundException NullPointerException, ArrayIndexOutOfBoundsException

Best Practices

  • Use checked exceptions for conditions the caller can reasonably recover from
  • Use unchecked exceptions for programming errors that should be fixed in code
  • Prefer specific exception types over generic Exception
  • Document exceptions with @throws in Javadoc

2. How does garbage collection work in Java?

Answer:

Garbage Collection (GC) is the JVM's automatic memory management process that reclaims memory occupied by objects no longer in use.

Generational Heap Structure

+------------------------------------------------------+
|                         HEAP                         |
+---------------------------+--------------------------+
|     Young Generation      |     Old Generation       |
+-----------+---------------+       (Tenured)          |
|   Eden    |   Survivor    |                          |
|           |   (S0 | S1)   |   Long-lived objects     |
|   New     |   Short-lived |   promoted from Young    |
|  objects  |   survivors   |                          |
+-----------+---------------+--------------------------+

GC Process (Mark-Sweep-Compact)

  1. Mark: Traverse from GC roots (stack, static fields) and mark all reachable objects
  2. Sweep: Remove unmarked (unreachable) objects and reclaim memory
  3. Compact: Move surviving objects together to reduce fragmentation

Available Garbage Collectors

Collector Optimized For Use Case
Serial GC Footprint Single-threaded, small heaps
Parallel GC Throughput Batch processing
G1 GC (default) Balanced Most applications (Java 9+)
ZGC / Shenandoah Low latency Large heaps, <10ms pauses

Key Points

  • System.gc() is only a suggestion -- JVM decides when to run GC
  • finalize() is deprecated (Java 9+) -- use try-with-resources or Cleaner
  • Monitor with tools: jstat, VisualVM, -Xlog:gc*

3. What are Java annotations, and how are they used?

Answer:

Annotations are metadata markers that provide information about code to the compiler, tools, or runtime frameworks without changing program behavior directly.

Built-in Annotations

Annotation Purpose
@Override Compiler checks method actually overrides superclass method
@Deprecated Marks element as obsolete; triggers compiler warnings
@SuppressWarnings Silences specific compiler warnings
@FunctionalInterface Ensures interface has exactly one abstract method
@SafeVarargs Suppresses heap pollution warnings for varargs

Retention Policies

  • RetentionPolicy.SOURCE -- Discarded at compile time (e.g., @Override)
  • RetentionPolicy.CLASS -- In bytecode but not available at runtime
  • RetentionPolicy.RUNTIME -- Available via reflection (e.g., Spring annotations)

Creating Custom Annotations

@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.METHOD, ElementType.TYPE})
public @interface Loggable {
    String value() default "INFO";
    boolean includeArgs() default false;
}

Framework Annotations Examples

  • Spring: @Component, @Autowired, @Transactional
  • JPA: @Entity, @Table, @Column
  • Validation: @NotNull, @Size, @Email
  • Testing: @Test, @BeforeEach, @Mock

4. Explain the concept of reflection in Java

Answer:

Reflection is a feature that allows programs to examine and modify their own structure and behavior at runtime using the java.lang.reflect package.

Core Reflection Classes

Class Purpose
Class<T> Represents a class; entry point for reflection
Method Invoke methods dynamically
Field Access and modify fields
Constructor Create instances dynamically

Common Operations

// Get Class object
Class<?> clazz = Class.forName("com.example.User");

// Create instance
Object obj = clazz.getDeclaredConstructor().newInstance();

// Access private field
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);  // Bypass access modifiers
field.set(obj, "John");

// Invoke method
Method method = clazz.getMethod("getName");
String name = (String) method.invoke(obj);

Use Cases

  • Frameworks: Spring IoC, Hibernate ORM, JUnit testing
  • Serialization: JSON/XML libraries (Jackson, Gson)
  • IDE Tools: Code completion, debugging
  • Plugin Systems: Loading classes dynamically

Trade-offs

Advantages Disadvantages
Dynamic class loading Performance overhead (slower than direct calls)
Framework flexibility No compile-time type safety
Access private members Breaks encapsulation

5. What is the difference between ArrayList and LinkedList?

Answer:

Both implement List interface but use different internal data structures with different performance characteristics.

Internal Structure

ArrayList:  [ elem0 | elem1 | elem2 | elem3 | ... ]  + Contiguous array

LinkedList: [prev|data|next] Y [prev|data|next] Y [prev|data|next]  + Doubly-linked nodes

Performance Comparison

Operation ArrayList LinkedList
Random access get(i) O(1) [OK] O(n)
Add at end add(e) O(1) amortized O(1)
Add at beginning [!] O(n) -- shift all O(1)
Insert/delete middle [!] O(n) -- shift elements [!] O(n) -- traverse first
Memory per element Low (just the reference) [OK] High (node + 2 pointers)
Cache locality Excellent (contiguous) [OK] Poor (scattered nodes)

When to Use

  • ArrayList (90%+ of cases): Random access, iteration, most general use
  • LinkedList: Frequent add/remove at both ends (use as Deque), or with ListIterator for local modifications

Tip: In practice, ArrayList is almost always faster due to CPU cache efficiency. Consider ArrayDeque for queue/stack operations instead of LinkedList.

6. What is the difference between HashMap, TreeMap, and LinkedHashMap?

Answer:

All three implement Map interface but differ in ordering, performance, and internal structure.

Comparison Table

Feature HashMap LinkedHashMap TreeMap
Structure Hash table Hash table + linked list Red-Black tree
Ordering [OK] No order Insertion order Sorted (natural/comparator)
get/put O(1) O(1) O(log n)
Null keys One allowed One allowed [OK] Not allowed
Thread-safe [OK] No [OK] No [OK] No

When to Use

  • HashMap: Default choice for key-value storage -- fastest performance
  • LinkedHashMap: Need predictable iteration order or LRU cache implementation
  • TreeMap: Need sorted keys, range queries (subMap(), headMap()), or navigation methods

LRU Cache with LinkedHashMap

Map<String, Object> lruCache = new LinkedHashMap<>(16, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<String, Object> eldest) {
        return size() > MAX_SIZE;  // Auto-remove oldest on overflow
    }
};

Thread-safe alternatives: Use ConcurrentHashMap or Collections.synchronizedMap()

7. What are generics in Java, and why are they useful?

Answer:

Generics enable types (classes, interfaces, methods) to operate on parameterized types, providing compile-time type safety and eliminating runtime casts.

Without vs With Generics

// Without generics (Java 1.4 and earlier)
List names = new ArrayList();
names.add("Alice");
names.add(123);  // [!] No compile error!
String name = (String) names.get(1);  // [!] ClassCastException at runtime!

// With generics (Java 5+)
List<String> names = new ArrayList<>();
names.add("Alice");
names.add(123);  // Compile error!
String name = names.get(0);  // No cast needed

Generic Syntax

Syntax Meaning Example
<T> Type parameter class Box<T>
<T extends Number> Upper bound T must be Number or subclass
<? extends T> Wildcard upper bound Read-only (producer)
<? super T> Wildcard lower bound Write-only (consumer)
<?> Unbounded wildcard List<?> -- any type

PECS Principle (Producer Extends, Consumer Super)

// Producer: read from -- use extends
void printAll(List<? extends Number> nums) {
    for (Number n : nums) System.out.println(n);
}

// Consumer: write to -- use super
void addNumbers(List<? super Integer> list) {
    list.add(1);
    list.add(2);
}

Type Erasure

Generic type information is removed at runtime (for backward compatibility):

  • List<String> becomes List at runtime
  • Cannot use primitives: List<int> [X] --> Use List<Integer>
  • Cannot check generic type: obj instanceof List<String> [X]

8. Explain the concept of multithreading in Java

Answer:

Multithreading enables concurrent execution of multiple threads within a single process, allowing better CPU utilization and responsive applications.

Thread Lifecycle

+-------+   start()   +-----------+   scheduler   +---------+
|  NEW  |------------>| RUNNABLE  |-------------->| RUNNING |
+-------+             +-----------+               +----+----+
                            ^                          |
         notify/timeout     |     wait/sleep/join      v
                      +---------+               +-----------+
                      | BLOCKED |               |  WAITING  |
                      |/WAITING |               |TIMED_WAIT |
                      +---------+               +-----------+
                                                      |
                                                      v
                                               +------------+
                                               | TERMINATED |
                                               +------------+

Creating Threads

// 1. Extend Thread (not recommended)
class MyThread extends Thread {
    public void run() { /* task */ }
}

// 2. Implement Runnable (better)
Runnable task = () -> System.out.println("Running");
new Thread(task).start();

// 3. ExecutorService (recommended)
ExecutorService executor = Executors.newFixedThreadPool(4);
executor.submit(() -> processTask());
executor.shutdown();

// 4. Callable with return value
Future<Integer> future = executor.submit(() -> computeResult());
Integer result = future.get();  // Blocking

Key Concurrency Concepts

Concept Description
Race Condition Multiple threads accessing shared data without synchronization
Deadlock Two threads waiting for each other's locks forever
Volatile Ensures visibility of changes across threads (no caching)
Atomic Thread-safe operations without locks (AtomicInteger)

Modern Concurrency (Java 21+)

  • Virtual Threads: Lightweight threads for high-throughput I/O
  • Structured Concurrency: Managing related tasks as a unit
  • Scoped Values: Thread-local alternative for virtual threads

9. What is the difference between synchronized methods and blocks?

Answer:

Both provide mutual exclusion (only one thread can execute at a time), but differ in scope and flexibility.

Synchronized Method vs Block

// Synchronized Method -- locks entire method
public synchronized void increment() {
    count++;  // Lock held for entire method
    // Other operations also locked unnecessarily
}

// Synchronized Block -- locks only critical section
public void increment() {
    // Non-critical code runs without lock
    synchronized (this) {
        count++;  // Only this is locked
    }
    // More non-critical code
}

Comparison

Aspect Synchronized Method Synchronized Block
Lock object this or Class.class Any specified object
Scope Entire method Only the block
Flexibility [OK] Coarse-grained Fine-grained
Readability Simple More verbose
Performance [OK] More contention Better concurrency

Best Practices

  • Prefer synchronized blocks with minimal scope
  • Use private lock objects to prevent external interference
  • Consider ReentrantLock for advanced features (tryLock, fairness)
  • Use java.util.concurrent classes when possible
// Best practice: private lock object
private final Object lock = new Object();

public void safeMethod() {
    synchronized (lock) {
        // Critical section
    }
}

10. What are Java Streams, and how do they differ from Collections?

Answer:

Streams (Java 8+) provide a declarative, functional-style API for processing sequences of elements through pipelines of operations.

Stream Pipeline Structure

List<String> result = users.stream()        // 1. Source
    .filter(u -> u.isActive())               // 2. Intermediate (lazy)
    .map(User::getName)                       // 2. Intermediate (lazy)
    .sorted()                                 // 2. Intermediate (lazy)
    .collect(Collectors.toList());           // 3. Terminal (triggers execution)

Streams vs Collections

Aspect Collections Streams
Purpose Store & manage data Process & transform data
Storage Hold elements [OK] Don't store data
Iteration External (for-each) Internal (pipeline)
Reusability Reusable [OK] Single-use (consumed)
Evaluation Eager Lazy (until terminal op)
Parallelism Manual Built-in (.parallelStream())
Modification Mutable [OK] Immutable (no side effects)

Stream Creation

// From collection
list.stream()

// From array
Arrays.stream(array)

// From values
Stream.of("a", "b", "c")

// Infinite streams
Stream.iterate(0, n -> n + 1)  // 0, 1, 2, 3...
Stream.generate(Math::random)   // random values

When to Use

  • Streams: Transformations, filtering, aggregations, parallel processing
  • Collections: Storing data, random access, modification, iteration multiple times

11. What is the Java Collections Framework?

Answer:

The Java Collections Framework (JCF) is a unified architecture of interfaces, implementations, and algorithms for representing and manipulating groups of objects.

Collections Hierarchy

                         Iterable
                            |
                        Collection
              +-------------+-------------+
              |             |             |
            List           Set          Queue
              |             |             |
    +---------+------+  +---+---+     +---+---+
 ArrayList  LinkedList  HashSet  TreeSet  PriorityQueue
                        |
                   LinkedHashSet

                          Map (separate hierarchy)
              +-----------+-----------+
          HashMap     TreeMap    LinkedHashMap
              |
      ConcurrentHashMap

Choosing the Right Collection

Need Use Why
Indexed access ArrayList O(1) get by index
Queue/Stack ArrayDeque Faster than LinkedList
No duplicates HashSet O(1) contains check
Sorted unique elements TreeSet NavigableSet methods
Key-value lookup HashMap O(1) get/put
Thread-safe map ConcurrentHashMap Lock striping for concurrent access
Priority-based processing PriorityQueue Heap-based ordering

Collections Utility Class

// Sorting
Collections.sort(list);
Collections.sort(list, Comparator.reverseOrder());

// Searching (list must be sorted)
int index = Collections.binarySearch(list, key);

// Immutable views
List<String> immutable = Collections.unmodifiableList(list);

// Thread-safe wrappers
List<String> syncList = Collections.synchronizedList(list);

// Utility
Collections.shuffle(list);
Collections.reverse(list);
Collections.frequency(list, element);

Best Practices

  • Program to interfaces: List<String> names = new ArrayList<>()
  • Use diamond operator: new HashMap<>()
  • Prefer immutable collections: List.of(), Set.of(), Map.of() (Java 9+)
  • Check for ConcurrentModificationException when iterating and modifying

12. What are common Stream API operations and best practices?

Answer:

Stream operations are categorized as intermediate (lazy, return Stream) or terminal (eager, trigger execution and produce result).

Intermediate Operations (Lazy)

Operation Purpose Example
filter() Select matching elements .filter(x -> x > 5)
map() Transform elements .map(String::toUpperCase)
flatMap() Flatten nested streams .flatMap(List::stream)
distinct() Remove duplicates .distinct()
sorted() Sort elements .sorted(Comparator.reverseOrder())
limit(n) / skip(n) Truncate / skip elements .skip(10).limit(5)

Terminal Operations (Eager)

Operation Returns Example
collect() Collection/value .collect(Collectors.toList())
toList() (Java 16+) Immutable List .toList()
reduce() Single value .reduce(0, Integer::sum)
count() long .count()
findFirst() / findAny() Optional .findFirst().orElse(null)
anyMatch() / allMatch() boolean .anyMatch(x -> x > 100)

Useful Collectors

// Group by
Map<Type, List<User>> byType = users.stream()
    .collect(Collectors.groupingBy(User::getType));

// Join strings
String names = users.stream()
    .map(User::getName)
    .collect(Collectors.joining(", "));

// Statistics
IntSummaryStatistics stats = numbers.stream()
    .collect(Collectors.summarizingInt(Integer::intValue));
// stats.getSum(), stats.getAverage(), stats.getMax()

Best Practices

  • Use method references: User::getName over u -> u.getName()
  • Avoid side effects in lambdas (no modifying external variables)
  • Use parallelStream() only for CPU-bound tasks on large datasets
  • Close I/O streams: try (Stream<String> lines = Files.lines(path))
  • [!] Don't reuse streams -- they can only be consumed once
  • [!] Avoid peek() for logic -- use only for debugging

13. What is a Thread in Java and explain the Thread Lifecycle?

Answer:

A Thread is the smallest unit of execution within a process. Java supports multithreading, allowing concurrent execution of multiple threads to improve performance and responsiveness.

Thread Lifecycle States

State Description How to Enter
NEW Thread created but not started new Thread()
RUNNABLE Ready to run or running (JVM manages scheduling) thread.start()
BLOCKED Waiting to acquire a monitor lock Entering synchronized block
WAITING Waiting indefinitely for another thread wait(), join(), park()
TIMED_WAITING Waiting for a specified time sleep(ms), wait(ms), join(ms)
TERMINATED Thread completed execution or died run() completes or exception thrown

Thread Lifecycle Diagram

                          start()
+-------+ ------------------------------------> +---------------+
|  NEW  |                                       |   RUNNABLE    |--------+
+-------+                                       |  (Ready/Run)  |        |
                                                +-------+-------+        |
                                                        |                |
                        +-------------------------------+----------------+
                        |                               |                |
                        v                               v                |
               +----------------+          +----------------+            |
               |    BLOCKED     |          |    WAITING     |            |
               | (waiting for   |          | wait()         |            |
               |  monitor lock) |          | join()         |            |
               +-------+--------+          +-------+--------+            |
                        |                          |                     |
                        | lock acquired            | notify/notifyAll    |
                        +--------------------------+---------------------+
                                                   |
                                                   v
                                            +------------------+
                                            |   TERMINATED     |
                                            | (run completed)  |
                                            +------------------+

Ways to Create Threads

// 1. Extend Thread class (not recommended -- uses inheritance)
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Running in: " + Thread.currentThread().getName());
    }
}
new MyThread().start();

// 2. Implement Runnable interface (preferred -- composition over inheritance)
Runnable task = () -> System.out.println("Running in: " + Thread.currentThread().getName());
new Thread(task).start();

// 3. Implement Callable (when you need a return value)
Callable<Integer> callable = () -> {
    return 42;  // Can return a value
};
ExecutorService executor = Executors.newSingleThreadExecutor();
Future<Integer> future = executor.submit(callable);
Integer result = future.get();  // Blocks until complete

// 4. ExecutorService (recommended for production)
ExecutorService executor = Executors.newFixedThreadPool(4);
executor.submit(() -> processTask());
executor.shutdown();

Important Thread Methods

Method Description
start() Starts the thread, JVM calls run()
run() Contains the code to execute (don't call directly!)
sleep(ms) Pauses current thread for specified milliseconds
join() Waits for the thread to complete
interrupt() Requests thread interruption
yield() Hints scheduler to give time to other threads
isAlive() Returns true if thread is started and not terminated

Platform Thread Characteristics

  • 1:1 mapping: Each Java thread maps to OS thread
  • Heavyweight: ~1MB stack memory per thread
  • Limited scalability: Practical limit ~10,000 threads
  • Blocking is expensive: Blocks the OS thread

14. What are Virtual Threads in Java (Project Loom)?

Answer:

Virtual Threads (introduced in Java 21 as a standard feature) are lightweight threads that dramatically reduce the effort of writing, maintaining, and observing high-throughput concurrent applications. They are part of Project Loom.

Virtual Threads vs Platform Threads

Aspect Platform Threads Virtual Threads
Mapping 1:1 with OS threads M:N (many virtual to few carrier threads)
Memory ~1MB stack per thread ~1KB initial, grows as needed
Creation Cost Expensive (kernel call) Cheap (JVM managed)
Scalability ~10,000 threads max Millions of threads possible
Blocking Blocks OS thread (expensive) Unmounts from carrier (efficient)
Best For CPU-bound tasks I/O-bound tasks (HTTP, DB, file)

How Virtual Threads Work

+----------------------------------------------------------------------+
|                    JVM Virtual Thread Scheduler                      |
+----------------------------------------------------------------------+
|  Virtual Thread Pool (millions of lightweight threads)               |
|  +-----+ +-----+ +-----+ +-----+ +-----+ +-----+ +-----+ +-----+     |
|  | VT1 | | VT2 | | VT3 | | VT4 | | VT5 | | VT6 | | VT7 | | ... |     |
|  +--+--+ +--+--+ +--+--+ +--+--+ +--+--+ +--+--+ +--+--+ +-----+     |
|     |       |       |       |       |       |       |               |
|     +-------+-------+-------+-------+-------+-------+               |
|                     |               |                               |
|              +------+------+ +------+------+                        |
|              |   Carrier   | |   Carrier   |  (Platform threads)    |
|              |   Thread 1  | |   Thread 2  |                        |
|              +------+------+ +------+------+                        |
+----------------------------------------------------------------------+
                      |               |
              +-------+-------++------+-------+
              |   OS Thread   ||  OS Thread   |
              +---------------++------+-------+

Creating Virtual Threads

// 1. Using Thread.startVirtualThread() -- fire and forget
Thread.startVirtualThread(() -> {
    System.out.println("Running in virtual thread");
});

// 2. Using Thread.ofVirtual() -- more control
Thread vThread = Thread.ofVirtual()
    .name("my-virtual-thread")
    .start(() -> {
        System.out.println("Running in: " + Thread.currentThread());
    });
vThread.join();

// 3. Using ExecutorService (recommended for production)
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    // Each task gets its own virtual thread
    IntStream.range(0, 100_000).forEach(i -> {
        executor.submit(() -> {
            // Simulate I/O operation
            Thread.sleep(Duration.ofSeconds(1));
            return fetchDataFromAPI(i);
        });
    });
}  // Executor auto-closes, waits for all tasks

// 4. Check if running on virtual thread
if (Thread.currentThread().isVirtual()) {
    System.out.println("I'm a virtual thread!");
}

Virtual Threads with Spring Boot

# application.yml (Spring Boot 3.2+)
spring:
  threads:
    virtual:
      enabled: true  # All request handling uses virtual threads

# This enables virtual threads for:
# - Tomcat/Jetty embedded servers
# - @Async methods
# - Scheduled tasks

When to Use Virtual Threads

Good Use Cases
  • HTTP client calls (REST APIs)
  • Database queries (JDBC)
  • File I/O operations
  • Message queue consumers
  • High-concurrency web servers
[OK] Avoid For
  • CPU-intensive computations (use platform threads)
  • Code that holds native locks (synchronized on short critical sections is OK)
  • Thread-local heavy applications (use Scoped Values instead)

Important Considerations

  • Don't pool virtual threads: They're cheap to create -- just make new ones
  • Avoid synchronized: Use ReentrantLock for long critical sections
  • ThreadLocal caution: Works but can cause memory issues with millions of threads
  • Carrier thread pinning: synchronized blocks pin the virtual thread to its carrier

15. What is the java.util.concurrent package?

Answer:

The java.util.concurrent package provides high-level concurrency utilities that simplify multithreaded programming, offering thread-safe collections, executors, synchronizers, and atomic operations.

Key Components Overview

java.util.concurrent
+-- Executors & Thread Pools
|   +-- ExecutorService, ScheduledExecutorService
|   +-- ThreadPoolExecutor, ForkJoinPool
|   +-- Executors (factory methods)
|
+-- Concurrent Collections
|   +-- ConcurrentHashMap, ConcurrentLinkedQueue
|   +-- CopyOnWriteArrayList, CopyOnWriteArraySet
|   +-- BlockingQueue (ArrayBlockingQueue, LinkedBlockingQueue)
|
+-- Synchronizers
|   +-- CountDownLatch, CyclicBarrier, Phaser
|   +-- Semaphore, Exchanger
|   +-- ReentrantLock, ReadWriteLock, StampedLock
|
+-- Atomic Variables
|   +-- AtomicInteger, AtomicLong, AtomicBoolean
|   +-- AtomicReference, AtomicStampedReference
|   +-- LongAdder, LongAccumulator (high contention)
|
+-- Future & CompletableFuture
    +-- Future, Callable
    +-- CompletableFuture (async programming)

ExecutorService Examples

// Fixed thread pool -- for CPU-bound tasks
ExecutorService fixedPool = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors()
);

// Cached thread pool -- for short-lived async tasks
ExecutorService cachedPool = Executors.newCachedThreadPool();

// Scheduled executor -- for periodic tasks
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);
scheduler.scheduleAtFixedRate(
    () -> System.out.println("Heartbeat"),
    0, 1, TimeUnit.SECONDS
);

// Virtual thread executor (Java 21+) -- for I/O-bound tasks
ExecutorService virtualPool = Executors.newVirtualThreadPerTaskExecutor();

// Always shutdown executors!
executor.shutdown();
executor.awaitTermination(30, TimeUnit.SECONDS);

Concurrent Collections

Collection Thread-Safe Alternative To Best For
ConcurrentHashMap HashMap High-concurrency read/write maps
CopyOnWriteArrayList ArrayList Read-heavy, rare writes (listeners)
BlockingQueue Queue Producer-consumer patterns
ConcurrentLinkedQueue LinkedList Non-blocking queue operations

Synchronizers

// CountDownLatch -- wait for N events to complete
CountDownLatch latch = new CountDownLatch(3);

// In worker threads:
latch.countDown();  // Decrement count

// In main thread:
latch.await();  // Block until count reaches 0

// ---------------------------------------------------

// Semaphore -- limit concurrent access
Semaphore semaphore = new Semaphore(3);  // Allow 3 concurrent

semaphore.acquire();  // Wait for permit
try {
    // Access limited resource (e.g., connection pool)
} finally {
    semaphore.release();
}

// ---------------------------------------------------

// CyclicBarrier -- threads wait for each other
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    System.out.println("All threads reached barrier!");
});

// In each thread:
barrier.await();  // Wait until all 3 threads arrive

CompletableFuture (Async Programming)

// Chain async operations
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> fetchUser(userId))       // Run in ForkJoinPool
    .thenApply(user -> user.getEmail())         // Transform result
    .thenCompose(email -> sendEmailAsync(email)) // Chain another async
    .exceptionally(ex -> {
        log.error("Failed", ex);
        return "fallback@email.com";
    });

// Combine multiple futures
CompletableFuture<Void> allOf = CompletableFuture.allOf(
    fetchUser(1), fetchUser(2), fetchUser(3)
);
allOf.join();  // Wait for all

// Get fastest result
CompletableFuture<User> fastest = CompletableFuture.anyOf(
    fetchFromCache(id), fetchFromDB(id)
).thenApply(obj -> (User) obj);

Atomic Operations

// AtomicInteger -- thread-safe counter without locks
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();      // ++counter
counter.getAndIncrement();      // counter++
counter.addAndGet(5);           // counter += 5
counter.compareAndSet(10, 20);  // CAS operation

// LongAdder -- better for high contention (Java 8+)
LongAdder adder = new LongAdder();
adder.increment();  // Many threads can increment simultaneously
long total = adder.sum();

// AtomicReference -- for reference types
AtomicReference<User> userRef = new AtomicReference<>(oldUser);
userRef.compareAndSet(oldUser, newUser);

Lock Interfaces

Lock Type Use Case Features
ReentrantLock Replace synchronized tryLock, fairness, interruptible
ReadWriteLock Read-heavy workloads Multiple readers, single writer
StampedLock Optimistic reads Better performance, validation stamp

Best Practices

  • Prefer ConcurrentHashMap over Collections.synchronizedMap()
  • Use ExecutorService instead of manually managing threads
  • Use CompletableFuture for async workflows
  • Use LongAdder for high-contention counters
  • [!] Always release locks in finally blocks
  • [!] Always shutdown executor services
