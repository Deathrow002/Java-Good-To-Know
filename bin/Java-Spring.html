2. Explain the concept of Object-Oriented Programming in Java

Answer:

Object-Oriented Programming (OOP) is a paradigm centered around objects that contain data (fields) and code (methods). Java is fundamentally built on OOP principles.

The Four Pillars of OOP

Pillar Definition Java Example
Encapsulation Bundle data + methods, hide internal state private fields + public getters/setters
Inheritance Create new class from existing class class Dog extends Animal
Polymorphism One interface, multiple implementations Method overriding, Animal a = new Dog()
Abstraction Hide complexity, show only essentials abstract class, interface

Code Examples

// Encapsulation: private data, controlled access
public class BankAccount {
    private double balance;  // Hidden state
    
    public void deposit(double amount) {
        if (amount > 0) balance += amount;  // Validation
    }
}

// Inheritance + Polymorphism
public abstract class Shape {
    public abstract double area();  // Abstraction
}

public class Circle extends Shape {
    private double radius;
    
    @Override
    public double area() {  // Polymorphism
        return Math.PI * radius * radius;
    }
}

Benefits of OOP

  • Modularity: Classes can be written and maintained independently
  • Reusability: Inheritance allows code reuse across classes
  • Flexibility: Polymorphism enables interchangeable implementations
  • Maintainability: Encapsulation isolates changes to specific classes

3. What is the difference between == and .equals() in Java?

Answer:

These are two different ways to compare objects in Java:

Comparison

Aspect == Operator .equals() Method
Compares Memory addresses (references) Object content (logical equality)
For Primitives Compares values [OK] Cannot be used
For Objects Same object in memory? Same content/value?
Can Override [OK] No Yes
Null Safe Yes [OK] Throws NPE if called on null

Code Examples

// String comparison
String s1 = new String("Hello");
String s2 = new String("Hello");
String s3 = "Hello";
String s4 = "Hello";

s1 == s2       // false -- different objects in heap
s1.equals(s2)  // true  -- same content

s3 == s4       // true  -- same object in String Pool (interned)
s3.equals(s4)  // true  -- same content

// Primitive comparison
int a = 5, b = 5;
a == b         // true  -- compares values directly

Best Practices

  • Use .equals() for object content comparison
  • Use == for primitive types and null checks
  • Use Objects.equals(a, b) for null-safe comparison
  • For strings, prefer "literal".equals(variable) to avoid NPE

4. What are hashCode() and equals() methods used for?

Answer:

These methods work together to determine object equality and enable efficient storage in hash-based collections.

The Contract

Method Purpose Return Type
equals() Defines logical equality between objects boolean
hashCode() Returns integer for hash-based collections int

Critical Rules

// THE CONTRACT:
1. If a.equals(b) == true  ->  a.hashCode() == b.hashCode()  MUST
2. If a.hashCode() == b.hashCode()  ->  a.equals(b) may be true or false
3. If a.equals(b) == false  ->  hashCodes MAY differ (recommended for performance)

// CONSEQUENCE:
[!] Override BOTH or NEITHER -- never just one!

How HashMap Uses Both

// When you do: map.put(key, value)
1. Calculate hashCode() -> determines bucket index
2. If bucket has entries -> use equals() to find exact match

// Visual:
HashMap Buckets:
[0] -> null
[1] -> [Entry("A")] -> [Entry("K")]  // Same hashCode, different objects
[2] -> [Entry("B")]
...

Proper Implementation

public class Employee {
    private Long id;
    private String name;
    
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Employee employee = (Employee) o;
        return Objects.equals(id, employee.id);  // Based on business key
    }
    
    @Override
    public int hashCode() {
        return Objects.hash(id);  // Same fields as equals()
    }
}

// Modern alternative: Use records (Java 14+)
public record Employee(Long id, String name) {}  // Auto-generates both!

5. Explain method overloading vs method overriding

Answer:

Both are ways to achieve polymorphism but at different stages and with different purposes.

Comparison

Aspect Overloading Overriding
Definition Same name, different parameters Same signature in subclass
Where Same class Parent-child classes
Binding Compile-time (static) Runtime (dynamic)
Return Type Can be different Must be same or covariant
Access Modifier Can be different Cannot be more restrictive
Annotation None @Override recommended

Code Examples

// OVERLOADING -- Same class, different parameters
public class Calculator {
    public int add(int a, int b) { return a + b; }
    public int add(int a, int b, int c) { return a + b + c; }  // Different params
    public double add(double a, double b) { return a + b; }    // Different types
}

// OVERRIDING -- Subclass provides specific implementation
public class Animal {
    public void speak() { System.out.println("Some sound"); }
}

public class Dog extends Animal {
    @Override  // Always use this annotation!
    public void speak() { System.out.println("Woof!"); }
}

// Runtime polymorphism
Animal animal = new Dog();
animal.speak();  // Output: "Woof!" -- Dog's version is called

Key Points

  • Overloading: Provides multiple ways to call the same method with different inputs
  • Overriding: Enables runtime polymorphism -- actual object type determines method
  • static, final, and private methods cannot be overridden

6. What is the difference between an abstract class and an interface?

Answer:

Both provide abstraction but serve different purposes and have different capabilities.

Comparison (Java 17+)

Feature Abstract Class Interface
Instantiation [OK] Cannot [OK] Cannot
Inheritance Single only (extends) Multiple (implements)
Constructors Yes [OK] No
Instance Fields Any type [OK] Only public static final
Concrete Methods Yes default, static, private
Access Modifiers Any Methods: public (or private)

Code Examples

// ABSTRACT CLASS -- Shared state + behavior
public abstract class Animal {
    protected String name;  // Instance state
    
    public Animal(String name) { this.name = name; }  // Constructor
    
    public void eat() { System.out.println(name + " eats"); }  // Concrete
    public abstract void speak();  // Abstract -- must implement
}

// INTERFACE -- Contract (capability)
public interface Flyable {
    int MAX_ALTITUDE = 10000;  // public static final
    
    void fly();  // Abstract
    
    default void land() {  // Default implementation (Java 8+)
        System.out.println("Landing...");
    }
}

// A class can extend ONE abstract class but implement MULTIPLE interfaces
public class Bird extends Animal implements Flyable, Comparable<Bird> {
    public Bird(String name) { super(name); }
    
    @Override public void speak() { System.out.println("Chirp!"); }
    @Override public void fly() { System.out.println("Flying!"); }
    @Override public int compareTo(Bird o) { return name.compareTo(o.name); }
}

When to Use

  • Abstract Class: "IS-A" relationship with shared state/behavior (e.g., Animal)
  • Interface: "CAN-DO" capability across unrelated classes (e.g., Comparable, Serializable)

8. What is the difference between final, finally, and finalize()?

Answer:

These three keywords/concepts sound similar but serve completely different purposes:

Comparison

Keyword Type Purpose Status
final Keyword Prevents modification/extension Use freely
finally Block Always executes after try/catch Use freely
finalize() Method Cleanup before GC s️ Deprecated (Java 9+)

Code Examples

// FINAL -- Prevents modification
final int MAX_SIZE = 100;       // Constant variable (cannot reassign)
final List<String> list = new ArrayList<>();  // Reference can't change
list.add("item");               // But object content CAN change!

final class ImmutableClass { }  // Cannot be extended
class Parent {
    final void method() { }     // Cannot be overridden
}

// FINALLY -- Cleanup block
try {
    connection = getConnection();
    // risky operations
} catch (SQLException e) {
    log.error("DB error", e);
} finally {
    // ALWAYS runs -- even if exception thrown or return in try
    if (connection != null) connection.close();
}

// Modern alternative: try-with-resources
try (Connection conn = getConnection()) {
    // Auto-closes conn, no finally needed
}

About finalize() -- [!] Deprecated

// DON'T USE -- Deprecated since Java 9, removed in Java 18
@Override
protected void finalize() throws Throwable {
    // Unpredictable timing, may never run!
}

// INSTEAD USE:
1. try-with-resources (AutoCloseable)
2. Cleaner API (Java 9+)
3. Explicit close() methods

9. Explain the concept of immutability in Java

Answer:

An immutable object is one whose state cannot be modified after creation. Once created, it remains constant for its lifetime.

Benefits of Immutability

Benefit Why It Matters
Thread Safety Can be shared between threads without synchronization
Cache Friendly Safe to cache -- value never changes
Hash Key Safe Perfect for HashMap/HashSet keys
Security Prevents accidental or malicious modification

How to Create Immutable Classes

public final class ImmutablePerson {  // 1. Make class final
    private final String name;         // 2. Make fields private and final
    private final List<String> hobbies;
    
    public ImmutablePerson(String name, List<String> hobbies) {
        this.name = name;
        this.hobbies = new ArrayList<>(hobbies);  // 3. Defensive copy
    }
    
    // 4. No setters -- only getters
    public String getName() { return name; }
    
    public List<String> getHobbies() {
        return Collections.unmodifiableList(hobbies);  // 5. Return copy/view
    }
}

// Modern alternative: Records (Java 14+)
public record Person(String name, int age) { }  // Automatically immutable!

Immutable Classes in Java

  • String -- most famous example
  • Integer, Double, all wrapper classes
  • LocalDate, LocalTime, Instant (java.time)
  • BigInteger, BigDecimal

Immutability Checklist

  • Class declared final (or all methods final)
  • All fields private final
  • No setters
  • Defensive copies of mutable objects (in constructor and getters)
  • No methods that modify internal state

11. What is the extends keyword in Java?

Answer:

The extends keyword is used to create inheritance relationships, allowing a class to inherit properties and methods from another class, or an interface to extend another interface.

Usage Scenarios

Context Syntax Limit
Class extends Class class Dog extends Animal Single inheritance only
Interface extends Interface interface A extends B, C Multiple allowed
Generic upper bound <T extends Number> Type constraint

Code Examples

// Class inheritance +" single parent only
public class Animal {
    protected String name;
    
    public void eat() {
        System.out.println(name + " is eating");
    }
}

public class Dog extends Animal {  // Dog IS-A Animal
    public void bark() {
        System.out.println(name + " says Woof!");
    }
}

// Interface extending multiple interfaces
public interface Swimmer { void swim(); }
public interface Runner { void run(); }

public interface Triathlete extends Swimmer, Runner {
    void cycle();  // Now has swim(), run(), and cycle()
}

// Generic type bound
public <T extends Comparable<T>> T findMax(List<T> list) {
    return Collections.max(list);  // T must be Comparable
}

What Gets Inherited

  • public and protected members
  • Package-private members (if same package)
  • [OK] private members (not accessible, but still exist)
  • [OK] Constructors (not inherited, but can be called via super())

Y' Note: Java doesn't support multiple class inheritance to avoid the "diamond problem" +" use interfaces instead.

12. What is the implements keyword in Java?

Answer:

The implements keyword is used when a class commits to providing implementations for all abstract methods defined in one or more interfaces.

Key Characteristics

Feature Description
Multiple Interfaces A class can implement many interfaces
Contract Fulfillment Must implement ALL abstract methods (or be abstract)
Default Methods Inherited automatically (can override)
Combined with extends extends comes first, then implements

Code Examples

// Define interfaces (contracts)
public interface Flyable {
    void fly();
    default void land() { System.out.println("Landing..."); }
}

public interface Swimmable {
    void swim();
}

// Implement multiple interfaces
public class Duck implements Flyable, Swimmable {
    @Override
    public void fly() {
        System.out.println("Duck is flying");
    }
    
    @Override
    public void swim() {
        System.out.println("Duck is swimming");
    }
    // land() is inherited from Flyable's default method
}

// Combine extends and implements
public class Penguin extends Bird implements Swimmable {
    @Override
    public void swim() {
        System.out.println("Penguin swims gracefully");
    }
}

// Using interfaces for polymorphism
Flyable flyer = new Duck();     // Interface reference
flyer.fly();                     // "Duck is flying"

List<Swimmable> swimmers = List.of(new Duck(), new Penguin());
swimmers.forEach(Swimmable::swim);  // Both swim!

extends vs implements

Aspect extends implements
Used with Classes, Interfaces Classes only
Relationship IS-A (inheritance) CAN-DO (capability)
Multiple [OK] One class only Many interfaces

13. What is the super keyword in Java?

Answer:

The super keyword refers to the parent (superclass) of the current object. It's used to access parent class members that are hidden or overridden by the child class.

Three Uses of super

Use Syntax Purpose
Call parent constructor super() or super(args) Initialize parent class
Access parent method super.methodName() Call overridden method
Access parent field super.fieldName Access hidden field

Code Examples

public class Animal {
    protected String name;
    
    public Animal(String name) {
        this.name = name;
        System.out.println("Animal constructor called");
    }
    
    public void speak() {
        System.out.println("Animal speaks");
    }
}

public class Dog extends Animal {
    private String breed;
    
    public Dog(String name, String breed) {
        super(name);  // 1. MUST be first line +" calls Animal(name)
        this.breed = breed;
        System.out.println("Dog constructor called");
    }
    
    @Override
    public void speak() {
        super.speak();  // 2. Call parent's speak() first
        System.out.println(name + " says Woof!");  // Then add behavior
    }
}

// Usage
Dog dog = new Dog("Buddy", "Golden Retriever");
// Output:
// Animal constructor called
// Dog constructor called

dog.speak();
// Output:
// Animal speaks
// Buddy says Woof!

super vs this

Keyword Refers To Common Use
this Current object Distinguish fields from params
super Parent class Access overridden members
this() This class constructor Constructor chaining
super() Parent constructor Initialize parent

Important Rules

  • super() must be the first statement in a constructor
  • If you don't call super(), Java inserts super() (no-arg) automatically
  • Cannot use super in static context (no instance)
  • super cannot skip generations +" only accesses immediate parent

4. Explain the concept of reflection in Java

Answer:

Reflection is a feature that allows programs to examine and modify their own structure and behavior at runtime using the java.lang.reflect package.

Core Reflection Classes

Class Purpose

// Get Class object

9. What is the difference between synchronized methods and blocks?

Answer:

    count++;  // Lock held for entire method
    // Other operations also locked unnecessarily
}

// Synchronized Block +" locks only critical section
public void increment() {
    // Non-critical code runs without lock
    synchronized (this) {
        count++;  // Only this is locked
    }
    // More non-critical code
}

9. How do you handle exceptions in Spring Boot REST APIs?

Answer:

1. Global Exception Handler (Recommended)

@RestControllerAdvice
public class GlobalExceptionHandler {
    
    // Handle specific exception
    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(ResourceNotFoundException ex) {
        return new ErrorResponse(
            HttpStatus.NOT_FOUND.value(),
            "NOT_FOUND",
            ex.getMessage(),
            LocalDateTime.now()
        );
    }
    
    // Handle validation errors
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(e -> e.getField() + ": " + e.getDefaultMessage())
            .toList();
        return new ErrorResponse(400, "VALIDATION_ERROR", errors.toString(), LocalDateTime.now());
    }
    
    // Catch-all for unexpected errors
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleAll(Exception ex) {
        log.error("Unexpected error", ex);
        return new ErrorResponse(500, "INTERNAL_ERROR", "An unexpected error occurred", LocalDateTime.now());
    }
}

@Component
public class JwtTokenProvider {
    
    @Value("${jwt.secret}")
    private String secretKey;
    
    @Value("${jwt.expiration:3600000}")  // 1 hour default
    private long validityInMs;
    
    private SecretKey key;
    
    @PostConstruct
    protected void init() {
        this.key = Keys.hmacShaKeyFor(secretKey.getBytes(StandardCharsets.UTF_8));
    }
    
    public String createToken(String username, List<String> roles) {
        Date now = new Date();
        Date validity = new Date(now.getTime() + validityInMs);
        
        return Jwts.builder()
            .subject(username)
            .claim("roles", roles)
            .issuedAt(now)
            .expiration(validity)
            .signWith(key)
            .compact();
    }
    
    public boolean validateToken(String token) {
        try {
            Jwts.parser().verifyWith(key).build().parseSignedClaims(token);
            return true;
        } catch (JwtException | IllegalArgumentException e) {
            return false;
        }
    }
    
    public String getUsername(String token) {
        return Jwts.parser().verifyWith(key).build()
            .parseSignedClaims(token).getPayload().getSubject();
    }
}

public class JwtAuthenticationFilter extends OncePerRequestFilter {
    
    private final JwtTokenProvider tokenProvider;
    
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) 
            throws ServletException, IOException {
        
        String token = resolveToken(request);
        
        if (token != null && tokenProvider.validateToken(token)) {
            Authentication auth = tokenProvider.getAuthentication(token);
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        
        filterChain.doFilter(request, response);
    }
    
    private String resolveToken(HttpServletRequest request) {
        String bearerToken = request.getHeader("Authorization");
        if (bearerToken != null && bearerToken.startsWith("Bearer ")) {
            return bearerToken.substring(7);
        }
        return null;
    }
}

5. What is CSRF, and how does Spring Security protect against it?

Answer:

CSRF (Cross-Site Request Forgery) is an attack that tricks authenticated users into performing unintended actions on a web application by exploiting the trust the site has in the user's browser.

CSRF Attack Flow

+-------------+          +-------------+          +-------------+
|   Victim    |          |   Attacker  |          |   Bank.com  |
|   Browser   |          |   Website   |          |   Server    |
+-------------+          +-------------+          +-------------+
      |                        |                        |
      |  1. User logs into Bank.com                     |
      |------------------------------------------------>|
      |<-- Session cookie set --------------------------+
      |                        |                        |
      |  2. User visits attacker's site                 |
      |------------------>|                             |
      |                   |                             |
      |  3. Hidden form auto-submits:                   |
      |  <form action="bank.com/transfer">              |
      |     <input name="to" value="attacker">          |
      |     <input name="amount" value="10000">         |
      |  </form>                                        |
      |                   |                             |
      |  4. Browser sends request WITH session cookie   |
      |-------------------------------------------------->|
      |                        |                        |
      |  5. Bank processes transfer (victim is logged in)|
      |                        |                        |

Spring Security CSRF Protection

Spring Security generates a unique token per session that must be included in state-changing requests (POST, PUT, DELETE).

Answer:

Spring provides declarative transaction management via @Transactional, abstracting the complexity of managing transactions across different data sources.

Transaction Flow

@Transactional method called
           |
           v
+----------------------+
| TransactionManager   |
| begins transaction   |
+----------------------+
           |
           v
+----------------------+
| Execute method body  |
|   - DB operations    |
|   - Business logic   |
+----------------------+
           |
    +------+------+
    |             |
    v             v
SUCCESS        EXCEPTION
    |             |
    v             v
+--------+   +-----------+
| COMMIT |   | ROLLBACK  |
+--------+   +-----------+

Transaction Propagation Types

Propagation Behavior
REQUIRED (default) Join existing TX, or create new if none exists
REQUIRES_NEW Always create new TX, suspend existing
MANDATORY Must run within existing TX, throw exception if none
SUPPORTS Use TX if exists, run non-transactional if not
NOT_SUPPORTED Suspend existing TX, run non-transactional
NEVER Throw exception if TX exists
NESTED Create savepoint within existing TX

Isolation Levels

Isolation Dirty Read Non-Repeatable Phantom
READ_UNCOMMITTED Possible Possible Possible
READ_COMMITTED Prevented Possible Possible
REPEATABLE_READ Prevented Prevented Possible
SERIALIZABLE Prevented Prevented Prevented

@Transactional Examples

@Service
public class OrderService {
    
    private final OrderRepository orderRepository;
    private final PaymentService paymentService;
    private final NotificationService notificationService;
    
    // Basic transaction - rollback on any RuntimeException
    @Transactional
    public Order createOrder(OrderRequest request) {
        Order order = new Order(request);
        order = orderRepository.save(order);
        paymentService.processPayment(order);  // Same transaction
        return order;
    }
    
    // Custom rollback rules
    @Transactional(
        rollbackFor = Exception.class,           // Rollback on checked exceptions
        noRollbackFor = NotificationException.class  // Don't rollback for this
    )
    public Order createOrderWithNotification(OrderRequest request) {
        Order order = orderRepository.save(new Order(request));
        notificationService.sendConfirmation(order);  // May throw NotificationException
        return order;
    }
    
    // Read-only transaction (optimization hint)
    @Transactional(readOnly = true)
    public List<Order> findAllOrders() {
        return orderRepository.findAll();
    }
    
    // Separate transaction for audit logging
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logAudit(String action, Long orderId) {
        // This commits independently, even if outer TX rolls back
        auditRepository.save(new AuditLog(action, orderId));
    }
    
    // Timeout handling
    @Transactional(timeout = 30)  // 30 seconds
    public void processLargeOrder(OrderRequest request) {
        // Long-running operation
    }
}

Common Pitfalls

  • Self-invocation: Calling @Transactional method from same class bypasses proxy
  • Checked exceptions: By default, only RuntimeException triggers rollback
  • Private methods: @Transactional doesn't work on private methods
  • Final classes/methods: Cannot be proxied (use interface or CGLIB)
// WRONG - self-invocation bypasses @Transactional
@Service
public class OrderService {
    public void process() {
        createOrder();  // @Transactional NOT applied!
    }
    
    @Transactional
    public void createOrder() { ... }
}

// CORRECT - inject self or use separate service
@Service
public class OrderService {
    @Autowired
    private OrderService self;  // Inject proxy
    
    public void process() {
        self.createOrder();  // Goes through proxy
    }
}

🔥 Apache Kafka

1. What is Apache Kafka, and what are its main use cases?

Answer:

Apache Kafka is a distributed event streaming platform designed for high-throughput, fault-tolerant, real-time data pipelines and streaming applications.

Key Characteristics

  • Distributed: Runs as a cluster across multiple servers/data centers
  • Scalable: Horizontally scalable by adding more brokers
  • Fault-tolerant: Data is replicated across brokers
  • High-throughput: Handles millions of messages per second
  • Durable: Messages are persisted to disk

Main Use Cases

  • Event Streaming: Real-time event processing and analytics
  • Messaging: Decoupling microservices communication
  • Log Aggregation: Collecting logs from distributed systems
  • Data Integration: ETL pipelines and data synchronization
  • CQRS/Event Sourcing: Storing domain events as source of truth
  • Activity Tracking: User behavior and metrics collection

2. What are the core components of Kafka architecture?

Answer:

  • Broker: A Kafka server that stores data and serves clients. A cluster consists of multiple brokers for scalability and fault tolerance.
  • Topic: A named category/feed to which messages are published. Topics are partitioned and replicated.
  • Partition: A topic is divided into partitions for parallelism. Each partition is an ordered, immutable sequence of messages.
  • Producer: Client that publishes messages to topics. Decides which partition to send messages to.
  • Consumer: Client that reads messages from topics. Tracks its position (offset) in partitions.
  • Consumer Group: A group of consumers that jointly consume a topic. Each partition is consumed by exactly one consumer in the group.
  • ZooKeeper/KRaft: Coordinates brokers, manages metadata, and handles leader election. KRaft (Kafka Raft) is replacing ZooKeeper.
  • Offset: A unique identifier for each message within a partition, used to track consumption progress.

3. How does Kafka ensure message ordering and partitioning?

Answer:

Message Ordering

  • Messages are ordered within a partition but not across partitions
  • Consumers read messages in the order they were written to a partition
  • To guarantee global ordering, use a single partition (sacrifices parallelism)

Partitioning Strategies

  • Key-based: Messages with the same key go to the same partition (ensures ordering for related messages)
  • Round-robin: Messages distributed evenly when no key is specified
  • Custom partitioner: Implement Partitioner interface for custom logic

Best Practices

  • Use meaningful keys (e.g., user ID, order ID) for related messages
  • Choose partition count based on throughput needs and consumer parallelism
  • Avoid changing partition count for existing topics (breaks key-based ordering)

4. What is the difference between at-most-once, at-least-once, and exactly-once delivery?

Answer:

  • At-most-once:
    • Messages may be lost but never redelivered
    • Consumer commits offset before processing
    • Fastest but risk of data loss
  • At-least-once:
    • Messages are never lost but may be redelivered (duplicates possible)
    • Consumer commits offset after processing
    • Default Kafka behavior; requires idempotent consumers
  • Exactly-once:
    • Messages are delivered exactly once with no loss or duplicates
    • Achieved via idempotent producers (enable.idempotence=true) and transactional APIs
    • Higher latency but strongest guarantee

Exactly-once in Kafka: Use isolation.level=read_committed on consumers and transactional producers with transactional.id.

5. How does Kafka handle fault tolerance and replication?

Answer:

Replication

  • Each partition has a configurable replication factor (e.g., 3)
  • One replica is the leader; others are followers
  • All reads and writes go through the leader
  • Followers replicate from the leader asynchronously

In-Sync Replicas (ISR)

  • ISR is the set of replicas fully caught up with the leader
  • A message is committed only when all ISR replicas acknowledge it
  • If a follower falls behind, it's removed from ISR

Leader Election

  • If the leader fails, a new leader is elected from ISR
  • min.insync.replicas ensures minimum replicas before acknowledging writes
  • unclean.leader.election.enable=false prevents data loss (only ISR members can become leader)

Producer Acknowledgments (acks)

  • acks=0: No acknowledgment (fastest, no durability)
  • acks=1: Leader acknowledgment only
  • acks=all: All ISR replicas acknowledge (strongest durability)

7. What are Consumer Groups, and how do they enable scalability?

Answer:

A Consumer Group is a set of consumers that cooperate to consume messages from topics.

How It Works

  • Each partition is assigned to exactly one consumer in the group
  • Multiple consumers share the workload (parallel processing)
  • If a consumer fails, its partitions are reassigned to other consumers (rebalancing)

Scalability

  • Add more consumers (up to the number of partitions) to increase throughput
  • If consumers > partitions, some consumers remain idle
  • Multiple consumer groups can independently consume the same topic (pub/sub pattern)

Offset Management

  • Kafka tracks the committed offset per partition per consumer group
  • Offsets stored in internal __consumer_offsets topic
  • Auto-commit or manual commit for precise control

Rebalancing

  • Triggered when consumers join/leave or partitions change
  • Can cause temporary processing pause
  • Use CooperativeStickyAssignor for incremental rebalancing

9. How do you handle schema evolution in Kafka?

Answer:

Schema evolution is managed using a Schema Registry (e.g., Confluent Schema Registry) with serialization formats like Avro, Protobuf, or JSON Schema.

Schema Registry

  • Centralized repository for schemas
  • Schemas are versioned and associated with topics
  • Producers/consumers register and retrieve schemas via REST API

Compatibility Modes

  • BACKWARD: New schema can read data written by old schema (add optional fields, remove fields with defaults)
  • FORWARD: Old schema can read data written by new schema
  • FULL: Both backward and forward compatible
  • NONE: No compatibility checking

Best Practices

  • Always provide default values for new fields
  • Never remove required fields; mark them as deprecated
  • Use BACKWARD compatibility for most use cases
  • Test schema changes in non-production environments first
  • Use Avro or Protobuf for efficient binary serialization with schema support

10. What are common Kafka performance tuning strategies?

Answer:

Producer Tuning

  • batch.size: Increase batch size for higher throughput (default 16KB)
  • linger.ms: Add small delay to allow batching (e.g., 5-10ms)
  • compression.type: Use lz4 or snappy for compression (reduces network I/O)
  • buffer.memory: Increase buffer for high-throughput producers
  • acks: Trade durability for latency (acks=1 vs acks=all)

Consumer Tuning

  • fetch.min.bytes: Minimum data to fetch (reduces requests)
  • fetch.max.wait.ms: Maximum wait time for fetch.min.bytes
  • max.poll.records: Control records per poll (balance throughput vs processing time)
  • max.partition.fetch.bytes: Maximum bytes per partition per fetch

Broker Tuning

  • num.io.threads: I/O threads for disk operations
  • num.network.threads: Threads for network requests
  • log.segment.bytes: Segment size affects retention and compaction
  • Disk: Use SSDs and separate disks for logs

Cluster Design

  • Partition count: More partitions = more parallelism (but more overhead)
  • Replication factor: Balance durability vs. resource usage (typically 3)
  • Monitor: Use JMX metrics, Kafka Manager, or Confluent Control Center
